## 📘 암호 구조

## 🔥 1️⃣ 암호 공격 전체 구조

암호 공격은 크게 3가지 축으로 나뉜다.

```
① 암호해석 공격 (Cryptanalysis)
② 공격 모델 분류 (Attacker Model)
③ 부채널 공격 (Side Channel Attack)
```

---

## 2️⃣ 암호해석 공격 (Cryptanalysis)

👉 알고리즘의 수학적 구조를 공격

| 한글        | 영어                             | 핵심             |
| --------- | ------------------------------ | -------------- |
| 무차별 대입 공격 | Brute Force Attack             | 키 전수조사         |
| 기지 평문 공격  | Known Plaintext Attack         | 평문 일부 알고 있음    |
| 선택 평문 공격  | Chosen Plaintext Attack (CPA)  | 원하는 평문 암호화 가능  |
| 선택 암호문 공격 | Chosen Ciphertext Attack (CCA) | 원하는 암호문 복호화 가능 |
| 차분 공격     | Differential Cryptanalysis     | 입력 차이 분석       |
| 선형 공격     | Linear Cryptanalysis           | 선형 근사          |

📌 포인트
CPA / CCA는 **부채널이 아니라 암호 모델 공격**이다.

---

## 3️⃣ 공격 모델 (Attacker Capability Model)

이건 “공격자가 무엇을 할 수 있느냐” 기준.

|한글|영어|
|---|---|
|암호문 단독 공격|Ciphertext-Only Attack|
|기지 평문 공격|Known Plaintext Attack|
|선택 평문 공격|Chosen Plaintext Attack|
|선택 암호문 공격|Chosen Ciphertext Attack|

시험에서 자주 섞어서 낸다.

---

## 4️⃣ 부채널 공격 (Side Channel Attack)

👉 수학이 아니라 **물리적 특성**을 이용

| 한글        | 영어                              | 분석 대상        |
| --------- | ------------------------------- | ------------ |
| 전력 분석 공격  | Power Analysis Attack           | 소비 전력        |
| 타이밍 공격    | Timing Attack                   | 연산 시간 비밀키 추론 |
| 전자기 분석 공격 | Electromagnetic Analysis Attack | 전자파          |
| 캐시 공격     | Cache Attack                    | 캐시 접근 패턴     |
| 오류 주입 공격  | Fault Injection Attack          | 고의 오류 유도     |

📌 특징

- 암호 수학 안 깨도 됨

- 하드웨어 보안에서 중요

- 스마트카드, HSM 문제에서 자주 나옴

### 📘 서비스 ↔ 보안기술 한 장 정리

| 서비스(기본 프로토콜)          | 평문 여부    | 보안 기술 (한글 / 영어)                                                            | 적용 계층   | 무엇을 보호하는가             | 시험 포인트              |
| --------------------- | -------- | -------------------------------------------------------------------------- | ------- | --------------------- | ------------------- |
| 웹 (HTTP)              | 평문       | **HTTPS (HTTP Secure)** = **TLS (전송계층보안, Transport Layer Security) , SSL** | 전송계층    | 기밀성, 무결성, 서버 인증       | 웹·전자결제는 TLS 기반      |
| 원격 접속 (Telnet)        | 평문       | **SSH (보안 셸, Secure Shell)**                                               | 응용계층    | 암호화 통신, 사용자 인증        | Telnet 대체는 SSH      |
| 전자우편 (SMTP/POP3/IMAP) | 평문       | **PGP (프리티 굿 프라이버시, Pretty Good Privacy)** / **S/MIME (보안 다목적 인터넷 메일 확장)** | 응용계층    | 메일 내용 암호화, 전자서명(부인방지) | 메일 보안 = PGP/S-MIME  |
| 파일 전송 (FTP)           | 평문       | **SFTP (SSH 파일전송)** / **FTPS (SSL/TLS 기반 FTP)**                            | 응용/전송계층 | 파일 전송 시 기밀성·무결성       | SFTP≠FTPS 구분        |
| 전자결제 (웹 기반)           | 평문(HTTP) | ** SET,** **TLS 기반 HTTPS**                                                 | 전송계층    | 카드정보 등 민감정보 보호        | 결제는 별도 프로토콜 아님, TLS |

## 📘 VPN (Virtual Private Network, 가상 사설망) 정리

### 1️⃣ VPN 개념

|항목|내용|
|---|---|
|한글|가상 사설망|
|영어|Virtual Private Network|
|목적|공용망(인터넷) 위에 사설망처럼 안전한 통신 구성|
|제공 요소|기밀성(Confidentiality), 무결성(Integrity), 인증(Authentication)|

---

### 2️⃣ VPN 동작 원리

```text
터널링(Tunneling) + 암호화(Encryption) + 인증(Authentication)
```

- 터널링: 패킷을 다른 패킷 안에 넣어 전달

- 암호화: 데이터 보호

- 인증: 사용자/장비 검증

---

### 3️⃣ VPN 종류

|구분|한글|영어|특징|
|---|---|---|---|
|사이트 간 VPN|Site-to-Site VPN|LAN ↔ LAN 연결|기업 지사 연결|
|원격 접속 VPN|Remote Access VPN|사용자 ↔ 회사망|재택근무|

---

### 4️⃣ VPN 주요 프로토콜

| OSI 계층    | 프로토콜 (한글 / 영어)                                       | 역할       | 암호화 여부     | 시험 포인트          |
| --------- | ---------------------------------------------------- | -------- | ---------- | --------------- |
| 7~5 응용~세션 | **SSL VPN (SSL Virtual Private Network)**            | 웹 기반 VPN | ⭕ (TLS 사용) | 브라우저 기반         |
| 4 전송      | **TLS (Transport Layer Security, 전송계층보안)**           | 암호화 채널   | ⭕          | HTTPS 기반        |
| 3 네트워크    | **IPSec (Internet Protocol Security, 인터넷 프로토콜 보안)**  | IP 패킷 보호 | ⭕          | Site-to-Site 대표 |
| 3 네트워크    | **IKE (Internet Key Exchange, 인터넷 키 교환)**            | 키 협상     | ⭕          | IPSec 구성요소      |
| 2 데이터링크   | **L2TP (Layer 2 Tunneling Protocol, 제2계층 터널링 프로토콜)** | 터널링      | ❌          | 암호화 없음          |
| 2 데이터링크   | **PPTP (Point-to-Point Tunneling Protocol)**         | 터널링      | 약함         | 구식·취약           |
| 2 데이터링크   | **PPP (Point-to-Point Protocol)**                    | 인증·링크 설정 | ❌          | L2TP 기반         |

## 🔐 5️⃣ IPSec (Internet Protocol Security) 구조

IPSec은 시험 단골이다.

### 구성 요소

| 구성  | 한글                                           | 역할                |
| --- | -------------------------------------------- | ----------------- |
| AH  | 인증 헤더 (Authentication Header)                | 인증(무결성)           |
| ESP | 캡슐화 보안 페이로드 (Encapsulating Security Payload) | 인증(무결성)+ 암호화(기밀성) |
| IKE | 키 교환                                         | 세션키 생성            |

---

### 동작 모드

|모드|설명|
|---|---|
|전송 모드 (Transport Mode)|원래 IP 헤더 유지, 데이터 부분만 보호|
|터널 모드 (Tunnel Mode)|원래 IP 패킷 전체를 새 패킷 안에 캡슐화|

📌 Site-to-Site는 보통 **터널 모드**

## 🔒 6️⃣ SSL VPN (TLS VPN)

|항목|설명|
|---|---|
|기반|TLS (Transport Layer Security)|
|접근 방식|웹 브라우저 기반|
|장점|방화벽 통과 쉬움|
|단점|네트워크 전체 연결에는 제한적|

옵시디언에 넣기 좋게 **한 개 핵심표 + 세부 정리** 형태로 정리했다.
(주요 내용은 IETF TLS 표준 RFC 기준)

### 🔥 빠른 분류 공식 (시험용)

```text
가용성 ↓  → DoS/DDoS
기밀성 ↓  → Sniffing, MITM, SQLi
무결성 ↓  → XSS, CSRF
인증 ↓    → Brute Force, Session Hijacking
```

### 🎯 기출에서 자주 묻는 포인트

- **증폭(Amplification) vs 플러딩(Flooding)** 구분

- **네트워크 공격 vs 웹 공격** 구분

- **OWASP Top 10 계열** (SQLi, XSS, CSRF)

- **MITM과 세션 하이재킹 차이**

- **암호 공격(CPA/CCA)과 부채널 공격 차이**

## 📘 DoS / DDoS 공격 기법 — 한 장 정리
| 계열   | 공격 기법 (한글 / 영어)         | 원리                                       | 핵심 키워드        |
| ---- | ----------------------- | ---------------------------------------- | ------------- |
| TCP  | **SYN Flood (SYN 플러딩)** | 3-way handshake 미완성                      | Half-open     |
| ICMP | **Smurf (스머프 공격)**      | ICMP Broadcast 증폭                        | Amplification |
| UDP  | **Fraggle (프래글 공격)**    | UDP Broadcast 증폭                         | Smurf의 UDP    |
| IP   | **Land (랜드 공격)**        | 출발지=목적지 IP                               | Loop 유발       |
| IP   | **Teardrop (티어드롭)**     | Fragment 겹침                              | Overlapping   |
| IP   | **Bonk (봉크)**           | Fragment Offset 조작<br>시퀀스번호 같은 값으로 반복    | 재조립 오류        |
| IP   | **Boink (보잉크)**         | Fragment 시퀀스번호 일부를 중간에 비정상으로 섞어보냄 봉크 변형ㄴ | 변형형           |
| ICMP | **Ping of Death**       | 비정상 큰 패킷                                 | 구형 취약         |
| 응용   | **Slowloris**           | 연결 유지                                    | 동시 연결 고갈      |
| 응용   | **HTTP Flood**          | 정상 요청 위장                                 | 탐지 어려움        |
| 증폭   | **DNS Amplification**   | 작은 요청→큰 응답                               | Reflection    |
| 증폭   | **NTP Amplification**   | monlist 악용                               | 대규모           |

### 🔥 핵심 구분 공식

```text
Flooding = 그냥 많이 보냄
Amplification = 작게 보내고 크게 돌아오게 함
Reflection = IP 위조 사용
```

---

### 🧠 시험 직전 기억 6줄

```text
SYN Flood = 3-way handshake 악용
Smurf = ICMP Broadcast
Fraggle = UDP Broadcast
Land = 출발지IP = 목적지IP
Slowloris = 연결 유지
DNS/NTP = 증폭 공격
```

### 📘  블루투스(Bluetooth) 공격 기법 총정리

|공격 기법 (한글 / 영어)|공격 유형|핵심 목표|특징 / 시험 키워드|
|---|---|---|---|

|**KNOB 공격 (Key Negotiation of Bluetooth)**|암호 약화|키 길이 축소|암호 강도 다운그레이드|
|**BIAS 공격 (Bluetooth Impersonation AttackS)**|인증 우회|기기 사칭|재연결 인증 취약|

### 🔥 **각 도구 설명**

#### 1️⃣ **리버스 엔지니어링** (Reverse Engineering)

- **목적**: **실행 파일**을 분석하여 **악성 코드**나 **취약점**을 찾아내는 데 사용

- **대표 도구**: **OllyDBG, IDA, Ghidra**

- **시험 키워드**: **디버깅**, **디스어셈블**
    -> 악성 코드를 분석하거나, **바이너리 분석**을 통해 취약점을 파악할 때 쓰인다.

#### 2️⃣ **패킷 분석기** (Packet Analyzer)

- **목적**: **네트워크 트래픽**을 **모니터링**하고 **분석**하는 도구

- **대표 도구**: **Wireshark, tcpdump**

- **시험 키워드**: **패킷 캡처**, **트래픽**
    -> **네트워크 패킷**을 **캡처**하고 분석하여 **보안 위협**을 파악한다.

#### 3️⃣ **취약점 스캐너** (Vulnerability Scanner)

- **목적**: **시스템**의 **취약점**을 **자동으로 검사**하여 **위험 요소**를 식별

- **대표 도구**: **OpenVAS, Nessus**

- **시험 키워드**: **취약점 자동 스캔**
    -> **네트워크**나 **시스템**에서 **자동으로 취약점을** 찾아내는 도구.

#### 4️⃣ **침입 탐지 시스템** (Intrusion Detection System)

- **목적**: **침입**이나 **비정상적인 활동**을 **탐지**하는 시스템

- **대표 도구**: **Snort, Suricata**

- **시험 키워드**: **시그니처 탐지**
    -> **네트워크**에서 발생하는 **침입**을 **탐지**하여 경고를 발생시킨다.

#### 5️⃣ **침입 방지 시스템** (Intrusion Prevention System)

- **목적**: **침입을 실시간으로 차단**하는 시스템

- **대표 도구**: **Snort IPS Mode**

- **시험 키워드**: **실시간 차단**
    -> **침입 탐지**와 **차단**을 동시에 처리하여, **네트워크 보안**을 실시간으로 강화한다.

#### 6️⃣ **웹 취약점 점검** (Web Vulnerability Scanner)

- **목적**: **웹 애플리케이션**의 **취약점**을 **검사**하여 **보안 리스크**를 식별

- **대표 도구**: **Burp Suite, Paros, OWASP ZAP**

- **시험 키워드**: **XSS**, **SQLi 테스트**
    -> **웹 취약점**(예: **SQL Injection**, **Cross-Site Scripting**)을 점검하는 도구.

#### 7️⃣ **무선 침투 테스트** (Wireless Pentest)

- **목적**: **Wi-Fi 네트워크**의 보안을 **테스트**하고 **취약점**을 찾는 데 사용

- **대표 도구**: **Aircrack-ng**

- **시험 키워드**: **WPA 크랙**
    -> **Wi-Fi 패스워드**를 **크랙**하거나, 무선 네트워크의 보안을 테스트한다.

#### 8️⃣ **포트 스캐너** (Port Scanner)

- **목적**: **서버**나 **네트워크**에서 **열린 포트**를 **스캔**하여 **서비스**를 확인

- **대표 도구**: **Nmap**

- **시험 키워드**: **포트 스캔**
    -> **네트워크 포트**의 **상태**를 **스캔**하여 **열린 포트**를 찾고 보안을 강화한다.

#### 9️⃣ **로그 분석** (Log Analysis)

- **목적**: **로그 데이터를 분석**하여 **보안 위협**을 탐지하고 **위협 패턴**을 파악

- **대표 도구**: **ELK Stack**

- **시험 키워드**: **SIEM 연계**
    -> **로그 분석**을 통해 **보안 위협**을 탐지하고, **경고**를 발생시킨다.

#### 🔟 **보안 정보 이벤트 관리** (SIEM)

- **목적**: **보안 이벤트**를 **통합적으로 분석**하여 **보안 사고를 실시간으로 파악**

- **대표 도구**: **Splunk**

- **시험 키워드**: **이벤트 상관분석**
    -> **보안 정보**를 **통합적으로 관리**하고, **이벤트 상관분석**을 통해 **보안 위협**을 탐지한다.

## 📘 리눅스 권한

## 🔥 1️⃣ 기본 구조

```
[u][g][o]
```

- u = 소유자

- g = 그룹

- o = 기타 사용자

각 자리 = 숫자 하나

---

## 🔥 2️⃣ 숫자 뜻

```
r = 4
w = 2
x = 1
```

더해서 계산.

예:

```
7 = rwx
6 = rw-
5 = r-x
4 = r--
0 = ---
```

---

## 🔥 3️⃣ 예시

```
chmod 640 file
```

→ 6 = rw-
→ 4 = r--
→ 0 = ---

👉 소유자: 읽기+쓰기
👉 그룹: 읽기
👉 기타: 없음

---

## 🔥 4️⃣ 특수권한 (앞자리 하나 더)

```
4xxx → SUID
2xxx → SGID
1xxx → Sticky
```

예:

```
chmod 4755 file
```

→ 4 = SUID
→ 755 = 일반권한

## 🔥 한줄 요약

```
권한 = [특수][u][g][o]
r=4 w=2 x=1
```

---

## 🔥 특수권한 3개

| 권한         | 숫자   | 의미                 | 기억법       |
| ---------- | ---- | ------------------ | --------- |
| **SUID**   | 4000 | 실행 시 **파일 소유자 권한** | root 프로그램 |
| **SGID**   | 2000 | 실행 시 **그룹 권한**     | 그룹 공유     |
| **Sticky** | 1000 | **파일 삭제 제한 자유게시판** | /tmp      |

---

## 🔥 예시

```bash
chmod 4755 file
```

|자리|의미|
|---|---|
|4|SUID|
|7|소유자 rwx|
|5|그룹 r-x|
|5|기타 r-x|

---

## 🔥 실제 표시

|권한|표시|
|---|---|
|SUID|`rws`|
|SGID|`r-s`|
|Sticky|`t`|

예:

```id="rympk7"
-rwsr-xr-x
```

|권한|시험 포인트|
|---|---|
|SUID|권한 상승 가능|

## 📘  Zero Trust (제로 트러스트)

> **어떤 사용자·기기·네트워크도 기본적으로 신뢰하지 않고 모든 접근을 지속적으로 검증하는 보안 모델**

📚 출처

- Forrester Research – John Kindervag

- NIST SP 800-207 (Zero Trust Architecture)

---

### 🧠 핵심 철학

```text
Never Trust, Always Verify
절대 신뢰하지 말고 항상 검증하라
```

---

### ⚔️ 기존 보안 vs Zero Trust

|구분|기존 보안 (Perimeter Security)|Zero Trust|
|---|---|---|
|신뢰 기준|내부망 신뢰|아무도 신뢰하지 않음|
|인증 방식|최초 인증|지속적 검증|
|접근 통제|네트워크 중심|사용자·기기 중심|
|보안 철학|외부 위험|내부도 위험|

---

### 📜 Zero Trust 핵심 원칙

|원칙|설명|
|---|---|
|🔎 Verify Explicitly|사용자·기기·위치 등을 계속 검증|
|🔒 Least Privilege|최소 권한만 부여|
|🚨 Assume Breach|이미 침해되었다고 가정|

(NIST SP 800-207)

---

### 🧾 검증 요소

접속 요청 시 다음 요소들을 평가한다.

|요소|설명|
|---|---|
|👤 사용자|누구인가|
|💻 기기|신뢰 가능한 장치인가|
|🌍 위치|접속 위치|
|⏰ 시간|접속 시간|
|📊 행위|이상 행동 여부|

---

### ⚙️ 구현 기술

|기술|역할|
|---|---|
|🔑 MFA|다중 인증|
|🪪 IAM|계정 및 접근 관리|
|🌐 ZTNA|Zero Trust Network Access|
|🧩 Micro-segmentation|네트워크 세분화|
|🛡 Endpoint Security|단말 보안|

---

### 🎯 한 줄 정리

> **Zero Trust = 내부망도 절대 신뢰하지 않는다**

---

좋은 질문이다. 결론부터 말하면 **저 정도보다 조금 더 알아두는 게 감리사 시험 범위에 안전하다.**
보안 솔루션은 보통 **네트워크 / 데이터 / 인증 / 단말 / 관제**로 묶어서 출제된다. (정보보호 개론 및 NIST 보안 아키텍처 분류에서도 유사하게 계층적으로 구분됨)

옵시디언용으로 **“시험 대비 핵심 세트”**만 정리해 준다.

---

## 📘 🛡 보안 솔루션 핵심 정리

### 🌐 네트워크 보안

| 솔루션                                     | 설명                                               |
| --------------------------------------- | ------------------------------------------------ |
| 🔥 **Firewall**                         | IP, 포트, 프로토콜 기준으로 **네트워크 트래픽을 허용/차단하는 기본 보안 장비** |
| 🚨 **IDS (Intrusion Detection System)** | 공격 패턴을 **탐지만 하고 차단하지는 않는 침입 탐지 시스템**             |
| ⚔ **IPS (Intrusion Prevention System)** | 공격을 **탐지하고 실시간으로 차단까지 수행하는 침입 방지 시스템**           |
| 🌐 **WAF (Web Application Firewall)**   | **웹 애플리케이션 공격(SQL Injection, XSS 등)**을 탐지·차단     |
| 🔐 **VPN (Virtual Private Network)**    | **암호화 터널을 통해 인터넷 구간에서 안전한 통신을 제공**               |
| 🧩 **NAC (Network Access Control)**     | **단말 인증과 보안 상태 검사 후 내부 네트워크 접속을 통제**             |
| 🌊 **Anti-DDoS**                        | 대량 트래픽 공격을 **분산 처리하거나 필터링하여 서비스 가용성을 보호**        |

---

### 📂 데이터 보안

|솔루션|설명|
|---|---|
|📤 **DLP (Data Loss Prevention)**|**중요 데이터가 이메일, USB, 클라우드 등을 통해 외부로 유출되는 것을 탐지·차단**|
|🔒 **DRM (Digital Rights Management)**|**문서의 열람, 복사, 인쇄 등의 사용 권한을 제어하여 콘텐츠 사용을 관리**|
|🗄 **DB 접근제어**|**데이터베이스 접근을 사용자·권한 기준으로 통제하고 로그를 관리**|
|🔑 **DB 암호화**|**데이터베이스 내 중요 데이터를 암호화하여 저장 데이터 보호**|

---

### 👤 인증 / 접근 관리

| 솔루션                                       | 설명                                      |
| ----------------------------------------- | --------------------------------------- |
| 🔐 **SSO (Single Sign-On)**               | **한 번의 인증으로 여러 시스템에 접근할 수 있는 통합 인증 체계** |
| 🪪 **IAM (Identity & Access Management)** | **사용자 계정, 권한, 인증 정책을 통합적으로 관리하는 시스템**   |
| 🔑 **MFA (Multi-Factor Authentication)**  | **지식·소유·생체 등의 여러 인증 요소를 결합한 다중 인증 방식**  |

---

### 💻 단말 보안

|솔루션|설명|
|---|---|
|🦠 **Anti-Virus**|악성코드를 **탐지하고 제거하는 기본적인 엔드포인트 보안 솔루션**|
|🛡 **EDR (Endpoint Detection and Response)**|단말에서 발생하는 **이상 행위를 탐지하고 대응하는 위협 대응 시스템**|

---

### 📊 보안 관제

|솔루션|설명|
|---|---|
|📡 **SIEM (Security Information and Event Management)**|다양한 보안 장비의 로그를 **통합 수집·분석하여 보안 이벤트를 탐지**|
|🤖 **SOAR (Security Orchestration, Automation and Response)**|보안 이벤트 대응 절차를 **자동화하여 빠르게 대응하도록 지원**|

## 1️⃣ Buffer Overflow (버퍼 오버플로우)

## 🔐 보안 핵심 구분 (MAC / 전자서명 / 해시 / 인증서 / PKI)

|🔑 구분|🎯 목적|🔐 사용 키|⚙️ 특징|🧠 시험 포인트|
|---|---|---|---|---|
|🧩 **Hash**|무결성 확인|없음|메시지를 고정 길이 해시값으로 변환|SHA-256, SHA-3|
|🔒 **MAC**|무결성 + 인증|**대칭키**|메시지와 비밀키로 MAC 생성|HMAC|
|✍️ **전자서명**|무결성 + 인증 + **부인방지**|**공개키**|개인키로 서명 / 공개키로 검증|RSA, ECDSA|
|📜 **인증서**|공개키 신뢰|CA 서명|공개키 + 사용자 정보 포함|X.509|
|🏛 **PKI**|인증 인프라|공개키 체계|인증서 발급/관리 시스템|CA, RA|

## ⚡ 핵심 차이 (시험용)

|구분|핵심|
|---|---|
|🧩 **Hash**|무결성만|
|🔒 **MAC**|무결성 + 인증 (**대칭키**)|
|✍️ **전자서명**|무결성 + 인증 + **부인방지 (공개키)**|

## 🧠 시험장에서 바로 찍는 법

|키워드|정답|
|---|---|
|🔑 대칭키 인증|**MAC**|
|🚫 부인방지|**전자서명**|
|🧩 무결성만|**Hash**|
|📜 공개키 신뢰|**인증서**|
|🏛 인증서 관리|**PKI**|

## 🧭 구조 이해

```text
Message → Hash → 전자서명(Private Key) → 검증(Public Key)
```

```text
Message + Secret Key → MAC 생성 → 동일 키로 검증
```

## 📘  블록 암호 운용 모드 (Block Cipher Mode)

블록 암호는 보통 **AES 같은 암호 알고리즘을 어떻게 운용할지**를 결정하는 방식이다.

예

```text
AES + 운용모드
```

---

## 주요 운용 모드

### 1️⃣ ECB (Electronic Codebook, 전자 코드북)

**특징**

- 같은 평문 → 같은 암호문

- 구조 노출

**특징 요약**

```text
보안 취약
거의 사용 안함
```

---

### 2️⃣ CBC (Cipher Block Chaining, 암호 블록 연쇄)

**특징**

- 이전 암호문과 XOR 후 암호화

- 패턴 숨김 가능

**단점**

```text
병렬 처리 불가
```

---

### 3️⃣ CFB (Cipher Feedback, 암호 피드백)

**특징**

- 블록 암호를 **스트림 암호처럼 사용**

---

### 4️⃣ OFB (Output Feedback, 출력 피드백)

**특징**

- 키 스트림 생성 후 XOR

- 오류 전파 없음

---

### 5️⃣ CTR (Counter Mode, 카운터 모드)

**특징**

- 카운터 값 암호화 후 XOR

- 병렬 처리 가능

- 고속

---

### 6️⃣ GCM (Galois Counter Mode)

**특징**

- CTR 기반

- **기밀성 + 인증(무결성) 제공**

---

### 7️⃣ CCM (Counter with CBC-MAC)

**특징**

- CTR + CBC-MAC 결합

- **기밀성 + 인증(무결성) 제공**

---

## 🎯 시험 핵심 구분
| 모드          | 한글명               | 동작 방식              | 특징                | 시험 포인트           | 오류 전파     |
| ----------- | ----------------- | ------------------ | ----------------- | ---------------- | --------- |
| **ECB**     | 전자 코드북            | 각 블록을 독립적으로 암호화    | 동일 평문 → 동일 암호문    | 패턴 노출 / 보안 취약    | 1블록만 영향   |
| **CBC**     | 암호 블록 연쇄          | 이전 암호문과 XOR 후 암호화  | 패턴 숨김             | 병렬 처리 불가         | 2블록 영향    |
| **CFB**     | 암호 피드백            | 이전 암호문을 입력으로 사용    | 블록 암호를 스트림처럼 사용   | 오류 전파 발생         | 일부 비트 영향  |
| **OFB**     | 출력 피드백            | 키 스트림 생성 후 XOR     | 스트림 암호처럼 동작       | 오류 전파 없음         | 동일 비트 위치  |
| **CTR**     | 카운터 모드            | 카운터 값을 암호화 후 XOR   | 병렬 처리 가능          | 고속 암호화           | 해당 블록만 영향 |
| ==**GCM**== | ==갈루아 카운터 모드==    | ==CTR + GHASH 인증== | ==기밀성 + 인증(무결성)== | ==AEAD 모드==      | 블록 영향     |
| ==**CCM**== | ==카운터 + CBC-MAC== | ==CTR + CBC-MAC==  | ==기밀성 + 인증(무결성)== | ==무선 네트워크에서 사용== | 블록 영향     |

## 🔎 주요 특징 비교

|특징|ECB|CBC|CFB|OFB|CTR|
|---|---|---|---|---|---|
|패턴 노출|있음|없음|없음|없음|없음|
|병렬 처리|가능|불가|불가|가능|가능|
|오류 전파|없음|있음|있음|없음|없음|

## 🔐 블록 암호 운용 모드 시험 암기표

### ① 기본 블록 모드

|모드|한글|특징|시험 포인트|
|---|---|---|---|
|**ECB**|전자 코드북|블록 독립 암호화|**패턴 노출 (가장 취약)**|
|**CBC**|암호 블록 연쇄|이전 암호문과 XOR|**가장 대표적 모드**|

핵심 기억

```text
ECB → 단순 / 패턴 노출
CBC → 체인 구조
```

---

### ② 스트림 방식 모드

|모드|한글|특징|시험 포인트|
|---|---|---|---|
|**CFB**|암호 피드백|이전 암호문 사용|스트림처럼 동작|
|**OFB**|출력 피드백|키 스트림 생성|오류 전파 없음|
|**CTR**|카운터 모드|카운터 암호화|**병렬 처리 가능 / 빠름**|

핵심 기억

```text
CFB / OFB / CTR
→ 스트림처럼 동작
```

특히

```text
CTR → 병렬 처리 가능
```

---

### ③ 인증 포함 모드 (AEAD)

| 모드      | 특징            | 시험 포인트            |
| ------- | ------------- | ----------------- |
| **GCM** | CTR + 인증      | **기밀성 + 인증(무결성)** |
| **CCM** | CTR + CBC-MAC | **기밀성 + 인증(무결성)** |

핵심 기억

```text
GCM / CCM
→ 인증까지 제공
```

| ==기밀성 + 인증(무결성)== | ==GCM== |

## 📘  🌐 VLAN (Virtual LAN, 가상 LAN)

**정의**

```text
물리적으로 하나의 네트워크를
논리적으로 여러 개의 네트워크로 분리하는 기술
```

즉

```text
스위치 하나 → 여러 개의 LAN처럼 사용
```

---

## 🎯 VLAN이 필요한 이유

기존 네트워크

```text
스위치 1개
→ 전체가 하나의 브로드캐스트 도메인
```

문제

- 브로드캐스트 트래픽 증가

- 보안 문제

- 네트워크 관리 어려움

---

## VLAN 적용

```
스위치
 ├ VLAN10 (개발팀)
 ├ VLAN20 (회계팀)
 └ VLAN30 (서버)
```

👉 **같은 스위치라도 서로 통신 불가**

(라우터 또는 L3 스위치 필요)

## 🛡 IDS / IPS / Snort / iptables 정리

### 1️⃣ IDS (Intrusion Detection System, 침입 탐지 시스템)

|특징|탐지만 하고 차단은 하지 않음|

|특징|실시간 트래픽 차단|

핵심

```text
IPS = 탐지 + 차단
```

### 3️⃣ Snort

**정의**

```text
오픈소스 네트워크 침입 탐지 시스템
```

기능

|기능|설명|
|---|---|
|패킷 분석|네트워크 트래픽 분석|
|공격 탐지|시그니처 기반 탐지|
|룰 기반|탐지 규칙 사용|

---

#### Snort 룰 구조

예

```text
alert tcp !192.168.0.1 21 any -> 192.168.0.1 21
```

구성

| 요소                  | 의미   |
| ------------------- | ---- |
| alert               | 경고   |
| tcp                 | 프로토콜 |
| !192.168.0.1 21 any | 출발지  |
| →                   | 방향   |
| 192.168.0.1 21      | 목적지  |
a ny = 모든 것(anything)

---

#### Snort 주요 액션

|명령|의미|
|---|---|
|alert|경고|
|log|로그 기록|
|pass|통과|
|drop|차단|

---

### 4️⃣ iptables

**정의**

```text
리눅스 방화벽 설정 도구
```

기능

|기능|설명|
|---|---|
|패킷 필터링|트래픽 허용 / 차단|

|포트 제어|포트 접근 제한|

## iptables 한 줄 구조

iptables -A INPUT -p tcp --dport 22 -j ACCEPT

이걸 **4개만 보면 된다.**

|부분|의미|
|---|---|
|-A|규칙 추가|
|INPUT|들어오는 패킷|
|--dport 22|목적지 포트 22|
|-j ACCEPT|허용|

|옵션|의미|
|---|---|

## 🔐 SSO (Single Sign-On)

## 🔐 메시지 인증 방식 비교

|구분|한글명|키 종류|제공 기능|
|---|---|---|---|
|MAC|메시지 인증 코드|대칭키|무결성 + 송신자 인증|
|HMAC|해시 기반 메시지 인증 코드|대칭키|무결성 + 송신자 인증|
|Digital Signature|전자서명|비대칭키|무결성 + 인증 + 부인방지|

---

## 1️⃣ MAC (Message Authentication Code)

### 구조

```text
MAC = Message + Secret Key
```

송신자

```
MAC = f(message, secret key)
```

수신자

```
MAC 다시 계산 → 비교
```

---

### 특징

|항목|설명|
|---|---|
|키|대칭키|
|무결성|가능|
|송신자 인증|가능|
|부인방지|불가능|

---

### 왜 부인방지가 안됨?

```text
송신자와 수신자가
같은 키를 공유
```

그래서

```text
누가 메시지를 만들었는지
증명 불가능
```

---

## 2️⃣ HMAC

### 정의

```text
해시 함수를 사용하는 MAC
```

공식

```text
HMAC = H((K ⊕ opad) || H((K ⊕ ipad) || message))
```

(RFC 2104)

---

### 특징

|항목|설명|
|---|---|
|기반|해시 함수|
|키|대칭키|
|무결성|제공|
|인증|제공|
|부인방지|없음|

---

## 3️⃣ Digital Signature

### 구조

송신자

```
signature = Sign(private key, message)
```

수신자

```
Verify(public key)
```

---

### 특징

|항목|설명|
|---|---|
|키|비대칭키|
|무결성|가능|
|인증|가능|
|부인방지|가능|

---

## 🔑 핵심 차이

|기능|MAC/HMAC|전자서명|
|---|---|---|
|무결성|O|O|
|인증|O|O|
|부인방지|X|O|

---

## 📌 시험 핵심 암기

```text
대칭키 인증
= MAC / HMAC
= 부인방지 없음

비대칭키 인증
= Digital Signature
= 부인방지 가능
```

## 🔐 인증 방식(Authentication Factors)

|인증 요소|영어|설명|예|
|---|---|---|---|
|**지식 기반 인증**|Knowledge Factor|사용자가 **알고 있는 정보**|비밀번호, PIN|
|**소유 기반 인증**|Possession Factor|사용자가 **가지고 있는 것**|OTP 토큰, 스마트카드, 보안카드|
|**생체 기반 인증**|Inherence Factor|사용자의 **신체 특징**|지문, 홍채, 얼굴|
|**행위 기반 인증**|Behavioral Factor|사용자의 **행동 패턴**|타이핑 패턴, 마우스 패턴|

## 🔐  IDS

먼저 이것을 고정해야 한다.

Positive = 공격으로 판단
Negative = 정상으로 판단

그리고

True = 판단이 맞음
False = 판단이 틀림

## 📊 IDS 판정표

|실제 상황|IDS 판단|결과|
|---|---|---|
|공격|공격|**True Positive (TP)**|
|정상|정상|**True Negative (TN)**|
|정상|공격|**False Positive (FP)**|
|공격|정상|**False Negative (FN)**|

## 🎯 시험에서 자주 나오는 두 개

|유형|의미|
|---|---|
|**False Positive**|정상인데 공격으로 오탐|
|**False Negative**|공격인데 정상으로 판단|

## 🎯 시험용 한 줄 암기

False Positive = 정상인데 공격이라고 오탐
False Negative = 공격인데 정상이라고 놓침

## 🔥 보안 시험에서 중요 포인트

|유형|위험|
|---|---|
|False Positive|가용성 문제|
|False Negative|보안 문제|

## 📊 TLS Handshake 핵심 흐름

|단계|메시지|설명|
|---|---|---|
|1|ClientHello|클라이언트가 지원하는 암호 알고리즘, TLS 버전 전송|
|2|ServerHello|서버가 사용할 암호 알고리즘 선택|
|3|Certificate|서버 인증서 전송|
|4|Key Exchange|키 교환 정보 전달|
|5|Finished|세션키 생성 완료|

## 블록암호 운용모드 비교 (시험용)

| 모드                              | 구조 / 원리               | 특징                      | 오류 전파     | 병렬 처리   |
| ------------------------------- | --------------------- | ----------------------- | --------- | ------- |
| **ECB** (Electronic Codebook)   | 블록을 **독립적으로 암호화**     | 동일 평문 → 동일 암호문 (패턴 노출)  | 1블록만 영향   | 가능      |
| **CBC** (Cipher Block Chaining) | 이전 **암호문과 XOR 후 암호화** | 가장 오래된 방식, 초기벡터(IV) 필요  | 2블록 영향    | 복호화만 가능 |
| **CFB** (Cipher Feedback)       | 이전 **암호문을 암호화하여 XOR** | 스트림처럼 동작                | 일부 비트 영향  | 불가      |
| **OFB** (Output Feedback)       | 이전 **출력값을 암호화하여 XOR** | 오류 전파 거의 없음             | 동일 비트 위치  | 가능      |
| **CTR** (Counter Mode)          | **카운터 암호화 후 XOR**     | 매우 빠름, 병렬 처리 가능         | 해당 블록만 영향 | 가능      |
| **GCM** (Galois/Counter Mode)   | **CTR + GHASH 인증**    | ==기밀성 + 무결성 제공 (AEAD)== | 블록 영향     | 가능      |
| **CCM** (Counter with CBC-MAC)  | **CTR + CBC-MAC 인증**  | ==기밀성 + 무결성 제공 (AEAD)== | 블록 영향     | 제한적     |

### Bell-LaPadula & Biba 보안 모델 규칙 정리

(출처: Bell & LaPadula, _Secure Computer Systems_, MITRE, 1973 / Biba, _Integrity Considerations for Secure Computer Systems_, MITRE, 1977)

|모델|보안 목적|규칙 이름|규칙 내용|
|---|---|---|---|
|Bell-LaPadula|기밀성(Confidentiality)|Simple Security Property|**No Read Up** → 높은 등급 읽기 금지|
|Bell-LaPadula|기밀성(Confidentiality)|Star Property (*-Property)|**No Write Down** → 낮은 등급 쓰기 금지|
|Bell-LaPadula|기밀성(Confidentiality)|Discretionary Security Property|ACL 등 재량적 접근제어 적용|
|Biba|무결성(Integrity)|Simple Integrity Property|**No Read Down** → 낮은 무결성 읽기 금지|
|Biba|무결성(Integrity)|Star Integrity Property|**No Write Up** → 높은 무결성 쓰기 금지|
|Biba|무결성(Integrity)|Invocation Property|낮은 무결성 주체가 높은 무결성 객체 호출 금지|

---

#### 시험 핵심 기억

```
Bell-LaPadula : No Read Up / No Write Down  → 기밀성
Biba          : No Read Down / No Write Up → 무결성
```

### SPN vs Feistel 간략 정리

| 구분      | **SPN (Substitution-Permutation Network)** | **Feistel Network**     |
| ------- | ------------------------------------------ | ----------------------- |

| 복호화     | **역연산 필요**                                 | **같은 구조 사용**            |
| 대표 알고리즘 | **AES**                                    | **DES**                 |

|구분|설명|
|---|---|
|최소 권한 (Least Privilege)|사용자는 수행해야 할 작업에 필요한 최소한의 권한만 가져야 하며, 더 이상 필요하지 않은 경우 해당 권한은 회수되어야 한다.|
|고장 안전 디폴트 (Fail-Safe Defaults)|명시적으로 허가되지 않은 모든 자원에 대한 접근은 기본적으로 거부되어야 한다.|
|매커니즘의 효율성 (Economy of Mechanism)|시스템 설계는 가능한 한 단순하게 구성되어야 하며, 복잡성은 오류 및 취약점 발생 가능성을 증가시킨다.|
|완벽한 조율 (Complete Mediation)|모든 접근 시도는 반드시 검사되어야 하며, 접근 시마다 권한 검증이 이루어져야 한다.|
|열린 설계 (Open Design)|보호 메커니즘은 공격자의 무지에 의존하지 않아야 하며, 공개되고 검증된 방식으로 설계되어야 한다.|
|권한 분리 (Separation of Privilege)|시스템은 단일 조건에 의해 권한을 부여해서는 안 되며, 다중 조건을 만족해야 권한이 부여되어야 한다.|
|최소 공통 메커니즘 (Least Common Mechanism)|공유 메커니즘의 수와 사용을 최소화해야 하며, 공유 객체는 정보 흐름 및 의도되지 않은 상호작용에 대한 잠재적 위험을 가진다.|
|사용의 편리성 (Psychological Acceptability)|사용자가 일반적으로 시스템을 사용하는 과정에서 자연스럽게 보안 메커니즘이 적용되도록 설계되어야 한다.|

## 📘 정성적 위험분석 vs 정량적 위험분석

### 1️⃣ 구분

|구분|정성적 위험분석|정량적 위험분석|
|---|---|---|
|개념|전문가 판단·경험 기반으로 위험 수준을 상대적으로 평가|수치·금액 기반으로 위험을 계량화하여 평가|
|표현|높음/중간/낮음 등 등급|금액, 확률, 기대손실 등 수치|
|목적|위험 우선순위 결정|비용-효과 분석, 투자 의사결정|
|정확도|상대적으로 낮음|상대적으로 높음|
|속도|빠름|느림 (데이터 필요)|
|데이터 요구|적음|많음|

## 📘 생체인증의 조건

|특성|설명|
|---|---|
|보편성|모든 사람이 가지고 있는 보편적 특징을 대상으로 해야 한다.|
|영구성|특징은 변하지 않아야 하며, 시간이 지나도 변형되지 않아야 한다.|
|독특성|동일한 특징을 가진 다른 사람이 존재하지 않아야 한다.|
|획득성|센서를 통해 쉽게 획득할 수 있고, 정량화가 가능해야 한다.|
|시스템 성능|처리 속도가 빠르고 처리량이 커야 하며, 알고리즘 및 시스템 구성의 효율성이 확보되어야 한다.|
|저항성|위조 및 변조가 불가능하거나 매우 어려워야 한다.|
|수용성|사용자가 인증 시스템 사용에 거부감을 느끼지 않아야 한다.|

## 📘 암호화 블록

| 모드          | 한글명               | 동작 방식              | 특징                | 시험 포인트           | 오류 전파     |
| ----------- | ----------------- | ------------------ | ----------------- | ---------------- | --------- |
| **ECB**     | 전자 코드북            | 각 블록을 독립적으로 암호화    | 동일 평문 → 동일 암호문    | 패턴 노출 / 보안 취약    | 1블록만 영향   |
| **CBC**     | 암호 블록 연쇄          | 이전 암호문과 XOR 후 암호화  | 패턴 숨김             | 병렬 처리 불가         | 2블록 영향    |
| **CFB**     | 암호 피드백            | 이전 암호문을 입력으로 사용    | 블록 암호를 스트림처럼 사용   | 오류 전파 발생         | 일부 비트 영향  |
| **OFB**     | 출력 피드백            | 키 스트림 생성 후 XOR     | 스트림 암호처럼 동작       | 오류 전파 없음         | 동일 비트 위치  |
| **CTR**     | 카운터 모드            | 카운터 값을 암호화 후 XOR   | 병렬 처리 가능          | 고속 암호화           | 해당 블록만 영향 |
| ==**GCM**== | ==갈루아 카운터 모드==    | ==CTR + GHASH 인증== | ==기밀성 + 인증(무결성)== | ==AEAD 모드==      | 블록 영향     |
| ==**CCM**== | ==카운터 + CBC-MAC== | ==CTR + CBC-MAC==  | ==기밀성 + 인증(무결성)== | ==무선 네트워크에서 사용== | 블록 영향     |
👉 암호화 알고리즘으로 복호화 가능 : CFB OFB CTR 
👉 암호화 복호화 병렬처리 : ECB CTR
👉 복호화만 병렬처리 가능 : CBC OFB

## 📘 해시함수 요구사항

| 특징                      | 설명                                                                       |
| ----------------------- | ------------------------------------------------------------------------ |
| 메시지 압축 (Message Digest) | 임의의 길이의 평문을 고정된 길이의 출력값으로 변환해야 한다.                                       |
| 계산의 용이성                 | 입력값 x가 주어지면 H(x)는 계산적으로 쉽게 구할 수 있어야 한다.                                  |
| 일방향성 (One-Way)          | 메시지(평문)에서 해시값을 구하는 것은 쉽지만, 해시값에서 원래 메시지를 구하는 것은 매우 어려워야 한다. (역상 저항성)     |
| 약한 충돌 회피성               | 입력값 x가 주어졌을 때 H(x') = H(x)인 x' (x ≠ x)를 찾는 것은 계산적으로 불가능해야 한다. (제2역상 저항성) |
| 강한 충돌 회피성               | H(x) = H(x')인 서로 다른 두 입력값 x, x'를 찾는 것은 계산적으로 불가능해야 한다. (충돌 저항성)          |

## 📘 디피-헬만 (Diffie-Hellman) 키 교환

### 1️⃣ 개념

|구분|내용|
|---|---|
|목적|공개된 채널에서 안전하게 **공유 비밀키 생성**|
|방식|이산로그 문제 기반|
|특징|중간에서 값이 보여도 **비밀키는 계산 불가**|

### 2️⃣ 계산 절차

|단계|설명|
|---|---|
|공통값 설정|큰 소수 p, 생성자 g 공개|
|A 선택|사용자 A가 비밀값 a 선택|
|B 선택|사용자 B가 비밀값 b 선택|
|A 공개값|A = g^a mod p|
|B 공개값|B = g^b mod p|
|A 계산|K = B^a mod p|
|B 계산|K = A^b mod p|

## 📘 포워드 프록시 vs 리버스 프록시

| 구분         | 포워드 프록시 (Forward Proxy) | 리버스 프록시 (Reverse Proxy) |
| ---------- | ----------------------- | ----------------------- |
| ==위치==         | ==클라이언트 앞단==                | ==서버 앞단==                   |
| 대상         | 내부 사용자 → 외부 서버          | 외부 사용자 → 내부 서버          |
| 목적         | 사용자 보호, 접근 제어           | 서버 보호, 부하 분산            |
| ==숨기는 대상== | ==클라이언트(IP)==           | ==서버(IP)==              |
| ==사용 예==   | ==기업 내부 인터넷 접속 통제==     | ==웹 서비스, CDN, 로드밸런서==   |
| ==대표 기능==  | ==캐싱, 필터링, 우회 접속==      | ==로드밸런싱, SSL 종료, 보안==   |

## 👉 핵심 특징

|구분|내용|
|---|---|
|포워드 프록시|클라이언트가 프록시를 인지하고 사용|

## 📘 OWASP LLM Top 10 (2025) — 영문 / 한글 / 개념

### 1️⃣ LLM01: Prompt Injection (프롬프트 인젝션)

- **정의:** 사용자 입력으로 시스템 프롬프트/정책을 우회하거나 모델 동작을 변경하는 공격
    
- **핵심:** 지시 탈취 / 정책 무력화
    

### 2️⃣ LLM02: Sensitive Information Disclosure (민감 정보 노출)

- **정의:** 모델이 학습 데이터, 프롬프트, API 키 등 민감 정보를 출력하는 문제
    
- **핵심:** 데이터 유출
    

### 3️⃣ LLM03: Supply Chain (공급망 취약점)

- **정의:** 외부 모델, 데이터셋, 라이브러리 등 의존 요소를 통한 공격
    
- **핵심:** 외부 의존 위험
    

### 4️⃣ LLM04: Data and Model Poisoning (데이터 및 모델 오염)

- **정의:** 학습 데이터, 파인튜닝, RAG 등을 통해 모델을 악의적으로 오염
    
- **핵심:** 응답 왜곡 / 백도어
    

### 5️⃣ LLM05: Improper Output Handling (부적절한 출력 처리)

- **정의:** LLM 출력값을 검증 없이 실행/처리하는 문제
    
- **핵심:** 출력 → 코드/명령 실행 위험
    

### 6️⃣ LLM06: Excessive Agency (과도한 권한 부여)

- **정의:** LLM에 시스템 제어, 결제, 실행 권한 등을 과도하게 부여
    
- **핵심:** 자동 실행 위험
    

### 7️⃣ LLM07: System Prompt Leakage (시스템 프롬프트 유출)

- **정의:** 내부 시스템 프롬프트 또는 정책이 외부로 노출되는 문제
    
- **핵심:** 내부 정책 노출
    

### 8️⃣ LLM08: Vector and Embedding Weaknesses (벡터/임베딩 취약점)

- **정의:** 벡터DB, 임베딩, RAG 과정에서 발생하는 보안 취약점
    
- **핵심:** 검색 기반 데이터 오염/유출
    

### 9️⃣ LLM09: Misinformation (허위 정보)

- **정의:** 모델이 부정확하거나 잘못된 정보를 생성하고 이를 신뢰하는 문제
    
- **핵심:** 환각(Hallucination) + 잘못된 판단 유도
    

### 🔟 LLM10: Unbounded Consumption (무제한 자원 소비)

- **정의:** 과도한 요청/토큰 사용으로 비용 및 자원을 무한 소비하게 하는 공격
    
- **핵심:** 비용 폭증 / 자원 고갈

## 📘  OWASP Top 10 (2025) — 영문 / 한글 / 개념

### 1️⃣ A01: Broken Access Control (접근 통제 실패)

- **정의:** 사용자가 권한을 넘어 데이터나 기능에 접근하는 문제
    
- **핵심:** 권한 검증 실패
    

### 2️⃣ A02: Security Misconfiguration (보안 설정 오류)

- **정의:** 잘못된 설정, 기본값 사용, 불필요한 서비스 노출
    
- **핵심:** 설정 실수
    

### 3️⃣ A03: Software Supply Chain Failures (소프트웨어 공급망 실패)

- **정의:** 외부 라이브러리, 의존성, 빌드/배포 과정에서 발생하는 취약점
    
- **핵심:** 공급망 공격
    

### 4️⃣ A04: Cryptographic Failures (암호화 실패)

- **정의:** 민감 데이터 보호를 위한 암호화가 적절히 적용되지 않은 경우
    
- **핵심:** 데이터 보호 실패
    

### 5️⃣ A05: Injection (인젝션)

- **정의:** 사용자 입력이 코드나 쿼리로 실행되는 취약점
    
- **핵심:** SQL, OS, LDAP Injection
    

### 6️⃣ A06: Insecure Design (취약한 설계)

- **정의:** 설계 단계에서 보안이 고려되지 않아 발생하는 구조적 문제
    
- **핵심:** 설계 결함
    

### 7️⃣ A07: Authentication Failures (인증 실패)

- **정의:** 인증 및 세션 관리의 취약점으로 계정 탈취 가능
    
- **핵심:** 인증 취약
    

### 8️⃣ A08: Software or Data Integrity Failures (무결성 실패)

- **정의:** 코드 또는 데이터의 변조를 방지하지 못하는 문제
    
- **핵심:** 무결성 훼손
    

### 9️⃣ A09: Security Logging & Alerting Failures (로깅 및 알림 실패)

- **정의:** 공격 탐지 및 대응을 위한 로그와 모니터링 부족
    
- **핵심:** 탐지 실패
    

### 🔟 A10: Mishandling of Exceptional Conditions (예외 처리 실패)

- **정의:** 오류나 예외 상황을 적절히 처리하지 못해 발생하는 취약점
    
- **핵심:** 에러 처리 취약

## 📘 네트워크 필수 명령어 세트

### 1️⃣ ping

- 목적: **연결 여부 확인**
    
- 방식: ICMP Echo Request/Reply
    
- 핵심: “통신 되냐 안되냐”
    

### 2️⃣ tracert

- 목적: **경로 확인**
    
- 방식: TTL + ICMP
    
- 핵심: “어디서 끊기냐”
    

### 3️⃣ ipconfig

- 목적: **내 PC 네트워크 정보 확인**
    
- 확인 가능:
    
    - IP 주소
        
    - 서브넷 마스크
        
    - 게이트웨이
        
    - DNS
        

👉 `/all` 옵션 자주 나옴

### 4️⃣ nslookup

- 목적: **도메인 → IP 변환 확인**
    
- 핵심: DNS 조회
    

👉 “DNS 문제냐?” 판단할 때 사용

### 5️⃣ netstat

- 목적: **현재 연결 상태 확인**
    
- 확인:
    
    - 열린 포트
        
    - 연결 상태 (ESTABLISHED 등)
        

👉 `/an` 옵션 자주 나옴

## 📘 인공지능 ISO/IEC 42001 vs ISO/IEC 23894

| 구분     | ISO/IEC 42001 (AIMS)                                   | ISO/IEC 23894 (Risk Management)                        |
| ------ | ------------------------------------------------------ | ------------------------------------------------------ |
| 정의     | 인공지능 경영시스템 (Artificial Intelligence Management System) | 인공지능 위험 관리 (Artificial Intelligence - Risk Management) |
| 성격     | 요구사항(Requirements) 기반 표준                               | 지침(Guidelines) 기반 표준                                   |
| 인증 가능성 | 인증 가능 (Certifiable)                                    | 인증 불가 (가이드로 활용)                                        |
| 주요 목적  | 조직의 AI 거버넌스 체계 구축 및 대외 신뢰성 증명                          | AI 수명 주기 전반의 위험 관리 프로세스 구체화                            |
| 기반 표준  | ISO/IEC 27001 (정보보호 경영시스템)                             | ISO 31000 (위험 관리 가이드라인)                                |
| 핵심 범위  | 정책, 리더십, 자원 관리, 성과 평가 등 조직 전체                          | 위험 식별, 분석, 평가, 처리 등 리스크 관리 실무                          |
| 적용 시점  | 조직의 AI 도입 및 운영 전략 수립 시                                 | AI 개발, 배포, 운영 등 실무 프로세스 단계                             |
| 비유     | 건축물의 전체 설계도 및 관리 규정                                    | 안전한 건축을 위한 상세 안전 시공 매뉴얼                                |
