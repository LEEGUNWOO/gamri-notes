|유형|발생 원인|발생 상황|해결 방법|핵심 포인트|
|---|---|---|---|---|
|구조적 해저드|하드웨어 자원 부족|동시에 같은 자원 접근 (메모리, ALU 등)|자원 분리, 파이프라인 stall|“자원 충돌”|
|데이터 해저드|명령어 간 데이터 의존성|이전 결과 나오기 전에 다음 명령 사용|Forwarding, Stall, 재배치|“값이 아직 없음”|
|제어 해저드|분기/점프|분기 결과 확정 전 다음 명령 실행|Branch prediction, Delay slot, Flush|“흐름이 바뀜”|

| 종류  | 의미            | 예시           |
| --- | ------------- | ------------ |
| RAW | Write 후 Read  | A=5 저장 전에 읽음 |
| WAR | Read 후 Write  | 읽기 전에 덮어씀    |
| WAW | Write 후 Write | 순서 꼬임        |

```text
1: R1 = R2 + R3
2: R4 = R1 + R5  → RAW 발생
```

👉 1번 결과 나오기 전에 2번이 R1 사용

# 📘 라우팅 프로토콜 종류

라우팅 프로토콜은 크게 두 가지 유형으로 나뉜다:

1. **내부 라우팅 프로토콜 (Interior Gateway Protocol, IGP)**

2. **외부 라우팅 프로토콜 (Exterior Gateway Protocol, EGP)**

### 1️⃣ **내부 라우팅 프로토콜 (IGP)**

- 같은 **자율 시스템(AS)** 내에서 라우팅을 처리하는 프로토콜

- **라우팅 정보**를 **자기 네트워크 안**에서만 교환

|프로토콜|설명|특징|예시|
|---|---|---|---|
|**RIP (Routing Information Protocol)**|거리 벡터 라우팅 프로토콜|홉 수(hop count) 기반, 최대 홉 수 15|작은 네트워크에서 사용|
|**OSPF (Open Shortest Path First)**|링크 상태 라우팅 프로토콜|Dijkstra 알고리즘 사용, 최단 경로 계산|대규모 네트워크에서 사용|
|**EIGRP (Enhanced Interior Gateway Routing Protocol)**|하이브리드 라우팅 프로토콜|거리 벡터 + 링크 상태, 빠른 수렴|Cisco 네트워크 장비에서 사용|
|**IS-IS (Intermediate System to Intermediate System)**|링크 상태 라우팅 프로토콜|OSPF와 유사, 대규모 네트워크에서 사용|대형 ISP와 기업 네트워크에서 사용|

### 2️⃣ **외부 라우팅 프로토콜 (EGP)**

- **다른 자율 시스템(AS)** 간의 라우팅을 처리하는 프로토콜

|프로토콜|설명|특징|예시|
|---|---|---|---|
|**BGP (Border Gateway Protocol)**|경로 벡터 라우팅 프로토콜|AS 간 라우팅, 경로 선택 기준으로 AS 경로, 정책 기반|인터넷 상의 라우팅에 사용|

## 🔥 각 라우팅 프로토콜의 특징

### **RIP (Routing Information Protocol)**

- **거리 벡터** 프로토콜로, **최대 15 홉까지** 라우팅 가능.

- 가장 오래된 라우팅 프로토콜이지만, 작은 네트워크에서 유용하다.

- **단점**: 느리고, 비효율적인 경로 계산.

### **OSPF (Open Shortest Path First)**

- **링크 상태** 프로토콜로, **최단 경로 우선(Dijkstra)** 알고리즘을 사용.

- 대규모 네트워크에서 효율적이며, **빠르고 정확한 수렴**이 가능.

- **단점**: 설정과 관리가 복잡할 수 있음.

### **EIGRP (Enhanced Interior Gateway Routing Protocol)**

- **하이브리드** 프로토콜로, **거리 벡터 + 링크 상태**를 결합한 방식.

- **빠른 수렴**과 **효율적인 경로 계산**이 특징.

- **Cisco 장비에서만** 사용 가능.

### **BGP (Border Gateway Protocol)**

- **경로 벡터** 프로토콜로, **자율 시스템(AS)** 간의 경로 정보를 교환.

- **인터넷**에서 주로 사용되며, **정책 기반 라우팅**을 지원.

- **단점**: 설정이 복잡하고, **수렴 속도**가 느림.

## 🔥 라우팅 프로토콜 비교

|특성|RIP|OSPF|EIGRP|BGP|
|---|---|---|---|---|
|유형|거리 벡터|링크 상태|하이브리드|경로 벡터|
|수렴 속도|느림|빠름|빠름|느림|
|확장성|낮음|높음|높음|매우 높음|
|복잡성|낮음|중간|중간|높음|
|네트워크 크기|소규모|대규모|중대규모|대규모|

# 📘 HBM vs HBF (핵심 비교)


■ HBM (High Bandwidth Memory)

- 정의: 3D 적층 구조 기반 고대역폭 DRAM
- 특징: TSV 적층 / 초고속 / 저전력
- 용도: GPU, AI 연산, HPC
- 핵심: 속도 중심 메모리

■ HBF (High Bandwidth Flash)

- 정의: 고대역폭 NAND Flash (HBM 구조 응용)
- 특징: 대용량 / HBM 대비 저속 / 적층 구조
- 용도: AI 저장/추론 보조
- 상태: 초기/개발 단계

|항목|HBM|HBF|
|---|---|---|
|메모리 종류|DRAM|NAND Flash|
|속도|매우 빠름|상대적으로 느림|
|용량|작음|매우 큼|
|용도|연산용 (GPU)|저장/추론 보조|
|상태|상용화|개발/초기 단계|

# 📘 IOT 통신 프로토콜 CoAP / MQTT / XMPP

■ CoAP (Constrained Application Protocol)

- 구조: REST 기반 요청/응답
    
- 전송: UDP
    
- 특징: HTTP 경량화, 저전력, 저사양 디바이스
    
- 메시지: Confirmable / Non-confirmable
    
- 신뢰성: 낮음 (재전송으로 보완)
    
- 용도: 센서, IoT 디바이스
    

■ MQTT (Message Queuing Telemetry Transport)

- 구조: Pub/Sub (브로커 중심)
    
- 전송: TCP
    
- 특징: 초경량, 저전력, 연결 유지
    
- 구성: Publisher / Broker / Subscriber
    
- QoS: 0 (최대 1회), 1 (최소 1회), 2 (정확히 1회)
    
- 용도: IoT, 모바일, 실시간 데이터
    

■ XMPP (Extensible Messaging and Presence Protocol)

- 구조: 메시지 스트림 (클라이언트-서버)
    
- 전송: TCP
    
- 특징: XML 기반, 확장성 높음
    
- 통신: Message / Presence / IQ
    
- 확장: Pub/Sub 가능 (XEP-0060)
    
- 용도: 채팅, IoT, 메시징 시스템
    

■ 핵심 비교

| 항목  | CoAP         | MQTT                | XMPP   |
| --- | ------------ | ------------------- | ------ |
| 구조  | REST         | Pub/Sub             | 메시지    |
| 전송  | UDP          | TCP                 | TCP    |
| 특징  | HTTP 유사      | 브로커 기반              | XML 확장 |
| 신뢰성 | 낮음           | QoS 지원              | 높음     |
| 적합  | 초저전력         | IoT 실시간             | 메시징/확장 |
| 방식  | 요청/응답 (REST) | **발행/구독** (Pub/Sub) | 메시지 교환 |

**한 줄 암기**

- **CoAP** = IoT용 경량 HTTP (UDP)
- **MQTT** = IoT 메시지 발행/구독 (브로커 필요)
- **XMPP** = XML 기반 메시지 (채팅 앱 기원)

시험에서 **MQTT 발행/구독 구조** 가 제일 자주 나와.

■ 시험 포인트

- CoAP → UDP + REST
    
- MQTT → 브로커 + QoS
    
- XMPP → XML + 확장성 + IQ
    

■ 한줄  
CoAP=UDP REST / MQTT=브로커 PubSub / XMPP=XML 메시징 + 확장

# 📘 TCP 연결/종료 절차

■ 3-Way Handshake (연결 설정)

|단계|송신자 → 수신자|내용|
|---|---|---|
|1|클라이언트 → 서버|SYN (연결 요청)|
|2|서버 → 클라이언트|SYN + ACK (요청 수락)|
|3|클라이언트 → 서버|ACK (확인)|

■ 핵심

- 연결을 “확립”하는 과정
    
- 양쪽 모두 송수신 준비 상태 확인
    

■ 한줄  
SYN → SYN/ACK → ACK

■ 4-Way Handshake (연결 종료)

|단계|송신자 → 수신자|내용|
|---|---|---|
|1|클라이언트 → 서버|FIN (종료 요청)|
|2|서버 → 클라이언트|ACK (확인)|
|3|서버 → 클라이언트|FIN (종료 요청)|
|4|클라이언트 → 서버|ACK (확인)|

■ 핵심

- 연결을 “종료”하는 과정
    
- 송신/수신이 각각 독립적으로 종료됨 (Full Duplex)
    

■ 한줄  
FIN → ACK → FIN → ACK

📌 시험 포인트

|구분|3-way|4-way|
|---|---|---|
|목적|연결 설정|연결 종료|
|특징|동시 확인|순차 종료|
|이유|상태 동기화|양방향 독립 종료|

📌 핵심 한줄  
연결은 3번 / 종료는 4번

# 📘스케줄링 알고리즘
## 선점 방식 (Preemptive Scheduling)

|알고리즘|특징|트리거|함정|
|---|---|---|---|
|SRT (Shortest Remaining Time)|남은 실행시간 가장 짧은 프로세스 선택|실행 중에도 더 짧은 작업 오면 교체|Starvation 발생|
|Round Robin (RR)|Time Quantum 기준 순환|타임슬라이스|Quantum 너무 크면 FCFS됨|
|Priority Scheduling (선점형)|우선순위 높은 프로세스가 선점|우선순위|낮은 프로세스 Starvation|
|Multilevel Queue (MLQ)|큐별 우선순위, 큐 간 선점 가능|시스템/인터랙티브 구분|큐 간 이동 없음|
|Multilevel Feedback Queue (MLFQ)|큐 이동 가능, 동적 우선순위|I/O vs CPU 패턴|설계 복잡|

## 비선점 방식 (Non-Preemptive Scheduling)

| 알고리즘                              | 특징            | 트리거      | 함정            |
| --------------------------------- | ------------- | -------- | ------------- |
| FCFS (First Come First Serve)     | 먼저 온 순서 처리    | 도착 순서    | Convoy Effect |
| SJF (Shortest Job First)          | 실행시간 가장 짧은 작업 | 실행시간     | 실제 시간 예측 어려움  |
| Priority Scheduling (비선점)         | 우선순위 높은 것 먼저  | 우선순위     | Starvation    |
| HRN (Highest Response Ratio Next) | (대기+실행)/실행    | Aging 반영 | 계산 필요         |

## 빠지는 경우 많이 틀리는 포인트

- RR → 선점이다 (무조건)
    
- SJF vs SRT →
    
    - SJF = 비선점
        
    - SRT = 선점
        
- Priority → 선점/비선점 둘 다 존재
    
- MLFQ → 선점 기반 + Aging 개념 포함
    
- MLQ vs MLFQ
    
    - MLQ = 이동 없음
        
    - MLFQ = 이동 있음 (핵심 차이)
        

## 시험 트리거 한줄

- “중간에 뺏는다” → 선점
    
- “끝날 때까지 실행” → 비선점
    
- “남은시간 기준” → SRT
    
- “응답시간 개선 / Aging” → HRN or MLFQ

# 📘 흐름제어 2계층

### [Stop-and-Wait ARQ]

■ 정의  
= 한 프레임 보내고 ACK 받을 때까지 대기

■ 윈도우  
= Ws = 1, Wr = 1

■ 시퀀스 번호  
= 0, 1 (1비트면 충분)

■ 특징

- 구조 단순 
    
- 효율 최악 (대기시간 큼)
    

■ 트리거  
= “하나씩 전송 / 기다림”

■ 함정  
= 윈도우 개념 약함 → 1로 고정

### [Go-Back-N ARQ]

■ 정의  
= 오류 발생 시 해당 프레임부터 이후 전부 재전송

**■ 윈도우**  
**= 송신윈도우= 2^m - 1**  
**= 수신윈도우= 1(고정)**

### [Selective Repeat ARQ]

■ 정의  
= 오류 난 프레임만 선택 재전송

**■ 윈도우**  
**= 송신윈도우= (2^m) / 2**  
==**= 수신윈도우= 1(고정)

■ 정의

- 충돌 없음

특징
- 충돌 발생
- 충돌 후 재전송

- Polling → 중앙이 순서 지정

- Reservation → 데이터 송신하기전 미리 예약
    
특징
- 충돌 없음
- 지연 발생 가능

# 📘 **IPV4 vs IPV6 헤더 구성 비교**

#### 1️⃣ **IPV4 헤더 구성 (기본 20바이트)**

|필드|크기|설명|
|---|---|---|
|**버전 (Version)**|4비트|**IPv4**일 경우 4|
|**헤더 길이 (Header Length)**|4비트|헤더의 길이를 **32비트 워드** 단위로 표시|
|**서비스 타입 (Type of Service, TOS)**|8비트|패킷의 **우선순위** 및 **서비스 요구사항**|
|**전체 길이 (Total Length)**|16비트|**패킷의 전체 길이** (헤더 + 데이터)|
|**식별자 (Identification)**|16비트|패킷을 **식별**하는 번호|
|**플래그 (Flags)**|3비트|패킷의 **분할 여부**를 나타내는 플래그|
|**조각 오프셋 (Fragment Offset)**|13비트|패킷의 **조각화**를 위한 오프셋 값|
|**TTL (Time to Live)**|8비트|패킷이 **네트워크에서 살아 있는 시간** (초 단위)|
|**프로토콜 (Protocol)**|8비트|**상위 계층 프로토콜** (TCP, UDP 등)|
|**헤더 체크섬 (Header Checksum)**|16비트|헤더의 **오류 검출**|
|**출발지 IP 주소 (Source IP Address)**|32비트|출발지의 **IPv4 주소**|
|**목적지 IP 주소 (Destination IP Address)**|32비트|목적지의 **IPv4 주소**|
|**옵션 (Options)**|가변 크기|**옵션 필드** (옵션을 포함할 경우에만 존재)|

- **기본 크기**: **20바이트**

- **옵션** 포함 시 **최대 60바이트**까지 확장

#### 2️⃣ **IPV6 헤더 구성 (고정 40바이트)**

| 필드                                  | 크기    | 설명                                |
| ----------------------------------- | ----- | --------------------------------- |
| **버전 (Version)**                    | 4비트   | **IPv6**일 경우 6                    |
| **트래픽 클래스 (Traffic Class)**         | 8비트   | **서비스 품질** 및 **우선순위** 정의          |
| **플로우 레이블 (Flow Label)**            | 20비트  | 특정 흐름에 대해 **처리 우선순위**를 지정         |
| **페이로드 길이 (Payload Length)**        | 16비트  | **데이터 부분**의 길이                    |
| **다음 헤더 (Next Header)**             | 8비트   | 상위 계층 프로토콜 (TCP, UDP 등)           |
| **홉 제한 (Hop Limit)**                | 8비트   | 패킷이 **네트워크에서 살아 있는 시간** (TTL과 동일) |
| **출발지 주소 (Source IP Address)**      | 128비트 | 출발지의 **IPv6 주소**                  |
| **목적지 주소 (Destination IP Address)** | 128비트 | 목적지의 **IPv6 주소**                  |

- **기본 크기**: **40바이트** (옵션 없음)

- **옵션**은 **확장 헤더**로 분리하여 처리

- 홀드다운: **180초**

# 📘  TCP 혼잡제어 알고리즘 정리



### 1. AIMD (Additive Increase / Multiplicative Decrease)

- 혼잡 윈도우(cwnd) 크기를 늘릴 때는 **1씩 선형 증가** (1 → 2 → 3 → 4 → ...)
- 혼잡이 감지되면 윈도우 크기를 **절반으로 감소**
- 늘어날 때는 **+1**, 줄어들 때는 **×1/2**
- **공평한 방식** — 네트워크에 늦게 참여한 노드도 시간이 지나면 네트워크 평균에 수렴
- **단점:** 윈도우 크기를 올리는 속도가 너무 느림

### 2. 느린 출발 (Slow Start)

- cwnd 값이 **1에서 시작**하여 **2, 4, 8…** 순으로 **지수적으로 증가**
- 한계치(ssthresh)에 도달할 때까지 지수 증가, 혼잡 감지 시 윈도우 크기를 **1로 감소**
- AIMD의 단점(선형 증가로 인한 속도 지연)을 개선

### 3. TCP 혼잡제어 정책

|정책|설명|
|---|---|
|**Tahoe**|가장 기본. Slow Start → 혼잡 감지 시 AIMD(혼잡 회피) 방식으로 전환|
|Reno / New Reno / Cubic|Tahoe를 발전시킨 정책들|

- 공통 전제: **혼잡 발생 시 윈도우 크기를 줄여 혼잡을 회피**

### 4. Tahoe & Reno정책 동작 흐름

```
cwnd = 1 (시작)
  ↓
[Slow Start 구간] 지수 증가
  ↓
ssthresh 도달
  ↓
[Congestion Avoidance 구간] 선형 증가 (AIMD)
  ↓
혼잡 감지 → cwnd = 1, ssthresh = W(loss) / 2 로 재설정
  ↓
다시 Slow Start부터 반복
```


### 5. 혼잡 감지 조건

- **타임아웃** 발생  -> Tahoe,Reno : cwnd 1로 감소 
- **3번의 중복 ACK** 수신  -> Tahoe : cwnd 1로 감소, **Reno : cwnd 1/2로 감소 (임계값이랑 동일해지기때문에 SlowStart 뛰어넘음)**


### 6. Slow Start 임계점 (ssthresh)

- **의미:** 해당 지점까지만 Slow Start를 사용하겠다는 기준값
- **설정 방법:** 송신측이 ssthresh 값을 **혼잡 윈도우 크기의 절반**으로 정함)

# 📘  데이터 압축 기법 정리

### 무손실 압축 (Lossless)

| 기법                          | 한글명           | 설명                                                  |
| --------------------------- | ------------- | --------------------------------------------------- |
| **Run-length Coding**       | 실행(반복) 길이 부호화 | 연속 반복되는 정보를 횟수+값으로 표현 (TEAAAAAAAAAALOV → TEA@12LOV) |
| **Dictionary Coding (LZW)** | 사전 부호화        | 반복 어구를 사전에 등록 후 위치 정보로 치환                           |
| **Huffman Coding**          | 허프만 코딩        | 출현 빈도 기반으로 각 단위 정보에 비트 수를 효율적으로 할당                  |
| **Arithmetic Coding**       | 산술 부호화        | 메시지 전체를 구간으로 나누어 이진으로 인코딩                           |

### 손실 압축 (Lossy)

| 기법                                  | 한글명             | 설명                                                   |
| ----------------------------------- | --------------- | ---------------------------------------------------- |
| **Delta Modulation**                | 델타 변조<br>(예측부호) | 양자화된 표본값을 그대로 저장하지 않고, 기준 표본값과 예측값의 **차이값**만 저장      |
| **DCT** (Discrete Cosine Transform) | 이산 코사인 변환(예측부호) | 수학적 변환(Transform Coding) 사용. JPEG, MPEG, H.261 등에 활용 |
| **Quantization**                    | 양자화             | 미리 정의된 quantum 정보를 이용하여 주어진 데이터를 코딩                  |

## 📘  블루투스 / BLE / 지그비 / 비콘

---

### 블루투스 (Bluetooth Classic)

- 표준: IEEE 802.15.1
- 주파수: 2.4GHz ISM 대역
- 거리: 일반적으로 **수 미터 ~ 수십 미터** (출력에 따라 다름)
- 속도: 최대 3Mbps (EDR 기준)
- 토폴로지: 피코넷(Piconet) — 마스터 1 + 슬레이브 최대 7
- 용도: 헤드셋, 키보드, 마우스, 스피커

### BLE (Bluetooth Low Energy)

- 블루투스 4.0부터 도입
- 속도: 규격상 1Mbps, 실효 처리량은 그보다 낮음
- 거리: 일반적으로 **~10m**, 블루투스 5 기준 저전력 모드에서 ~40m
- Classic 대비 **전력 소모 대폭 감소**, 간헐적 소량 데이터 전송에 최적
- 용도: 스마트워치, 헬스케어 센서, 비콘

### 지그비 (ZigBee)

- 표준: IEEE 802.15.4 기반
- 주파수: 2.4GHz
- 거리: **최대 100m** (직접 시퀀스 확산 스펙트럼 방식)
- 속도: **최대 250kbps**
- 특징: 초저전력, 메시(Mesh) / 스타 / 트리 토폴로지 지원
- 용도: 스마트홈 센서, 공장 자동화, 스마트 조명

### 비콘 (Beacon)

- BLE 기반 **일방향 신호 브로드캐스트**
- 대표 규격: iBeacon(Apple), Eddystone(Google)
- 거리: **~5m~50m (안정적 거리 20~30m, 최대 70m~100m)**
- 용도: 실내 측위, 근거리 마케팅
- 특징: 송신만 하고 응답 안 받음 → 전력 소모 극히 낮음

|---|---|---|---|---|

|전력|중|저|초저|초저|
|속도|최대 3Mbps|1Mbps (규격상)|250kbps|-|
|토폴로지|피코넷|스타/메시|메시/스타/트리|브로드캐스트|

## SDN (Software Defined Networking)

### 한 줄 정의

**네트워크 제어(Control)와 데이터 전달(Data)을 분리**해서 소프트웨어로 네트워크를 중앙 집중 관리하는 기술


### 기존 네트워크 vs SDN

| 구분      | 기존 네트워크   | SDN       |
| ------- | --------- | --------- |
| **제어 위치**   | **각 장비에 분산**  | **중앙 컨트롤러**   |
| 설정 방식   | 장비별 개별 설정 | 소프트웨어로 일괄 |
| 유연성     | 낮음        | 높음        |
| 대표 프로토콜 | OSPF, BGP | OpenFlow  |

**컨트롤 플랜(제어기)과 데이터 플랜(하드웨어 스위치)을 나누자 컨트롤플랜을 장비마다가 아닌 1개만 중앙에서 관리하여 모든 데이터플랜을 제어하자!**

SDN에서 데이터 플레인 장비를 **스위치**라고 부르는 이유는

기존 라우터는 **스스로 경로 판단** (3계층 라우팅)을 했는데, SDN에서는 그 판단 기능을 컨트롤러가 다 가져가버리니까 남은 장비는 그냥 **패킷 전달만 하는 스위치 역할**만 하게 되는 거야.

즉

- **기존 라우터** = 경로판단 + 패킷전달 (둘 다)
- **SDN 데이터플레인 장비** = 패킷전달만 (스위치처럼)

### 3계층 구조

```
┌─────────────────────────┐
│     Application Layer    │  ← 네트워크 앱 (방화벽, 로드밸런서 등)
├─────────────────────────┤
│      Control Layer       │  ← SDN 컨트롤러 (뇌)
│   (SDN Controller)       │    경로 계산, 정책 결정
├─────────────────────────┤
│  Infrastructure Layer    │  ← 스위치·라우터 (근육)
│  (Data Plane)            │    단순 패킷 전달만 수행
└─────────────────────────┘
```

**계층 간 통신** 거야.

```
[Application Layer]  ← 앱들 (방화벽, 로드밸런서 등)
        ↕  Northbound API  (컨트롤러 ↔ 앱)
[Control Layer (= Network Control Layer)]      ← SDN 컨트롤러
        ↕  Southbound API  (컨트롤러 ↔ 스위치)
[Data Layer (=Infrastructure Layer)]         ← 스위치들
```

- 컨트롤러 ↔ 앱 : **Northbound API**
- 컨트롤러 ↔ 장비 : **Southbound API (OpenFlow)**


### 핵심 개념

**Control Plane (제어 평면)**

- 경로 결정, 라우팅 테이블 계산
- SDN에서는 컨트롤러가 담당

**Data Plane (데이터 평면)**

- 실제 패킷 전달
- SDN에서는 스위치가 담당
- 컨트롤러 지시에 따라 **Flow Table** 기반으로 동작

**OpenFlow**

- 컨트롤러 ↔ 스위치 간 표준 프로토콜
- 컨트롤러가 스위치의 Flow Table을 직접 제어


### 시험 포인트

- **핵심 = Control/Data Plane 분리**
- **컨트롤러 = 중앙집중식 뇌** (OSPF처럼 분산 아님)
- **OpenFlow = Southbound API 대표**
- **장점** : 유연성↑, 중앙관리, 프로그래밍 가능
- **단점** : 컨트롤러 단일장애점(SPOF), 컨트롤러 과부하 위험

| 구분      | 기술                | 핵심                    |
| ------- | ----------------- | --------------------- |
| 가상화     | NFV, 하이퍼바이저, 컨테이너 | 물리 자원을 논리적으로 추상화      |
| 제어방식 변경 | **SDN**           | Control/Data Plane 분리 |
|         |                   |                       |

근데 시험에서 SDN이 가상화 챕터에 같이 묶여서 나오는 이유는 **"네트워크 자원을 소프트웨어로 유연하게 다룬다"** 는 맥락에서 NFV랑 함께 묶이기 때문이야.

**NFV(네트워크 기능 가상화)랑 SDN은 자주 같이 나오는데**

- **SDN = 네트워크 제어를 소프트웨어로**
- **NFV = 네트워크 장비를 소프트웨어**

## OpenFlow

**정의:** 제어 평면과 데이터 평면을 분리하고 이들을 연결하는 표준 인터페이스

**동작 방식:** 컨트롤러가 스위치에 명령 → 스위치는 명령에 따라 패킷을 전송·수정·폐기. 스위치에 등록된 플로우 엔트리 없는 패킷은 컨트롤러에 문의 후 처리

### Flow 테이블 구조

|구성|설명|
|---|---|
|**조건 (Match fields)**|처리할 조건. 물리적 포트번호 ~ L4 TCP/UDP 포트까지. 이 정보를 **튜플(tuple)** 이라고 부름|
|**처리 (Actions)**|전송·폐기·지정 필드값 수정 등 처리 방법|
|**통계 (Counters)**|플로우 엔트리별 송수신 트래픽 양·패킷 수·바이트 수. 경로 우회 등에 활용|

### Action 종류

1. Forward packet to port(s)
2. Encapsulate and forward to controller
3. Drop packet
4. Send to normal processing pipeline
5. Modify Fields

### OpenFlow 테이블 처리

- 스위치 내 플로우 테이블에 번호 부여, **0번부터 시작**
- 모든 입력 패킷은 **0번 플로우 테이블부터 비교**
- Table 0 → Table 1 → ... → Table n → Packet Out

## 하이퍼바이저 (Hypervisor)

**정의:** 호스트 컴퓨터에서 다수의 OS를 동시 실행시키는 가상 플랫폼 = VMM(Virtual Machine Monitor)

### Type 1 vs Type 2

|구분|Type 1 (베어메탈)|Type 2 (호스티드)|
|---|---|---|
|구조|하드웨어 → 하이퍼바이저 → VM|하드웨어 → OS → 하이퍼바이저 → VM|
|동작|하드웨어 위에서 직접 구동|일반 프로그램처럼 OS 위에서 실행|
|성능|높음|낮음|
|설치|어려움 (드라이버 직접 설정)|쉬움|
|제품|Xen, ESX 서버|VMware, 버추얼PC|

### 전가상화 vs 반가상화

| 구분      | 전가상화                          | 반가상화                      |
| ------- | ----------------------------- | ------------------------- |
| 가상화 범위  | 하드웨어 **전체**                   | 하드웨어 **일부**               |
| OS 수정   | 불필요                           | 커널 **일부 수정** 필요           |
| 하드웨어 제어 | 게스트OS가 직접 통제하는 것처럼 동작         | 일부는 하이퍼바이저가 통제            |
| 성능      | CPU-VT 기술 필요, I/O 복잡 → **낮음** | I/O 단순, RT OS 지원 → **높음** |
| 주요 제품   | VMware ESX 서버                 | XENExpress                |
| 특이사항    | RT OS 지원 **불가**               | HyperCall로 기능 활용          |

### 도커

**컨테이너 실행/관리**

- 이미지로 컨테이너 띄우고, 시작/중지/삭제 관리
- 각 컨테이너는 독립된 프로세스로 격리 실행

**네트워킹**

- 컨테이너 간 통신을 위한 가상 네트워크 구성
- 포트 매핑으로 외부 접근 제어

**볼륨(스토리지) 관리**

- 컨테이너는 꺼지면 데이터 사라짐
- 볼륨으로 호스트와 데이터 공유·영속화

근데 시험 관점에서 도커의 핵심 포인트는 **VM vs 컨테이너 차이**야.

|구분|VM|컨테이너(Docker)|
|---|---|---|
|가상화 대상|하드웨어|OS 프로세스|
|OS|게스트OS 별도 필요|호스트OS 공유|
|크기|무거움 (GB)|가벼움 (MB)|
|시작속도|느림|빠름|
|격리수준|높음|낮음|

## 컨테이너 오케스트레이션 (Container Orchestration)

**정의:** 다수의 컨테이너를 자동으로 배포·확장·관리·복구하는 기술

### 주요 기능

|기능|설명|
|---|---|
|자동 배포|컨테이너를 적절한 서버에 자동 배치|
|오토스케일링|부하에 따라 컨테이너 수 자동 증감|
|로드밸런싱|트래픽을 컨테이너들에 분산|
|자가복구|죽은 컨테이너 자동 재시작|
|서비스 디스커버리|컨테이너 간 자동으로 서로 찾아 통신|


프로비저닝 및 배포 / 구성 및 일정 조정 / 리소스 할당 / 컨테이너 가용성 / 워크로드 밸런싱 기반 컨테이너 스케일링·제거 / 로드 밸런싱 및 트래픽 라우팅 / 컨테이너 상태 모니터링 / 애플리케이션 설정 / 컨테이너 간 보안 유지

### 주요 기능

|기능|설명|
|---|---|
|**서비스 디스커버리**|컨테이너 생성·배치·이동 시 IP·Port 정보 자동 업데이트로 서비스 탐색 지원|
|**스케일링 - 로드밸런싱**|컨테이너의 컴퓨팅 자원 사용량 설정 및 자동 배분|
|**스케일링 - 스케줄링**|늘어난 컨테이너를 적합한 서버에 배포, 서버 다운 시 다른 서버에서 자동 구동|
|**클러스터링**|여러 서버를 묶어 하나의 서버처럼 사용, 가상네트워크로 산재된 서버 연결|
|**로깅/모니터링**|여러 서버를 한 곳에서 상태 모니터링 및 로그 관리|
### 대표 도구

|도구|특징|
|---|---|
|**Kubernetes (k8s)**|업계 표준, 구글 개발, 가장 많이 사용|
|Docker Swarm|도커 내장, 설정 단순, 기능 제한적|
|Apache Mesos|대규모 클러스터, 복잡|

### Kubernetes 핵심 구조

```
Master Node (제어)
├── API Server   ← 모든 요청 처리
├── Scheduler    ← 컨테이너 배치 결정
└── etcd         ← 클러스터 상태 저장

Worker Node (실행)
└── Pod          ← 컨테이너 실행 최소 단위
```


### 구성 요소

|구성|설명|
|---|---|
|**Master**|설정 환경 저장, 클러스터 전체 관리. API서버·스케줄러·컨트롤러매니저·etcd로 구성. kubectl로 세팅|
|**Worker Node**|마스터 명령을 받아 실제 작업 수행. Pod들이 배치되는 서비스 컴포넌트|
|**Pod**|컨테이너 배포·관리 단위. 하나 이상의 컨테이너 포함. 같은 Pod 내 컨테이너는 로컬통신 가능 + 디스크 공유|
|**ReplicaSet**|여러 Pod를 필요한 수만큼 미리 실행시켜 두는 장치|
|**kubelet**|각 노드에서 Pod 상태 관리 에이전트|
|**kube-proxy**|각 노드의 네트워크 규칙 관리|

### Master 내부 구성

|컴포넌트|역할|
|---|---|
|API Server|모든 요청의 진입점|
|Scheduler|컨테이너 배치 결정|
|Controller-Manager|클러스터 상태 유지|
|etcd|클러스터 설정·상태 저장 (분산 키-값 저장소)|

### 핵심 특징

**Fault-tolerance:** 기존 서비스 중단 없이 점진적 수시 업데이트 가능. 특정 컨테이너 장애 발생 시 즉시 복제 컨테이너 생성해 서비스 유지

### 시험 포인트

- **오케스트레이션 = 컨테이너 자동 관리**
- **Kubernetes = 사실상 표준**
- **Pod = k8s에서 컨테이너 묶음 단위**
- **도커 단독 vs 오케스트레이션** 차이 → 도커는 단일 호스트, 오케스트레이션은 **멀티 호스트** 관리

## 클라우드 컴퓨팅 서비스 모델

### IaaS / PaaS / SaaS 기본 구조

|구분|제공 범위|예시|
|---|---|---|
|**IaaS**|인프라(서버,스토리지,네트워크)|AWS EC2, Azure VM|
|**PaaS**|실행환경+인프라|Google App Engine|
|**SaaS**|애플리케이션 전체|Gmail, Office365|
|**BaaS**|백엔드 기능을 API로 제공|Firebase, AWS Amplify|
|**FaaS**|함수 실행환경 제공|AWS Lambda, Azure Functions|

### 서버리스 = BaaS + FaaS

**BaaS (Backend as a Service)**

- 로그인·DB·파일저장·푸시알림 등 대중적 백엔드 기능을 API로 제공
- 내 코드 없이 갖다 쓰기만 함

**FaaS (Function as a Service)**

- 내가 짠 코드를 함수 단위로 올리면 호출될 때만 실행
- 실행 시간·횟수만큼만 과금
- 제약 : 실행시간 제한, 메모리 제한, 무상태

**실제 사용 패턴**

```
대중적 기능 (로그인, DB, 알림) → BaaS 갖다 쓰기
기업 고유 비즈니스 로직       → FaaS로 함수 직접 작성
```

### PaaS vs 서버리스 차이

|구분|PaaS|서버리스|
|---|---|---|
|배포 단위|애플리케이션 전체|함수 단위|
|과금|서버 켜있는 시간|실행될 때만|
|스케일링|개발자 설정|자동|
|서버 관리|일부 개발자|완전 사업자|

### 시험 포인트

- **서버리스 = BaaS + FaaS**
- **FaaS = 함수 단위 실행, 호출시만 과금**
- **BaaS = 흔한 백엔드 기능 API 제공**
- 서버리스는 PaaS보다 한 단계 더 추상화
- 복잡한 비즈니스 로직은 FaaS로 직접 구현

## API 게이트웨이 (API Gateway)

**정의:** 클라이언트와 백엔드 서비스 사이에서 **모든 API 요청의 단일 진입점** 역할을 하는 서버

### 구조

```
클라이언트
    ↓
API Gateway  ← 단일 진입점
    ↓ ↓ ↓
서비스A  서비스B  서비스C  (마이크로서비스들)
```

### 주요 기능

|기능|설명|
|---|---|
|**인증/인가**|토큰 검증, 접근 권한 확인|
|**라우팅**|요청을 적절한 서비스로 전달|
|**로드밸런싱**|트래픽 분산|
|**요청/응답 변환**|프로토콜 변환, 데이터 포맷 변환|
|**속도 제한 (Rate Limiting)**|과도한 요청 차단|
|**로깅/모니터링**|요청 기록 및 추적|
|**캐싱**|응답 캐싱으로 성능 향상|

### MSA에서 API Gateway가 필요한 이유

MSA는 서비스가 수십~수백 개로 분산되어 있어서 클라이언트가 각 서비스를 직접 호출하면

- 서비스 주소를 다 알아야 함
- 인증을 서비스마다 각각 처리해야 함
- 관리가 복잡해짐

→ API Gateway 하나로 **통합 관리**

### 시험 포인트

- **단일 진입점 (Single Entry Point)** 이 핵심
- **MSA와 항상 묶어서** 나옴
- 인증·라우팅·로드밸런싱·모니터링 기능 담당
- SDN의 컨트롤러랑 역할이 비슷한 느낌 (중앙에서 제어)
## 프록시 (Proxy)

**정의:** 클라이언트와 서버 사이에서 **대신 통신해주는 중계 서버**

### 구조

```
클라이언트 → 프록시 서버 → 인터넷 → 목적지 서버
```

### 종류

|구분|위치|역할|
|---|---|---|
|**Forward Proxy**|클라이언트 앞|클라이언트 대신 요청. IP 숨김, 접근제어|
|**Reverse Proxy**|서버 앞|서버 대신 응답. 로드밸런싱, 서버 IP 숨김|

### Forward vs Reverse 한 줄 차이

- **Forward** = 클라이언트 신원 숨김 (내가 누군지 서버가 모름)
- **Reverse** = 서버 신원 숨김 (어느 서버인지 클라이언트가 모름)

### 주요 기능

- 캐싱 (같은 요청 반복 시 저장된 응답 반환)
- 접근 제어 (특정 사이트 차단)
- 로드밸런싱 (Reverse Proxy)
- 보안 (실제 서버 IP 노출 방지)

### API Gateway vs Reverse Proxy

|구분|Reverse Proxy|API Gateway|
|---|---|---|
|목적|트래픽 중계·분산|API 통합 관리|
|기능|단순|인증·변환·모니터링 등 복잡|
|대상|일반 서버|주로 MSA|


```
Reverse Proxy
    + 인증/인가
    + 라우팅
    + Rate Limiting
    + 로깅/모니터링
    + 요청변환
= API Gateway
```

그래서 API Gateway를 **"스마트 Reverse Proxy"** 라고 보면 딱 맞아.

Reverse Proxy는 그냥 대신 받아서 넘겨주는 거고, API Gateway는 넘겨주면서 **이것저것 다 처리**하는 거야.


### 시험 포인트

- **Reverse Proxy = 로드밸런서 역할도 함**
- **API Gateway는 Reverse Proxy의 확장판** 느낌
- 캐싱으로 서버 부하 감소가 자주 출제 포인트

## 하둡 (Hadoop)

**정의:** 아파치 재단의 빅데이터 저장·분석·활용 기술. 대용량 데이터 처리를 위한 대규모 분산 컴퓨팅 지원 프레임워크

### 빅데이터 처리 순서

수집(Collecting) → 저장(Store) → 처리·분석(Analysis) → 시각화(Visualizing)

### 핵심 2가지 (시험 必)

**HDFS (Hadoop Distributed File System)**

- 파일을 **블록(64M) 단위**로 나눠 다중 노드에 분산 저장
- **리플리케이션** : 블록을 여러 노드에 복제 (기본 3개) → 장애 대응
- **Namenode** : 메타데이터 관리, 클라이언트 접근 처리 (단일 마스터)
- **Datanode** : 실제 파일 블록 저장
- 순차처리 특성 (스트리밍 접근)

**MapReduce**

- 구글이 제안한 분산 처리 프레임워크
- **Map** : 흩어진 데이터를 (key, Value) 형태로 분류·묶기
- **Shuffle** : 정렬, 파티셔닝
- **Reduce** : 중복 제거하고 원하는 데이터 추출
- 순서 : Map → Shuffle → Reduce

### 하둡 에코시스템 (분야별)

| 분야  | 도구            | 설명                             |
| --- | ------------- | ------------------------------ |
| 저장  | **HDFS**      | 분산 파일 저장                       |
| 저장  | **HBase**     | 빅데이터용 컬럼기반 DB                  |
| 처리  | **MapReduce** | 분산 배치 처리                       |
| 질의  | **Hive**      | SQL 유사 HiveQL, MapReduce 대신 수행 |
| 질의  | **Pig**       | Pig Latin 언어로 데이터 질의           |
| 분석  | **Mahout**    | 머신러닝 (분류/군집/추천)                |
| 관리  | **Zookeeper** | 분산 서버 조정, 로드밸런싱, 동기화           |
| 관리  | **Ambari**    | 하둡 클러스터 모니터링                   |
| 수집  | **Flume**     | 분산환경 Log 수집                    |
| 수집  | **Sqoop**     | RDBMS ↔ 하둡 데이터 포팅              |

### 하둡 버전 비교

| 구분   | Hadoop 1.0    | Hadoop 2.0                                          |
| ---- | ------------- | --------------------------------------------------- |
| 처리방식 | 배치만 가능        | 배치·실시간·스트리밍 모두                                      |
| 자원관리 | JobTracker 단독 | **YARN** (Resource Manager + Application Master 분리) |
| 확장성  | 4천대 이상 불가     | 최대 10000개 노드                                        |
| 처리엔진 | MapReduce만    | Spark, Storm 등 다양한 엔진 지원                            |

### Storm (실시간 처리)

**정의:** 실시간 데이터 처리를 위한 분산 환경 기반 스트리밍 시스템

|구성|설명|
|---|---|
|**Nimbus**|마스터 노드, 중앙 컨트롤러|
|**Supervisor**|워커 노드, 실제 작업 실행|
|**Zookeeper**|노드 상태 모니터링·공유|
|**Spout**|데이터 소스 (Kafka 등에서 데이터 수신)|
|**Bolt**|실제 데이터 처리 수행|

### 하둡 3.0 핵심

**Erasure Coding (EC)**

- 기존 : 1G 데이터 → 3개 복제 → 3G 사용
- EC : 1G 데이터 → n등분 → 패리티 셀 생성 → **1.5G만 사용**
- 저장 효율 대폭 향상, Fault Tolerance 유지

### 시험 포인트

- **HDFS + MapReduce = 하둡의 핵심 2가지**
- **Map → Shuffle → Reduce 순서**
- **Hive = SQL 유사 (HiveQL)**, MapReduce 대체
- **YARN = Hadoop 2.0 자원관리 플랫폼**
- **Storm = 실시간**, MapReduce = 배치
- **Namenode = 단일 마스터 → SPOF 주의**

## NoSQL

**정의:** Not Only SQL. SQL만 고집하지 말고 서비스에 특화된 DB를 선택하자는 아이디어. RDBMS 대체가 아닌 **RDBMS가 적합하지 않은 곳에서의 대안**

### RDBMS 한계 (NoSQL 등장 이유)

- 급격한 데이터 증가를 스케일업으로 감당 어려움
- 고비용
- 모든 데이터가 트랜잭션·강한 일관성이 필요한 건 아님
- 비정형 데이터 처리 요구 증가

### NoSQL 특징

- 고정된 스키마 없음 → 확장성 뛰어남
- 원자적 트랜잭션 지원 어려움
- 조인 지원 안 함 → 앱에서 직접 구성
- 무결성보다 **확장성** 우선

### NoSQL 4가지 유형

|유형|구조|대표 DB|
|---|---|---|
|**키-값 스토어**|Key : Value|Redis|
|**컬럼형**|Key : 여러 Column-Value|HBase, Cassandra|
|**문서형**|Key : JSON/XML Document|MongoDB, CouchDB|
|**그래프형**|노드와 노드 간 관계|Neo4J|

### CAP 이론 (시험 必)

**분산 컴퓨팅 환경은 3가지 특성 중 2가지만 만족 가능**

|특성|설명|
|---|---|
|**C (Consistency)**|어느 노드로 접근해도 같은 데이터|
|**A (Availability)**|노드 장애 시에도 정상 요청 처리 가능|
|**P (Partition Tolerance)**|네트워크 장애 시에도 서비스 수행 가능|

**CAP 조합별 대표 DB**

|조합|대표|
|---|---|
|CA|RDBMS (MySQL, PostgreSQL)|
|CP|MongoDB, HBase, BigTable|
|AP|Cassandra, CouchDB, Riak|
|CP (키-값)|Redis, MemcacheDB|

### 시험 포인트

- **NoSQL = RDBMS 대체 아님**, 보완적 관계
- **CAP 이론 = 3가지 중 2가지만 가능**
- **키-값 → Redis / 컬럼 → HBase, Cassandra / 문서 → MongoDB / 그래프 → Neo4J**
- **컬럼형 = 열 중심**, 관계형 = 행 중심

## 가용성 계산

**가용성 공식**

```
가용성 = MTTF / (MTTR + MTTF) × 100
```

### 용어 정리

| 용어       | 풀네임                       | 의미                                         |
| -------- | ------------------------- | ------------------------------------------ |
| **MTTF** | Mean Time to Failure      | 정상 가동 시작 후 다음 고장까지 **평균 사용 시간**            |
| **MTTR** | Mean Time to Repair       | 고장을 **수리하는 평균 시간**                         |
| **MTBF** | Mean Time Between Failure | 고장 시작 후 다음 고장까지 **평균 고장 간격** = MTTF + MTTR |

### 예제 풀이

**문제:** 40시간 운영 → 10시간 수리 → 20시간 운영 → 10시간 수리

```
MTTF = (40 + 20) / 2 = 30시간  (정상 운영 평균)
MTTR = (10 + 10) / 2 = 10시간  (수리 시간 평균)

가용성 = 30 / (10 + 30) = 30 / 40 = 0.75 = 75%
```

### 시험 포인트

- **공식 = MTTF / (MTTR + MTTF)**
- **MTTF ↑, MTTR ↓ → 가용성 ↑**
- **MTBF = MTTF + MTTR** (고장 간격)
- 여러 번의 데이터 나오면 **평균값** 먼저 구하고 공식 적용

## 고가용성 HA (High Availability)

**정의:** 서버·네트워크·프로그램 등 정보시스템이 오랜 기간 지속적으로 정상 운영 가능한 성질

**5 nines (99.999%)** = 1년에 5분 15초 이하 장애시간 허용 → 고품질 데이터센터 목표

### HA 구성방식 3가지

|구분|Hot Standby|Mutual Takeover|Concurrent Access|
|---|---|---|---|
|방식|Active + Standby 구성. 장애 시 Standby로 Failover|2개 시스템이 각각 고유 기능 수행하다 장애 시 상대 자원 인수하여 2개 업무 동시 수행|다수 시스템이 동일 기능 수행 (로드밸런싱 유사). 장애 시 Failover 불필요|
|장점|Failover 후에도 동일 성능 보장|자원 효율적 이용|Failover 불필요|
|단점|자원 낭비 (하나는 항상 대기)|장애 후 한 서버에 업무 몰릴 수 있음|로드밸런싱 위한 추가 장비 필요|

### HA 다운타임 (암기)

| 가용성         | Nines           | 1년 중 다운타임  |
| ----------- | --------------- | ---------- |
| 90%         | one nine        | 36.53일     |
| 99%         | two nines       | 3.65일      |
| **99.9%**   | **three nines** | **8.77시간** |
| 99.99%      | four nines      | 52.60분     |
| **99.999%** | **five nines**  | **5.26분**  |
| 99.9999%    | six nines       | 31.56초     |

### 시험 포인트

- **5 nines = 99.999% = 1년에 5분 15초** 이하
- **Hot Standby = Active/Standby** 구조
- **Mutual Takeover = 장애 시 상대 자원 인수**
- **Concurrent Access = 로드밸런싱 유사, Failover 불필요**
- 용량산정 3가지 방법 이름과 특징

## 성능 계산

### 용어 정리

| 용어                          | 설명                             |
| --------------------------- | ------------------------------ |
| **응답시간 (Response Time)**    | 사용자가 요청 보내고 응답 받을 때까지 시간       |
| **Think Time**              | 응답 받은 후 다음 요청 전까지 대기 시간        |
| **요청간격 (Request Interval)** | 응답시간 + Think Time              |
| **활성사용자 (Active User)**     | 지금 응답시간에 속해 있는 사용자             |
| **동시사용자 (Concurrent User)** | PC 앞에 앉아 있는 사용자 (요청간격에 속한 사용자) |
| **TPS**                     | 초당 처리 건수                       |

### 핵심 공식 (Little's Law)

```
TPS = 활성사용자 / 평균응답시간(초)
TPS = 동시사용자 / 서비스요청간격(초)

활성사용자 = TPS × 응답시간
동시사용자 = TPS × 요청간격
```

**관계식**

- Concurrent Users(동시사용자) = Active Users(활성사용자 ) + Inactive Users(대기 사용자)
- TPS = Concurrent Users(동시사용자) ÷ Request Interval(요청간격)

### 예제

**햄버거 가게 : 1시간에 360명 방문, 평균 식사시간 10분**

```
TPS = 360명 ÷ 3600초 = 0.1 TPS
평균응답시간 = 600초 (10분)
활성사용자 = 0.1 × 600 = 60명
```

### 시험 포인트

- **활성사용자 = TPS × 응답시간**
- **동시사용자 = TPS × 요청간격**
- **요청간격 = 응답시간 + Think Time**
- 동시사용자 > 활성사용자 (Think Time만큼 더 넓음)

## 스토리지 DAS / NAS / SAN

|구분|DAS|NAS|SAN|
|---|---|---|---|
|풀네임|Direct Attached Storage|Network Attached Storage|Storage Area Network|
|연결|서버에 **직접** (SCSI, FC)|**LAN** (TCP/IP)|**전용망** (Fiber Channel)|
|접근단위|블록|**파일**|블록|
|공유|불가 (서버 각각 관리)|가능 (파일서버 통해)|가능 (FC Switch 통해)|
|속도|빠름|보통 (접속 증가시 저하)|**빠름** (고속전송)|
|확장성|낮음 (연결 수 한계)|높음|높음|
|비용|저렴|중간|고가|
|특징|서버가 직접 파일시스템 관리|파일서버가 중간 매체 역할|데이터 전용망, 장거리 연결 가능|

### NAS 통신 경로

```
서버 → TCP/IP(LAN) → 파일서버 → SCSI/FC → Storage
```

### 시험 포인트

- **NAS = 파일 단위, SAN = 블록 단위**
- **NAS = LAN 공유, SAN = 전용망**
- **NAS 단점 = 접속 증가시 성능 저하**
- **IP SAN** = SAN을 FC 대신 IP로 구현 (저렴)
- DAS → NAS → SAN 순으로 공유·확장성 높아짐

| MPL 2.0    | 약함       | 모질라 공중 라이센스 / 수정 파일만 공개 / 실행파일은 독점 배포 가능 |
| CDDL       | 약함       | MPL 유사 / 썬 마이크로시스템즈가 MPL 개조              |
| EPL / CPL  | 약함       | IBM이 이클립스 등에서 만든 라이센스 / GPL 유사하나 완화      |
| GPL 2.0    | 강함       | 수정·링크 시 소스 공개 필수 / 2차 저작물도 GPL 적용        |
| GPL 3.0    | 강함       | GPL 2.0 + 설치정보 제공 의무 + 특허 개선자 면책 조항      |
| AGPL 3.0   | 가장 강함    | 네트워크 서비스 제공 시에도 소스 공개 의무                 |

### TTA 표준 분류 기준

|분류|종류|해당 라이센스|
|---|---|---|
|허용적 (Permissive)|라이센스 변경 허용|Apache 2.0, BSD-2, BSD-3, MIT|

## 분산처리 시스템

개념: 중앙집중식과 대비 / 분산된 시스템이 작업 처리 후 결과 상호 교환 / 사용자는 전체를 하나의 컴퓨터로 인식

### 분산처리 시스템의 투명성 (Transparency)

목적: 다수의 컴퓨터를 마치 한 대의 컴퓨터처럼 보이게 하는 것

|투명성|영문|핵심 내용|
|---|---|---|
|접근 투명성|Access|원격/내부 관계없이 동일한 접근 방식으로 접속|
|위치 투명성|Location|자원이 어느 컴퓨터에 있는지 알 필요 없이 이용|
|병행(동시수행) 투명성|Concurrency|다수 트랜잭션 동시 수행해도 결과 일관성 유지|
|중점(복사) 투명성|Replication|동일 자원이 여러 곳에 존재해도 하나로 보임|
|고장(장애) 투명성|Failure|구성 요소 장애 발생해도 서비스 정상 제공|
|이주(이동) 투명성|Migration|자원이 다른 컴퓨터로 이동해도 사용자는 의식 못함|
|성능 투명성|Performance|시스템 재구성해도 사용자는 의식 못함|
|규모 투명성|Scaling|구성 요소 추가/제거 등 규모 변화를 의식 못함|
|이기종 투명성|Heterogeneity|다른 H/W·S/W 구성이어도 사용자는 의식 못함|

## 머신러닝 / 딥러닝 핵심 정리

### 학습 모드 & 알고리즘 매핑

|학습 방식|딥러닝 알고리즘|사용 분야|
|---|---|---|
|지도학습|CNN, RNN|컴퓨터 비전(CNN) / 자연어처리(NMT)|
|비지도학습|오토인코더|-|
|강화학습|Q-Learning, DQN|게임 (알파고)|
|생성모델|GAN|이미지 생성|

### 학습 단위

|용어|설명|
|---|---|
|epoch|전체 데이터셋을 신경망이 한 번 통과한 횟수 / 작으면 과소적합, 크면 과적합|
|batch size|전체 데이터를 나눈 소그룹의 데이터 수 / 크면 학습 느림·메모리 부족, 작으면 훈련 불안정|
|iteration|1 epoch을 마치는데 필요한 미니배치 수 (= 스텝)|

### CNN 핵심

|개념|설명|
|---|---|
|Convolution|Input × Filter → Feature Map 생성|
|Stride|필터 순회 간격|
|Padding|Feature Map 크기 축소 방지 / 외곽을 0으로 채움|
|Pooling|Feature Map 크기 축소 / Max·Average·Min Pooling|
|출력크기 공식|**O = (I - F + 2P) / S + 1**|

### GAN 문제점

|문제|설명|
|---|---|
|Vanishing Gradients|Discriminator가 너무 잘 학습 → Generator gradient 소실|
|Mode Collapse|Generator가 다양한 데이터 못 만들고 비슷한 것만 생성|

DCGAN: GAN의 fully-connected를 convolution으로 대체 / 배치 정규화로 학습 안정화

### RNN 핵심

|구분|내용|
|---|---|
|특징|순환구조 / 과거 정보를 현재 학습에 반영 / 시계열·순서 데이터에 특화|
|활성화 함수|tanh (하이퍼볼릭 탄젠트) / ReLU 쓰면 오버플로우 발생|
|한계|기울기 소실 / 과적합 / 병렬처리 어려움 / 장기 의존성 학습 어려움|

### LSTM (RNN 한계 보완)

|게이트|역할|
|---|---|
|망각 게이트 (Forget)|셀 정보 지울지 말지 결정 / 0=지움, 1=기억|

GRU = LSTM 간소화 버전

### Transformer / BERT / GPT

|구분|GPT|BERT|
|---|---|---|
|풀네임|Generative Pre-trained Transformer|Bidirectional Encoder Representations from Transformers|

|강점|문장 생성|문장 의미 추출|
