![[Pasted image 20260330164607.png]]정답 **①**

n은 공개키 {e, n}에도 포함되어 있어. 즉 **n은 공개값**이야. 개인키의 구성요소이기도 하지만 공개하지 않는다는 건 틀려.

**RSA 핵심 정리**

| 항목  | 내용                       |
| --- | ------------------------ |
| n   | p × q (두 소수의 곱), **공개값** |
| e   | 공개키 지수, 공개값              |
| d   | 개인키 지수, 비밀값              |
| 공개키 | {e, n}                   |
| 개인키 | {d, n}                   |
| 암호화 | C = P^e mod n (공개키)      |
| 복호화 | P = C^d mod n (개인키)      |
**P = 평문 (Plaintext)**

**C = 암호문 (Ciphertext)**

암호화: 평문 P를 공개키로 암호화 → C = P^e mod n

복호화: 암호문 C를 개인키로 복호화 → P = C^d mod n

비밀로 유지해야 하는 건 **d** (개인키 지수)와 **p, q** (원래 두 소수) 야.

**풀이**

주어진 규칙의 의미:

|옵션|의미|
|---|---|
|`-A INPUT`|INPUT 체인에 규칙 추가|
|`-s 192.168.0.8/32`|**출발지(source) IP**가 192.168.0.8인 패킷|
|`-j DROP`|해당 패킷을 차단|

이 호스트 자신의 IP가 192.168.0.8이므로, 이 규칙은 **"자기 자신을 출발지로 위장한 패킷이 들어오면 차단"** 하는 규칙이다.

**Land Attack**은 공격자가 패킷의 **출발지 IP와 목적지 IP를 동일하게** (여기서는 둘 다 192.168.0.8) 설정하여 전송하는 공격으로, 피해자 시스템이 자기 자신에게 계속 응답을 생성하며 자원을 고갈시킨다.

위 iptables 규칙은 바로 이 상황, 즉 **자신의 IP가 출발지로 설정된 패킷**을 DROP함으로써 Land Attack을 방어한다.

**나머지 오답 이유**

|선택지|오답 이유|
|---|---|
|① IP Fragmentation|패킷 분열(단편화) 공격으로, 출발지 IP 차단으로 방어하지 않음|
|③ ARP Spoofing|L2(데이터링크) 계층 공격이며, iptables는 L3/L4 계층에서 동작하므로 방어 불가|
|④ ICMP Redirect|라우팅 경로 조작 공격으로, 출발지 IP 필터링과 무관|

**각 선택지 분석**

|---|---|---|
|① SSL/TLS는 부인방지 기능을 제공한다|**오답**|SSL/TLS가 제공하는 보안 서비스는 **기밀성, 무결성, 인증(서버/클라이언트)**이며, 부인방지(Non-repudiation)는 제공하지 않는다. 부인방지는 상위 애플리케이션 계층의 전자서명으로 구현|
|② 전송데이터가 암호화되므로 IP주소가 노출되지 않는다|**오답**|SSL/TLS는 **애플리케이션 페이로드(L7)**를 암호화하며, **IP 헤더(L3)**는 암호화하지 않는다. IP 주소는 여전히 평문으로 노출됨. IP까지 숨기려면 VPN(IPsec 등) 필요|

**TLS 서브 프로토콜 정리 (RFC 5246)**

|서브 프로토콜|역할|
|---|---|

|Handshake Protocol|세션 협상, 인증서 교환, 키 교환|
|Change CipherSpec Protocol|암호화 파라미터 전환 신호|
|Alert Protocol|오류 및 경고 메시지 전달|

**정답: ③ 클라이언트 인증서**

**Client Hello 포함 항목 (RFC 5246 기준)**

|포함 항목|선택지 대응|판정|
|---|---|---|
|client_version (TLS 버전)|—|포함|
|random (32바이트 랜덤값)|① 클라이언트 랜덤값|**포함 ✓**|
|cipher_suites (암호 스위트 목록)|② 지원 가능한 암호화 알고리즘 리스트|**포함 ✓**|
|compression_methods|④ 메시지 압축 여부|**포함 ✓**|
|session_id|—|포함|
|extensions (optional)|—|포함|
|**클라이언트 인증서**|③ 클라이언트 인증서|**미포함 ✗**|

**③이 오답인 이유**

클라이언트 인증서는 Client Hello에 포함되지 않는다. TLS Handshake 흐름에서 클라이언트 인증서는 다음 순서로 전송된다:

```
Client Hello →
                ← Server Hello
                ← Server Certificate
                ← Certificate Request (서버가 클라이언트 인증 요구 시)
                ← Server Hello Done
Client Certificate  ← 이 시점에 전송
Client Key Exchange →
...
```

즉, 클라이언트 인증서는 서버가 **CertificateRequest** 메시지를 보낸 경우에만, **별도의 Certificate 메시지**로 전송된다. (RFC 5246 §7.4.6)

**정답: ② Active 모드로 동작하는 경우 데이터 전송을 위해 클라이언트가 서버에게 접속을 시도한다.**

**② 오답 이유 — Active vs Passive 모드 비교**

|구분|제어 채널(Control)|데이터 채널(Data) 연결 방향|
|---|---|---|
|**Active 모드**|클라이언트 → 서버:21|**서버:20 → 클라이언트(임의 포트)**|
|**Passive 모드**|클라이언트 → 서버:21|**클라이언트 → 서버(임의 포트)**|

Active 모드에서는 클라이언트가 PORT 명령으로 자신의 포트를 서버에 알려주면, **서버가 클라이언트에게** 데이터 채널 접속을 시도한다. 선택지 ②는 주체가 반대로 서술되어 있으므로 틀렸다.

**나머지 선택지 검증**

|선택지|판정|이유|
|---|---|---|
|① FTP는 평문 전송 → 스니핑 취약|**옳음**|FTP는 암호화 없이 ID·PW·데이터를 평문 전송 (RFC 959)|
|③ Bounce Attack 취약|**옳음**|FTP의 PORT 명령에 제3의 호스트 IP를 지정할 수 있어, 서버를 중계자로 악용하는 FTP Bounce Attack 가능|
|④ TFTP는 UDP, 인증 없음|**옳음**|TFTP는 UDP 69번 포트 사용, 인증 기능 없음 (RFC 1350)|


**FTP Bounce Attack**

FTP의 **Active 모드 PORT 명령어**의 설계 취약점을 악용한 공격이다.

**정상적인 Active 모드 동작**

```
클라이언트 → 서버에게 PORT 명령 전송
"나의 IP(1.1.1.1), 포트(4000)로 데이터 연결해줘"
→ 서버:20 → 클라이언트:4000 으로 데이터 채널 연결
```

**Bounce Attack 악용 방식**

PORT 명령에 **클라이언트 자신의 IP가 아닌 제3의 피해자 IP**를 지정할 수 있다는 점을 악용한다.

```
공격자 → FTP 서버에게 PORT 명령 전송
"데이터 채널을 피해자(3.3.3.3):포트로 연결해줘"
→ FTP 서버:20 → 피해자:해당포트 로 연결 시도
```

**공격 목적 2가지**

|목적|설명|
|---|---|
|**포트 스캔**|FTP 서버가 피해자의 포트에 연결 시도 → 응답으로 열린 포트 파악. 공격자의 IP가 아닌 **FTP 서버 IP로 스캔**되므로 추적 회피 가능|
|**방화벽 우회**|공격자가 직접 접근 못하는 내부 시스템에, **신뢰받는 FTP 서버를 중계자로 이용**하여 접근|

**핵심 요약**

> FTP 서버를 **"대리인(Bounce)"** 으로 이용하여 제3의 시스템을 공격하거나 스캔하는 기법. 공격 출처가 FTP 서버로 위장되므로 공격자 추적이 어렵다.

**대응 방법**

PORT 명령에서 지정한 IP가 **실제 클라이언트 IP와 다를 경우 거부**하도록 FTP 서버를 설정하면 차단 가능하다. (대부분의 현대 FTP 서버는 기본적으로 차단)

```
1  String cmd = args[0];          ← 외부 입력값을 검증 없이 그대로 사용
7  ps = Runtime.getRuntime().exec(cmd);  ← OS 명령어로 직접 실행
```

**핵심 문제점**

`args[0]`으로 받은 **외부 입력값을 아무런 검증·필터링 없이** `Runtime.exec()`에 직접 전달하고 있다. 공격자가 다음과 같은 값을 입력하면 임의 OS 명령어가 실행된다.

```
# 정상 입력
ls

# 공격 입력 예시
ls; rm -rf /
ls && cat /etc/passwd
```

**각 선택지 검토**

|선택지|판정|이유|
|---|---|---|
|① 경로 조작 및 자원 삽입|**오답**|파일 경로(`../` 등)를 조작하여 비인가 파일에 접근하는 취약점. 이 코드는 파일 경로가 아닌 **명령어 자체**를 실행함|
|**② 운영체제 명령어 삽입**|**정답**|외부 입력을 검증 없이 `Runtime.exec()`에 전달 → OS 명령어 임의 실행 가능 (CWE-78)|
|③ 취약한 API 사용|**오답**|`Runtime.exec()` 자체가 취약한 API이기도 하나, 이 문제의 핵심은 입력값 미검증으로 인한 **명령어 삽입**이 주된 취약점|
|④ 중요한 자원에 대한 잘못된 권한설정|**오답**|파일·디렉토리 등의 접근 권한 설정 오류에 해당하며, 이 코드와 무관|

**대응 방법**

- `Runtime.exec()` 사용 자체를 지양
- 불가피한 경우 **허용 목록(whitelist)** 방식으로 입력값 검증
- 명령어와 인자를 분리하여 배열 형태로 전달 (`exec(String[])`)

## 2025년 105번 — SSL/TLS 핸드세이크

**정답: ③ HTTP 전송 데이터에 대한 MAC 계산 및 암호화**

|선택지|판정|근거|
|---|---|---|
|① 클라이언트 인증|✅ 핸드세이크 작업|선택적이지만 핸드세이크에서 수행|
|② 서버 인증|✅ 핸드세이크 작업|인증서 교환으로 서버 인증|
|③ HTTP 전송 데이터 MAC 계산 및 암호화|❌ 핸드세이크 아님|이건 핸드세이크 **이후** 데이터 전송 단계|
|④ 암호 알고리즘 협상|✅ 핸드세이크 작업|ClientHello/ServerHello에서 협상|

**SSL/TLS 핸드세이크 순서**

```
1. ClientHello     → 암호 알고리즘 목록 전송
2. ServerHello     → 알고리즘 선택(협상)
3. 인증서 교환     → 서버 인증
4. 키 교환         → 세션키 생성
5. (클라이언트 인증) → 선택적
6. Finished        → 핸드세이크 완료

→ 이후부터 데이터 암호화 전송 (MAC 계산은 여기서)
```

**핵심**

```
핸드세이크 = 협상 + 인증 + 키교환
실제 데이터 암호화/MAC = 핸드세이크 이후
```

**SSL/TLS 4가지 프로토콜**

|프로토콜|역할|
|---|---|
|Handshake|협상 + 인증 + 키교환|
|Change Cipher Spec|"이제부터 암호화 시작" 신호|
|**Record**|**실제 데이터 암호화/MAC 계산 전송**|
|Alert|오류/경고 메시지|

**이 문제 연결**

```
③번 "HTTP 데이터 MAC 계산 및 암호화"
= Record 프로토콜이 하는 일
= Handshake 프로토콜이 아님
→ 정답
```

**한 줄 암기**

```
Handshake  = 준비
Change     = 시작 신호
Record     = 실제 데이터
Alert      = 에러
```

감리 관련 법령 문제니까 검색 먼저 할게.**102. 개인정보보호법 시행령 제33조에서 대통령령으로 정하는 기준으로써, 개인정보보호법 시행령 제35조(==개인정보 영향평가의 대상==)에 대한 설명으로 가장 부적합한 것은 무엇인가?**

① 구축·운용 또는 변경하려는 개인정보파일로서 **5만명 이상의 정보주체에 관한 민감정보 또는 고유식별정보의 처리**가 수반되는 개인정보파일 
② 구축·운용하고 있는 개인정보파일을 해당 공공기관 내부 또는 외부에서 구축·운용하고 있는 **다른 개인정보파일과 연계하려는 경우로서 연계 결과 10만명 이상의 정보주체**에 관한 개인정보가 포함되는 개인정보파일 
③ 구축·운용 또는 변경하려는 **개인정보파일로서 100만명 이상의 정보주체**에 관한 개인정보파일 ④ 법 제33조제1항에 따른 **개인정보 영향평가를 받은 후에 개인정보 검색체계 등 개인정보파일의 운용체계를 변경하려는 경우 그 개인정보파일. 이 경우 영향평가 대상은 변경된 부분으로 한정한다.**

**정답: ②**

시행령 제35조 제2호는 연계 결과 **50만명** 이상의 정보주체에 관한 개인정보가 포함되는 경우임.

②에서 **10만명**이라고 했으니 틀린 것. 실제는 **50만명**.

|기준|숫자|
|---|---|
|민감정보·고유식별정보|**5만명** 이상|
|연계 결과|**50만명** 이상|
|일반 개인정보|**100만명** 이상|

**104. 개인정보의 안전성 확보조치 기준에 따른 제8조(접속기록의 보관 및 점검)에 포함되지 않은 것은 무엇인가?**

① 개인정보처리자는 개인정보처리시스템에 접속한 자(다만, 정보주체는 제외한다)의 접속기록을 1년 이상 보관·관리하여야 한다.

| ----------------------- | -------------- | --------- | ------- | ----------------- |
| Client Hello            | 클라이언트 헬로       | 클라이언트     | 필수      | -                 |
| Server Hello            | 서버 헬로          | 서버        | 필수      | -                 |
| Server Certificate      | 서버 인증서         | 서버        | 선택적     | 익명 키 교환 시 생략      |
| Server Key Exchange     | 서버 키 교환        | 서버        | ==선택적== | ==RSA 방식 시 생략==   |

| Server Hello Done       | 서버 헬로 완료       | 서버        | 필수      | -                 |

| Finished                | 완료             | 양방향       | 필수      | -                 |


**핵심:**

SSL Handshake에서 클라이언트는 세션 키 생성을 위한 **pre-master secret**을 서버에 전달해야 하는데, 이게 **Client Key Exchange** 메시지를 통해 전송됨. 어떤 키 교환 알고리즘을 쓰든 이 과정은 반드시 필요함.

**정확한 명칭 정리:**

| 특징                      | 설명                                                                       |

## 103번 문제 

**개인정보의 안전성 확보조치 기준에 따른 제5조(접근 권한의 관리)에 포함되지 않는 것은 무엇인가?**

① 개인정보처리자는 정당한 권한을 가진 자만이 개인정보처리시스템에 접근할 수 있도록 일정 횟수 이상 인증에 실패한 경우 개인정보처리시스템에 대한 접근을 제한하는 등 필요한 **조치를 할 수 있다.**   (해야한다 ㅎ)

② 개인정보처리자는 개인정보처리시스템에 대한 접근 권한을 업무 수행에 필요한 최소한의 범위로 차등 부여하여야 한다.

③ 개인정보처리자는 개인정보취급자 또는 개인정보취급자의 업무가 변경되었을 경우 지체 없이 개인정보처리시스템의 접근 권한을 변경 또는 말소하여야 한다.

④ 개인정보처리자는 제1항 및 제2항에 의한 권한 부여, 변경 또는 말소에 대한 내역을 기록하고, 그 기록을 최소 3년간 보관하여야 한다.

## 정답: ① ✅

**선택지 판정표**

|선택지|판정|근거|
|---|---|---|
|①|❌ 포함 안 됨|인증 실패 접근 제한은 **제5조 제6항** 내용이긴 하나, 문제 지문 표현 중 **"할 수 있다"가 오류** — 원문은 "하여야 한다" (의무사항). 또한 이 내용은 제6조(접근통제) 성격에 가까움|
|②|✅ 포함됨|제5조 제1항 — 최소한의 범위로 차등 부여|
|③|✅ 포함됨|제5조 제2항 — 업무 변경 시 지체 없이 변경/말소|
|④|✅ 포함됨|제5조 제3항 — 내역 기록, 최소 3년 보관|

**핵심 포인트**

①의 함정은 **"할 수 있다"** — 재량규정처럼 써놨는데 원문은 **"하여야 한다"** (강행규정). 감리사 시험에서 법령 문제는 **조동사 하나** 바꿔서 오답 만드는 패턴 자주 나옴. "할 수 있다" vs "하여야 한다" 꼭 체크해.

## 104번 문제 복원

**일회용 패드(One Time Pad)에 대한 설명 중 가장 부적절한 것은 무엇인가?**

① 평문과 랜덤(Random)한 비트열과의 XOR만을 취하는 단순한 암호 기법이다.

② 과거에 사용한 랜덤한 비트열의 키를 재사용할 수 있어 실용적이다.

③ 일회용 패드가 해독불가능하다는 것은 Shannon에 의해 수학적으로 증명되었다.

④ 일회용 패드의 아이디어는 스트림 암호에 활용되고 있으며, 의사난수생성기 등을 사용하여 강력한 암호를 구축할 수 있다.

## 정답: ② ✅

**선택지 판정표**

|선택지|판정|이유|
|---|---|---|
|①|✅ 적절|OTP = 평문 XOR 랜덤 키스트림. 정확한 설명|
|②|❌ 부적절|OTP의 핵심 조건이 **키 절대 재사용 금지**. 재사용하면 해독 가능해져서 OTP의 완전 비밀성(Perfect Secrecy) 붕괴|
|③|✅ 적절|Shannon이 1949년 정보이론으로 OTP의 완전 비밀성 수학적 증명|
|④|✅ 적절|스트림 암호가 OTP 개념 차용. 실제로는 의사난수생성기(PRNG)로 키스트림 생성|

**OTP 3대 조건** (이거 어기면 OTP 아님)

|조건|내용|
|---|---|
|키 길이 = 평문 길이|키가 평문보다 짧으면 안 됨|
|완전 랜덤 키|의사난수 아닌 진짜 난수|
|**키 재사용 절대 금지**|← ②번이 바로 이걸 위반|

|③|✅ 적절|역할 = 직무/책임 기반 추상화 개체 맞음|
|④|✅ 적절|권한 = 조직 차원 오퍼레이션 관계 추상화 개체. 맞는 설명|

**핵심**

```
사용자 → [역할] → 권한
         ↑
      이게 매개체
```

매개체는 **역할**이지 권한이 아님. ②가 역할과 권한 위치를 바꿔놓은 함정.

## 110번 문제 복원

**다음에서 설명하는 암호공격으로 가장 적합한 것은?**

> 이 공격방식은 일정량의 평문에 대응하는 암호문을 알고 있는 상태에서 암호문과 평문의 관계로부터 **키를 추정하**여 해독하는 공격 방법으로써, Linear Attack과 같은 사례가 있다.

① 암호문 단독 공격(ciphertext only attack) 
② 알려진 평문 공격(known plaintext attack)
③ 선택 평문 공격(chosen plaintext attack) 
④ 선택 암호문 공격(chosen ciphertext attack)

## 정답: ② ✅

**선택지 판정표**

| 선택지      | 판정   | 이유                                                     |
| -------- | ---- | ------------------------------------------------------ |
| ① 암호문 단독 | ❌    | 암호문만 가진 상태. 평문 정보 없음                                   |
| ② 알려진 평문 | ✅ 정답 | 평문+암호문 쌍을 **이미 알고 있는** 상태에서 키 추정. Linear Attack이 대표 사례 |
| ③ 선택 평문  | ❌    | 공격자가 **원하는 평문을 선택**해서 **암호화 가능한** 상태. 더 강력한 공격         |
| ④ 선택 암호문 | ❌    | 공격자가 **원하는 암호문을 선택**해서 **복호화 가능한 상태**. 가장 강력한 공격       |

**암호 공격 강도 순서**

|공격 유형|공격자가 가진 것|강도|
|---|---|---|
|암호문 단독|암호문만|최약|
|알려진 평문|평문+암호문 쌍 (수동)|↑|
|선택 평문|원하는 평문 선택 → 암호화|↑|
|선택 암호문|원하는 암호문 선택 → 복호화|최강|

**"알려진"(known) vs "선택"(chosen)** 차이가 핵심 — 수동적으로 갖고 있냐, 능동적으로 고르냐.

## 111번 문제 복원

**다음에서 설명하는 공격기법으로 가장 적합한 것은?**

> 웹서버와 TCP 연결 시, 정상적인 HTTP format의 GET 요청을 하지만 window size를 작게 조작하여 Server가 이 요청에 대한 응답을 천천히 하게 만들어 Connections를 장시간 유지하게 함으로써 웹서버의 자원을 소모시키는 공격

① Slow HTTP POST DoS
② HTTP Get Flooding with Cache-Control (CC Attack) 
③ Hash DoS
④ Slow HTTP Read DoS

## 정답: ④ ✅

**선택지 판정표**

|선택지|판정|이유|
|---|---|---|
|① Slow HTTP POST DoS|❌|POST 본문을 **아주 천천히 전송**해서 서버 연결 점유. 전송 속도 조작|
|② CC Attack|❌|Cache-Control 헤더로 캐시 우회 강제 → 서버 직접 부하. 윈도우 크기와 무관|
|③ Hash DoS|❌|HTTP 파라미터 해시 충돌 유발 → CPU 부하. 전혀 다른 공격|
|④ Slow HTTP Read DoS|✅ 정답|**TCP window size를 작게** 조작 → 서버가 응답을 조금씩 나눠 보내야 함 → 연결 장시간 점유 → 자원 소모|

**Slow HTTP 계열 3종 비교**

| 공격                      | 조작 대상               | 핵심                |
| ----------------------- | ------------------- | ----------------- |
| Slow HTTP POST          | 요청 본문 전송 속도         | 업로드를 아주 천천히       |
| Slow HTTP Read          | **TCP Window Size** | 수신 버퍼 작게 → 응답 천천히 |
| Slow Loris(Slow Header) | HTTP 헤더 전송 속도       | 헤더를 끝없이 조금씩       |

## 113번 문제 복원

**다음 중 SSL 프로토콜의 동작에 대한 설명으로 가장 적합한 것은 무엇인가?**

① 통신 중에 일시 중지한 세션을 다시 재시작하려면 반드시 마스터 비밀값(master secret)을 생성해야 한다.

② 클라이언트의 데이터 송신용(write) 암호키와 수신용(read) 암호키가 반드시 동일해야 정상적인 비밀통신 가능하다.

③ 핸드쉐이크(handshake) 과정에서 서버 측에서는 반드시 클라이언트 측의 인증서를 요구해야 한다.

④ 전송 데이터에 대한 압축이 필요한 경우는 반드시 암호화 이전에 압축을 수행해야 한다.

## 정답: ④ ✅

**선택지 판정표**

| 선택지 | 판정   | 이유                                                                           |
| --- | ---- | ---------------------------------------------------------------------------- |
| ①   | ❌    | 세션 재개(Session Resumption) 시 기존 세션 ID로 master secret **재사용** 가능. 새로 생성 안 해도 됨 |
| ②   | ❌    | 송신/수신 키는 **서로 다름**. 클라이언트→서버, 서버→클라이언트 각각 별도 키 사용                            |
| ③   | ❌    | 클라이언트 인증서는 **선택사항**. **서버 인증서만 필수**. 클라이언트 인증은 상호인증(Mutual TLS) 때만           |
| ④   | ✅ 정답 | SSL Record Protocol 처리 순서: **압축 → MAC → 암호화** 순서가 맞음                         |

**SSL Record Protocol 처리 순서**

```
원본 데이터
   ↓ 단편화 (Fragment)
   ↓ 압축 (Compress)
   ↓ MAC 추가
   ↓ 암호화 (Encrypt)
전송
```

압축이 암호화보다 반드시 먼저 와야 하는 이유 → 암호화된 데이터는 랜덤처럼 보여서 압축이 전혀 안 됨.

## 114번 문제 복원

**다음 중 데이터베이스 보안 통제 기법으로 가장 거리가 먼 것은 무엇인가?**

① Inference Control ② Risk Control ③ Access Control ④ Flow Control

## 정답: ② Risk Control ✅

**선택지 판정표**

|선택지|판정|이유|
|---|---|---|
|① Inference Control|✅ DB 보안 통제|추론 공격 방지 — 여러 쿼리 조합으로 민감정보 유추하는 것 차단|
|② Risk Control|❌ DB 보안 통제 아님|위험관리(Risk Management) 개념. DB 보안 통제 기법 분류에 포함 안 됨|
|③ Access Control|✅ DB 보안 통제|접근 권한 통제 — DB 보안의 핵심 기법|
|④ Flow Control|✅ DB 보안 통제|정보 흐름 통제 — 높은 보안등급→낮은 보안등급으로 정보 유출 방지|

**DB 보안 통제 3대 기법**

| 기법                      | 역할                        |
| ----------------------- | ------------------------- |
| **Access Control(접근통제)**    | 누가 어떤 데이터에 접근 가능한지 통제     |
| **Flow Control(흐름통제)**      | 데이터 흐름 방향 통제 (보안등급 기반)    |
| **Inference Control(추론통제)** | 쿼리 조합을 통한 추론으로 민감정보 유출 차단 |

Risk Control은 일반 정보보안/위험관리 용어지 DB 보안 통제 기법 분류가 아님.

## 116번 문제 복원

**다음에서 설명하는 것은 무엇인가?**

> 기업에 물리적 설비, 전력, 난방, 통풍 및 온도 조절기 등을 제공하는 백업 설비로 조직이 이를 이용할 경우 다른 백업 설비보다 가장 비용이 적게 드는 장점을 가진다.

① 콜드 사이트(cold site) ② 미러 사이트(Mirror site) ③ 핫 사이트(hot site) ④ 웜 사이트(warm site)

## 정답: ① Cold Site ✅


**사이트 유형 비교**

|유형|준비 수준|복구 시간|비용|

## 119번 문제 복원

**"LLM 애플리케이션을 위한 OWASP Top 10 2025"를 기준으로 아래에서 설명하는 것은 무엇인가?**

> LLM에서 발생하는 이것은 LLM 모델을 활용하는 애플리케이션의 핵심적인 보안 취약점 중 하나입니다. LLM이 신뢰할 수 있는 것처럼 보이지만 실제로는 잘못되거나 오해를 유발하는 정보를 생성할 때 발생합니다. 이러한 취약점은 보안 침해, 평판 손상, 법적 책임과 같은 심각한 문제를 초래할 수 있습니다. 주요 원인 중 하나는 환각입니다. 환각은 LLM이 사실처럼 보이는 허구의 콘텐츠를 생성하는 현상을 의미합니다. 이는 모델이 훈련 데이터에서 부족한 정보를 통계적 패턴을 기반으로 보완하려고 할 때 발생하며, 모델이 내용을 실제로 이해하지 못한 채 출력을 생성하는 경우입니다.

① 프롬프트 인젝션(Prompt Injection) 
② 민감 정보 유출(Sensitive Information Disclosure) 
③ 부적절한 출력 처리(Improper Output Handling) 
④ 허위 정보(Misinformation)

## 정답: ④ Misinformation ✅

| ① 프롬프트 인젝션(Prompt Injection)                 | ❌   | 악의적 입력으로 LLM 조작. 환각과 무관                                    |
| ② 민감 정보 유출(Sensitive Information Disclosure) | ❌   | 훈련 데이터의 민감정보 노출. 환각과 무관                                    |
| ③ 부적절한 출력 처리(Improper Output Handling)       | ❌   | LLM 출력을 다운스트림 시스템에 안전하게 전달 못하는 문제                          |

PIP가 SIEM, 위협인텔, ID관리 등 각종 데이터를 수집해서 PE(정책엔진)에 공급 → PE가 허용/거부 결정 → PA(정책관리자)가 그 결정을 PEP에 전달 → PEP가 실제 접근을 차단하거나 허용.

PIP는 정보를 "생성·전달"하는 역할이고 결정은 안 함. 이게 핵심 구분.

**102. Diffie-Hellman 알고리즘에서 p=11, g=5를 사용하고, A의 개인키가 3, B의 개인키는 2일 때 공유키는 무엇인가?**

① 1　② 5　③ 11　④ 15

**[풀이]**

**Step 1. 각자 공개키 계산 ==(g^개인키 mod p)**==

|주체|계산|공개키|
|---|---|---|
|A|5³ mod 11 = 125 mod 11|**4**|
|B|5² mod 11 = 25 mod 11|**3**|

125 mod 11: 11×11=121, 125-121=**4** ✓ 25 mod 11: 11×2=22, 25-22=**3** ✓

**Step 2. 공유키 계산 (상대방 공개키^내 개인키 mod p)**

|주체|계산|결과|
|---|---|---|
|A가 계산|3³ mod 11 = 27 mod 11|**5**|
|B가 계산|4² mod 11 = 16 mod 11|**5**|

둘 다 **5** → 공유키 = **5**

**정답: ②**

네가 틀린 이유는 메모 보니까 공개키 계산에서 mod 연산 실수한 거임. 125 mod 11 = 4인데 다르게 계산한 듯. DH는 계산 순서 외워두면 됨:

> g^내개인키 mod p → 공개키 교환 → 상대공개키^내개인키 mod p → 공유키

**103. 다음 중 스트림 암호 방식에 해당하는 알고리즘은 무엇인가?**

① AES　② RC4　③ DES　④ RSA

**[선택지 판정]**

|선택지|암호 방식|판정|이유|
|---|---|---|---|
|① AES|블록 암호|❌|128bit 블록 단위 암호화|
|**② RC4**|**스트림 암호**|**✅**|1바이트씩 XOR 연산, 대표적 스트림 암호|
|③ DES|블록 암호|❌|64bit 블록 단위 암호화|
|④ RSA|비대칭 암호|❌|공개키 암호, 블록/스트림 분류 자체가 다름|

**정답: ②**

**[핵심 개념]**

| 알고리즘 | 대칭/비대칭  | 블록/스트림 |
| ---- | ------- | ------ |
| AES  | **대칭**  | 블록     |
| RC4  | **대칭**  | 스트림    |
| DES  | **대칭**  | 블록     |
| RSA  | **비대칭** | -      |

대칭키 = 암호화/복호화 같은 키 사용 비대칭 = 공개키/개인키 쌍

블록/스트림은 대칭키 안에서의 세부 분류임. RSA는 비대칭이라 블록/스트림 분류 자체가 해당 없음.
RC4는 SSL/WEP에서 쓰다가 보안 취약점으로 퇴출된 알고리즘임. 시험엔 스트림 암호 대표로 자주 나옴.

**116. 재해복구 계획에서 RPO가 60분일 때 가장 적합한 백업 정책은 무엇인가?**

① 주 1회 테이프백업 후 원격지 소산 백업 
② 시간 단위 백업 및 원사이트 운영 
③ 일 1회 백업과 콜드사이트 운영 
④ 실시간 또는 준 실시간 데이터 복제

**[핵심 개념]**

RPO (Recovery Point Objective) = **최대 허용 데이터 손실 시간**

RPO 60분 = 최대 60분치 데이터 손실까지 허용

**[선택지 판정]**

|선택지|데이터 손실 가능 범위|판정|
|---|---|---|
|① 주 1회 백업|최대 **7일**치 손실|❌ RPO 60분 초과|
|**② 시간 단위 백업**|최대 **60분**치 손실|**✅ RPO 60분 딱 맞음**|
|③ 일 1회 백업|최대 **24시간**치 손실|❌ RPO 60분 초과|
|④ 실시간 복제|손실 **거의 없음**|❌ RPO 60분보다 과도한 수준|

**정답: ②**

④번이 헷갈릴 수 있는데 — 실시간 복제는 RPO가 거의 0분 수준이라 60분 요구사항보다 **과잉 스펙**임. 비용 대비 비효율.


**재해복구 핵심 지표 정리**

|지표|풀네임|한글명|의미|
|---|---|---|---|
|**RPO**|Recovery Point Objective|복구 목표 시점|최대 허용 **데이터 손실** 시간|
|**RTO**|Recovery Time Objective|복구 목표 시간|재해 후 **서비스 재개**까지 허용 시간|
|**RCO**|Recovery Capacity Objective|복구 목표 용량|복구 시 확보해야 할 **최소 처리 용량**|

RPO 1시간 = 오후 2시 데이터까지는 살아있어야 함 
RTO 4시간 = 오후 7시까지는 서비스 재개해야 함
