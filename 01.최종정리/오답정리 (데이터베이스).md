### 선택지 판정 테이블

|보기|주장|판정|이유|

|ㄱ|함수 종속 = 두 속성이 **서로** 결정|❌ 틀림|FD는 **단방향** — X→Y가 성립해도 Y→X는 별개|
|ㄴ|부분 함수 종속은 2NF뿐 아니라 3NF도 위반|✅ 옳음|2NF 미달이면 3NF도 자동 미달 (정규형은 포함 관계)|
|ㄷ|X→Y, XW→Z이면 의사이행규칙으로 YW→Z 성립|❌ 틀림|의사이행규칙 방향이 **반대**|
|ㄹ|FD={AB→C, B→D}이면 완전 함수 종속 불만족|✅ 옳음|B→D가 부분 함수 종속|

### ㄷ이 틀린 이유 — 의사이행규칙 정확한 방향

Armstrong 공리의 의사이행규칙(Pseudotransitivity):

```
올바른 방향: X → Y  이고  YW → Z  이면  XW → Z
문제의 주장: X → Y  이고  XW → Z  이면  YW → Z  ← 역방향, 성립 안 함
```

### ㄹ 검증

```
후보키 찾기: AB⁺ = {A,B} → C → {A,B,C} → D → {A,B,C,D} → AB가 후보키

B → D:
  B는 후보키 AB의 진부분집합(proper subset)
  D가 키의 일부(B)에만 종속 → 부분 함수 종속 존재
  → 완전 함수 종속성 불만족 ✅
```

### 정규형 포함 관계 (ㄴ 관련)

```
1NF ⊂ 2NF ⊂ 3NF ⊂ BCNF

부분 함수 종속 존재 → 2NF 위반
→ 2NF 위반이면 3NF도 당연히 위반
```

### 기초 분석

**후보키 찾기:**


- B→CD: B는 슈퍼키 아님 → 이행적 종속 → 3NF 위반

### 선택지 판정 테이블

|번호|분해안|무손실 조인|FD 보존|3NF|판정|
|---|---|---|---|---|---|
|①|R1(A,B,D), R2(B,C)|△|△|B→C 분리됐으나 AB→D 중복|❌|
|**②**|**R1(A,B), R2(B,C,D)**|**✅**|**✅**|**✅**|**✅ 정답**|

|④|R1(B,C,D), R2(A,D)|❌|❌|아래 설명|❌|

### ④가 틀린 이유

```
무손실 조인 검증:
R1(B,C,D) ∩ R2(A,D) = {D}

무손실 조인 조건:
공통속성 D가 R1 또는 R2의 슈퍼키여야 함
→ D는 어느 쪽의 키도 아님 ❌
→ 조인 시 spurious tuple(가짜 투플) 발생
→ 원본 R 복원 불가
```

또한 A→B FD가 보존 안 됨 (A, B가 같은 릴레이션에 없음)

### ②가 정답인 이유

```
R1(A,B): 후보키=A, A→B ✓ 3NF 만족
R2(B,C,D): 후보키=B, B→CD ✓ 3NF 만족

무손실 조인: R1∩R2 = {B}, B는 R2의 슈퍼키 ✅
FD 보존: A→B (R1에서), B→CD (R2에서) ✅
```

문제 44 정답은 ④

[정답]  
④ 계층구조가 변경(추가, 삭제)되거나 중간 계층이 새로 생기면, 전체 데이터를 갱신해 주어야 하므로 유지보수가 어려운 단점이 있다.

이게 가장 거리가 멀다.

왜 ④가 틀렸냐

순환 관계(Recursive Relationship)는  
같은 엔터티가 자기 자신과 관계를 맺는 구조다.

예:  
직원(Employee) — 상사(Manager)  
부서(Department) — 상위부서(Parent Department)  
카테고리(Category) — 상위카테고리(Parent Category)

이 방식의 장점은  
계층을 별도 엔터티 여러 개로 쪼개지 않고  
하나의 엔터티 + 자기참조 FK로 표현해서  
계층 추가/삭제/변경에 비교적 유연하다는 점이다.

즉,  
중간 계층이 새로 생겨도 보통은  
전체 데이터를 전부 뜯어고치는 게 아니라  
부모-자식 참조값만 조정하면 된다.

그래서 ④의  
“전체 데이터를 갱신해야 해서 유지보수가 어렵다”  
이건 순환 관계의 일반적 특성으로 보기 어렵다.

선택지 판정표

|번호|판정|이유|
|---|---|---|
|①|O|순환 관계는 동일 엔터티 내부에서 상하위 관계를 표현하므로 계층형 데이터를 하나의 구조로 통합해 설계할 때 활용 가능|
|②|O|순환 관계에서는 상위가 없을 수도 있고(루트), 하위가 없을 수도 있어 일반적으로 선택 참여로 표현 가능|
|③|O|여러 계층을 하나의 엔터티로 통합 표현하므로 각 계층에서 쓰이던 속성을 하나의 엔터티가 포괄하는 형태로 설계 가능|
|④|X|순환 관계는 오히려 계층 변경에 비교적 유연한 편이다. “전체 데이터 갱신 필요”는 과장된 설명|

핵심 개념만 박아두면

[순환 관계 / Recursive Relationship]

■ 정의  
동일 엔터티가 자기 자신과 관계를 맺는 관계

■ 예시  
직원-상사  
부서-상위부서  
카테고리-상위카테고리

■ 특징  
하나의 엔터티로 계층 구조 표현 가능  
자기참조 외래키(Self-referencing Foreign Key) 사용  
계층 변경 시 구조적으로 비교적 유연

■ 함정  
“계층 변경 시 전체 데이터 다 수정해야 한다”  
→ 순환 관계의 일반 특징으로 보기 어려움  
→ 오답 포인트

이유  
부채꼴 함정(Fan Trap)의 핵심은 “일부 관계가 누락돼서 부채꼴 형상이 된다”가 아니다.

선택지 판정표

|번호|판정|이유|
|---|---|---|

정답은 **④**

핵심은 **자연 조인(Natural Join)** 의 정의다.

**자연 조인 = 공통 속성(Z)에 대해 동등 조인(Equi Join)한 뒤, 중복된 조인 속성을 한 번만 남기는 것**이다.

즉 식으로 쓰면

**R ⋈N S = ΠX∪Y ( R ⋈R.Z = S.Z S )**

여기서 의미는:

- 먼저 **R.Z = S.Z** 조건으로 조인하고
- 그다음 결과에서 **중복된 조인 속성 하나를 제거**해서
- 최종 속성 집합을 **X ∪ Y** 로 만든다
    

선택지 판정표

| 번호    | 판정    | 이유                                                                        |
| ----- | ----- | ------------------------------------------------------------------------- |
| **①** | **X** | 자연 조인 결과 속성은 **X∩Y가 아니라 X∪Y** 여야 한다.                                      |
| **②** | **X** | 왼쪽이 **세미조인**인데, 오른쪽은 **자연조인 형태**라 의미가 다름. 게다가 **ΠX∪Y(S)** 자체도 보통 성립하기 어렵다 |
| **③** | **X** | **(R×S) ⋈N R** 는 자연 조인의 일반식이 아니다. 의미도 맞지 않음(오류)                           |
| **④** | **O** | **동등 조인 후 중복 속성 제거**라는 자연 조인의 정의에 맞음                                      |

직관적으로 보면

- **세타 조인(Theta Join)**: 조건 걸고 그냥 붙임
- **자연 조인(Natural Join)**: 공통 속성 같게 붙이고, **같은 컬럼 두 개 생긴 건 하나로 정리**
- 그래서 **자연 조인 = 동등 조인 + 중복 조인 속성 제거**

**교집합(X∩Y)** = **공통 속성** = **조인 조건에 쓰는 컬럼**

**합집합(X∪Y)** = **전체 결과 속성** = **프로젝션에 남길 컬럼**

예:  
R(a,b,c), S(c,d)

- **X∩Y = {c}**
- **X∪Y = {a,b,c,d}**
- 
자연조인:  
**R ⋈N S = Πa,b,c,d ( R ⋈R.c=S.c S )**

한 줄:  
**교집합은 붙이는 기준, 합집합은 남기는 결과**

한 줄 암기

**자연 조인 = 동등 조인 후, 공통 속성 중복 제거**

이 문제는 **①이 제일 흔한 함정**이다.  
조인 결과 속성을 **교집합(X∩Y)** 로 착각하면 바로 틀린다.  
실제는 **합집합(X∪Y)** 다.

    - 자식 없음
    - **LEFT OUTER JOIN** 이므로 **NULL 붙여서 1행**

핵심은 **집합 연산(EXCEPT(차집합), INTERSECT(교집합))은 중복 제거**되고, 마지막에만 **UNION ALL** 로 그냥 이어 붙인다는 점이다.

먼저 데이터에서 Sno만 보면

STUDENT = **{1001, 1002, 1003, 1004, 1005, 1006}**  
ENROLLMENT = **{1001, 1001, 1003, 2006}** → 집합 연산에 들어가면 **{1001, 1003, 2006}**

SQL 분해

```sql
SELECT COUNT(Sno)
FROM (
  (SELECT Sno FROM ENROLLMENT
   EXCEPT
   SELECT Sno FROM STUDENT)
  UNION ALL
  (SELECT Sno FROM STUDENT
   INTERSECT
   SELECT Sno FROM ENROLLMENT)
);
```

1. **ENROLLMENT EXCEPT STUDENT (차집합연산)** 
    
    - {1001, 1003, 2006} - {1001,1002,1003,1004,1005,1006}
    - 결과 = **{2006}**
        
2. **STUDENT INTERSECT ENROLLMENT(교집합연산)**
    
    - 공통값 = **{1001, 1003}**
    - 
3. **UNION ALL**
    
    - {2006} + {1001,1003}
    - 결과 행 = **3개**
        
4. **COUNT(Sno)**
    
    - 결과 = **3**

선택지 판정표

|선택지|판정|이유|
|---|---|---|
|**① 2**|X|INTERSECT 결과를 1개로 잘못 세면 나오는 값|
|**② 3**|**O**|**{2006} UNION ALL {1001,1003} = 3행**|
|**③ 4**|X|ENROLLMENT의 중복 1001을 그대로 계산한 오류|
|**④ 5**|X|집합 연산에서 중복 제거를 무시한 오류|

한 줄로 박으면  
**EXCEPT/INTERSECT는 중복 제거 후 계산, 마지막 UNION ALL로 1개 + 2개 = 3개**

정답: **가 = NOT EXISTS, 나 = EXISTS**

즉 완성형은 이렇게 된다.

```sql
SELECT 학생번호, 이름
FROM 학생
WHERE NOT EXISTS
 ( SELECT 과목번호 FROM 과목
   WHERE EXISTS
    ( SELECT 수강.과목번호
      FROM 수강
      WHERE 수강.과목번호 = 과목.과목번호
        AND 수강.학생번호 = 학생.학생번호 ));
```

핵심 해석

- 안쪽 `EXISTS` = **이 학생이 이 과목을 수강한 적 있냐**
- 바깥 `NOT EXISTS` = **그런 과목이 하나도 없냐**
- 따라서 전체 의미는 **한 번도 수강하지 않은 학생 찾기**
    

현재 데이터로 보면

- 학생 100 → 수강 있음
- 학생 200 → 수강 있음
- 학생 300 → **수강 없음**
    

그래서 결과는 **학생 300, 학생3** 이다.

선택지 판정 기준표

| 위치    | 들어갈 것          | 이유                                      |
| ----- | -------------- | --------------------------------------- |
| **가** | **NOT EXISTS** | 수강한 과목이 **하나도 없는 학생**을 찾으려면 바깥에서 부정해야 함 |
| **나** | **EXISTS**     | 특정 과목에 대해 그 학생의 수강 기록이 **존재하는지** 확인     |
9. 초압축 예시

| 서브쿼리 결과     | EXISTS | NOT EXISTS |
| ----------- | ------ | ---------- |
| **1행 이상**   | **참**  | **거짓**     |
| **0행(공집합)** | **거짓** | **참**      |
한 줄 포인트

**“한 번도 없다” = 바깥은 NOT EXISTS, 안쪽은 EXISTS**

**핵심 개념: 인덱스 스캔 방식 분류**

존재하는 인덱스 목록부터 정리:

|테이블|인덱스|대상 컬럼|유일성|
|---|---|---|---|
|DEPT|PK 자동생성|DEPTNO|Unique|
|EMP|PK 자동생성|EMPNO|Unique|
|EMP_IDX01|명시 생성|ENAME|Non-Unique|
|EMP_IDX02|명시 생성|DEPTNO|Non-Unique|

**각 선택지 스캔 방식 판정:**

|선택지|사용 인덱스|조건|스캔 방식|정답 여부|
|---|---|---|---|---|
|① `DEPT WHERE DEPTNO > 1001`|DEPT PK|부등호(범위)|**Range Scan**|나머지 셋|
|② `EMP WHERE DEPTNO = 2001`|EMP_IDX02|= 이지만 Non-Unique|**Range Scan** (복수 행 가능)|나머지 셋|
|③ `EMP WHERE EMPNO = 10007`|EMP PK|= + **Unique**|**Unique Scan** ← 이게 다름|**정답**|
|④ `EMP WHERE ENAME LIKE '김%'`|EMP_IDX01|전방일치(범위)|**Range Scan**|나머지 셋|

**왜 ③만 다른가?**

Range Scan vs Unique Scan의 본질적 차이:

```
Range Scan  → "이 범위 어딘가에 있겠지, 끝까지 찾아봐"
              → 복수 행 가능성 있으므로 계속 스캔

Unique Scan → "딱 1건만 존재 보장 (Unique 인덱스 + = 조건)"
              → 1건 찾으면 즉시 중단, 추가 스캔 없음
```

②가 헷갈릴 수 있는 포인트: `DEPTNO = 2001`처럼 `=` 조건이어도, EMP_IDX02는 **Non-Unique 인덱스**라 동일 DEPTNO를 가진 여러 직원이 존재할 수 있음 → Range Scan으로 처리됨.

③은 EMPNO가 **PK(= Unique 인덱스)** + `=` 조건의 조합 → 유일하게 **Unique Scan**.

**핵심 개념: 각 조인 기법의 특성**

|선택지|내용|판정|이유|
|---|---|---|---|
|① NL 조인 - 드라이빙 테이블 범위가 속도에 영향|✅ 맞음|outer 테이블 행 수만큼 inner 반복 → 드라이빙 테이블이 작을수록 유리||
|② Single-Loop 조인 - 후행 테이블에 인덱스 필수|✅ 맞음|인덱스로 inner 테이블 1건씩 액세스하는 방식이 전제||
|③ Sort-Merge 조인 - 선후 우선순위 무관, 전체범위 처리|✅ 맞음|양쪽 모두 정렬 후 병합 → 대칭적, 전체범위 불가피||
|④ Hash 조인 - 조인 컬럼 가공하면 느려진다|❌ 틀림 (정답)|아래 상세 설명||

**④가 틀린 이유:**

```
NL/Single-Loop 조인에서 조인 컬럼 가공:
  WHERE TO_CHAR(dept_id) = '100'
  → 인덱스 사용 불가 → Full Scan 유발 → 급격히 느려짐  ← 가공의 타격이 큼

Hash 조인에서 조인 컬럼 가공:
  → 애초에 인덱스를 사용하지 않음
  → DB가 hash function을 컬럼 값에 적용해 bucket 구성
  → 가공된 값이라도 hash table 구성 자체는 동일하게 동작
  → 인덱스 기반 조인에 비해 컬럼 가공에 훨씬 덜 민감
```

즉, "조인 컬럼 가공 → 느려진다"는 설명은 **인덱스를 전제로 한 NL/Single-Loop 조인의 특성**이지, Hash 조인의 특성이 아님.

Hash 조인의 실제 성능 저하 요인은 조인 컬럼 가공이 아니라 **hash area(메모리) 부족 → 디스크 spill 발생** 여부임.

CBO(Cost-Based Optimizer)는 테이블 통계(행 수, 컬럼 분포, 인덱스 선택도 등)를 기반으로 작동하는데, 이 통계 수집 방식과 내부 비용 계산 공식이 **DBMS 제품 및 버전마다 상이**함. Oracle 19c와 12c의 CBO도 동작이 다르고, Oracle CBO ≠ PostgreSQL planner. ②번은 이 사실을 완전히 반대로 서술한 것.

## RBO vs CBO 완전 정리

### 비유로 먼저 이해하기

**RBO** = 베테랑 고참 직원

> "이 경우엔 무조건 인덱스 써. 규정집에 그렇게 나와 있거든. 데이터가 몇 건인지는 안 봐."

**CBO** = 데이터 분석가 출신 신입

> "일단 테이블 통계 보고... 인덱스 쓰는 게 비용 3이고, Full Scan이 비용 1이네. Full Scan 씁니다."

### 핵심 비교표

| 구분          | RBO                    | CBO                |
| ----------- | ---------------------- | ------------------ |
| 판단 기준       | 규칙(우선순위)               | 비용(통계 기반)          |
| 통계정보        | 무시                     | 핵심                 |
| 실행계획 예측     | 가능 (항상 동일)             | 어려움 (통계 따라 변동)     |
| DB 제품/버전 영향 | 낮음                     | **높음** ← ②번 오답 포인트 |
| 대용량 성능      | 취약                     | 우수                 |
| 현재 사용       | Oracle 10g+ deprecated | 현재 모든 주요 DBMS 표준   |
| 힌트 필요성      | 낮음                     | 통계 오류 시 힌트로 보정     |

### ②번이 왜 틀렸는지 - 핵심 원리

**지연 갱신(Deferred Update)** 의 동작:

```
트랜잭션 실행 중
    → DB에 직접 안 씀
    → 로그에만 기록 (new value만)
    → 커밋 시점에 로그를 실제 DB에 반영
```

|상황|처리|
|---|---|
|커밋 전 실패|DB에 아무것도 안 썼으므로 **그냥 로그 버리면 끝** → undo 불필요|
|커밋 후 재시작|로그에 있는 new value를 DB에 다시 써야 함 → **redo 필요**|

**즉, 지연 갱신 = redo만 / undo 불필요**


### 즉시 vs 지연 비교표

|구분|즉시 갱신 (Immediate)|지연 갱신 (Deferred)|
|---|---|---|
|DB 쓰는 시점|트랜잭션 실행 중 (커밋 전도 가능)|커밋 시점에만|
|로그 기록|old value + new value|new value만|
|실패 시|undo 필요|그냥 폐기 (undo 불필요)|
|재시작 시|redo 필요|redo 필요|
|필요 연산|**undo + redo**|**redo만**|
|구현 복잡도|복잡|단순|


### 기억법

> **지연** = 커밋 전엔 DB 안 건드림 = 되돌릴 게 없음 = **undo 필요 없음** **즉시** = 커밋 전에도 DB 건드림 = 되돌려야 할 수도 있음 = **undo 필요**

### ②번 오답 핵심

4가지 정책 정확히 구분해야 함:

| 정책           | 의미                             | 관련 연산    |
| ------------ | ------------------------------ | -------- |
| **Steal**    | 커밋 전에도 dirty page를 디스크에 쓸 수 있음 | UNDO 필요  |
| **No-Steal** | 커밋 전엔 디스크에 절대 안 씀              | UNDO 불필요 |
| **Force**    | 커밋 시점에 반드시 디스크에 씀              | REDO 불필요 |
| **No-Force** | 커밋해도 디스크에 즉시 안 써도 됨            | REDO 필요  |

②번이 설명한 **"미완료 상태에서 디스크에 안 씀"** = **No-Steal** 정의임 No-Force는 **"커밋했어도 디스크에 즉시 안 써도 됨"** 이 맞는 설명

### 실무 DBMS 조합

대부분의 DBMS (Oracle, PostgreSQL 등):

```
Steal + No-Force 조합 사용
→ UNDO 필요 (Steal 때문에)
→ REDO 필요 (No-Force 때문에)
→ 그래서 WAL(Write-Ahead Logging) 로그 필수
```

**정답: ③**

먼저 스케줄 실행 흐름부터 정리

```
시간순:
1. T1: read(x)           → T1이 x 읽음 (x=100이라 가정)
2. T2: read(x), x=x-50  → T2가 x 읽고 x=50 계산
3. T2: read(y), y=y+50  → T2가 y 읽고 y=150 계산
4. T1: read(y), display(x+y) → T1이 y 읽음
                              → T1의 x=100(1번에서 읽은값), y=100(아직 T2 write 전)
                              → display = 200 출력
5. T2: write(x), write(y)    → 이제 x=50, y=150 디스크에 씀
```

x+y 실제값 = 50+150 = **200** → T1이 출력한 200과 일치하므로 **일관성은 유지됨** → 이 스케줄은 **직렬 가능(serializable)**

### ① 2PLP(비관적)로 생성 가능한가?

2PLP(2단계 로킹) 규칙:

- **확장 단계:** 락 획득만 가능, 해제 불가
- **수축 단계:** 락 해제만 가능, 획득 불가

이 스케줄을 2PLP로 재현하려면:

```
T1이 read(x) 후 → lock 유지한 채
T2가 read(x) 시도 → T1이 x lock 안 풀었으면 T2 대기해야 함
```

근데 스케줄을 보면 T1이 x를 읽은 뒤 T2가 x를 읽고 있음 → T1이 x lock을 **이미 해제**했다는 의미 → 그런데 T1은 이후 read(y)를 또 획득함 → **해제 후 재획득 = 2PLP 위반**

**2PLP로는 생성 불가**

### ② 낙관적 병행 제어로 생성 가능한가?

낙관적 기법은:
### 3단계 구조

```
읽기 단계 → 검증 단계 → 쓰기 단계
(Read)      (Validate)   (Write)
```

- 실행 중엔 아무 락도 안 걸고 자유롭게 실행
- **커밋 직전** **검증(validation) 단계**에서 충돌 여부 확인
- 충돌 없으면 커밋, 있으면 롤백

이 스케줄은 T1/T2가 x를 동시에 읽고 있지만 **T1은 x를 쓰지 않음** → T2의 write(x)와 T1의 read(x) 사이에 실제 충돌(dirty read 등) 없음 → 검증 단계 통과 가능

==**낙관적 병행제어는 직렬가능하기만 해도 된다고 볼 수 있음**==


### 최종 판정표

|구분|판정|이유|
|---|---|---|
|직렬 가능 스케줄인가|✅|x+y 합계 일관성 유지|
|2PLP로 생성 가능|❌|T1이 lock 해제 후 재획득 → 2PLP 위반|
|낙관적 기법으로 생성 가능|✅|검증 단계에서 충돌 없음|

→ **"2PLP 불가, 낙관적 가능"** = **③번**

## 정답: ④

**핵심 오류:** Strict(엄밀) 2PL은 **배타(exclusive) 로크**를 트랜잭션 완료 시까지 유지하는 것이지, **공유(shared) 로크**가 아님

|선택지|판정|이유|
|---|---|---|
|① 타임스탬프 복귀 트랜잭션 → 새 타임스탬프 부여|✅ 옳음|타임스탬프 순서 기법에서 롤백된 트랜잭션은 재시작 시 새 타임스탬프를 받아 순서 재정렬됨|
|② Conservative 2PL → 교착상태 없음|✅ 옳음|실행 전 모든 로크를 선점 획득 → 대기 사이클 자체가 발생 안 함|
|③ Rigorous 2PL → 연쇄복귀 방지, 교착상태 가능|✅ 옳음|커밋까지 모든 로크(공유+배타) 보유 → cascading rollback 방지, 단 deadlock은 여전히 가능|
|④ Strict 2PL → **공유** 로크를 완료까지 유지|❌ **틀림**|**배타(exclusive) 로크**를 완료까지 유지하는 것임. 공유 로크는 중간에 해제 가능|

**2PL 변형 3종 비교 정리**

| 종류                   | 특징                       | 교착상태 | 연쇄복귀 |
| -------------------- | ------------------------ | ---- | ---- |
| Conservative(보수) 2PL | 실행 전 모든 로크 선점            | ✅ 방지 | 가능   |
| Strict (엄밀)2PL       | **배타 로크**만 커밋까지 유지       | 가능   | ✅ 방지 |
| Rigorous(엄격) 2PL     | **모든 로크**(공유+배타) 커밋까지 유지 | 가능   | ✅ 방지 |

④에서 "공유 로크"라고 쓴 게 핵심 함정이야. Strict는 **exclusive lock**을 끝까지 잡는 거고, 공유 로크는 일찍 풀 수 있어서 오히려 Rigorous 2PL보다 동시성이 높음.

## 정답: ④

**핵심 오류:** Wait-Die ↔ Wound-Wait의 선점/비선점이 **반대로** 서술됨

| 선택지                             | 판정       | 이유                                               |
| ------------------------------- | -------- | ------------------------------------------------ |
| ① Wait-Die: Ti가 더 오래된 트랜잭션이면 대기 | ✅ 옳음     | Ti(old) vs Tj(new) → Ti가 기다림. Ti가 newer면 die(롤백) |
| ② 복귀한 트랜잭션은 재시작 시 기존 타임스탬프 유지   | ✅ 옳음     | starvation(기아) 방지 위해 원래 TS 그대로 유지 **==(핵심)==**   |
| ③ 둘 다 starvation 없음, 불필요한 롤백 가능 | ✅ 옳음     | TS 기반이라 순환 대기 자체가 없어 기아 없음. 단 불필요 롤백은 단점         |
| ④ Wait-Die=선점, Wound-Wait=비선점   | ❌ **틀림** | **정반대**                                          |

**올바른 정리:**

|기법|선점 여부|동작 방식|
|---|---|---|
|**Wait-Die**|**비선점** (no preemption)|old→wait, new→die(롤백). 자기 자신이 롤백됨|
|**Wound-Wait**|**선점** (preemption)|old→상대방을 wound(롤백)시킴, new→wait|

**암기 팁:**

- **Wound**-wait = old가 new를 **찌름(선점)** → 선점
- wait-**Die** = new가 스스로 **죽음** → 비선점
- **Wound = 상처를 입히다, 찌르다** 
- Wound = old가 new를 **찌름** (타동사, 선점)
- Die = new가 스스로 **죽음** (자동사, 비선점)

> **지지도(support)** = A∩B 거래수 / 전체 거래수

> **신뢰도(confidence)** = A∩B 거래수 / A 거래수

> **향상도(lift)** = 신뢰도(A→B) / 지지도(B) , 지지도(A→B) / 지지도(A) * 지지도(B)

|lift 값|의미|
|---|---|
|**lift > 1**|양의 상관관계 (같이 살 가능성 ↑)|
|**lift = 1**|독립 (아무 관계 없음)|
|**lift < 1**|음의 상관관계 (같이 살 가능성 ↓)|

**각 선택지 계산:**

① 기저귀→맥주 신뢰도

```
기저귀∩맥주 = 3
기저귀 = 4
→ 3/4 ✅
```

② 당근→양파 지지도

```
당근∩양파 = 2
전체 = 6
→ 2/6 = 1/3 ✅
```

③ 기저귀→맥주 향상도

```
신뢰도(기저귀→맥주) = 3/4
지지도(맥주) = 4/6 = 2/3
→ (3/4) ÷ (2/3) = 9/8 ✅
```

④ 분유→두부 신뢰도

```
분유∩두부 = 3  (2,3,6번)
분유 = 4       (2,3,5,6번)
→ 3/4 ≠ 1/2 ❌
```

|선택지|계산|판정|
|---|---|---|
|① 기저귀→맥주 신뢰도|3/4 ✅|✅ 옳음|
|② 당근→양파 지지도|2/6 = 1/3 ✅|✅ 옳음|
|③ 기저귀→맥주 향상도|(3/4) ÷ (4/6) = 9/8 ✅|✅ 옳음|
|④ 분유→두부 신뢰도|3/4 인데 1/2라고 함|❌ **틀림**|

**④ 계산:**

- 분유 있는 거래: 2, 3, 5, 6 = **4건**
- 분유∩두부: 2, 3, 6 = **3건**
- 신뢰도 = 3/4 → 1/2 아님

| ③ 연속(Sequence) = **일정 시간 간격** 데이터 패턴 분석 | ❌ 틀림 | "일정 시간 간격"은 **시계열(Time Series)** 특징. Sequence는 시간 간격 상관없이 **순서 패턴** 분석 |
| ④ {A,B}→{C,D} 신뢰도 ≤ {A,B,C}→D 신뢰도       | ✅ 옳음 | 수학적으로 맞음 (아래 증명)                                                       |

**④ 증명:**

```
신뢰도 = support(전체) / support(조건)

{A,B}→{C,D}  = support(A,B,C,D) / support(A,B)
{A,B,C}→D    = support(A,B,C,D) / support(A,B,C)

분자 동일, 분모는:
support(A,B) ≥ support(A,B,C)  ← 항목 많을수록 support 작아짐

∴ {A,B}→{C,D} 신뢰도 ≤ {A,B,C}→D 신뢰도 ✅
```

조건이 더 구체적일수록(항목 추가) 신뢰도는 올라가거나 같아짐

**Apriori 알고리즘 완전 정리**

**핵심 원리: 비단조적 성질(Antimonotone)**

> 빈발 항목집합의 **모든 부분집합은 반드시 빈발** 반대로 비빈발 항목집합의 **모든 상위집합은 반드시 비빈발**

```
{A,B,C}가 빈발이면 → {A,B}, {A,C}, {B,C}, {A}, {B}, {C} 모두 빈발

```

|단계|동작|
|---|---|

|3|남은 것끼리 조합 → 2-itemset 생성|
|4|또 지지도 계산 → 미만이면 버림|
|5|반복...|
|6|최종 빈발 항목집합으로 연관 규칙 생성|

```
= 60 / 80
= 0.75
```

- 특이도 = 실제 정상 중 **맞게 음성 판정한 비율** (TN/실제음성)

|② 분류 + 회귀 모두 사용 가능|✅ 옳음|Random Forest 등 둘 다 됨|
|③ 의사결정트리 활용 → **해석 용이**|❌ 틀림|모델 여러 개 합치면 **오히려 해석 어려움** (블랙박스)|
|④ 정확도 높아지지만 계산량 증가|✅ 옳음|앙상블 단점 그대로|

**③이 틀린 이유:**

의사결정트리 **단일** 모형은 해석 쉬움

근데 앙상블은 트리를 **수백 개** 합치니까

→ "이게 왜 이런 결과가 나왔지?" 설명 불가 = **블랙박스** ㅋㅋ
    앙상블은 항상 해석 어렵다.


[참고]

**앙상블 = 여러 모델 합쳐서 더 좋은 결과 내기**

**비유:**

> 전문가 1명한테 물어보는 것보다 전문가 100명한테 물어보고 다수결 내는 게 더 정확함

■ 정의 = 여러 모델을 결합해 **성능 향상**

■ 핵심 = 약한 모델 여러 개 → 강한 모델 하나

■ 장점 = 정확도↑

■ 단점 = 계산량↑, 해석 어려움(블랙박스)

■ 함정 = 단일 모델 아님 / 의사결정트리 단독 사용 아님

**대표 기법 3가지:**

**배깅 Bagging(Bootstrap Aggregating)** 

```
■ 정의 = 데이터 랜덤 샘플링 후 독립적으로 학습 → 평균/다수결

■ 구조 = 복원추출 → 여러 모델 병렬 학습 → 평균/투표

■ 특징 = 병렬 처리 / 분산(Variance) 감소 / 과적합 방지

■ 대표 = Random Forest

■ 함정 = 순차 아님
```

**부스팅 Boosting**

```
■ 정의 = 이전 모델 오답을 보완하며 순차적으로 학습

■ 구조 = 모델1 → 틀린 거 가중치↑ → 모델2 → 반복

■ 특징 = 순차 처리 / 편향(Bias) 감소 / 정확도 높음

■ 대표 = XGBoost, AdaBoost

■ 함정 = 병렬 아님 / 과적합 위험
```

**스태킹 Stacking**

```
■ 정의 = 여러 모델 결과를 또 다른 모델이 학습해서 최종 결과

■ 특징 = 가장 복잡함

■ 함정 = Bagging/Boosting 아님
```

**Random Forest:**

Bagging + 의사결정트리 조합

```
■ 정의 = 배깅 + 결정트리 + 랜덤 feature 선택

■ 구조 = 복원추출 → 트리마다 변수 랜덤 선택 → 수백개 트리 → 다수결/평균

■ 특징 = 과적합 감소 / 변수 중요도 파악 가능 / 안정적 성능

■ 함정 = Boosting 아님 (Bagging 계열)
```

랜덤 포레스트 특징:

- 과적합에 강함
- 변수 중요도 파악 가능
- 단일 트리보다 정확도 높음
- 해석은 어려움 (블랙박스)

**LOD ⊂ 시맨틱 웹 ⊃ 온톨로지** 관계입니다.

| 구분       | 설명                                             |
| -------- | ---------------------------------------------- |

## 2025년 52번 — 무손실 분해 (Lossless Join)

**정답: ③ or ④ (둘 다 손실 분해)**

**무손실 분해 조건**

```
R1 ∩ R2 (공통속성) 가 R1 또는 R2의 슈퍼키여야 함
```

**각 선택지 분석**

|-|공통속성|슈퍼키 여부|판정|
|---|---|---|---|
|① R1(A,B,C,D) R2(C,D)|{C,D}|R2의 전체 = 슈퍼키 ✅|무손실|
|② R1(A,B,C) R2(C,D)|{C}|C→D이므로 R2 슈퍼키 ✅|무손실|
|③ R1(A,C) R2(A,B)|{A}|—|**손실**|
|④ R1(A,B) R2(A,C,D)|{A}|—|**손실**|

**③이 손실인 이유**

```
R1(A,C) + R2(A,B) = A,B,C 만 있음
원본 R(A,B,C,D)의 D가 아예 없음! → 복구 불가
```

**④이 손실인 이유**

```
공통속성 {A}의 클로저:
FDs: (A,B)→C, C→D, C→A
A만으로는 B도 C도 결정 불가
A+ = {A} → 슈퍼키 아님 → 손실
```

## 2025년 54번 — 2NF 정규화

**정답: ② or ③ (2개 or 3개)**

**기본키 파악**

```
(EmpID, ProjID) → DeptID → DeptName
(EmpID, ProjID) → EmpName (EmpID만으로도 결정)
```

**기본키 = (EmpID, ProjID)**

**부분 함수 종속 찾기 (2NF 위반)**

```
EmpID → EmpName     ← EmpID만으로 결정 = 부분 종속 ❌
DeptID → DeptName   ← DeptID만으로 결정 = 부분 종속 ❌
(EmpID, ProjID) → DeptID  ← 완전 종속 ✅
```

**2NF 분해 — 부분 종속 제거**

|릴레이션|속성|
|---|---|
|R1|EmpID, ProjID, DeptID|
|R2|EmpID, EmpName|
|R3|DeptID, DeptName|

→ **3개** 생성

**②가 정답인 경우:**

```
R1(EmpID, ProjID, DeptID, DeptName) + R2(EmpID, EmpName)
= DeptID→DeptName을 이행 종속으로 보고 2NF에선 허용
= 2개
```

**③이 정답인 경우:**

```
DeptID→DeptName도 부분종속으로 보고 분리
= 3개
```

**핵심**

```
2NF = 부분 함수 종속 제거
3NF = 이행 함수 종속 제거 (DeptID→DeptName은 여기서 제거)
```

엄밀히는 **③ 3개**가 더 정확해. DeptID→DeptName도 부분종속이거든 👍

## 2025년 58번 — SQL 분석

**정답: ① **

**SQL 구조 분석**

```sql
SELECT ENO FROM E
WHERE NOT EXISTS (
    (SELECT PNO FROM P WHERE DNO=1)   -- 1번 부서 프로젝트 전체
    EXCEPT
    (SELECT PNO FROM W WHERE E.ENO=ENO) -- 해당 직원이 참가한 프로젝트
)
```

**EXCEPT 의미**

```
A EXCEPT B = A에는 있고 B에는 없는 것

1번 부서 프로젝트 EXCEPT 직원이 참가한 프로젝트
= 직원이 참가 안 한 1번 부서 프로젝트
```

**NOT EXISTS 의미**

```
NOT EXISTS (참가 안 한 1번 부서 프로젝트)
= 참가 안 한 프로젝트가 없다
= 1번 부서 프로젝트 모두 참가함
```

**선택지 판정**

|선택지|판정|근거|
|---|---|---|
|① 1번 부서 프로젝트에 **참가하지 않은** 직원|❌ 틀림|반대. **모두 참가한** 직원을 구하는 쿼리|
|② 1번 부서 프로젝트에 **모두 참가한** 직원|✅ 맞음|NOT EXISTS + EXCEPT = 나눗셈 연산과 동일|
|③ 두 번째 부속질의는 상관 질의|✅ 맞음|E.ENO=ENO → 외부 E 참조 = 상관 질의|
|④ 관계 대수 나눗셈과 동일|✅ 맞음|÷ 연산 = "모든 ~에 참가한" 구현|

**핵심 패턴 암기**

```
NOT EXISTS (A EXCEPT B) = A를 모두 포함하는 B
= 관계대수 나눗셈(÷)
= "모든 ~에 참가한"
```

## 2025년 69번 — 베이즈 정리

**정답: ④ 0.6**

**주어진 값**

```
P(C1) = 0.5,  P(C2) = 0.5
P(X|C1) = 0.6
P(X|C2) = 0.4
```

**베이즈 공식**

```
P(C1|X) = P(X|C1) × P(C1) / P(X)
```

**P(X) 계산 (전체 확률)**

```
P(X) = P(X|C1)×P(C1) + P(X|C2)×P(C2)
     = 0.6×0.5 + 0.4×0.5
     = 0.3 + 0.2
     = 0.5
```

**P(C1|X) 계산**

```
P(C1|X) = 0.6×0.5 / 0.5
         = 0.3 / 0.5
         = 0.6
```

**직관으로 이해**

```
C1, C2 사전확률 동일(0.5) → 사전확률 상쇄
→ P(X|C1) / (P(X|C1) + P(X|C2))
= 0.6 / (0.6+0.4)
= 0.6 / 1.0
= 0.6
```

## 2025년 70번 — 즉시 변경 복구

**정답: ③ C : 8**

**트랜잭션 상태 분류**

```
checkpoint 기준으로 판단

T1: start→commit (checkpoint 전) → 제외
T2: start (checkpoint 전), commit 없음 → UNDO
T3: start (checkpoint 후), commit 있음 → REDO
```

| 트랜잭션 | 상태                  | 처리   |
| ---- | ------------------- | ---- |
| T1   | checkpoint 전 commit | 제외   |
| T2   | commit 없이 crash     | UNDO |
| T3   | checkpoint 후 commit | REDO |

**REDO 처리 (변경 후 값으로)**

```
T3 REDO: C = 9  (8→9), D = 14 (12→14)
```

**UNDO 처리 (변경 전 값으로)**

```
T2 UNDO: B = 5  (5←7, 원래값으로)
         A = 15 (15←20, 원래값으로)
```

**최종 복구 값**

| 데이터 | 복구 값  | 이유                          |
| --- | ----- | --------------------------- |
| B   | 5     | T2 UNDO(5←7)                |
| C   | **9** | T3 REDO(9)                  |
| D   | 14    | T3 REDO(14)                 |

**③이 틀린 이유**

```
C는 REDO 대상 → 변경 후 값 9
문제에서 C:8 이라고 했으니 틀림
```


```
T1 REDO: A = 15  (이미 15니까 그대로)
T2 UNDO: A = 15  (20→15로 되돌림)
```

**T1 REDO는 사실상 no-op**

```
checkpoint 전 commit = 이미 디스크에 반영
→ REDO 해봤자 같은 값 덮어쓰는 것
→ 결과 변화 없음
```

**실질적으로 A를 바꾸는 건 T2 UNDO** 👍

이게 checkpoint의 존재 이유야 — checkpoint 전 commit은 안전하다고 보장해주는 거거든.


**즉시 변경(Immediate Update) 복구 순서**

```
① UNDO 먼저  → commit 없는 트랜잭션 롤백
② REDO 나중  → commit 완료 트랜잭션 재실행
```

**왜 UNDO 먼저냐면**

```
crash 시점 디스크에는
- 미완료 트랜잭션 데이터가 섞여있음 (즉시 변경이라)
- 이걸 먼저 제거해야 깨끗한 상태

그 다음 REDO로 완료된 것만 재반영
```

**지연 변경(Deferred Update)은 반대**

```
지연 변경 → UNDO 불필요, REDO만
(commit 전엔 디스크에 안 쓰니까)
```

즉시 변경이라서 UNDO가 필요하고 먼저 하는 거야 👍

## 2025년 74번 — Gini Index / 분류 오류

**정답: ④ 3개, 3개면 Gini Index = 1이다 (틀림)**

**Gini Index 공식**

```
Gini = 1 - Σ(pi²)
p1 = 클래스1 비율, p2 = 클래스2 비율
```

**각 선택지 계산**

| -   | 구성     | 계산                                         | 결과      | 판정           |
| --- | ------ | ------------------------------------------ | ------- | ------------ |
| ①   | 0개, 6개 | 1-(0²+1²) = 1-1                            | **0**   | ✅ 맞음         |
| ②   | 1개, 5개 | p1=1/6, p2=5/6<br>1-(1/36+25/36) = 1-26/36 | **>0**  | ✅ 맞음         |
| ③   | 3개, 3개 | 분류오류 = 1-max(p) = 1-0.5                    | **0.5** | ✅ 맞음         |
| ④   | 3개, 3개 | 1-(0.5²+0.5²) = 1-0.5                      | **0.5** | ❌ 틀림 (1이 아님) |

**Gini Index 범위**

```
완전 순수 (0개, 6개) → Gini = 0    (최소)
완전 혼합 (3개, 3개) → Gini = 0.5  (최대, 이진분류)
절대 1이 될 수 없음!
```

**max(p) = 가장 많은 클래스의 비율**

```
분류 오류 = 1 - max(p1, p2)
```

**직관**

```
노드에서 다수결로 분류할 때
틀릴 확률 = 소수 클래스 비율
```

**예시**

```
3개, 3개 → p1=0.5, p2=0.5
max(0.5, 0.5) = 0.5
분류오류 = 1 - 0.5 = 0.5

1개, 5개 → p1=1/6, p2=5/6
max(1/6, 5/6) = 5/6
분류오류 = 1 - 5/6 = 1/6

0개, 6개 → p1=0, p2=1
max(0, 1) = 1
분류오류 = 1 - 1 = 0  (완벽 분류)
```

**한 줄 요약**

```
다수결 분류 시 틀리는 비율 = 분류 오류
```

**분류 오류 공식**

```
max(p) = 가장많은 클래스
분류 오류 = 1 - max(p1, p2)
3개,3개 → max = 0.5 → 오류 = 0.5  ✅
```

**57. 데이터베이스에서 사용하는 인덱스에 대한 설명 중 가장 잘못된 것은 무엇인가?**

① 도메인 인덱스는 컬럼의 특정 타입을 위해 만들어진 인덱스로서 갱신이 적은 큰 문서의 Full-text 검색에 유리하며, 내용이 갱신되었을 경우에는 인덱스 REBUILD 작업을 별도로 진행하여야 한다. 
② 함수 기반 인덱스의 장점은 입력 값이 동일해도 결과가 다른 경우에도 사용할 수 있다는 것이다. 
③ 밀집 인덱스(dense index)는 데이터 레코드 하나에 대해 하나의 인덱스 엔트리가 만들어지는 인덱스이다. 
④ 집중 인덱스(clustered index)는 데이터 레코드의 물리적 순서가 인덱스 엔트리 순서와 동일하게 유지하도록 구성된 인덱스이다.

**정답: ②**

|선택지|판정|이유|
|---|---|---|
|① 도메인 인덱스|✅|Full-text 검색용, REBUILD 필요 → 맞음|
|② 함수 기반 인덱스|❌|**반대**. 입력값 동일하면 결과도 동일해야 사용 가능. 비결정적 함수(SYSDATE, RANDOM 등)는 사용 불가|
|③ 밀집 인덱스|✅|레코드 1개 = 엔트리 1개 → 맞음|
|④ 집중 인덱스|✅|물리적 순서 = 인덱스 순서 → 맞음|

**함수 기반 인덱스 핵심**

사용 가능 조건: **결정적 함수(Deterministic)** 만 가능 → 같은 입력 → 항상 같은 출력이어야 인덱스로 쓸 수 있음

```
UPPER(name)   → 가능 (항상 동일 결과)
SYSDATE       → 불가 (호출할 때마다 다름)
DBMS_RANDOM   → 불가 (호출할 때마다 다름)
```

**인덱스 종류 전체 정리**

| 인덱스                              | 핵심 특징                                        | 비고                       |
| -------------------------------- | -------------------------------------------- | ------------------------ |
| 밀집 인덱스 (Dense Index)             | 레코드 1개 = 엔트리 1개. 모든 레코드에 인덱스 존재              | 검색 빠름, 공간 많이 씀           |
| 희소 인덱스 (Sparse Index)            | 일부 레코드에만 엔트리 존재 (블록 단위)                      | 공간 절약, 검색 느림             |
| 집중 인덱스 (Clustered Index)         | 물리적 저장 순서 = 인덱스 순서. 테이블당 1개만 가능              | 범위 검색에 유리                |
| 비집중 인덱스 (Non-clustered Index)    | 물리적 순서와 무관. 테이블당 여러 개 가능                     | 포인터로 실제 데이터 참조           |
| 함수 기반 인덱스 (Function-based Index) | 컬럼 값이 아닌 함수 결과값으로 인덱스 생성. ==**결정적 함수만 가능**== | UPPER(name) 같은 변환 검색에 유리 |
| 도메인 인덱스 (Domain Index)           | 특정 데이터 타입 전용. Full-text, 공간 데이터 등            | 갱신 시 ==REBUILD== 필요      |
| 비트맵 인덱스 (Bitmap Index)           | 값의 종류가 적은 컬럼에 적합 (성별, 등급 등). 비트열로 저장         | OLAP에 유리, OLTP엔 부적합      |
| 결합 인덱스 (Composite Index)         | 두 개 이상 컬럼을 묶어서 인덱스 생성                        | 선두 컬럼 조건 없으면 인덱스 안 탐     |
| 해시 인덱스 (Hash Index)              | 해시 함수로 버킷에 저장. 동등 검색(=)에 최적                  | 범위 검색 불가                 |

**시험 단골 비교**

|구분|밀집 vs 희소|집중 vs 비집중|
|---|---|---|
|차이|엔트리 수|물리적 저장 순서 일치 여부|
|개수|둘 다 여러 개 가능|집중은 테이블당 **1개만**|
|범위검색|밀집이 유리|집중이 유리|

**61. 데이터베이스의 성능을 향상시키기 위한 방법 중 하나인 쿼리오프로딩(Query-off Loading)에 대한 설명 중 가장 부적절한 것은?**

① Horizontal Scale-out 방식이다. 
② 물리적으로 다른 데이터베이스에 데이터를 샤드(Shard)라는 각각의 개별 파티션으로 분할하여 분산 저장하고 조회하는 방법이다. 
③ Update 트랜잭션과 Read 트랜잭션을 분리하여 DB 성능을 향상시키는 기법이다. 
④ Master DB는 Update 트랜잭션(Create/Delete/Update)만 수행하고 Staging DB는 Master DB에서 바로 다수의 Slave DB로 복제 시 성능 저하 방지하기 위한 중간 경유지 역할을 한다.

**정답: ②**

|선택지|판정|이유|
|---|---|---|
|① Horizontal Scale-out|✅|읽기 DB 여러 개로 수평 확장 맞음|
|② 샤딩(Sharding) 설명|❌|이건 **샤딩** 설명. Query-off Loading 아님|
|③ Update/Read 트랜잭션 분리|✅|Query-off Loading 핵심 맞음|
|④ Staging DB = 중간 경유지|✅|Master→Staging→Slave 구조 맞음|

**Query-off Loading vs 샤딩 차이:**

|기법|설명|
|---|---|
|Query-off Loading|Master(쓰기) / Slave(읽기) **역할 분리**|
|Sharding|데이터를 **물리적으로 분할**해서 여러 DB에 저장|

②는 완전히 다른 개념인 샤딩을 설명한 거야. 

|기법|설명|위치|
|---|---|---|
|파티셔닝 (Partitioning)|하나의 DB 안에서 테이블을 행 기준으로 분할|**같은 DB 서버**|
|샤딩 (Sharding)|**물리적으로 다른 DB 서버**에 데이터 분산|**다른 DB 서버**|

```sql
   FROM PLAYER X
   WHERE X.TEAM_ID = 'K02'
   AND NOT EXISTS (SELECT 1 FROM PLAYER Y
                   WHERE Y.PLAYER_ID = X.PLAYER_ID
                   AND POSITION = 'MF');

④ SELECT NAME, POSITION, BACK_NO, HEIGHT
   FROM PLAYER
   WHERE TEAM_ID = 'K02'
   AND PLAYER_ID NOT IN (SELECT PLAYER_ID
                         FROM PLAYER
                         WHERE POSITION <> 'MF');
```

**정답: ④**

|선택지|의미|판정|
|---|---|---|
|① MINUS|K02팀 중 MF 제외|✅|
|② AND POSITION <> 'MF'|K02팀 중 MF 아닌 것|✅|
|③ NOT EXISTS|K02팀 중 MF인 레코드 없는 것|✅|
|④ NOT IN 서브쿼리|K02팀 중 **POSITION이 MF가 아닌** PLAYER_ID 제외|❌|

**MINUS = 차집합**

```sql
A MINUS B = A에서 B에 해당하는 것 빼기
```

이 문제에서:

```sql
K02팀 전체
MINUS
MF 전체
= K02팀 중 MF 아닌 것
```

**집합 연산자 정리:**

| 연산자       | 의미          | 비고                     |
| --------- | ----------- | ---------------------- |
| UNION     | 합집합 (중복 제거) |                        |
| UNION ALL | 합집합 (중복 포함) |                        |
| INTERSECT | 교집합         |                        |
| MINUS     | 차집합         | Oracle                 |
| EXCEPT    | 차집합         | SQL Server, PostgreSQL |

Oracle은 MINUS, 다른 DB는 EXCEPT 씀.

**④가 다른 이유:**

서브쿼리가 `WHERE POSITION <> 'MF'` 즉 **MF가 아닌** 선수들의 PLAYER_ID를 반환함.

그걸 NOT IN으로 제외하면 → **MF가 아닌 선수들을 제외** = **MF만 남음**

나머지 ①②③은 모두 **K02팀에서 MF 제외**인데, ④는 반대로 **MF만 남기는** 결과가 됨.

**67. 다음 중 검사시점(checkpoint) 회복 알고리즘에 대한 설명으로 옳은 것만 모두 고른 것은?**

> ㄱ. 체크포인트 시 기록되는 로그 레코드 `<checkpoint L>`의 L에는 체크포인트 이전에 이미 완료된 모든 트랜잭션 리스트가 담긴다.

> ㄹ. 회복 작업 시 로그를 순차 검색하는 과정에서 `<Ti, Commit>` 로그 레코드를 만나면, 해당 트랜잭션 Ti를 UNDO-list에서 삭제하고 REDO-list에 삽입한다.

② ㄴ, ㄷ  
③ ㄴ, ㄹ  
④ ㄱ, ㄹ

선택지 판정표

|선택지|판정|이유|
|---|---|---|
|① ㄱ, ㄷ|❌|ㄱ, ㄷ이 틀림|
|② ㄴ, ㄷ|❌|ㄷ이 틀림|
|③ ㄴ, ㄹ|✅ 정답|ㄴ, ㄹ이 맞음|
|④ ㄱ, ㄹ|❌|ㄱ이 틀림|

문항별 판정표

|항목|판정|이유|
|---|---|---|

|ㄴ|✅|최종 체크포인트 시점에 **활동 중이던 트랜잭션들**은 초기 **UNDO-list**에 넣음|
|ㄷ|❌|`<Ti, Start>`를 만나면 **UNDO-list** 관련 처리 대상이지, REDO-list에 넣지 않음|
|ㄹ|✅|`<Ti, Commit>`를 만나면 **UNDO-list에서 제거**하고 **REDO-list에 삽입**|

핵심 포인트

| 항목                        | 내용                                          |
| ------------------------- | ------------------------------------------- |
| 체크포인트 로그 `<checkpoint L>` | **그 시점에 실행 중인 트랜잭션 목록**                     |
| UNDO-list 초기값             | 최종 체크포인트 당시 **활동 중 트랜잭션**                   |
| `<Ti, Commit>` 처리         | **UNDO-list 제거 + REDO-list 추가**             |
| `<Ti, Start>` 처리          | 역방향/순차 처리 문맥에서 **미완료 트랜잭션 식별용**, REDO 대상 아님 |

개념 포함 표

|개념|포함|
|---|---|
|UNDO-list|장애 시 **취소해야 할 미완료 트랜잭션**|
|REDO-list|장애 후 **재실행해야 할 완료 트랜잭션**|
|Checkpoint|회복 시작 지점을 줄여 검색 범위 축소|
|Commit|완료된 트랜잭션 → REDO 후보|

한 줄 정리

|로그를 만났을 때|처리|
|---|---|
|`<checkpoint L>`|L에 있는 트랜잭션들을 초기 **UNDO-list**에 넣음|
|`<Ti, Start>`|아직 종료 안 된 시작 트랜잭션으로 보고 **UNDO-list** 후보로 봄|
|`<Ti, Commit>`|**UNDO-list에서 제거**하고 **REDO-list에 넣음**|


| 상황            | 처리       |
| ------------- | -------- |
| **Commit 없음** | **UNDO** |
| **Commit 있음** | **REDO** |

**71. 다음 중 트랜잭션 병행제어에 대한 설명으로 옳은 것만을 모두 고른 것은?**

> ㄱ. strict 2PL(2-Phase Locking) 기법에서는 트랜잭션이 실행되기 전 모든 로크를 한꺼번에 획득한다.  
> ㄴ. rigorous 2PL 기법은 트랜잭션의 직렬 가능성을 보장하지만 교착 상태(deadlock)가 발생할 수 있다.  
> ㄷ. 트랜잭션의 격리 수준(isolation level) 중 부정 판독(dirty read)이 발생할 수 있는 것은 Read Committed이다.  
> ㄹ. 트랜잭션의 격리 수준이 더 제한적일수록 트랜잭션의 일관성을 보존하나, 동시에 수행 중인 트랜잭션들의 처리율은 떨어진다.

① ㄱ, ㄷ  
② ㄴ, ㄹ  
③ ㄱ, ㄹ  
④ ㄴ, ㄷ

정답: **②**

선택지 판정표

|선택지|판정|이유|
|---|---|---|
|① ㄱ, ㄷ|❌|ㄱ, ㄷ 모두 틀림|
|② ㄴ, ㄹ|✅ 정답|ㄴ, ㄹ이 맞음|
|③ ㄱ, ㄹ|❌|ㄱ이 틀림|
|④ ㄴ, ㄷ|❌|ㄷ이 틀림|

문항별 판정표

|항목|판정|이유|
|---|---|---|
|ㄱ|❌|실행 전에 모든 락을 한꺼번에 얻는 건 **보수적 2단계 로킹(Conservative 2PL / Static 2PL)** 설명이다. **Strict 2PL**은 보통 **배타 락(X-lock)을 커밋/롤백까지 유지**하는 방식|
|ㄴ|✅|**Rigorous 2PL**은 모든 락을 종료 시점까지 유지하므로 **직렬 가능성 보장**, 하지만 **교착상태는 발생 가능**|
|ㄷ|❌|**Dirty Read(부정 판독)** 는 **Read Uncommitted**에서 가능하다. **Read Committed**에서는 불가|
|ㄹ|✅|격리 수준이 높아질수록 일관성/고립성은 좋아지지만, 동시성은 줄어들어 처리율이 떨어질 수 있음|

핵심 포인트

|항목|내용|
|---|---|
|Conservative 2PL|시작 전에 필요한 락을 **전부 획득**|
|Strict 2PL|**배타 락(X-lock)** 을 트랜잭션 종료까지 유지|
|Rigorous 2PL|**모든 락(S/X)** 을 트랜잭션 종료까지 유지|
|Dirty Read 가능 수준|**Read Uncommitted**|
|격리수준 상승 효과|일관성↑, 동시성↓, 처리율↓|

짧게 외우면

|개념|한 줄|
|---|---|
|실행 전 락 전부 획득|**Conservative 2PL**|
|Dirty Read 가능|**Read Uncommitted**|
|Rigorous 2PL|**직렬 가능 + 교착 가능**|

**73. 두 개의 트랜잭션 T1, T2가 다음과 같이 동시에 수행될 때 발생하는 문제점에 대한 설명으로 옳은 것은?**  
**(단, T1의 timestamp = 10, T2의 timestamp = 5 이고, 숫자가 작을수록 더 오래된 트랜잭션이라고 가정한다.)**

문제 조건

|트랜잭션|timestamp|의미|
|---|--:|---|
|T1|10|더 젊음(younger)|
|T2|5|더 오래됨(older)|

수행순서 표

| T1          | T2          |
| ----------- | ----------- |
|             | lock(x)     |
|             | read(x)     |
| lock(y)     |             |
| read(y)     |             |
|             | x = x + 200 |
| y = y - 100 |             |
|             | write(x)    |
|             | lock(y)     |
| write(y)    |             |
| lock(x)     |             |
대기 관계 표

|트랜잭션|보유 자원|요청 자원|상태|
|---|---|---|---|
|T2 (ts=5, 더 오래됨)|x|y|대기|
|T1 (ts=10, 더 젊음)|y|x|대기|

wait-die 적용

규칙

| 요청자 vs 보유자              | 처리                |
| ----------------------- | ----------------- |
| 오래된 트랜잭션이 젊은 트랜잭션 자원 요청 | **wait**          |
| 젊은 트랜잭션이 오래된 트랜잭션 자원 요청 | **die(rollback)** |

| 시점                                       | 상황                    | 처리  |
| ---------------------------------------- | --------------------- | --- |
| T2(older=5)가 y 요청, y는 T1(younger=10)이 보유 | **T2 waits**          |     |
| T1(younger=10)가 x 요청, x는 T2(older=5)가 보유 | **T1 dies(rollback)** |     |

결론: **wait-die에서는 T1이 기다리는 게 아니라 롤백된다.**

wound-wait 적용

규칙

| 요청자 vs 보유자              | 처리                        |
| ----------------------- | ------------------------- |
| 오래된 트랜잭션이 젊은 트랜잭션 자원 요청 | **wound = 젊은 쪽 rollback** |
| 젊은 트랜잭션이 오래된 트랜잭션 자원 요청 | **wait**                  |

적용 표

| 시점                                       | 상황              | 처리    |
| ---------------------------------------- | --------------- | ----- |
| T2(older=5)가 y 요청, y는 T1(younger=10)이 보유 | **T1 rollback** |       |
| 이후                                       | T2가 계속 진행       | 교착 회피 |

결론: **wound-wait에서는 T1이 롤백된다.**

선택지 판정표

| 선택지                                                                      | 판정  | 이유                                     |
| ------------------------------------------------------------------------ | --- | -------------------------------------- |
| ① 갱신 손실(lost update)이 발생하므로, 트랜잭션 독립성 레벨을 Read Committed로 설정해서 해결할 수 있다. | ❌   | 이 상황은 갱신 손실이 아니라 **교착상태**              |
| ② Wound-wait 기법을 사용하면, 트랜잭션 T1이 롤백된다.                                    | ✅   | T2가 더 오래된 트랜잭션이므로 T1을 wound시켜 롤백       |
| ③ Wait-die 기법을 사용하면, 트랜잭션 T1은 기다리기만 하면 된다.                               | ❌   | wait-die에서는 **T1이 die(rollback)** 해야 함 |
| ④ 2단계 로킹 규약(2PL)을 따르면 문제를 해결할 수 있다.                                      | ❌   | 2PL은 직렬가능성은 보장하지만 **교착상태는 막지 못함**      |

핵심 포인트

| 항목            | 내용                         |
| ------------- | -------------------------- |
| older         | timestamp 작을수록 오래됨         |
| wait-die      | older wait / younger die   |
| wound-wait    | older wound / younger wait |
| 이 문제의 older   | **T2**                     |
| 이 문제의 younger | **T1**                     |

개념 포함 표

|개념|포함|
|---|---|
|교착상태(Deadlock)|순환 대기|
|Wait-die|비선점, younger rollback|
|Wound-wait|선점, younger rollback 가능|
|2PL|deadlock 가능|

한 줄 정리

|항목|내용|
|---|---|
|이론상 정답|**②**|
|이유|**wound-wait면 T1 rollback, wait-die여도 T1은 wait가 아니라 die**|
