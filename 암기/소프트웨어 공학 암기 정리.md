## 1. 클래스 다이어그램 (Class Diagram) 제일중요!!!

**정의**: 클래스의 구조(속성/연산/가시성)와 클래스 간 관계(연관/의존/상속/구현/집합/합성)를 표현

**필수 요소(3)**  
- Class(클래스) / Relationship(관계) / Multiplicity(다중도)

**선/기호(대표)**  
- 연관(Association): `-->` (필드로 지속 참조)  
- 의존(Dependency): `..>` (메서드에서 일시 사용)  
- 상속(Generalization): `--|>` (extends)  
- 구현(Realization): `..|>` (implements)  
- 집합(Aggregation): `o--` (생명주기 독립)  
- 합성(Composition): `*--` (생명주기 종속)

```mermaid
classDiagram
direction TB

class Customer {
  +String name
  +placeOrder()
}
class Order {
  +String orderNo
  +pay(p: Payment)
}
class Payment {
  +process()
}
class Animal
class Dog
class Flyable {
  <<interface>>
  +fly()
}
class Bird
class Team
class Player
class House
class Room

Customer "1" --> "0..*" Order : Association
Order ..> Payment : Dependency
Dog --|> Animal : Generalization
Bird ..|> Flyable : Realization
Team o-- "0..*" Player : Aggregation
House *-- "1..*" Room : Composition
```

**함정(3)**  
1) 연관 vs 의존: **필드면 연관 / 메서드(매개변수·지역변수)면 의존**  
2) 집합 vs 합성: **부분의 생명주기 독립=집합 / 종속=합성**  
3) 화살표 방향: **사용자→피사용자**, 상속/구현은 **자식→부모**

## 2. 객체 다이어그램 (Object Diagram)
**정의**: 특정 시점의 객체(인스턴스)와 링크를 스냅샷으로 표현

**필수 요소(3)**  
- Object(객체) / Slot(속성값) / Link(링크)

```mermaid
flowchart TB
  classDef obj fill:#ffffff,stroke:#999,stroke-width:1px,rx:10,ry:10;

  c1["c1:Customer<br/>name=Gunu"]:::obj
  o1["o1:Order<br/>orderNo=O-1001"]:::obj
  o2["o2:Order<br/>orderNo=O-1002"]:::obj

  o1 --> c1
  o2 --> c1
```

**함정(3)**  
1) 설계도가 아니라 **런타임 예시**  
2) 절차/흐름은 없음(활동/시퀀스로)  
3) 다중도 규칙은 클래스에서, 객체는 **실제 상태**

---

## 3. 컴포넌트 다이어그램 (Component Diagram)
**정의**: 배포/빌드 단위 모듈(컴포넌트) 간 의존 및 인터페이스 관계 표현

**필수 요소(3)**  
- Component(컴포넌트) / Provided(제공) / Required(요구)

```mermaid
flowchart TB
  classDef c fill:#ffffff,stroke:#666,stroke-width:1px,rx:12,ry:12;
  classDef tag fill:#ffffff,stroke:#aaa,stroke-width:1px,rx:10,ry:10,color:#444;

  UI["Web UI"]:::c
  API["API Server"]:::c
  AUTH["Auth Service"]:::c
  DB[("Database")]

  t1["<<component>>"]:::tag
  t2["<<component>>"]:::tag
  t3["<<component>>"]:::tag

  UI --- t1
  API --- t2
  AUTH --- t3

  UI -->|"REST"| API
  API -->|"uses"| AUTH
  API -->|"JDBC"| DB
```

**함정(3)**  
1) 클래스(설계)와 다름 → 컴포넌트는 **모듈/배포 단위**  
2) 배치(물리 노드)와 혼동 금지  
3) 내부 클래스 구조는 클래스 다이어그램으로

---

## 4. 배치 다이어그램 (Deployment Diagram)
**정의**: 물리 노드(Node) 위에 소프트웨어(Artifact)가 어떻게 배치되는지 표현

**필수 요소(3)**  
- Node(노드) / Artifact(아티팩트) / Communication Path(통신 경로)

```mermaid
flowchart TB
  classDef n fill:#ffffff,stroke:#666,stroke-width:1px,rx:14,ry:14;
  classDef tag fill:#ffffff,stroke:#aaa,stroke-width:1px,rx:10,ry:10,color:#444;

  subgraph Client["Client (Node)"]
    B["Browser"]:::n
    aB["<<artifact>>"]:::tag
    B --- aB
  end

  subgraph Web["Web Server (Node)"]
    W["webapp.war"]:::n
    aW["<<artifact>>"]:::tag
    W --- aW
  end

  subgraph DBNode["DB Server (Node)"]
    D[("RDBMS")] 
    aD["<<artifact>>"]:::tag
    D --- aD
  end

  B -->|"HTTPS 443"| W
  W -->|"TCP 1521"| D
```

**함정(3)**  
1) 컴포넌트(모듈)와 다름 → 배치는 **물리 위치**  
2) 네트워크 상세 과다 금지(핵심 경로만)  
3) “어디에 배치되나”가 핵심

## 5. 패키지 다이어그램 (Package Diagram)
**정의**: 요소를 논리 묶음(패키지)으로 분리하고 의존 방향을 표현

**필수 요소(3)**  
- Package(패키지) / Dependency(의존) / Layering(레이어 규칙)

```mermaid
flowchart TB
  classDef p fill:#ffffff,stroke:#666,stroke-width:1px,rx:14,ry:14;
  classDef tag fill:#ffffff,stroke:#aaa,stroke-width:1px,rx:10,ry:10,color:#444;

  UI["ui"]:::p --- t1["<<package>>"]:::tag
  SVC["service"]:::p --- t2["<<package>>"]:::tag
  DOM["domain"]:::p --- t3["<<package>>"]:::tag
  INF["infrastructure"]:::p --- t4["<<package>>"]:::tag

  UI --> SVC
  SVC --> DOM
  SVC --> INF
  INF --> DOM
```

**함정(3)**  
1) 폴더 그림이 아니라 **의존 방향**이 핵심  
2) 순환 의존은 설계 냄새(시험 단골)  
3) 패키지로 클래스 설계를 대체 불가

## 6. 복합구조 다이어그램 (Composite Structure Diagram)
**정의**: 한 구조(클래스/컴포넌트) 내부의 Part/Port/Connector 연결을 표현

**필수 요소(3)**  
- Part(부품) / Port(포트) / Connector(커넥터)

```mermaid
flowchart TB
  classDef boundary fill:#fff9db,stroke:#d6c77a,stroke-width:1px,rx:14,ry:14;
  classDef port fill:#ffffff,stroke:#555,stroke-width:1px,rx:10,ry:10;
  classDef part fill:#ffffff,stroke:#777,stroke-width:1px,rx:12,ry:12;
  classDef tag fill:#ffffff,stroke:#aaa,stroke-width:1px,rx:10,ry:10,color:#444;

  subgraph OS["OrderService"]
    style OS fill:#fff9db,stroke:#d6c77a,stroke-width:1px,rx:14,ry:14;

    T["<<composite>>"]:::tag
    P["port<br/>placeOrder()"]:::port
    R["part<br/>OrderRepository"]:::part
    G["part<br/>PaymentGateway"]:::part

    T --> P
    P --> R
    P --> G
  end
```

**함정(3)**  
1) 클래스 간 관계가 아니라 **내부 구조**  
2) 컴포넌트 다이어그램(모듈 간)과 다름  
3) 너무 상세하면 구현 문서화(핵심 part만)

## 7. 프로파일 다이어그램 (Profile Diagram)
**정의**: UML을 도메인에 맞게 스테레오타입/태그/제약으로 확장하는 규칙 표현

**필수 요소(3)**  
- Profile / Stereotype / Tagged Value(or Constraint)

```mermaid
classDiagram
class SecureProfile {
  <<profile>>
}
class encrypted {
  <<stereotype>>
}
class UserData {
  +name:String
  +rrn:String
}
SecureProfile <.. encrypted : defines
encrypted <.. UserData : applies
```

**함정(3)**  
1) “설계 그림”이 아니라 **확장 규칙**  
2) 스테레오타입 남발 금지  
3) 표준/합의 없이 쓰면 혼란

## 9. 활동 다이어그램 (Activity Diagram)
**정의**: 처리 절차의 흐름(분기/병렬/동기화)을 표현

**필수 요소(3)**  
- Action / Decision-Merge / Start-End

**Fork/Join**: 병렬 시작/동기화(굵은 막대가 교재 표준 기호)

```mermaid
flowchart TB
  classDef act fill:#ffffff,stroke:#666,stroke-width:1px,rx:12,ry:12;
  classDef bar fill:#111,stroke:#111,color:#111,rx:2,ry:2;

  S([Start]) --> A[결제 완료]:::act
  A --> F[" "]:::bar
  F --> D[배송 요청]:::act
  F --> N[알림 발송]:::act
  D --> J[" "]:::bar
  N --> J
  J --> E([End])
```

## 11. 시퀀스 다이어그램 (Sequence Diagram)
**정의**: 시간 흐름에 따른 참여자 간 메시지 순서를 표현

**필수 요소(3)**  
- Participant / Lifeline / Message

```mermaid
sequenceDiagram
  autonumber
  actor C as Customer
  participant UI as WebUI
  participant S as OrderService
  participant P as PaymentGateway
  participant R as OrderRepository

  C->>UI: 주문 클릭
  UI->>S: placeOrder(cart)
  S->>P: authorize(amount)

  alt 승인 성공
    P-->>S: ok(authCode)
    S->>R: save(order)
    R-->>S: saved(orderId)
    S-->>UI: success(orderId)
    UI-->>C: 완료
  else 승인 실패
    P-->>S: fail(reason)
    S-->>UI: error(reason)
    UI-->>C: 실패
  end
```

## 12. 커뮤니케이션 다이어그램 (Communication Diagram)

## 14. 상호작용 개요 다이어그램 (Interaction Overview Diagram)


# (5) SysML

## 1) Requirement Diagram (REQ) — 요구사항/추적의 핵심

- SysML 전용, 시험에서 “추적(Traceability)” 관계를 자주 물음
    
- 자주 쓰는 관계: 
    
    - **deriveReqt**(파생), **refine**(정제), **satisfy**(만족: 설계/블록이 요구를 만족), **verify**(검증: 테스트가 요구를 검증), **trace**(추적)
        

```mermaid
flowchart LR
  classDef req fill:#fff,stroke:#666,rx:10,ry:10;
  classDef blk fill:#fff,stroke:#444,rx:12,ry:12;
  classDef test fill:#fff,stroke:#444,rx:12,ry:12;

  R0["<<requirement>>\nREQ-0: 주문 시스템\ntext=주문/결제/배송을 제공"]:::req
  R1["<<requirement>>\nREQ-1: 결제 승인\ntext=카드 결제 승인 처리"]:::req
  R2["<<requirement>>\nREQ-2: 배송 요청\ntext=승인 후 배송 요청"]:::req

  B1["<<block>>\nOrderService"]:::blk
  B2["<<block>>\nPaymentGateway"]:::blk
  T1["<<testCase>>\nTC-1: 결제승인 테스트"]:::test

  R0 -->|deriveReqt| R1
  R0 -->|deriveReqt| R2

  B1 -->|satisfy| R2
  B2 -->|satisfy| R1

  T1 -->|verify| R1

  R1 -.->|trace| R2
  R1 -.->|refine| R1
```

**시험에서 자주 나오는 포인트**

- “설계요소가 요구를 **만족**” = `satisfy`
    
- “테스트가 요구를 **검증**” = `verify`

## 2) Block Definition Diagram (BDD) — SysML의 “클래스 다이어그램 대체”

- UML Class Diagram 확장/대체 성격
    
- Block(블록) = 시스템 구성요소(하드/소프트/인력/데이터 등 다 포함 가능)
    

```mermaid
classDiagram
direction LR

class OrderSystem {
  <<block>>
  +maxUsers: int
}
class OrderService {
  <<block>>
  +placeOrder()
}
class PaymentGateway {
  <<block>>
  +authorize()
}
class DeliveryService {
  <<block>>
  +requestShip()
}

OrderSystem *-- "1" OrderService : composition
OrderService --> PaymentGateway : uses
OrderService --> DeliveryService : uses
```

**포인트**

- BDD는 “타입/구조(블록 정의)”
    
- “내부 연결(포트/흐름)”은 다음 IBD로 간다.
    

---

## 3) Internal Block Diagram (IBD) — 블록 내부 “파트/포트/흐름” 연결

- SysML 전용, UML Composite Structure 확장
    
- 구성요소(Part) + 포트(Port) + 흐름(Item Flow)을 보여주는 게 핵심
    

```mermaid
flowchart TB
  classDef part fill:#fff,stroke:#444,rx:12,ry:12;
  classDef port fill:#fff,stroke:#777,rx:10,ry:10;

  subgraph OS["<<block>> OrderSystem (IBD)"]
    direction TB

    p1["part: orderSvc:OrderService"]:::part
    p2["part: payGw:PaymentGateway"]:::part
    p3["part: deliv:DeliveryService"]:::part

    oPort["port: orderOut"]:::port
    payPort["port: payReq/payRes"]:::port
    shipPort["port: shipReq"]:::port

    p1 --- oPort
    p2 --- payPort
    p3 --- shipPort

    p1 -->|ItemFlow: paymentRequest| p2
    p2 -->|ItemFlow: paymentResult| p1
    p1 -->|ItemFlow: shipRequest| p3
  end
```

**시험 포인트**

- “블록 **정의**”는 BDD, “블록 **내부 배선/흐름**”은 IBD
    
- IBD에서 **포트/흐름** 키워드 나오면 거의 정답.
    

---

## 4) Parametric Diagram (PAR) — 제약식(Constraint)로 성능/공학 계산

- SysML 전용
    
- 핵심은 **Constraint Block(제약 블록)** + **바인딩(binding) 연결**로 “변수들이 수식으로 묶인다”를 보여주는 것
    

예: 응답시간 `T = Proc + Net`

```mermaid
flowchart LR
  classDef c fill:#fff,stroke:#444,rx:12,ry:12;
  classDef v fill:#fff,stroke:#777,rx:10,ry:10;

  CB["<<constraint>>\nResponseTime\nT = Proc + Net"]:::c

  T["value: T"]:::v
  Proc["value: Proc"]:::v
  Net["value: Net"]:::v

  Proc -->|binding| CB
  Net -->|binding| CB
  CB -->|binding| T
```

**시험 포인트**

- PAR은 “다이어그램이 계산한다”기보다 **제약/수식을 모델에 박아 추적 가능**하게 만드는 용도.

## 5) 나머지 5개는 “UML과 거의 동일”이라 그림은 네 UML꺼 그대로 써도 됨

네 파일 표에서도 SysML에서 **ACT/SD/STM/UC/PKG는 UML 공통(거의 동일)**로 정리돼 있음

- Package Diagram (PKG) = UML 패키지와 동일
    
- Activity / Sequence / State Machine / Use Case = UML 그림 그대로 사용

# 📘 ✅ SOLID 원칙 (객체지향 설계 5대 원칙)

> 제안: **Robert C. Martin (Uncle Bob)**

---

## 1️⃣ S — **단일 책임 원칙 (SRP, Single Responsibility Principle)**

**정의**

> 하나의 클래스는 **오직 하나의 변경 이유(Responsibility)** 만 가져야 한다.

**핵심 키워드**

- 변경 이유 1개
    
- 역할 분리
    
- 응집도(High Cohesion) 향상
    

**잘못된 예**

- 주문 클래스가 결제 + DB저장 + 이메일 발송 다 함
    

**의도**

- 변경 영향 최소화
    
- 유지보수 용이성 증가
    

---

## 2️⃣ O — **개방-폐쇄 원칙 (OCP, Open-Closed Principle)**

**정의**

> 확장(Extension)에는 열려 있고, 수정(Modification)에는 닫혀 있어야 한다. (변하는것과 변하지 않는것 분리)

**핵심 키워드**

- 기존 코드 수정 X
    
- 인터페이스 / 추상화 사용
    
- 다형성(Polymorphism)
    

**예시**

```java
interface DiscountPolicy { 
    int discount(int price);
}
```

새 할인 정책 추가 시 기존 코드 수정 없이 구현 클래스만 추가

---

## 3️⃣ L — **리스코프 치환 원칙 (LSP, Liskov Substitution Principle)**

**정의**

> 자식 클래스는 부모 클래스를 대체(Substitute)할 수 있어야 한다.

**핵심 키워드**

- 행위 일관성
    
- 계약(Contract) 유지
    
- 오버라이딩 시 규약 위반 금지
    

**대표 사례**

- `Rectangle`을 상속한 `Square` 문제
    

**시험 포인트**

- "부모 타입으로 참조 가능해야 한다"
    
- "기능 축소 금지"
    

---

## 4️⃣ I — **인터페이스 분리 원칙 (ISP, Interface Segregation Principle)**

**정의**

> 클라이언트는 사용하지 않는 인터페이스에 의존하면 안 된다.

**핵심 키워드**

- 인터페이스 분리
    
- Fat Interface 지양
    
- 필요한 기능만 제공
    

**예시**

- 프린터 인터페이스에 scan(), fax() 다 넣지 말고 분리
    

---

## 5️⃣ D — **의존 역전 원칙 (DIP, Dependency Inversion Principle)**

**정의**

> 고수준(잘 바뀌지 않는것) 모듈은 저수준 모듈에 의존하면 안 되고,  
> 둘 다 **추상화(Abstraction)** 에 의존해야 한다.

**핵심 키워드**

- 구현체 → 인터페이스 의존
    
- DI (Dependency Injection)
    
- 결합도 감소
    

**잘못된 예**

```java
OrderService service = new OrderRepository();  // 구현체 직접 의존
```

**올바른 예**

```java
OrderService service = new OrderService(OrderRepositoryInterface repo);
```

# (중요) Strategy vs State vs Template Method — 3초 구분표

|구분 질문|Strategy|State|Template Method|
|---|---|---|---|
|바뀌는 것|**알고리즘**|**상태에 따른 행동**|**흐름(순서)**는 고정|
|교체 주체|외부에서 주입/교체|내부 전이(상태가 바뀜)|상위가 골격 제공|
|키워드|“알고리즘 교체”|“상태 전이”|“골격/훅”|

## 🔥 디자인 패턴 정리 (옵시디언 표 형식)

| 패턴 이름                                  | 키워드               | 설명                                                    |
| -------------------------------------- | ----------------- | ----------------------------------------------------- |
| **싱글턴 패턴 (Singleton)**                 | 단일 인스턴스           | 시스템 전체에서 **하나의 객체만**을 유지하고 **글로벌 접근** 제공              |
| **팩토리 메서드 (Factory Method)**           | 객체 생성, 인터페이스      | **객체 생성**을 하위 클래스에 위임하여, **구체 클래스**의 결합도를 낮춤          |
| **추상 팩토리 (Abstract Factory)**          | 객체 그룹, 생성을 일관성 있게 | **유사 객체 그룹을 생성**할 수 있도록 하는 인터페이스 제공                   |
| **빌더 패턴 (Builder)**                    | 객체 생성, 복잡한 객체     | **복잡한 객체**를 단계별로 만들 수 있도록 분리하여 생성하는 패턴                |
| **퍼사드 패턴 (Facade)**                    | 단순화, 인터페이스        | 복잡한 하위 시스템을 **단순한 인터페이스**로 감춰서 제공                     |
| **프록시 패턴 (Proxy)**                     | 대리자, 접근 제어        | **실제 객체**에 접근하는 **대리자**를 두고, **제어**하거나 **지연 로딩**      |
| **컴포지트 패턴 (Composite)**                | 트리 구조, 부분과 전체     | **부분**과 **전체**를 동일하게 다룰 수 있도록 하는 계층 구조                |
| **데코레이터 패턴 (Decorator)**               | 기능 확장, 객체 수정      | **기능을 동적으로** 추가할 수 있도록 **객체를 감싸는** 패턴                 |
| **어댑터 패턴 (Adapter)**                   | 인터페이스 변환          | 기존 객체를 **다른 인터페이스**로 변환하여 사용할 수 있도록 하는 패턴             |
| **브리지 패턴 (Bridge)**                    | 구현 분리, 추상화        | **구현**을 **추상화**와 분리하여 독립적으로 변화할 수 있도록 분리              |
| **합성 패턴 (Composite)**                  | 트리 구조, 객체 합성      | **부분**과 **전체**를 동일하게 다룰 수 있도록 계층적 구조로 결합              |
| **플라이웨이트 패턴 (Flyweight)**              | 공유, 객체 최소화        | **많은 객체를 공유**하여 메모리 소비를 줄이면서 객체 개수를 최소화               |
| **비지터 패턴 (Visitor)**                   | 객체 구조 변경 없이 기능 추가 | 객체의 **구조**는 그대로 두고, **기능**만 **외부에서 추가**하는 패턴          |
| **템플릿 메서드 패턴 (Template Method)**       | 알고리즘 뼈대, 하위 구현    | 알고리즘의 뼈대는 상위 클래스에서 정의하고, **세부 구현**은 하위 클래스에서 처리       |
| **상태 패턴 (State)**                      | 상태 기반 동작, 전이      | **상태**에 따라 **행동이 달라지는** 패턴, **상태 객체**가 동작을 처리         |
| **전략 패턴 (Strategy)**                   | 알고리즘 교체, 다형성      | 알고리즘을 **동적으로 교체**할 수 있게 해주는 패턴                        |
| **옵저버 패턴 (Observer)**                  | 이벤트 시스템, 구독       | **상태 변경 시 자동**으로 **구독자에게 알림**을 주는 패턴                  |
| **커맨드 패턴 (Command)**                   | 요청 객체화, 큐         | 요청을 **객체로 캡슐화**하여, 실행을 **지연**시키거나 **취소**할 수 있게 만드는 패턴 |
| **책임 연쇄 패턴 (Chain of Responsibility)** | 요청 처리, 책임 분배      | 여러 처리 객체가 **연결**되어, **요청을 처리할 수 있는 객체**에 전달되는 패턴      |
| **메디에이터 패턴 (Mediator)**                | 중앙 집중, 상호작용 중재    | 객체들 간의 **복잡한 상호작용**을 **중앙에서 중재**하는 패턴                 |
| **미멘토 패턴 (Memento)**                   | 상태 저장, 복원         | 객체의 **상태**를 **저장**하고, 필요시 **복원**할 수 있게 하는 패턴          |
| **가변 객체 패턴 (Null Object)**             | 기본 구현, 처리         | **Null 대신 기본 구현 객체**를 사용하는 패턴                         |
| **인터프리터 패턴 (Interpreter)**             | 문법 처리, 표현식        | **언어**의 문법을 처리하는 **해석기**를 정의하는 패턴                     |

# 📘 리팩토링 기법 24개 치트시트 (한글·영문·키워드·예시)

> 예시는 “패턴만 보이게” 아주 짧게 썼다. (시험용 식별에 집중)

| 한글명            | 영문명                                           | 코드에서 찾는 키워드/징후               | 미니 예시 (Before → After)                                      | 한 줄 설명          |
| -------------- | --------------------------------------------- | ---------------------------- | ----------------------------------------------------------- | --------------- |
| 메서드 추출         | Extract Method                                | 긴 코드 블록/중복 블록                | `...10줄...` → `calc();`                                     | 의미 단위로 메서드로 분리  |
| 메서드 인라인        | Inline Method                                 | “호출만 하는” 얇은 메서드              | `a=b();` → `a=...;`                                         | 불필요한 위임/추상 제거   |
| 임시변수 인라인       | Inline Temp                                   | temp가 한 번만 쓰임                | `t=f(); use(t)` → `use(f())`                                | 불필요한 temp 제거    |
| 임시변수 질의로 대체    | Replace Temp with Query                       | 계산값 저장(temp) 반복              | `t=price();` → `price()`                                    | 계산을 메서드(질의)로    |
| 임시변수 분리        | Split Temporary Variable                      | 같은 temp가 “다른 의미”로 재사용        | `t=a+b; t=c+d;` → `t1=...; t2=...;`                         | 의미가 다른 temp는 분리 |
| 매개변수 할당 제거     | Remove Assignments to Parameters              | 파라미터 직접 변경                   | `x++;` → `y=x; y++;`                                        | 파라미터는 불변처럼 취급   |
| 변수 추출          | Extract Variable                              | 복잡한 표현식                      | `if(a*b+c>d)` → `v=a*b+c; if(v>d)`                          | 가독성/의미 부여       |
| 매직넘버 상수화       | Replace Magic Number with Symbolic Constant   | 숫자 리터럴 남발                    | `if(x>86400)` → `if(x>SECONDS_PER_DAY)`                     | 의미를 이름으로        |
| 알고리즘 치환        | Substitute Algorithm                          | 로직이 복잡/대체 쉬움                 | `for(...)...` → `Collections.sort(...)`                     | 더 단순/명확한 알고리즘으로 |
| 메서드 이름 변경      | Rename Method                                 | 이름이 역할을 못함                   | `get()` → `getTotalPrice()`                                 | 이름이 문서          |
| 필드 캡슐화         | Encapsulate Field                             | `public field`               | `obj.x` → `obj.getX()`                                      | 정보은닉 강화         |
| 메서드 이동         | Move Method                                   | 다른 클래스 데이터 더 씀(Feature Envy) | `A.f()` → `B.f()`                                           | 관심사가 있는 곳으로 이동  |
| 필드 이동          | Move Field                                    | 필드가 다른 클래스에 더 어울림            | `A.x` → `B.x`                                               | 데이터의 “주인” 정리    |
| 클래스 추출         | Extract Class                                 | 한 클래스가 책임 과다(God Class)      | `User{addr,card,...}` → `User+Address+Payment`              | SRP로 분리         |
| 클래스 인라인        | Inline Class                                  | 클래스가 너무 빈약                   | `A{delegate}` + `B{...}` → `B{...+delegate}`                | 불필요 분리 제거       |
| 매개변수 객체 도입     | Introduce Parameter Object                    | 파라미터가 너무 많음                  | `f(a,b,c)` → `f(new P(a,b,c))`                              | 인자 묶어 응집        |
| 전체 객체 전달       | Preserve Whole Object                         | 객체에서 값만 쪼개 전달                | `f(o.x,o.y)` → `f(o)`                                       | 파라미터 폭발 감소      |
| 메서드 객체로 대체     | Replace Method with Method Object             | 긴 메서드+많은 지역변수                | `longMethod()` → `new MethodObj().run()`                    | 지역변수/단계 분리 쉬움   |
| 조건문 분해         | Decompose Conditional                         | if 조건/then/else가 복잡          | `if(complex){...}else{...}` → `if(isA())doA(); else doB();` | 조건/행동을 메서드화     |
| 중복 조건절 통합      | Consolidate Conditional Expression            | `if(a) .. else if(b)..` 공통   | `if(a                                                       |                 |
| 조건 중복 코드 통합    | Consolidate Duplicate Conditional Fragments   | then/else에 동일 코드             | `if..{x;}else{x;}` → `x; if..{..}else{..}`                  | 중복 제거           |
| 가드 절로 중첩 조건 대체 | Replace Nested Conditional with Guard Clauses | if 중첩이 깊음                    | `if(a){if(b){...}}` → `if(!a) return; if(!b) return; ...`   | 조기 반환으로 평탄화     |
| 조건을 다형성으로 대체   | Replace Conditional with Polymorphism         | type code + switch/if        | `switch(type){A...B...}` → `A.calc()/B.calc()`              | 분기 제거, 확장 용이    |
| 중간자 제거         | Remove Middle Man                             | 단순 위임 메서드만 존재                | `a.getB().getC()` (또는 `a.getC()->b.getC()`)                 | 불필요 위임 제거       |

## 🔷 안정성 지표 공식 정리 (시험 계산용)

|항목|공식|의미|
|---|---|---|
|**Instability (I)**|I = Fan-out / (Fan-in + Fan-out)|0에 가까울수록 안정|
|**Abstractness (A)**|A = Na / Nc|추상 클래스 비율|
|**Distance (D)**|D =|A + I - 1|
- **Fan-out**: **패키지**에서 **다른 패키지**로의 의존성 수 (의존성이 많이 퍼져 있을수록 높음)
- **Fan-in**: **다른 패키지**에서 **현재 패키지**로의 의존성 수 (현재 패키지가 얼마나 많이 참조되는지)
- **Na**: **추상 클래스**의 수
- **Nc**: **전체 클래스**의 수
- **D = 0**: **완전히 구체적**이고, **안정적인 시스템**이다.
- **D = 2**: **추상화 수준**이 높지만 **불안정성**도 높다.
- **D = -0.5**: **구체적**이고 **안정적**인 시스템.

# 📘   Cause-Effect Graph 제약 정리 

| 순서     | 제약명 (영문)             | 한글 의미  | 적용 대상           | 의미 (시험용 한 줄)             |
| ------ | -------------------- | ------ | --------------- | ------------------------ |
| **원인** | **Exclusive**        | 배타     | Cause 집합        | 여러 원인 중 동시에 참 불가 (최대 1개) |
| **원인** | **Inclusive**        | 하나 이상  | Cause 집합        | 여러 원인 중 적어도 하나는 참        |
| **원인** | **One and Only One** | 정확히 하나 | Cause 집합        | 여러 원인 중 딱 하나만 참          |
| **원인** | **Requires**         | 요구     | Cause ↔ Cause   | A가 참이면 B도 반드시 참          |
| **결과** | **Mask**             | 마스크    | Effect ↔ Effect | 하나의 결과가 다른 결과를 무효화       |
# 🔥 헷갈리는 구분

| 제약               | 참 차이 핵심                     |
| ---------------- | --------------------------- |
| Exclusive        | 0개 또는 1개 가능 **(둘 다 거짓 가능)** |
| Inclusive        | 1개 이상 **(동시 참 가능)**         |
| One and Only One | 반드시 1개 **(둘 다 참이거나 거짓 불가)** |
| Requires         | A → B (논리적 의존)              |
| Mask (결과)        | 결과 간 제약 (Effect 전용)         |

# 📘  Combinatorial Test Design
# 📌 예시 조건표 (공통 예시)

|요인(Factor)|수준(Level)|개수|
|---|---|---|
|글꼴 이름|궁서, 굴림, 바탕, 고딕|4|
|글꼴 크기|10, 12, 14, 16, 18|5|
|스타일|밑줄, 기울임, 보통|3|

👉 요인 수 = 3  
👉 수준 수 = 4, 5, 3

# 🔥 Combinatorial Test Design 암기표

| 영문                   | 한글     | 개념                       | 계산식                              | 예시 결과 (4×5×3)     |
| -------------------- | ------ | ------------------------ | -------------------------------- | ----------------- |
| **All Combinations** | 전체 조합  | 모든 가능한 조합을 전부 테스트        | $TC = n_1 \times n_2 \times n_3$ | 4×5×3 = **60**    |
| **Each Choice**      | 이치 초이스 | 각 수준(Level)이 최소 1번 이상 등장 | $TC \ge \max(n_i)$               | 최소 **5개 이상**      |
| **Pairwise (2-way)** | 2요인 조합 | 모든 2요인 조합이 최소 1회 포함      | 공식 없음 (알고리즘 기반)                  | 약 **15~20개 수준**   |
| **Base Choice**      | 기준 선택  | 기준 1개 + 한 요인씩 변경         | $TC = 1 + \sum (n_i - 1)$        | 1+(3+4+2)= **10** |

---

# 🔥 계산 예시

### 1️⃣ 전체 조합

4 × 5 × 3 = 60  

---

### 2️⃣ Each Choice

가장 많은 수준 수 = 5  
→ 최소 **5개 이상**

---

### 3️⃣ Base Choice

TC = 1 + (4-1) + (5-1) + (3-1)  
  
= 1 + 3 + 4 + 2 = 10  


---

# 🔥 난이도 순서 (시험 감각용)

Each Choice < Base Choice < Pairwise < All Combinations

---

# 🔥 한 줄 암기

- 곱하면 → 전체 조합
    
- max(n) → Each Choice
    
- 1+Σ(n-1) → Base Choice
    
- 2개씩 보장 → Pairwise

# 📘 순환복잡도 (Cyclomatic Complexity) — 

## 🔹 정의

> **독립 경로 수 = 최소 테스트 케이스 수**

제안자: McCabe  
기준: **제어 흐름(분기)**

---

## 🔹 계산 공식 3개

|방식|공식|
|---|---|
|결정문 기반|**CC = 결정문 수 + 1**|
|그래프 기반|**CC = E − N + 2**|
|영역 기반|**CC = 폐구간 수 + 1**|

---

## 🔹 핵심 개념

- 분기 없으면 1
    
- 분기 하나 추가될 때마다 +1
    

즉

> 기본 1 + 분기 수

---

# ⚠️ 시험 함정 포인트 (중요)

## 1️⃣ “조건 수”가 아니다 ❌

## 2️⃣ “결정문 수”다 ⭕

### 예시

```java
if (A && B)
```

- 조건식은 2개 (A, B)
    
- 하지만 **결정문은 1개**
    

👉 순환복잡도는 +1만 증가

---

# 📘 MC/DC와 비교

|구분|순환복잡도|MC/DC|
|---|---|---|
|기준|**결정문 수**|**조건 수**|
|목적|경로 복잡도|조건 영향 검증|
|계산|분기 수 +1|각 조건 독립 영향 확인|
|증가 기준|if 개수|A, B, C 개수|

---

### 다시 강조

```java
if (A && B)
```

|항목|값|
|---|---|
|순환복잡도|+1|
|MC/DC|조건 2개 → 최소 3개 이상 테스트 필요|

## 📘 Defect-Based Metrics (결함 기반 메트릭) 통합 정리표

| 구분   | 영문                                          | 한글        | 계산식(개념)                             | 목적          | 좋을수록 |
| ---- | ------------------------------------------- | --------- | ----------------------------------- | ----------- | ---- |
| 제품   | **Defect Density**                          | 결함 밀도     | 총 결함 수 / 규모(KLOC, FP)               | 제품 품질 수준 비교 | 낮음   |
| 공정   | **PCE (Process Containment Effectiveness)** | 공정 구속성 효율 | 해당 단계 제거 결함 / 해당 단계 발생 결함           | 단계 내 결함 억제  | 높음   |
| 공정   | **DRE (Defect Removal Efficiency)**         | 결함 제거 효율  | 해당 단계 제거 결함 / (해당 단계 제거 + 이후 단계 발견) | 단계 품질       | 높음   |
| 테스트  | **DDE (Defect Detection Efficiency)**       | 결함 검출 효율성 | 테스트 발견 / (테스트 발견 + 운영 발견)           | 테스트 효과성     | 높음   |
| 공정   | **Defect Leakage**                          | 결함 유출률    | 이후 단계 발견 결함 / 전체 결함                 | 유출 정도       | 낮음   |
| 공정   | **Defect Detection Ratio**                  | 결함 검출 비율  | 특정 단계 발견 / 전체 결함                    | 단계 기여도 분석   | 높음   |
| 프로젝트 | **Defect Arrival Rate**                     | 결함 유입률    | 단위 기간 결함 수                          | 품질 안정화 추세   | 낮음   |

# 📘   객체지향 개념 종합 식별표

| 구분  | 개념 (한글 / 영어)                 | 핵심 의미         | 코드에서 보이는 신호              | 시험 판별 공식     | 미니 코드 예시                                      |
| --- | ---------------------------- | ------------- | ------------------------ | ------------ | --------------------------------------------- |
| ①   | **추상화 (Abstraction)**        | 공통 규약 정의      | `interface`, `abstract`  | 틀만 있고 구현 없음  | `interface Shape { double area(); }`          |
| ②   | **캡슐화 (Encapsulation)**      | 데이터 보호        | `private`, getter/setter | 데이터 직접 접근 차단 | `private int r; public int getR(){return r;}` |
| ③   | **상속 (Inheritance)**         | 부모 기능 확장      | `extends`, `implements`  | is-a 관계      | `class Circle extends Shape {}`               |
| ④   | **다형성 (Polymorphism)**       | 같은 호출, 다른 실행  | 부모타입=자식객체                | 런타임에 실행 결정   | `Shape s = new Circle(); s.area();`           |
| ⑤   | **오버라이딩 (Overriding)**       | 부모 메서드 재정의    | `@Override`              | 시그니처 동일      | `@Override double area(){...}`                |
| ⑥   | **오버로딩 (Overloading)**       | 이름 같고 매개변수 다름 | 파라미터 개수/타입 다름            | 컴파일타임 다형성    | `sum(int a); sum(int a,int b);`               |
| ⑦   | **의존 (Dependency)**          | 사용하면 의존       | 매개변수 타입 사용               | “쓰면 의존”      | `void draw(Rect r){}`                         |
| ⑧   | **연관 (Association)**         | 객체가 객체를 참조    | 멤버 변수 참조<br>**생성하고 사용**  | has-a        | `class Car { Engine e; }`                     |
| ⑨   | **집합 (Aggregation)**         | 약한 포함         | 외부에서 주입                  | 독립 생명주기      | `Car(Engine e){this.e=e;}`                    |
| ⑩   | **합성 (Composition)**         | 강한 포함         | 내부에서 new                 | 같이 생성/소멸     | `Car(){ e=new Engine(); }`                    |
| ⑪   | **동적 바인딩 (Dynamic Binding)** | 런타임 결정        | 부모타입 호출                  | 다형성 기반       | `Shape s=new Circle();`                       |
| ⑫   | **정적 바인딩 (Static Binding)**  | 컴파일 시 결정      | `static`, `final`        | 다형성 약화       | `static void f(){}`                           |

# 🔥 4대 특징 한 번에 묶은 코드 예시

```java
interface Shape {              // ① 추상화
    double area();
}

class Circle implements Shape {   // ③ 상속(구현)
    private double r;             // ② 캡슐화

    @Override
    public double area() {        // ⑤ 오버라이딩
        return r * r * 3.14;
    }
}

Shape s = new Circle();           // ④ 다형성
s.area();                         // ⑪ 동적 바인딩
```

# 🔥 시험장에서 빠르게 찍는 요령

- `interface` → 추상화
    
- `private` → 캡슐화
    
- `extends` → 상속
    
- 부모타입 = 자식객체 → 다형성
    
- 매개변수로 쓰면 → 의존
    
- 내부 `new` → 합성

자바 ArrayList랑 비슷하지만

# 📘 SW Process 품질 & SW 품질 관련 표준 정리 (영문/한글 병기)

# 1️⃣ SW Process 품질 (Software Process Quality)

---

## 🔹 1. CMMI (Capability Maturity Model Integration, 능력성숙도 통합모델)

### 📌 목적

- 조직의 **소프트웨어 개발 프로세스 성숙도(Maturity)** 향상
    

### 📌 성숙도 단계 (Maturity Level, 성숙도 수준)

|Level|명칭|의미|
|---|---|---|
|1|Initial (초기)|비정형, 개인 역량 의존|
|2|Managed (관리)|프로젝트 단위 관리|
|3|Defined (정의)|조직 표준 프로세스 정립|
|4|Quantitatively Managed (정량적 관리)|통계적·정량적 통제|
|5|Optimizing (최적화)|지속적 프로세스 개선|

### 📌 핵심 포인트

- 조직 전체 성숙도 모델
    
- 단계적(Level) 향상 구조
    
- Level 4부터 정량 관리
    

---

## 🔹 2. ISO/IEC 15504 (SPICE, Software Process Improvement and Capability Determination, 소프트웨어 프로세스 개선 및 능력결정)

### 📌 목적

- 프로세스 **능력수준(Capability Level, 능력 수준)** 평가
    

### 📌 능력 수준

|Level|명칭|의미|
|---|---|---|
|0|Incomplete (불완전)|수행 안됨|
|1|Performed (수행)|프로세스 수행|
|2|Managed (관리)|계획·관리됨|
|3|Established (정립)|조직 표준화|
|4|Predictable (예측)|정량 통제|
|5|Optimizing (최적화)|지속 개선|

### 📌 특징

- 2차원 구조: 프로세스 차원(Process Dimension) + 능력 차원(Capability Dimension)
    
- CMMI는 성숙도(Maturity), SPICE는 능력수준(Capability) 중심
    

# 🔥 프로세스 모델 비교

| 구분        | 초점                  | 관점    |
| --------- | ------------------- | ----- |
| CMMI      | 조직 성숙도(Maturity)    | 단계 모델 |
| ISO 15504 | 프로세스 능력(Capability) | 수준 모델 |
| SP 인증     | 국내 인증 체계            | 실무 적용 |

### 📌 핵심

## 🔹 3. ISO/IEC 5055 (Automated Source Code Quality Measure, 자동화 소스코드 품질 측정)

### 📌 측정 방식

- 정적 분석 (Static Analysis, 정적 분석)

### 📌 평가 품질 특성

|품질 특성 (한글 / 영어)|

|신뢰성 (Reliability)|

### 📌 핵심

- 코드 수준 품질 측정 표준
    
- 기술부채(Technical Debt) 관리 가능
    
- ISO/IEC 25010과 연계

## 📘 ISO/IEC 5055 정리

# 🔎 1️⃣ 개요

|항목|내용|
|---|---|
|정식 명칭|ISO/IEC 5055|
|주제|자동화된 소스코드 기반 소프트웨어 품질 측정|
|목적|**구조적 품질(Structural Quality)**을 객관적·정량적으로 평가|
|평가 방식|**정적 분석(Static Analysis)** 기반|

👉 실행 테스트가 아니라 **코드 자체의 구조적 위험(결함 가능성)**을 평가하는 표준.

## 📌 Structural Quality (구조적 품질)

# 🎯 3️⃣ 4대 품질 특성

| 품질 특성                              | 의미        | 위험 요소 예         |
| ---------------------------------- | --------- | --------------- |

## 📘 SP 인증 (Software Process Certification, 소프트웨어 프로세스 인증)


SP 인증은 **정보통신산업진흥원**이 운영하는 국내 소프트웨어 프로세스 인증제도이다.

```
영역 → 평가항목 → 세부평가항목
```

|평가항목|핵심 내용|

# 📘 IFPUG 정리 (Function Point 분석)

## 🔷 IFPUG 개요

International Function Point Users Group (IFPUG)는  
**기능점수(Function Point, FP)** 기반의 소프트웨어 규모 측정 표준을 관리·보급하는 국제 기관이다.

- 목적: SW **기능 규모 측정**
    
- 활용: 사업대가 산정, 생산성 측정, 견적 산정
    
- 표준 지침: **CPM (Counting Practices Manual)**
    

---

# 1️⃣ 기능점수(Function Point, FP) 개념

FP는 **사용자 관점의 기능 규모**를 측정한다.

✔ LOC(코드 라인 수)와 무관  
✔ 기술 독립적  
✔ 개발 전 요구사항 단계에서 측정 가능

---

# 2️⃣ 기능 유형 (5대 기능 요소)

IFPUG는 기능을 **데이터 기능 2개 + 트랜잭션 기능 3개**로 구분한다.

## 🔷 (1) 데이터 기능 (Data Functions)

|구분|의미|
|---|---|
|**ILF (Internal Logical File)**|내부에서 유지·관리하는 논리 데이터|
|**EIF (External Interface File)**|외부 시스템이 관리하는 참조 데이터|

---

## 🔷 (2) 트랜잭션 기능 (Transactional Functions)

|구분|의미|
|---|---|
|**EI (External Input)**|외부 → 내부 입력|
|**EO (External Output)**|내부 → 외부 출력 (가공 있음)|
|**EQ (External Inquiry)**|단순 조회 (가공 없음)|

---

# 3️⃣ 기능점수 산정 절차

## 🔥 ① 미조정 기능점수(UFP) 계산

각 기능 유형에 **복잡도(저·중·고)**를 적용하여 가중치 합산

### 🔷 가중치 표

|유형|저|중|고|
|---|---|---|---|
|ILF|7|10|15|
|EIF|5|7|10|
|EI|3|4|6|
|EO|4|5|7|
|EQ|3|4|6|

👉 가중치 합 = UFP

---

14개 일반시스템특성(GSC)을 평가

```id="d84ksp"
VAF = 0.65 + (총 GSC 점수 × 0.01)
```

최종 FP 계산:

```id="x73jd9"
FP = UFP × VAF
```
