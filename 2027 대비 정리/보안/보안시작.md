# ⭐ 정보보호개론 

## 📘정보보호 기본

### 📌정보보호의 개념

정보자산을 공개/노출, 변조/수정, 지체/재난 등의 위협으로부터 보호하여 정보의 기밀성, 무결성, 가용성을 확보 정보의 수집, 가공, 저장, 전송 중에 정보의 훼손, 변조, 유출 등을 방지하기 위한 관리적, 기술적 수단

### **📌 정보보호의 목적**

#### **✔️ 정보보호 주요 목적**

| 구분      | 설명                                                                            |
| ------- | ----------------------------------------------------------------------------- |
| **기밀성** | **Confidentiality**. 비인가된 개인·단체·프로세스 등으로부터 중요한 정보를 보호하는 것, 즉 노출되지 않는다는 확신을 제공 |
|         | 위협: 스니핑, 네트워크 모니터링, 엿보기                                                       |
|         | 대응책: 암호화, 접근통제                                                                |

|         | 위협: 트로이목마, 바이러스, 해킹, 파일 변조                                                    |
|         | 대응책: 인증, 디지털 서명, 접근통제                                                         |
| **가용성** | **Availability**. 인가를 받은 사용자가 정보나 서비스를 시기 적절하게 접근/사용할 수 있는 것                  |
|         | 위협: DOS 공격, 통신방해, 자연재해                                                        |
|         | 대응책: 이중화, Fault Tolerant 시스템, 백업/DRS                                          |

# ⭐ 암호학 
## 📘**암호화의 목적**

| 목적   | 내용                                               |
| ---- | ------------------------------------------------ |
| 기밀성  | 허가 받지 않은 사람이 부당한 방법으로 정보를 입수해도 내용을 알 수 없도록 하는 기능 |
| 무결성  | 전송 데이터나 DB의 데이터가 악의적인 목적으로 위/변조되는 것 방지           |
| 인증   | 데이터의 송/수신자가 상대방의 신원을 확인, 식별하는 기능                 |
| 부인방지 | 데이터의 송/수신자가 송/수신 사실을 추후에 부정하지 못하게 하는 기능          |

- **가용성** → 암호화랑 관계없음. 이중화/백업으로 구현
- **책임추적성** → 로깅/감사로 구현, 암호화 기술 범위 밖

암호화는 **기밀성·무결성·인증·부인불가** 


## **📘**암호화의 목적

| 암호                       | 설명                              | 예시                                                            |
| ------------------------ | ------------------------------- | ------------------------------------------------------------- |
| **전치 (Transposition)**   | 평문을 d×n 행렬로 구획 후 n×d 행렬로 전치     | HELLO → H E L / L O ? → 열로 읽으면 HL EO L                        |
|                          | 암호해독: 일정 간격으로 뛰어읽어 패턴 발견 (주기분석) |                                                               |
| **대입/치환 (Substitution)** | 평문의 각 문자를 대응되는 다른 문자로 대입        | - 카이사르 암호: A→D, B→E (3칸 밀기) →  HELLO→KHOOR<br>- 영문자판으로 한글치기 등 |
|                          | 암호해독: 철자 출현빈도 언어학적 통계방법 사용      |                                                               |
| **적암호 (Product cipher)** | 대체+전치 두 방법 반복 사용 → 강력한 암호 생성    | AES: SubBytes(치환) → ShiftRows(전치) → 반복                        |
| **대수화 (Algebraic)**      | 평문의 각 글자를 숫자로 바꿔 수학적 처리         | A=1, B=2... → 행렬 곱셈으로 암호화 (Hill cipher)                       |

| 개념             | 설명                                                                                              | 해당 원리      |
| -------------- | ----------------------------------------------------------------------------------------------- | ---------- |
| 혼돈 (Confusion) | 평문-암호문 관계를 알기 어렵게                                                                               | **대입(치환)** |
| 확산 (Diffusion) | 평문의 통계적 특성을 암호문 전체에 퍼뜨림<br>(1234 와 1233 이란걸 암호화했을 때 마지막만 틀리다거나 하면 안되고 암호문 전체에 확산되어 알아 볼 수 없어야함) | **전치**     |

## **📘**암호화 방식의 분류

- 암호화를 수행하는 정보 단위에 따른 분류 : 스트림 암호 / 블록 
- 암호 키의 형태에 따른 분류 : 대칭키 암호 / 비대칭키 암호

| 구분                     | -   | 설명                                                                   |
| ---------------------- | --- | -------------------------------------------------------------------- |
| **대칭키 알고리즘 (비밀키 암호화)** | **스트림** | - 한 번에 한 비트 또는 바이트 단위로 암호화 하는 알고리즘 (XOR 연산 사용)                       |
|                        | **블록**  | - 고정된 크기의 입력 블록을 고정된 크기의 출력 블록으로 변경하는 알고리즘                           |
| **비대칭키 알고리즘 (공개키 암호)** |     | - 암호화 시의 키와 복호화 시의 키가 서로 다른 암호화 알고리즘 <br>- 소인수 분해, 이산 대수 계산의 어려움에 기반 |
| **해쉬 알고리즘**            |     | - 임의의 길이를 가진 메시지를 입력으로 받아 고정된 길이의 문자열로 변환하는 암호 알고리즘                  |

## **📘** 대칭키 기본


### **📌 대칭키(=비밀키) 암호화 개념**

- 암호화 키와 복호화 키가 동일한 암호화 방식
- **키 명칭: 비밀키(Secret Key), 세션키(Session Key), 대칭키(Symmetric Key), 관용키(Conventional Key)**

### **📌 대칭키 암호화의 조건**

- 통신하려는 두 Party는 비밀리에 같은 키를 공유해야 함 (암호화 알고리즘은 공개 가능)
- 가장 문제가 되는 것은 **키를 어떻게 안전하게 공유할 것인가**

### **📌키 개수 공식(여러가지로 쓰임)**

노드 n개일 때 필요한 키 수 =
#### ==**n×(n-1)/2**==

예) 노드 5개 → 5×4/2 = **10개**

조합론의 **nC2** (n개 중 2개 선택)

쓰이는 곳:
- 대칭키 암호 - 필요한 키 개수
- 프로젝트 관리(PMBOK) - **의사소통 채널 수**
- 네트워크 - 완전 연결 시 필요한 링크 수 P2P
- 소프트웨어 테스트 - 인터페이스 수

**N개의 노드가 1:1로 전부 연결될 때 필요한 쌍의 수**를 구하는 거라 어디서든 같은 공식

### 📌대칭키 암호화 알고리즘의 두 방식

| 구분          | 스트림 암호 (Stream Cipher)                                                | 블록 암호 (Block Cipher)                       |
| ----------- | --------------------------------------------------------------------- | ------------------------------------------ |
| **개념**      | 하나의 비트 또는 하나의 바이트 단위로 암호화                                             | 여러 개의 비트를 묶은 블록을 단위로 암호화                   |
| **암호화 방법**  | 평문을 **XOR**로 1비트씩 암호화<br>**(평문에 XOR할 대상을 키 + IV → PRNG → 키스트림으로 만듬)** | 블록 단위로 치환, 대입을 반복하여 암호화                    |
| **장점**      | 빠른 속도, 실시간 암/복호화                                                      | 대용량의 평문 암호화                                |
| **대표 알고리즘** | RC4, SEAL, LFSR                                                       | DES, 3DES, AES, IDEA, Blowfish, SEED, ARIA |

SEED는 국산 알고리즘이고 AES 국내 대응이 SEED·ARIA 


## **📘** 대칭키 키 교환 (KDC)

### 📌키교환 개념

- 대칭키 암호화를 사용하기 위해 같은 비밀키를 통신 양편이 안전하게, 제3자 유출 없이 서로 공유하게 하는 것
- N명이 통신할 경우 필요한 키 수 : **N(N-1)/2**개

### 📌키교환의 어려운 점

- 유출 문제 : 키를 교환할 때 중간에 누군가가 가로챌 수 있음
- 관리 문제 : 키를 공유하는 사람이 많을수록 암호화 통신 횟수가 많아질수록 키 관리가 어려워짐 (키주인 파악, 키 저장 문제)

### 📌키 분배 센터 (KDC, Key Distribution Center)

- 위 키교환의 어려운 점을 어느 정도 해결
- 비밀키를 만들어서 대칭키 암호화 통신을 하려는 사람들에게 분배
- 사용자는 키를 만들/전달/관리할 필요 없음
- KDC를 신뢰한다는 가정이 있어야 함
- KDC와 사용자 사이의 비밀키 교환이 먼저 이루어져야 함
- 사용자가 많아질수록 KDC의 키 관리가 힘들어짐
- 공개키 암호화는 이 모든 키 교환 문제를 쉽게 해결

#### 🔥 **세션키**
- KDC가 각 구성원들 사이의 기밀 통신 지원을 위해 생성하는 비밀키
- 발급된 세션키는 통신이 종료된 뒤 폐기

#### 🔥 **KDC 키 분배 순서**
① Alice가 Bob과 통신하기 위해 필요한 세션키를 KDC에 요청 
② KDC는 Alice의 요청을 Bob에 전달 
③ Bob이 동의하면 두 사람 사이의 세션키 생성 
④ KDC가 Alice와 Bob에게 각각 세션키를 암호화하여 전달 
⑤ Alice와 Bob이 세션키로 암호화 통신


### 📌**공개키 암호화를 이용한 키 교환**

#### 🔥 키 합의 (Key Agreement)
- 외부에 공개해도 되는 공개키를 서로 교환한 후, 자신의 키 값과 상대의 키 값을 특수한 알고리즘으로 하나의 대칭키로 만듦
- 대표 알고리즘 : Diffie-Hellman

#### 🔥 키 전송 (Key Transport)
- 한 편이 대칭키(비밀키)를 생성하고, 상대방의 공개키로 암호화하여 전달
- 상대방은 자신의 개인키로 복호화하여 대칭키 획득
- 결국 대칭키가 안전하게 공유됨

| 구분    | 키 합의           | 키 전송         |
| ----- | -------------- | ------------ |
| 키 생성  | 양쪽이 함께 만듦      | 한쪽이 생성 후 전달  |
| 대표 방식 | Diffie-Hellman | RSA 기반 키 전송  |
| 공통 목적 | 대칭키를 안전하게 공유   | 대칭키를 안전하게 공유 |

## **📘** 대칭키 스트림 암호 방식

```
평문:  H        E        L        L        O
      01001000 01000101 01001100 01001100 01001111  < 평문
키:   K        E        Y        K        E
      01001011 01000101 01011001 01001011 01000101  < KEY
XOR:  00000011 00000000 00010101 00000111 00001010  < 암호문


복호화 (암호문 XOR 같은 키):
      00000011 00000000 00010101 00000111 00001010 < 암호문 
XOR   01001011 01000101 01011001 01001011 01000101 < KEY
    = 01001000 01000101 01001100 01001100 01001111 < 평문 (복호화됨)
      H        E        L        L        O  ✅
(키와 평문을 ASCII로 변환 한 것으로 2진 처리)
```

같은 키로 두 번 XOR하면 원래 평문으로 돌아오는 거 확인됨.

#### 🔥 스트림 암호 핵심 구조

대칭키를 그대로 XOR하지 않고, 키로 **키스트림을 생성**해서 XOR

```
대칭키 (+IV) → PRNG(키스트림 생성기) → 키스트림
평문 XOR 키스트림 = 암호문
```

**왜 키 그대로 안 쓰냐**

- 키 길이만큼만 암호화 가능
- 패턴 반복 → 취약
- 키 그대로 쓰는 건 OTP(일회용 패드), 실용적이지 않음

**IV 재사용 금지** 같은 키 + 같은 IV → 같은 키스트림 → 평문 XOR 평문 노출

## **📘** 대칭키 블록 암호 방식

### 📌블록암호 운용모드 
- 블록 암호는 보통 **AES 같은 암호 알고리즘을 어떻게 운용할지**를 결정하는 방식이다. 
- 스트림 암호처럼 비트 또는 바이트 단위가 아닌 128bit 또는 256bit 같이 블록단위로 처리함

#### 🔥**블록 암호화 모드 비교**

|모드|영문|키스트림 입력|IV|병렬처리|오류전파|활용 사례|
|---|---|---|---|---|---|---|
|ECB|Electronic Code Book|없음 (평문 직접 암호화)|❌|암복호 모두 가능|없음|짧은 자료 전송, 난수/IV 생성|
|CBC|Cipher Block Chaining|이전 암호문 블록|✅|복호화만 가능|손상 블록 + 다음 블록 1개|범용 블록형 전송 인증|
|CFB|Cipher Feedback|이전 암호문 블록 → 암호화 → 키스트림|✅|복호화만 가능|손상 블록 + 다음 블록 1개|범용 스트림형 전송 인증|
|OFB|Output Feedback|이전 키스트림 → 암호화 → 키스트림|✅|암복호 모두 불가|손상 블록만|잡음 있는 채널 스트림형 전송|
|CTR|Counter|카운터값 암호화 → 키스트림|✅ (카운터)|암복호 모두 가능|손상 블록만|고속 암복호화|

**핵심 구분 포인트**
- 스트림 암호 방식 **: CFB, OFB, CTR**

#### 🔥 병렬처리 & 오류전파

| 구분     | ECB   | CBC | CFB | OFB   | CTR   |
| ------ | ----- | --- | --- | ----- | ----- |
| 암호화 병렬 | ✅     | ❌   | ❌   | ❌     | ✅     |
| 복호화 병렬 | ✅     | ✅   | ✅   | ❌     | ✅     |

### 📌 ECB (Electronic Code Book)


- 가장 단순한 블록 암호화 모드
- 각 평문 블록을 **독립적으로** 암호화 (블록 간 의존성 없음)
- **동일한 평문 블록 → 동일한 암호문 블록 생성**
- IV 사용 없음
- 병렬 처리 가능
 
**단점**

- 동일 평문 패턴이 암호문에 그대로 노출 (혼돈은 있으나 확산 미흡)
- 해커가 암호를 해독하지 않고 암호문 블록 순서 재배열만으로 평문 조작 가능
- 안전하지 않아 실제 암호화 용도로 잘 사용되지 않음

![[Pasted image 20260610092518.png]]

### 📌 CBC (Cipher Block Chaining Mode)

- 이전 암호문 블록과 현재 평문 블록을 **XOR** 후 암호화 수행
- 초기화 벡터(IV) 필요 — 첫 번째 블록은 이전 암호문이 없으므로 대신할 랜덤 비트열 사용, 암호화마다 다른 IV 이용
- 암호화는 순차처리만 가능, **복호화만 병렬처리** 가능

**특징**
- 평문 블록1과 2의 값이 같아도 암호문 블록1과 2는 다름 → ECB 패턴 노출 문제 해결
  - 단 아래와 같음
	- 변경 블록 **이전** → 암호문 동일
	- 변경 블록 **이후** → 전부 달라짐
- 도중의 평문 블록만 뽑아내서 암호화 불가
- 암호문 블록 1개 손상 시 **2개의 평문 블록**에 영향
  - (복호화할때 손상된 블록 부분과 그 다음 블록이 암호화 블록으로 XOR하기때문에 손상됨)

**사용예**
- IPSec (통신 기밀성 보호 용도)

### 📌 CFB (Cipher FeedBack mode)

- 1단계 앞의 암호문 블록을 암호 알고리즘의 **입력**으로 사용 
- 초기화 벡터(IV) 필요
- 암호 알고리즘으로 키 스트림을 생성하는 의사난수 생성기로 동작, IV가 의사난수 생성기의 **seed**에 해당
- 암호화는 순차처리만 가능, **복호화만 병렬처리** 가능
- 암호문 블록 1개 손상 시 **2개의 평문 블록**에 영향
- ==암/복호화 시 모두 **암호 모드**로 동작 (복호화 시에도 블록 암호의 암호화 함수 사용)==
  (CFB는 **블록 암호 알고리즘을 사용**하지만, **동작 방식은 스트림 암호**처럼 동작)

**CBC vs CFB 핵심 차이**
- 이전 암호문 → 암호화 함수 → 키스트림 생성 → 평문과 XOR → 암호문
- **CBC**: 평문을 XOR 먼저 → 그 결과를 암호화
- **CFB**: 이전 암호문을 암호화 먼저 → 그 결과(키스트림)를 평문과 XOR

**블록 단위 처리 여부**
- CBC는 반드시 블록 단위로만 암호화
- CFB는 1비트씩도 가능 (스트림처럼 동작) (암호블록을 먼저 암호화하여 키스트림 생성하니 비트단위 XOR가능)

**복호화 방향**
- CBC 복호화: 복호화 함수 사용
- CFB 복호화: 암호화 함수 사용 (복호화 함수 불필요)

| 항목     | CBC           | CFB               |
| ------ | ------------- | ----------------- |
| XOR 위치 | 암호화 함수 입력 전   | 키스트림 생성 후 평문과 XOR |
| 평문 처리  | 암호화 함수에 직접 입력 | 키스트림에 XOR만 함      |
| 성격     | 블록 암호         | 스트림 암호            |
| 복호화 병렬 | 가능            | 가능                |

### 📌 OFB (Output FeedBack mode)

- 암호 알고리즘의 **출력(키스트림)을 다음 암호 알고리즘의 입력으로 피드백
- 키스트림 생성이 평문/암호문과 **완전히 독립** → 키스트림 사전 생성 가능
- 키스트림을 **미리 생성** 가능 → 평문 없어도 키스트림 준비 가능
- 암호화와 복호화 구조 동일
- 암호화, 복호화 모두 **병렬처리 불가**
- 복호화 시 암호문 1개 블록 오류 → 대응하는 평문 **1개 블록만** 오류 (오류 전파 없음)
- 잡음(Noise)에 강한 Stream Mode → 위성통신 등에 사용
- IV → 암호화 → 키스트림1 → 암호화 → 키스트림2 → 암호화 → 키스트림3...

**CFB vs OFB 핵심 차이**

| 항목         | CFB           | OFB                     |
| ---------- | ------------- | ----------------------- |
| 암호화 함수 입력  | 이전 **암호문** 블록 | 이전 **키스트림** (암호화 출력)    |
| 키스트림 사전 생성 | 불가 (암호문 필요)   | 가능 (평문 독립)              |
| 오류 전파      | 있음            | 없음(암호블록 없이 스트림만 봄으로.,.) |
| 병렬처리       | 복호화만 가능       | **둘 다 불가(키스트림 순차 생성)**  |


![[Pasted image 20260610101606.png]]


### 📌 CTR (Counter mode)

- 1씩 증가하는 카운터를 암호화해서 키스트림 생성, 평문과 XOR → 암호문
- 암호화/복호화 구조 동일 → 프로그램 구현 간단
- 패딩 불필요
- 암호화/복호화 모두 **병렬처리 가능** (카운터로 블록번호를 독립 계산 가능)
- OFB 단점 보완 : OFB는 키스트림이 우연히 반복될 수 있으나 CTR은 카운터가 매번 달라 반복 없음

**카운터 구성**

- **128비트(16바이트) 기준 앞 8바이트: 암호화마다 다른 초기값, 뒤 8바이트: 블록번호 카운트**
- 매 블록마다 카운터 값이 달라지므로 키스트림도 블록마다 다른 비트열


**OFB vs CTR 핵심 차이**

| 항목         | OFB        | CTR        |
| ---------- | ---------- | ---------- |
| 키스트림 입력    | 이전 키스트림 출력 | 독립적인 카운터 값 |
| 병렬처리       | 불가         | 가능         |
| 키스트림 반복 위험 | 있음         | 없음         |

### 📌 CCM / GCM (CRT 모드를 쓰는 .. AEAD 인증방식)

**AEAD (Authenticated Encryption with Associated Data)**

- 암호화 + 인증 태그 생성을 동시에 처리하는 방식
- **기밀성 +  인증을 한 번에 제공**
- 연관 데이터(Associated Data) : 암호화는 하지 않지만 인증 태그 계산에 포함 → 평문이어야 하지만 위변조는 막아야 하는 데이터 (예: 네트워크 패킷 헤더)

**대표 알고리즘**
- AES-GCM : Galois Field 기반 GHASH로 인증 태그 생성
- AES-CCM : CBC-MAC으로 인증 태그 생성
- ChaCha20-Poly1305 

**CCM / GCM 모드 (2024년 기출)**

| 항목    | CCM (Counter with CBC-MAC) | GCM (Galois/Counter Mode)            |
| ----- | -------------------------- | ------------------------------------ |
| 기반    | Counter + CBC-MAC          | Counter + Galois Field 해시            |
| 인증 방식 | CBC 모드로 마지막 블록을 인증정보로 활용   | 갈로아 유한체 기반 Hash 함수로 인증 태그 생성(MAC 역활) |
| 공통    | AEAD (암호화 + MAC 동시 제공)     |                                      |

## **📘** **공개키 암호화 (비대칭키 암호화)**

**개념**
- 암호화 키와 복호화 키가 다른 암호화 방식

**등장 배경**

| 문제       | 설명                                    |
| -------- | ------------------------------------- |
| 키관리 문제   | 비밀키의 배분, 공유문제, 수많은 키의 저장, 관리 문제       |
| ==**인증**==   | ==**메시지의 주인(작성자 또는 발신자)을 인증할 필요**==       |
| ==**부인방지**== | ==**메시지의 주인(작성자 또는 발신자)이 아니라고 부인함을 방지**== |

**특징**
- 모든 사용자는 항상 (개인키=사설키, 공개키) 쌍을 가짐
- 개인키는 외부 유출없이 안전하게 개인이 저장, 공개키는 외부에 공개해도 됨
- 암호 효율과 강도는 대칭키 암호화에 비해 현격하게 떨어짐

**동작 방식**
- Alice가 Bob의 공개키로 평문을 암호화 → 전송 → Bob이 자신의 개인키로 복호화

| 구분      | 대칭키 암호화                     | 공개키 암호화                |
| ------- | --------------------------- | ---------------------- |
| 키 개수    | 1개 [전체 : N(N-1)/2]          | 2개 [전체 : 2N]           |
| 키 보관    | 개인이 비밀리에 보관                 | 개인키는 비밀 보관, 공개키는 외부 공개 |
| 키 교환    | 어렵고 위험함                     | 매우 쉬움                  |
| 키 길이    | 128, 256bit (작음)            | 512, 1024, 2048bit (큼) |
| 암복호화 속도 | 매우 빠름                       | 매우 느림 (100~1000배 이상)   |
| 평문 길이   | 대용량 가능                      | 대용량 어려움 (시간, 연산 제약)    |
| 기밀성     | ✅                           | ✅                      |
| 인증      | ❌                           | ✅                      |
| 부인방지    | ❌                           | ✅                      |
| 주 사용처   | 일반 평문 암호화, 네트워크 통신 데이터 암복호화 | 키 교환, 메시지 인증(전자서명)     |

## **📘** **암호해독을 위한 공격 유형**

| 한글                | 영어                             | 핵심                                    |
| ----------------- | ------------------------------ | ------------------------------------- |
| ==**암호문 단독 공격**== | Ciphertext Only Attack         | 정보없음. **암호문만으로** 평문/키 추출<br>일반적으로 불가능 |
| ==**알려진 평문 공격**== | Known Plaintext Attack         | **평문이랑 그에 대응하는 암호문**을 **이미 알고 있는** 상태 |
| ==**선택 평문 공격**==  | Chosen Plaintext Attack (CPA)  | 원하는 평문 암호화 가능 (암호기 접근 가능)             |
| ==**선택 암호문 공격**== | Chosen Ciphertext Attack (CCA) | 원하는 암호문 복호화 가능 (복호화기 접근 가능)           |
| 무차별 대입 공격         | Brute Force Attack             | 키 전수조사                                |
| 차분 공격             | Differential Cryptanalysis     | 입력 차이 분석                              |
| 선형 공격             | Linear Cryptanalysis           | 선형 근사                                 |
| **부채널 공격**            | **Side Channel Attack**            | **전자기 변화, 연산 속도 등 간접 정보 이용**              |

## 📘 해시 함수 (메시지 다이제스트 함수)

### 📌 개념

- 복호화가 불가능한 암호화 방식의 일종
- 메시지 다이제스트라고도 불림
- 해시함수로 생성되는 값(암호화된 값)을 **MD값** 혹은 **해시값**이라 부름

### 📌 해시함수 요구사항

|특징|설명|
|---|---|
|메시지 압축 (Message Digest)|임의의 길이 평문 → 고정된 길이 출력값으로 변환|
|계산의 용이성|입력값 x가 주어지면 H(x)는 계산적으로 용이해야 함|
|일방향 (One-Way)|평문→해시값은 쉽지만, 해시값→평문은 사실상 불가. **역상 저항성**|
|약한 충돌 회피성|x가 주어졌을 때 H(x')=H(x)인 x'(≠x) 찾기 불가. **제2역상 저항성**|
|강한 충돌 회피성|H(x')=H(x)인 임의의 두 입력 x, x' 찾기 불가. **충돌저항성**|

### 📌 해시함수 사용

- 메시지 다이제스트는 **무결성 서비스만** 제공
- 수신 메시지 변조 여부, 다운로드 프로그램 변조 여부 확인 가능
- 주로 (공개키 암호화 + 해시함수) 조합으로 전자서명에 사용

| 구분                    | 종류                                   |
| --------------------- | ------------------------------------ |
| 키 없이 사용 (HASH 단독)     | **MIC (메시지 무결성 코드), MDC (수정 검출 코드)** |
| 키와 함께 사용 (HASH + 대칭키) | MAC (메시지 인증 코드)                      |

### 📌 MAC (Message Authentication Code)

![[Pasted image 20260612100819.png]]

🔥**정의**
- 송신자와 수신자가 공유한 비밀키로 만든 고정 길이 인증값 
- 메시지 사이에 MD값을 원본 메시지 사이에 끼워 넣는 것


> **메시지 + 공유된 비밀키** → 알고리즘 → MAC값 
> 받는 쪽도 똑같이 **메시지 + 공유된 비밀키** → 알고리즘 → MAC값  비교 → **무결성 + 인증 확인**

🔥**MAC 종류**

| 종류      | 기반            | 특징                                          |
| ------- | ------------- | ------------------------------------------- |
| HMAC    | 해시 함수         | 해시함수와 비밀키를 결합하여 인증 태그 생성                    |
| CBC-MAC | 블록암호 CBC      | CBC 모드로 마지막 블록을 인증 태그로 활용, 가변길이 취약          |
| **CMAC**    | **CBC-MAC 개선**    | **CBC-MAC 취약점 보완, 가변길이 메시지 안전, NIST 권장**        |
| **CCM**     | **CTR + CBC-MAC** | **CTR로 암호화 + CBC-MAC으로 인증 동시 처리 (WiFi, IoT)**   |
| GCM     | CTR + GHASH   | CTR로 암호화 + 갈루아 유한체 기반 GHASH로 인증 태그 생성 (TLS) |

### 📌 해시함수 알고리즘 종류(MDC)

|알고리즘|설명|출력값 길이|블록 길이|
|---|---|---|---|
|MD5|가장 많이 사용, 출력값 작은 편|128 bit|512 bit|
|SHA-1|미국 표준, MD4 기반, MD5보다 안전|160 bit|512 bit|
|SHA-256|SHA-1 기반, 해시값 증가|256 bit|512 bit|

#### 🔥 SHA-3

- SHA3-224, SHA3-256, SHA3-384, SHA3-512 4개 해시함수
- SHAKE128, SHAKE256 2개의 확장 가능한 출력함수(XOF)로 구성
- **스펀지 구조**: 입력 메시지 → bitrate에 따라 f함수 출력과 XOR → 흡수(absorbing) → 압착(squeezing) → 최종 해시값 출력

|종류|출력 크기(bit)|블록 크기(bit)|
|---|---|---|
|SHA3-224|224|1152|
|SHA3-256|256|1088|
|SHA3-384|384|832|
|SHA3-512|512|567|

### 📌 솔트 (Salt)

- 해시값 생성 시 원래 문자열에 **추가 정보를 더해** 메시지 다이제스트 생성
- Rainbow Table을 활용한 전수공격 방어 목적

> Salt 미적용: hash('1234') Salt 적용: hash('TEST1234'), hash('test1234test') 등

#### 🔥 **솔트 vs 페퍼**

|-|솔트|페퍼|
|---|---|---|
|저장 위치|DB에 저장|DB 밖 (코드/환경변수/HSM)|
|사용자마다 다른가|✅ 각자 다름|❌ 전체 공통값|
|비밀값인가|❌ 공개|✅ 비밀|
|목적|레인보우 테이블 방어|DB 탈취 시 추가 방어|
**같이 쓰면**

```
hash(비밀번호 + 솔트 + 페퍼)
```

DB 털려도 페퍼 모르면 복원 불가.
## 📘 전자서명

### 📌 개념

- 메시지 주체라고 주장하는 사용자가 맞는지 제3자가 확인 가능한 디지털 암호화 방식
- 현재는 주로 **RSA 등 공개키 암호화 기법**으로 구현

> 전자서명법 제2조: 사용자 신원과 문서변경 확인 가능, 비대칭 암호화방식으로 전자서명키 생성

### 📌 전자서명 특성

|특징|설명|
|---|---|
|서명자 인증 용이|누구든지 전자서명 생성한 서명인 검증 가능|
|부인방지|서명인은 후에 자신이 서명한 사실 부인 불가|
|위조불가|서명인의 개인키 없으면 서명 위조 불가|
|변경불가|이미 한 서명 변경 불가|
|재사용 불가|한 문서 서명을 다른 문서 서명으로 재사용 불가|

### 📌 전자서명 생성/확인 과정 (RSA 기반)
**정의**

> 송신자의 개인키로 생성한 서명값  
> 무결성 + 인증 + 부인방지 제공


![[Pasted image 20260612102158.png]]
**(1) 서명 순서 (앨리스)**

1. 해시함수로 메시지 압축 → 압축값(Message Digest) 생성
2. 앨리스의 **개인키로 암호화** → 전자서명 생성
3. 전자서명 + 원본 메시지 → 수신인에게 전송

**(2) 확인 순서 (밥)**

1. 수신 메시지에서 메시지 다이제스트 추출
2. 동일한 해시함수로 수신 메시지 해싱 → 압축값 생성
3. 앨리스의 **공개키로 전자서명 복호화** → 압축값 추출
4. 두 압축값 비교 → 일치하면 인증

### 📌 전자서명 요소기술

- 공개키 암호화 알고리즘 (RSA 등)
- X.509 인증서
- 해시함수 (메시지 다이제스트)

### 📌 전자서명 알고리즘


| 종류      | 기반                | 특징                  |
| ------- | ----------------- | ------------------- |
| RSA     | 소인수분해             | 메시지 복구 기능, 가장 대표적   |
| ElGamal | 이산대수              | 암호문이 평문의 2배, 높은 안전성 |
| DSA/DSS | 이산대수 (ElGamal 변형) | NIST 표준, SHA와 함께 사용 |
| ECDSA   | 타원곡선              | 짧은 키로 높은 보안         |

### 📌 전자서명 종류

|종류|설명|
|---|---|
|은닉 서명 (Blind Signature)|서명자에게 메시지 보여주지 않고 서명 획득. 전자투표·전자화폐에 활용|
|이중 서명 (Dual Signature)|결제정보·주문정보 각각 서명. 판매자/결제기관 교차 검증, 상호 비노출 (SET 프로토콜)|
|대리(위임) 서명|다른 사용자에게 권한 위임하여 대신 서명|
|다중서명|하나의 문서에 여러 사람이 서명|
|그룹서명|그룹 대표로 서명. 그룹 서명인 확인 가능, 누가 서명했는지는 불가|


## 📘 하이브리드 암호 시스템

## 📘 PKI (Public Key Infrastructure)

### 📌 개념

- 공개키 암호 기술 기반으로 인증서를 생성, 관리, 저장, 분배, 말소하는 하드웨어, 소프트웨어, 인력, 정책 등의 집합체
- 공개키의 생성, 공개, 분배, 검색, 인증을 효과적으로 구현하는 절차 및 시스템 구조

### 📌 PKI 특징

|특징|설명|
|---|---|
|공개키 인증|사용자의 공개키를 인증해주는 인증기관들의 네트워크|
|키 생성/전달|개인키, 공개키를 생성하고 배달해주는 서비스|
|인증서|인증서로 공개키와 주체의 신원을 연결, 인증|
|인증서 관리|인증서 생성, 전달, 삭제, 인증하는 관리 기반구조|
|하이브리드 시스템|대칭키, MD, 공개키 등 모든 암호화 방법이 혼합된 구조|

### 📌 PKI 구성요소

| 구성요소           | 설명                                                                       |
| -------------- | ------------------------------------------------------------------------ |
| **CA (인증기관)**      | **인증 정책 수립, 인증서 및 CRL 생성/공개/취소 관리**                                          |
| **RA (등록기관)**      | **사용자 신분 확인, 인증서 요구 승인, CA에 발급 요청, CA와 앱 간 인터페이스**                           |
| 공인 인증서         | X.509 v3 표준, 전자서명 검증키와 소유자 관계 증명하는 전자 파일                                 |
| 디렉토리 서버        | 공개키 저장/조회, CA 발급 인증서와 CRL 저장. **LDAP** 사용 (계층형 DB)                       |
| CRL (인증서 취소목록) | X.500 명명방식, 갱신일 타임스탬프 포함                                                 |
| KGS (키 생성 시스템) | 인증서 발급에 사용되는 전자서명 키 생성/관리                                                |
| 암호화 알고리즘       | 공개키(RSA, DSA, KCDSA, ECC, DH), 대칭키(DES, 3DES, IDEA, SEED, ARIA), 해시(SHA) |

### 📌 인증서 (Certificate)

#### 🔥 개념

- 공개키와 해당 공개키 소유자를 연결짓는 전자문서
- 일반적으로 **CA(Certification Authority)** 가 전자서명으로 생성
- 대부분 **X.509 버전 3 표준** 사용

#### 🔥 X.509 버전 3 구조

| 구분   | 항목                                                 |
| ---- | -------------------------------------------------- |
| 필수사항 | 버전, 일련번호, 알고리즘 ID, 발행자(Issuer), 유효성, 주체, 주체 공개키 정보 |
| 선택사항 | 발행자 유일번호(v1.0), 주체 유일번호(v2.0), 확장 필드(v3.0)         |

- 공개키와 그 소유자 정보가 저장된 문서
- 인증서를 발급한 **CA의 서명값** 포함

#### 🔥 인증서 구성 항목

| 항목          | 설명                                |
| ----------- | --------------------------------- |
| 일련번호        | 인증기관 내 유일번호                       |
| 버전          | X.509 버전. 현재 3.0 사용               |
| 발급자         | 인증서를 발급한 CA의 DN (X.500 표준에 따라 명명) |
| 유효기간        | 시작일과 만료일로 구성                      |
| 소유자         | 인증서 소유자 DN                        |
| 소유자의 공개키 정보 | 공개키 값, 알고리즘 식별자 등                 |
| 서명 알고리즘     | 인증서 발급 시 사용된 알고리즘 식별자             |
| CA 서명       | 앞의 목록들에 대한 서명값                    |

| ------------------------------ | ----------------------- |
| Version                        | 0=v1, 1=v2, 2=v3        |
| Serial Number                  | 발행자가 생성한 인증서 유일 식별자     |
| CA Signature Algorithm         | 발행자가 서명에 사용한 알고리즘       |
| Issuer Name                    | 발행자 ID (X.500 명명방식)     |
| Validity Period                | 시작/끝 시간 (UTCT 형식)       |
| Subject Name                   | 공개키 소유자 ID (X.500 명명방식) |
| Subject Public Key Information | 공개키 + 알고리즘 파라미터         |
| Issuer Unique Identifier       | 선택, v2 이상               |
| Subject Unique Identifier      | 선택, v2 이상               |
| Extension Field                | 선택, 인증 정책 등             |
| Issuer's Signature             | 앞의 목록들에 대한 서명값          |

### 📌 인증서 서명/검증 과정

**(1) CA 서명 과정**

```
인증서 내용 → 해시 → 해시값 → CA 개인키로 암호화 → 서명값
```

**(2) 인증서 검증 과정**

```
인증서 내용 → 해시 → 해시값A
서명값 → CA 공개키로 복호화 → 해시값B
A == B ? → 인증서 유효
```

### 📌 CA 체인 (인증서 체인)

- 인증서는 체인으로 연결되어 서명/인증 가능
- Root CA → 중간 CA → 사용자 인증서 순으로 신뢰 체인 형성

### 📌 PKI 아키텍처

| 구분  | 계층 구조                                    | 네트워크 구조                              | 하이브리드 구조 |
| --- | ---------------------------------------- | ------------------------------------ | -------- |
| 개념  | 모든 CA가 Root CA 하위에 계층적 연결                | CA들이 상호 인증, 각자 독립 도메인                | 두 방식 혼합  |
| 장점  | 인증경로 탐색 간단, 인증 빠름, 확장성 큼                 | 상업적 신뢰관계 구축 용이, Root CA 비밀키 분실 복구 용이 | 두 방식의 장점 |
| 단점  | Root 비밀키 분실 시 전체 재발급, 중간 CA 멈추면 하위 인증 불가 | 인증 경로 복잡, 다수 인증 경로 존재                | 구성의 복잡성  |
### 📌 PKI 요소기술

| 요소기술         | 내용                                     |
| ------------ | -------------------------------------- |
| 전자서명         | 해시함수(MD5, SHA-1), RSA, ECC, DSA        |
| 키분배          | RSA                                    |
| 문서 암호화       | SEED, DES, Triple-DES                  |
| **디렉토리 서비스** | **X.500 DAP/LDAP**                     |

| 인증서 유효성 확인   | **CRL (24시간 주기 배포), OCSP (실시간 상태 검증)** |
#### 🔥 CRL (Certificate Revocation List, 인증서 취소목록)

**개념** CA에서 해지되었거나 더 이상 유효하지 않은 인증서의 시리얼 번호 목록. 두 가지 상태 — 해지(영구 무효)와 효력중지(임시 무효).

**CRL 포함 항목** 해지된 인증서의 일련번호, 해지 날짜(타임스탬프 형식), 해지 이유

|항목|설명|
|---|---|
|Version|v1 또는 v2|
|서명 알고리즘|CRL 서명에 사용된 알고리즘|
|발행자 (Issuer)|CRL을 발행한 CA|
|발행일 (This Update)|이 CRL이 발행된 날짜|
|다음 갱신일 (Next Update)|다음 CRL 발행 예정일|
|폐지된 인증서 목록|시리얼 번호 + 폐지일 + 폐지 사유|
|CA 서명|CRL 자체에 대한 CA 전자서명|

#### 🔥 OCSP (Online Certificate Status Protocol)

**개념** X.509 디지털 인증서의 해지 상태를 실시간으로 확인하기 위한 프로토콜. CRL의 주기적 업데이트 한계를 극복하기 위해 만들어짐.

**동작 방식**

```
클라이언트 → OCSP 서버에 "이 인증서 유효해?" 요청
OCSP 서버 → Good / Revoked / Unknown 응답
```

**응답 상태**

|상태|의미|
|---|---|
|Good|유효한 인증서|
|Revoked|폐지된 인증서 (폐지일, 폐지 사유 포함)|
|Unknown|해당 인증서 정보 없음|

#### 🔥 CRL vs OCSP 비교

| -     | CRL                  | OCSP            |
| ----- | -------------------- | --------------- |
| 방식    | 목록 다운로드 후 로컬 확인      | 실시간 서버 요청       |
| 실시간성  | ❌ 주기적 (보통 24시간)      | ✅               |
| 다운로드  | Full CRL / Delta CRL | 개별 요청만 (~2.5kb) |
| 프라이버시 | ✅                    | ❌ 서버가 요청 내역 파악  |
| 표준    | RFC 5280             | RFC 6960        |

### 📌보안 핵심 구분
하이브리드 암호시스템이 이미 전자봉투 구조니까 따로 행이 필요없어. 그리고 전자서명은 선택적으로 추가하는  거고.

| 🔑 구분              | 🎯 목적               | 🔐 사용 키   | ⚙️ 특징                                            | 🧠 시험 포인트                 |
| ------------------ | ------------------- | --------- | ------------------------------------------------ | ------------------------- |
| 🧩 **Hash**        | 무결성 확인              | 없음        | 메시지를 고정 길이 해시값으로 변환(단방향), **단독으로는 무결성만** (인증 불가) | SHA-256, SHA-3            |
| 🔒 **MAC**         | 무결성 + 인증            | **대칭키**   | 메시지와 비밀키로 MAC 생성, **부인방지 불가** (대칭키라 양쪽 다 키 보유)   | HMAC(Hash기반) CMAC(블록암호기반) |
| ✍️ **전자서명**        | 무결성 + 인증 + **부인방지** | **공개키**   | 개인키로 서명 / 공개키로 검증, **해시(MD) 필수**, 기밀성 없음         | RSA, ECDSA                |
| 📦 **전자봉투**        | 세션키 안전 전달           | 수신자 공개키   | 세션키를 수신자 공개키로 암호화한 것. 메시지는 세션키로 별도 암호화           | **기밀성 보조수단**              |
| 🔀 **하이브리드 암호시스템** | 기밀성                 | 대칭키 + 공개키 | **메시지→세션키 암호화 + 세션키→공개키 암호화(전자봉투).**             | PGP, SSL/TLS              |
| 📜 **인증서**         | 공개키 신뢰              | CA 서명     | 공개키 + 사용자 정보 포함                                  | **X.509 v3** 표준           |
| 🏛 **PKI**         | 인증 인프라              | 공개키 체계    | 인증서 발급/관리 시스템                                    | CA(인증기관), RA(등록기관)        |

## 📘 SSL/TLS

### 📌 개념 및 등장배경

|구분|내용|
|---|---|
|개념|•암호화 요소를 네트워크 통신에 적용, 비보안 공개 통신라인에 보안 서비스(기밀성·무결성·인증) 제공하는 보안 프로토콜|
|위치|•트랜스포트 계층의 Socket Library — 응용 프로그램과 독립적, 통신경로 전체 암호화|
|역사|•Netscape 개발(SSL V1.0→2.0→3.0, 1995)<br>•MS사 PCT 발표 → IETF가 두 개 혼합해 TLS로 표준화·관리|
|등장배경|•기밀성: 쇼핑몰 카드번호·비밀번호 노출 방지<br>•무결성: 증권거래 금액 무결성 보장<br>•인증: 사용자·서버 인증 필요|
|**부인방지**|•**SSL/TLS는 부인방지 기능 없음** (별도 전자서명 방안 필요)|

### 📌 SSL 특징

| 항목        | 내용                                                                                                 |
| --------- | -------------------------------------------------------------------------------------------------- |
| 인증기반      | •공개키 인증서 기반 인증 방법 사용                                                                               |

| 무결성       | •메시지인증코드(MAC) 제공                                                                                   |

### 📌 관련 프로토콜

|구분|내용|
|---|---|
|SSL|•Netscape사 개발 (V1.0, V2.0, V3.0)<br>•IETF에서 TLS 버전 공개 후 이후 버전 없음|
|TLS|•IETF가 SSL 기반으로 설계한 공개버전, 전송계층 보안 제공<br>•Handshake Protocol: 보안 협상, 브라우저/서버간 인증<br>•Data Exchange Protocol: 무결성·기밀성 제공|
|WTLS|•무선인터넷 환경용<br>•TLS 기반으로 개발|

### 📌 SSL 프로토콜 구성요소

|구성요소|기능|
|---|---|
|Record Layer Protocol|•분배된 대칭키로 데이터 암호화 수행<br>•TCP 위에서 동작, Handshake에서 협의된 알고리즘 사용<br>•데이터 분할 → 압축 → 암호화 → 무결성 검증 수행|
|ChangeCipherSpec Protocol|•ChangeCipherSpec 단계 시작을 알리는 신호<br>•이후부터 협의된 암호 알고리즘(비밀키·메시지다이제스트·공개키) 적용하여 통신 시작|
|Alert Protocol|•오류 및 경고 상황을 상대방에게 알림 (2바이트 구조)<br>•SSL 통신 중 문제 발생 시 해당 내용 전달|
|Handshake Protocol|•사용할 암호 알고리즘 결정 및 키 교환 수행<br>•Record Layer 상위에서 동작<br>•PKI 기반 사용자 인증 수행<br>•비밀키 생성 및 설정<br>•서버·클라이언트 인증 수행|

> 동작 계층 순서: **Handshake → ChangeCipherSpec → Record**


### 📌 SSL Handshake 메시지 교환 순서

|Client|방향|메시지 / 설명|방향|Server|
|:-:|:-:|---|:-:|:-:|
|●|→|==**① ClientHello** — SSL 시작 알림 · 랜덤값 · 지원 알고리즘 리스트==|||
|||==**② ServerHello** — 서버 랜덤값 · 선택 알고리즘 전달==|←|●|
|||**③ Certificate** — 서버 인증서 전송 (옵션)|←|●|
|||**④ ServerKeyExchange** — 서버 키 교환 파라미터 전달 (옵션, DHE/ECDHE만)|←|●|
|||**⑤ CertificateRequest** — 클라이언트 인증서 요청 (옵션)|←|●|
|||==**⑥ ServerHelloDone** — 초기화 협상 완료 알림==|←|●|
|●|→|**⑦ Certificate** — 클라이언트 인증서 전송 (옵션)|||
|●|→|==**⑧ ClientKeyExchange** — Pre-Master Secret 생성 → 서버 공개키로 암호화 전달==|||
|●|→|**⑨ CertificateVerify** — 인증서 소유 증명 (옵션)|||
|●|→|==**⑩ ChangeCipherSpec** — 암호화 파라미터 실제 적용==|||
|●|→|==**⑪ Finished** — 클라이언트 측 협상 종료==|||
|||==**⑫ ChangeCipherSpec** — 서버 암호화 파라미터 적용==|←|●|
|||==**⑬ Finished** — 서버 측 협상 종료==|←|●|

### 📌 TLS 1.2 vs TLS 1.3 비교

| 구분           | TLS 1.2                       | TLS 1.3                                               |
| ------------ | ----------------------------- | ----------------------------------------------------- |
| Handshake    | •2 RTT                        | •1 RTT                                                |
| 0-RTT        | •없음                           | •지원 (재연결 시)                                           |
| Key Exchange | •RSA / DHE / ECDHE            | **•ECDHE 전용 (RSA 제거)**<br>•타원곡선 기반 이산대수               |
| 암호 알고리즘      | •RC4, MD5, SHA-1 등 취약 알고리즘 포함 | •취약 알고리즘 전부 제거<br>•AES-GCM, AES-CCM, SHA-256, SHA-384 |
| 인증서 전송       | •평문                           | •암호화                                                  |
| PFS          | •선택적                          | •필수                                                   |
| 하위 호환성       | •-                            | •직접 호환 불가<br>•버전 협상으로 TLS 1.2 fallback 지원             |
| 보안성          | •높음                           | •더 높음                                                 |

> RSA 제거 이유: 서버 개인키 유출 시 과거 트래픽 전체 복호화 가능 → PFS 미보장 → ECDHE 강제 대체

#### 🔥 TLS 1.3 주요 변경사항

| 항목                          | TLS 1.2                                                                                        | TLS 1.3                                      |
| --------------------------- | ---------------------------------------------------------------------------------------------- | -------------------------------------------- |
| Handshake RTT               | •2 RTT<br>•1 RTT: ClientHello / ServerHello + 인증서<br>•2 RTT: 키교환 + ChangeCipherSpec + Finished | •1 RTT<br>•ClientHello에 키 교환 정보 포함으로 왕복 단축   |
| ChangeCipherSpec            | •핸드셰이크 필수 메시지                                                                                  | •프로토콜 설계상 제거<br>•middlebox 호환성 목적 더미 메시지만 잔존 |
| 핸드셰이크 메시지                   | •ServerHelloDone,ServerKeyExchange,ClientKeyExchange 존재                                        | •위 3개 메시지 제거                                 |
| 암호화 스위트                     | •취약 알고리즘 포함 (RC4, MD5 등)                                                                       | •취약 알고리즘 전부 제거<br>•5개 AEAD 스위트만 유지           |
| SNI(Server Name Indication) | •평문 전송                                                                                         | •ECH(Encrypted Client Hello)로 암호화 지원         |
| 0-RTT                       | •없음                                                                                            | •재연결 시 지원                                    |
※ SNI : 하나의 IP에 여러 도메인이 있을 때, TLS 핸드셰이크 시 클라이언트가 목적지 도메인명을 서버에 알려주는 확장 필드. TLS 1.2까지는 평문 노출, TLS 1.3 ECH에서 암호화.



#### 🔥 TLS 버전별 RTT 비교

**TLS 1.2 (2 RTT)**

|Client|방향|메시지|방향|Server|
|:-:|:-:|---|:-:|:-:|
|●|→|ClientHello (암호 목록 제안)|||
|||ServerHello + 인증서|←|●|
|●|→|ClientKeyExchange + ChangeCipherSpec|||
|||ServerFinished (암호화 시작)|←|●|
|●|⇢|Application Data 전송|⇢|●|

**TLS 1.3 (1 RTT)**

|Client|방향|메시지|방향|Server|
|:-:|:-:|---|:-:|:-:|
|●|→|ClientHello + KeyShare + 암호목록 (한번에)|||
|||ServerHello + KeyShare + 인증서 + Finished (한번에)|←|●|
|●|⇢|Application Data 전송|⇢|●|

※ TLS 1.3 핵심: ClientHello 시점에 KeyShare 동봉 → 서버가 바로 암호화 응답 → 왕복 1번에 완료
※ 0-RTT : 재연결 시 이전 세션 티켓 재사용으로 핸드셰이크 생략, 즉시 데이터 전송. **재전송 공격(Replay Attack)** 에 취약

## 📘 IPSec (Internet Protocol Security)

### 📌 개념 및 목적

|구분|내용|
|---|---|
|개념|•TCP/IP 구조적 결함 극복, IP 프로토콜 수준에서 인증·암호화를 제공하는 인터넷 표준 확장 기술|
|목적|•Inbound/Outbound 패킷 단위로 암호·인증 서비스 제공하여 상위 계층 보호<br>•주소 부족·느린 속도·보안취약성 등 TCP/IP 보안성 강화|
|활용 영역|•인증, 기밀성, 키 관리 3개 영역|

### 📌 특징

|특징|내용|
|---|---|
|IETF 표준화|•RFC 2401~2412로 문서화|
|TCP/IP 보안취약성 방지|•호스트 사이 암호화된 안전한 경로 설정 통신|
|환경변화에 유연|•IP보안 프로토콜과 키관리 매커니즘 독립적 설계|
|필요한 보안 서비스 선택사용|•인증(AH)과 암호화(ESP) 별도 프로토콜로 선택적 사용 가능|
|다양한 암호·인증 알고리즘|•각 보안 프로토콜에 새로운 보안 모드 추가 용이|

### 📌 구성요소

| 구성요소                                  | 내용                                                                        |
| ------------------------------------- | ------------------------------------------------------------------------- |
| SA (Security Association)             | •두 호스트가 사용할 보안 매커니즘·보안모드·인증·암호 알고리즘·키 값 등 매개변수에 대한 논리적 설정                 |
| AH (Authentication Header)            | •IP패킷 무연결 무결성·데이터 발신지 인증·선택적 재전송 방지 제공<br>•HMAC 메시지 인증코드 사용               |
| ESP (Encapsulating Security Payload)  | •IP패킷 기밀성 보안서비스 제공<br>•선택적으로 AH 제공 서비스 포함 가능                              |
| IKE (Internet Key Exchange) **(키교환)** | •IPSec 통신을 위한 호스트 인증·키관리를 SA를 통해 협상<br>•SA 설정·변경·삭제 절차와 형식을 정의한 키 관리 프로토콜 |

### 📌 AH vs ESP 보안 서비스 비교

|보안 서비스|AH|ESP|
|---|:-:|:-:|
|접근 제어 (Access Control)|O|O|
|비연결형 무결성|O|O|
|데이터 근원지 인증|O|O|
|재전송 방지 (Anti-Replay)|O|O|
|**비밀성 (Confidentiality)**|**✕**|**O**|
|**제한적 트래픽 흐름 비밀성**|**✕**|**O**|

※ AH와 ESP 차이 핵심: **비밀성(암호화)은 ESP만 제공**
### 📌 AH vs ESP 헤더 구조
![[Pasted image 20260614133503.png]]
#### 🔥 핵심 포인트 정리

**공통 필드**

SPI(Security Parameters Index)와 Sequence Number는 AH·ESP 둘 다 갖고 있어. SPI는 수신측이 SA(Security Association)를 찾는 32비트 인덱스고, Sequence Number는 재전송 공격 방지용 카운터야.

**AH만의 특징**

- Next Header, Payload Length, Reserved 필드가 맨 앞에 붙음
- **IP 헤더의 불변 필드까지 인증 범위에 포함** → 이게 NAT에서 망하는 이유 (NAT가 IP주소 바꾸면 ICV 검증 실패)
- 암호화 없음 → 도청 가능, 기밀성 제로

**ESP만의 특징**

- Payload 전체를 암호화 (블록 암호 정렬용 Padding 필드 존재)
- Pad Length + Next Header가 **암호화된 영역 끝에** 붙음 (Trailer)
- ICV는 암호화 이후 맨 뒤에 위치
- IP 헤더는 인증 범위 제외 → NAT 통과 가능 (NAT-T: UDP 4500 포트 캡슐화)

**시험 출제 포인트**

|구분|암기 키워드|
|---|---|
|AH|51번, 인증만, IP헤더 포함, NAT 불가|
|ESP|50번, 암호화+인증, IP헤더 제외, NAT-T 가능|
|둘 다|SPI + Sequence Number + ICV|
|ESP Trailer|Padding + Pad Len + Next Header (암호화 영역 내)|

## 📘 5. Kerberos

### 📌 개념 및 특징

|항목|내용|
|---|---|
|출처|•MIT Athena Project 개발<br>•SSO 시스템의 일종|
|핵심|•신뢰하는 제3의 컴퓨터(KDC)가 클라이언트 인증<br>•서버는 클라이언트 사용자가 올바른 사용자인지 확인<br>•서로간 비밀통신 가능|
|암호방식|•대칭키 암호화<br>•종단간 암호화|
|보안효과|•통신 내용 암호화로 기밀성·무결성 보장<br>•Spoofing·Sniffing 차단|

### 📌 구성요소

|구성|설명|
|---|---|
|KDC|•모든 사용자·서비스의 암호화키 보관<br>•당사자와 서비스에 티켓 발행<br>•TGS + AS 두 가지 부분으로 구성|
|AS|•Authentication Service<br>•사용자 신원 최초 인증<br>•TGT 발행|
|TGS|•Ticket Granting Service<br>•서비스 접근용 티켓 발행|



### 📌 인증절차 (6단계)

|단계|방향|내용|
|---|---|---|
|①|사용자 → AS|•TGT(Ticket Granting Ticket) 요청|
|②|AS → 사용자|•암호화된 Session Key와 TGT 응답|
|③|사용자 → TGS|•TGT 이용하여 서비스 Ticket 요청|
|④|TGS → 사용자|•Ticket과 Session Key 응답|
|⑤|사용자 → App Server|•응용서버에 세션키로 암호화한 Ticket 전달|
|⑥|App Server → 사용자|•Ticket 검증 후 인증완료|
![[Pasted image 20260615094741.png]]

| 키워드        | 내용                             |
| ---------- | ------------------------------ |
| **TGT**    | 로그인 1회로 발급, KDC 재접근 증명서(마스터티켓) |
| **서비스 티켓** | 특정 서비스 접근권, TGS가 발급            |
| **SSO 원리** | TGT로 서비스 티켓 자동 발급 → 재로그인 불필요   |
| **암호화 주체** | AS는 사용자 비밀키로, TGS는 서비스 키로 암호화  |
### 📌 장단점

| 구분  | 내용                                                                                                                                                                                                         |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 장점  | •통신 내용 암호화 → 기밀성·무결성 보장<br>•Spoofing·Sniffing 차단                                                                                                                                                           |
| 단점  | •**KDC에 문제 발생 시 전체 서비스 중단 (단일장애점,**Single Point of Failure**, 약어는 **SPOF**)**<br>•사용자 비밀키가 시스템에 임시저장 → 침입자에 의해 유출 가능<br>•세션키도 사용자 시스템에 임시저장 → 침입에 취약<br>•Password Guessing 공격에 취약<br>•패스워드 변경 시 비밀키도 변경 필요 |

### 📌 개방형 분산환경 위협 3가지

| 위협             | 내용                                                                              | Kerberos 대응                              |
| -------------- | ------------------------------------------------------------------------------- | ---------------------------------------- |
| 가장(Masquerade) | •임의 사용자가 특정 워크스테이션에 접근하여<br>•그 워크스테이션상에서 작업하는 다른 사용자인 척 가장                      | •TGT/서비스 티켓으로 신원 증명<br>•비밀키 없이는 티켓 위조 불가 |
| IP 스푸핑         | •임의 사용자가 워크스테이션 네트워크 주소 변경<br>•변경된 워크스테이션에서 발송한 요청이 정상 워크스테이션에서 요청한 것처럼 보이도록 만듬 | •티켓이 암호화되어 있어<br>•IP 변조만으로는 인증 불가        |
| 재전송 공격         | •임의 사용자가 통신교환을 도청하고 재전송 공격<br>•특정 서버에 대한 접근허락을 받아내거나 통신과정 방해                    | •타임스탬프 + 티켓 유효기간으로<br>•만료된 티켓 재사용 차단     |

### 📌 S/MIME

### 📌 S/MIME 제공 서비스

|제공 서비스|쉬운 설명|
|---|---|
|Enveloped Data|내용을 **암호화**한 것. 수신자 공개키로 봉투 씌운 것 (전자봉투)|

|Signed and Enveloped Data|**암호화 + 서명 동시에** 적용|
※ base64: SMTP가 7bit ASCII만 지원하므로 바이너리 데이터를 ASCII 텍스트로 변환하는 부호화 (암호화 아님)

| **장점**     | **•트래픽 제어 및 QoS 기능 제공, 네트워크 레벨에서 VPN 기능 제공<br>•뛰어난 관리 편의성, 낮은 도입/관리 비용 (장비도입 필요 없음)** |
| **단점**     | **•동일 ISP 내부에서만 운영 가능, 고객사 고유정책 반영 미약<br>•공중망 전송 시 암호화 기능 미약함, 대역폭에 비해 고비용 구조**       |
※ **ISP(Internet Service Provider)**  : 인터넷 통신사(KT, SK브로드밴드, LG유플러스)
### 📌 SSL VPN

|구분|내용|

## 📘 9. 무선 네트워크 보안

### 📌 무선 LAN 서비스 흐름

|단계|내용|
|---|---|
|①|•PDA/노트북 → AP(Access Point): 서비스 요청|
|②|•AP → 인증서버: 인증 요청|
|③|•인증서버 → AP: 사용자 인증|
|④|•AP → PDA/노트북: 서비스 제공|
|구간|•무선(PDA↔AP): WEP/WPA/WPA2/WPA3 등<br>•유선(AP↔인증서버): RADIUS 등|

### 📌 무선 LAN 보안기술 진화
| 구분   | 내용                                                                                                                                            |
| ---- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| SSID | Service Set Identifier  <br>무선랜에서 네트워크 이름을 의미하며 기본적인 접속 제어 기능 제공  <br>AP가 비콘 신호에 SSID를 포함하여 브로드캐스트  <br>SSID만으로는 보안 기능이 약하며 접속 허용/차단 기준으로는 취약 |
| WEP  | Wired Equivalent Privacy  <br>무선 데이터 보안을 위한 초기 암호화 방식  <br>대칭키 구조 사용  <br>RC4 (40bit key) 암호 알고리즘 사용  <br>취약점 존재로 현재 사용 비권장                   |
| WPA  | Wi-Fi Protected Access  <br>WEP 보완 기술  <br>RC4 (128bit key) 사용  <br>TKIP 적용하여 동적 키 사용(무결성: Michael 알고리즘  *※MIC*)                              |
| WPA2 | AES-CCMP 암호화 사용  (TKIP도 지원)<br>인증 방식: IEEE 802.1X (기업) / PSK (개인)  <br>                                                                       |
| WPA3 | SAE 인증 방식 적용 (PSK 개선)  <br>GCMP-256(128) 암호화 사용  <br>PMF (Protected Management Frames) 적용                                                     |
※ PSK (Pre-Shared Key): 사전에 공유된 비밀번호(Wi-Fi 비밀번호)로 인증. 비밀번호 알면 누구나 접속 가능하고 오프라인 사전공격에 취약.
※ SAE (Simultaneous Authentication of Equals): WPA3에서 PSK 대체. 비밀번호가 같아도 매번 다른 키 생성(Dragonfly 핸드셰이크). 오프라인 사전공격 차단, PFS 지원. 
**비밀번호로 접속은 동일하나, 비밀번호 탈취 시에도 과거 트래픽 복호화 불가.**

| WPA-EAP | 사람마다 자기 계정으로 인증      |

## 📘 방화벽 (Firewall)

### 📌 개념 및 특징

| 구분    | 내용                                                                                                                                                                                                           |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 개념    | •보호된 네트워크와 다른 네트워크 사이의 엑세스(Access)를 제한하는(통제하는) 구성요소의 집합<br>•네트워크를 통해 흐르는 Packet을 미리 정해놓은 구축에 따라 차단하거나 전달하는 기능을 수행하는 **SW 또는 HW**<br>•허가된 사용자만 내부 네트워크에 접근가능하고 허가되지 않은 사용자는 내부 네트워크에 접근이 불가능 하도록 하기 위한 기술     |
| 주요 특징 | •**IP Address 및 Port 번호 별 서비스(TCP/UDP) 제어 가능 (방화벽의 본질)**<br>•NAT(Network Address Translation) 기반 사설IP와 공인IP 변환 가능 (방화벽 장비에 **통합**된 것)<br>•암호화 기술을 추가해 VPN(Virtual Private Network) 구현 가능 (방화벽 장비에 **통합**된 것) |

### 📌 방화벽 역할

|역할|설명|
|---|---|
|취약한 서비스의 보호|•안전하지 않은 서비스를 필터링 함으로써 보호<br>•선택된 프로토콜만 내부 네트워크로 들어오게 함으로써 내부 네트워크의 서비스를 보호함|
|호스트의 접근제어|•내부 여러 호스트 시스템(메일서버, 웹서버, DB서버, 개인 호스트 등)의 외부에 대한 접근제어<br>•원하지 않은 접근은 차단|
|보안의 집중|•인증 시스템 및 로깅, 접근 제어 등의 보안 기능을 하나의 호스트에 구현<br>•경제적으로 보안 구현 가능|
|확장된 프라이버시|•외부에서 내부 네트워크의 DNS정보 및 IP 주소를 보이지 않게 할 수 있음|
|로깅 및 통계자료|•시스템에 대한 모든 엑세스가 방화벽을 통과 하므로 접근에 대한 로깅을 할 수 있고 네트워크 사용에 대한 통계 자료를 제공 가능|
|쉬운 정책 구현|•네트워크 접근 통제 정책(사용자의 서비스에 대한 접근 제어)의 간편하고도 쉬운 구현이 가능케 함|

### 📌 방화벽 단점

| 단점                                     |
| -------------------------------------- |
| •전체적인 네트워크 성능이 감소(Overhead 증가)         |
| **•Single Point Of Failure 가 될 수 있음**  |
| **•바이러스, 트로이목마 등은 막기 어려움(L7 방화벽은 가능)** |
| •인가된 사용자 및 서비스의 악의적 사용은 막기 어려움         |

### 📌 동작 Layer별 방화벽 종류

| 종류                                                 | 동작 계층             | 핵심 특징                                                                                                                                                 |
| -------------------------------------------------- | ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| 패킷 필터링 (Packet Filtering)                          | •L3(IP)+L4(TCP)   | •**처리속도 가장 빠름<br>•패킷의 헤더정보만으로 차단 유무를 결정하며, 본문 내용(Payload)는 분석하지 않음**                                                                                  |
| Application Gateway (Forward Proxy)                | •L7 (Application) | •**보안성이 뛰어나고** **Logging과 Audit 기능** 제공<br>•Proxy Gateway 라고도 함<br>•매우 높은 보안정책 실현 및 바이러스 검사 등 부가기능 제공<br>•전용Gateway에 따른 어플리케이션의 유연성 부족하며 H/W에 의존적     |
| 상태 검사 패킷 필터<br>(Stateful Inspection packet filter) | •L3~L4            | •상태 검사 패킷 필터. **단순 IP, Port 뿐만 아니라 접속 Session 단위로 검증 수행**<br>•내부 사용자의 요청에 의한 응답(Response)만 인바운드 허용                                                    |
| Circuit Gateway(Forward Proxy)                     | •L5~L7 (5~7계층)    | •전용 Gateway가 아닌 **하나의 일반 Gateway로 모든 서비스 처리가능**<br>•**내부의 IP주소를 숨기는 것이 가능**<br>•Gateway의 사용을 위해 수**정된 클라이언트 모듈**이 필요<br>•**지원 불가능한 프로토콜의 존재 가능성이 있음** |
| Hybrid                                             | •혼합               | •**대부분의 상용 방화벽의 채택 방식임**<br>•Packet Filtering + Application Gateway 방식<br>•내부의 보안정책, 어플리케이션 등에 맞추어 선택적 보안설정 가능                                        |

#### 🔥SOCKS 프록시 = Circuit Gateway

| 구분  | Application Gateway | Circuit Gateway (SOCKS) |
| --- | ------------------- | ----------------------- |
| 방식  | 프로토콜마다 전용 프록시       | 프로토콜 상관없이 하나로 처리        |

### 📌 방화벽 구성 종류 (구성상 분류)

| 구성                                  | 구성              | 핵심 특징                                                                                                                                                                                                                                                                                                                                                                       | 장점                                                             | 단점                                                                              |
| ----------------------------------- | --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| 스크리닝 라우터<br>(Screening Router)      | 라우터 단독          | •패킷 필터링 라우터라고도 함, 주로 패킷을 필터링 하는 역할<br>•3,4 계층에서 수행(네트워크, 전송)되므로 IP, TCP, UDP 헤더만 분석 가능<br>•외부→내부, 내부→외부 패킷에 대해 IP주소나 Port 정보를 기반으로 필터링                                                                                                                                                                                                                                      | •필터링 속도가 빠르고 비용이 적게 듦, 사용자 투명성 보장<br>•유연성 높아 새로운 서비스 적용 쉬움     | •3,4 계층에 입각한 서비스만 필터링 가능함<br>•패킷 내의 데이터(내용)을 기반으로 차단 못함<br>•로그 남기기 힘듦(양이 너무 많음) |
| 베스천 호스트<br>(Bastion Host)           | 호스트 단독          | •방화벽 구성이라기 보다는, 게이트웨이 형식의 방화벽 SW가 설치되는 호스트를 말함<br>•보호된 네트워크에서 유일하게 외부의 공격에 노출된 컴퓨터 시스템으로써 내부 네트워크과 외부 네트워크 사이에 위치하는 게이트웨이임<br>•외부 사용자가 내부 네트워크에 접근 하려면 우선 Bastion Host를 통과 해야 함<br>•유일한 외부 접근점이기 때문에 해킹의 주요 대상임, 따라서 보안관리를 철저히 해야 함(모든 사용자 계정 지움, 파일, 유틸리티, IP 포워딩 파일 삭제)<br>•강력한 인증 기법을 구현하여야 하며, Bastion 호스트는 내부 네트워크로의 접근에 대한 기록(log), 감사 추적을 위한 기록 및 모니터링 기능을 가지고 있어야 함 | •데이터에 대한 확실한 방어, 로그 정보의 생성 및 관리 용이                             | •베스천 호스트가 손상되면 내부 방어가 불가능                                                       |
| 듀얼 홈드 게이트웨이<br>(Dual-Homed Gateway) | 배스천호스트(NIC 2개)  | •Bastion Host가 두 개의 네트워크 카드를 가지는 경우, 하나는 외부, 하나는 내부 연결<br>•양 네트워크간의 라우팅은 존재하지 않는다. 따라서 양 네트워크간의 직접적인 접근은 허용되지 않으므로 항상 인증을 받아야 내부 네트워크 접근 가능(프록시 개념임)                                                                                                                                                                                                                        | •설치 및 유지보수가 비교적 쉬움<br>•세밀한 분석가능 하므로 보안성이 뛰어나며 사용자 인증 등 부가기능    | •장애의 단일 지점 (SPOF)                                                               |
| Screened Host Gateway               | 라우터 + 베스천호스트    | •스크리닝 라우터(Screening Router)와 배스천 호스트(Bastion Host)로 구성한 방화벽<br>•스크리닝 라우터를 지나는 패킷은 무조건 배스천호스트로 향하며, 여기를 통과 해야 비로소 내부 네트워크로 갈 수 있음<br>•1차 방어는 스크리닝 라우터, 2차 방어는 배스천 호스트임                                                                                                                                                                                                       | •2단계로 방어 하기 때문에 안전(네트워크와 응용 계층 모두에서 방어)<br>•가장 많이 사용됨, 융통성이 좋음 | •방화벽 구축 비용이 증가                                                                  |
| Screened Subnet Gateway             | 라우터 + 베스천 + 라우터 | •스크리닝 라우터-Dual-Homed 게이트웨이-스크리닝 라우터 로 3단계 방어<br>•스크리닝 라우터 사이의 Subnet(DMZ)를 구성 할 수 있음                                                                                                                                                                                                                                                                                        | •3단계로 방어 하기 때문에 매우 안전                                          | •설치 하기 어렵고 관리 비용이 많이 듦, 구축 비용 증가                                                |

```
【스크리닝 라우터】
외부 ──[라우터]──────────────── 내부
        IP/Port만 필터링

【베스천 호스트】
외부 ──[베스천]──────────────── 내부
        서버 한대, 방화벽SW 설치

【듀얼 홈드】
외부 ──[NIC1 | 베스천 | NIC2]── 내부
        라우팅 OFF, 프록시만 통과

【스크린드 호스트 G/W】
외부 ──[라우터]──[베스천]─────── 내부
        2중 방어

【스크린드 서브넷 G/W】
외부 ──[라우터]──[베스천]──[라우터]── 내부
         ↑     └─DMZ─┘     ↑
       외부망              내부망
        가장 안전, 고비용
```

#### 🔥**시험 포인트**

- 보안 강도 순서: 스크리닝 라우터 < 듀얼홈드 < 스크린드호스트 < 스크린드서브넷
- DMZ = 스크린드 서브넷 G/W
- 배스천호스트 = 공격자 최우선 목표 (내/외부 접점이라)
- 스크리닝 라우터 단점 = **패킷 내용(페이로드) 검사 불가**

**일반 라우터**: 목적지 IP 보고 경로만 결정해서 전달
**스크리닝 라우터**: 거기에 더해서 **ACL(Access Control List)** 로 패킷 필터링 기능 추가


#### 🔥**ACL (Access Control List) 접근 통제**

|구분|내용|
|---|---|
|정의|네트워크 장비(라우터, 스위치)에서 패킷 필터링을 통해 트래픽을 허용/차단하는 접근 제어 목록|
|동작 위치|라우터 인터페이스 (인바운드/아웃바운드)|
|적용 방향|Inbound (들어오는 트래픽), Outbound (나가는 트래픽)|

|ACL 종류|배치 위치|
|---|---|
|표준 ACL|목적지에 **가깝게**|
|확장 ACL|출발지에 **가깝게**|

### 📌 iptables (리눅스 방화벽)

|구분|내용|
|---|---|
|개념|•리눅스 커널에 내장된 트래픽 제어 관리 도구로 Rule 기반 패킷 필터링 제공<br>•TCP 상태 기반의 추적 기능 제공(Stateful Inspection)<br>•NAT 기능을 제공하여 FORWARD 트래픽도 제어 가능|
|명령 구조|•`iptables [체인옵션] [패킷매칭옵션] [패킷처리옵션]`|
|주요 옵션|•--source (-s): 출발지 IP 주소 혹은 네트워크 이름<br>•--destination (-d): 목적지 IP 주소 혹은 네트워크 이름<br>•--sport: 출발지 port 번호 (ex --sport 0:65535)<br>•--dport: 목적지 port 번호<br>•--protocol (-p): 패킷의 프로토콜 ex> tcp, icmp<br>•--in-interface (-i): 입력 N/W 인터페이스 ex> eth0<br>•--out-interface (-o): 출력 N/W 인터페이스<br>•--state: 연결상태 ex> ESTABLISHED, RELATED<br>•-j: 매칭되는 패킷을 지정한 대상으로 이동(Accept, Drop, return 등 패킷 처리 옵션용)<br>•--tcp-flags: TCP 플래그 설정(ACK, FIN, PSH, RST, SYN, URG, ALL, NONE)<br>•-m: 확장 모듈 로드. Ex> -m state|
|예시|•`iptables -A INPUT -p tcp --dport 22 -j ACCEPT`|

## 📘 IDS (Intrusion Detection System)

### 📌 개념 및 주요 기능

|구분|내용|
|---|---|
|개념|•조직 IT시스템의 기밀성, 무결성, 가용성을 침해하고, 보안정책을 위반하는 침입사건을 사전 또는 사후에 감시, 탐지, 대응하는 보안 시스템<br>•컴퓨터 시스템의 비정상적인 사용, 오용, 남용 등을 실시간으로 탐지하는 시스템|
|보안 서비스|•기밀성·무결성: 비인가자 침입 탐지, 백도어 탐지, Sniffing, 바이러스, 내부자 불법 행위<br>•가용성: DOS공격, 시스템변경, 인터넷 웜 탐지|
|주요 기능|•저수준 필터링, 고수준 필터링을 이용한 감사자료 수집 가능<br>•오용탐지 기능, 비정상 침입탐지 기능<br>•서비스 거부 탐지기능, 자동침입 분류기능, 침입탐지 경보 및 대응, 시스템 취약성 점검 및 복구기능, 침입자 역추적<br>•새로운 침입 패턴 분석|

### 📌 IDS 분류 (1) - Information Source에 의한 분류

|구분|NIDS (Network based IDS)|HIDS (Host based IDS)|
|---|---|---|
|동작|•네트워크에 흐르는 패킷들을 검사 하여 침입을 판단함|•시스템상에 설치 되어 사용자가 시스템에서 행하는 행위나 파일의 체크를 통해 침입을 판단함|
|자료원|•promiscuous 모드로 동작하는 네트워크 카드나 스위치|•시스템 로그, 시스템 콜, 이벤트 로그|
|탐지 가능 공격|•스캐닝, 서비스 거부 공격(DOS), 해킹|•부적절한 로그인 시도, 비 인가자의 시스템 사용, 바이러스, 웜|
|장점|•네트워크 자원의 손실 및 패킷의 변조가 없음(캡처만 하기 때문)<br>•거의 실시간으로 탐지가 가능함<br>•감시 영역이 하나의 네트워크 서브넷으로써, HIDS에 비해 큼|•침입의 성공여부 식별이 가능함<br>•실제 해킹 및 해킹 시도 판단이 용이<br>•주로 SW적으로 서버같은 시스템에 인스톨 되며, 설치 및 관리가 간단함|
|단점|•부가 장비가 필요함(스위치 등)<br>•Gbit 네트워크 환경에서는 정상적으로 작동하기 힘들 수 있음<br>•암호화된 패킷은 분석 불가<br>•False Positive 가 높음(양치기 소년)|•감시 영역이 하나의 시스템으로 한정됨<br>•탐지 가능한 공격에 한계가 있음(주로 이벤트 로그로만 탐지)|

### 📌 IDS 분류 (2) - 자료원 분석 방법에 의한 분류

|구분|오용탐지 (Misuse)|비정상탐지 (Anomaly)|
|---|---|---|
|동작방식|•시그니처(signature)기반 = Knowledge 기반|•프로파일(Profile)기반 = Behavior 기반 = Statistical 기반|
|침입 판단 방법|•미리정의된 Rule에 매칭<br>•이미 정립된 공격 패턴을 미리 입력하고, 이것에 매칭됨|•미리학습된 사용자패턴에 어긋남<br>•정상적, 평균적 상태를 기준으로 하여 이에 급격한 변화가 있을때 침입 판단|
|사용기술|•패턴비교, 전문가시스템|•신경망, 통계적방법, 특징추출|
|장점|•빠른 속도, 구현이 쉬움<br>•사용자의 이해가 쉬움<br>•False Positive 가 낮음|•알려지지않은 공격(Zero Day Attack)대응 가능<br>•사용자가 미리 공격 패턴을 정의할 필요 없음|
|단점|•False Negative 큼<br>•알려지지 않은 공격 탐지불가<br>•대량의 자료를 분석 하는 데는 부적합|•어떤 행위가 정상인지, 비정상인지 결정할 수 있는 임계치를 정하기 힘듦<br>•False Positive 가 큼<br>•구현이 어려움|

> **False Positive**: 공격 아닌데도 공격이라 오판 하는 것 — **2024년 기출** **False Negative**: 공격인데도 공격이 아니라 오판 하는 것

### 📌 IDS 분류 (3) - 대응방법에 따른 분류

|구분|설명|
|---|---|
|수동적 대응 (Passive Response)|•IDS는 관리자에게 침입 정보만 제공<br>•실제 대응행동은 제공된 정보를 기초로 관리자가 수행<br>•종류: 알림 및 경보 등 (예: E-Mail, SMS)|
|능동적 대응 (Active Response)|•실제 대응행동을 IDS가 자동적으로 수행<br>•종류: 환경변경, 침입자에 대한 역공격 등 (예: Firewall과 연동, 세션차단)|

### 📌 방화벽 vs IDS 비교

|구분|방화벽 (FIREWALL)|침입탐지 (IDS)|
|---|---|---|
|목 적|•접근통제 및 인가, 방어|•실시간 탐지, 분석, 대응|
|위치|•외부와 내부 경계에서 외부 출입자 방어(수위실)|•내부에서 불법사용자 감시(CCTV)|
|운영 정책|•명백하게 허용하는 것만 허용|•명백하게 금지하는 것만 금지|
|장 점|•엄격한 Access Control<br>•인가된 트래픽만 허용<br>•보안정책 적용가능|•실시간 침입탐지<br>•통계적 분석 및 리포팅<br>•사후분석 대응기술|
|단 점|•Dial-Up등은 방화벽 통과 많음<br>•다양한 접근통제는 트래픽 부하 유발<br>•바이러스 침투나 암호화된 해킹 프로그램 차단불가|•변형된 침입탐지 패턴 취약<br>•트래픽 양이 떨어지는 경우 탐지율 낮음<br>•오판율이 높게 나타남|

|구분|내용|
|---|---|
|개념|•패킷 수집 라이브러리인 libpcap에 기반한 실시간 트래픽 분석과 IP 네트워크 상에서 패킷 로깅이 가능한 네트워크 침입탐지시스템(N-IDS)|
|룰 구조|•`action protocol 출발지IP 출발지PORT 패킷흐름 목적지IP 목적지PORT (매칭옵션)`|
|Action 종류|•Alert(경고+로그), log(로그), pass(무시), dynamic(로그), activate(경고+dynamic규칙 활성화)|
|프로토콜|•tcp, udp, ip, icmp|
|패킷 흐름|•`->` 단방향, `<>` 양방향|
|주요 옵션|•msg: 경고에서 출력할 메시지<br>•content: 패킷 내용 비교<br>•threshold: 발생 횟수 기반 탐지 - threshold:type threshold(패킷발생량), limit(임계시간), both(최초 패킷만 기록) / track by_src(출발지 기준), track by_dst(목적지 기준)<br>•flags: TCP 플래그(S=SYN, A=ACK, F=FIN 등)<br>•sid: 규칙 고유 ID|
|룰 예시|•`Alert tcp $EXTERNAL_NET any -> $HOME_NET 21 (msg:"tcp syn flooding"; content:"GET / HTTP1.1"; flags:S; threshold:type threshold, track by_dst, count 10000, seconds 2; sid:1000000;)`|

## 📘 IPS (Intrusion Prevention System)

### 📌 개념 및 필요성

|구분|내용|
|---|---|
|개념|•수동적인 침입 탐지에서 나아가, 보다 능동적으로 침입 행위를 차단 할 수 있는 시스템<br>•침입탐지시스템(IDS)의 오판(False Positive)과 미탐(Miss Detection) 문제 해결 위해 등장|
|(1) 공격 방식의 변화|•System Attack(Sniffing, IP Spoofing)<br>•→ Service Attack(DOS, Worm)<br>•→ Infrastructure Attack(알려지지 않은 공격)|
|(2) 패러다임의 변화요구|•시스템이나 장비의 기밀성, 무결성 요구<br>•→ 전체 기반구조의 가용성|
|(3) 기존 보안 솔루션의 한계|•IDS: 탐지 이후, 방화벽 연동에 의한 차단 외에는 적절한 차단 대책 없음<br>•방화벽: 인가된 사용자/시스템의 악의적인 행위 차단이 어려움<br>•바이러스/웜 차단: 취약성을 통해 전파(Technical Attack)되는 공격에 대응 불능|
|(4) 통합화 요구|•보안, 시스템, 장비 시장의 성숙<br>•보안 기술 통합을 통한 시너지 효과<br>•해킹 기법의 복잡화, 다양화<br>•관리 편의성 요구|

### 📌 IPS 특징 및 기능

|구분|내용|
|---|---|
|특징|•해킹(Hacking)의 실시간 자동 방어<br>•알려지지 않은 신규 바이러스, 해킹형태 차단<br>•OS 레벨 실시간 탐지 및 방어<br>•암호화 된 공격방어 및 응용 통한 불법 방지|
|기능|•정상 패턴 중 허용된 트래픽만 통과시켜 침입 방어<br>•웜, 비정상적 프로토콜, DDoS의 사전 탐지/차단<br>•침입의 방지와 차단을 동시에 실행|

### 📌 IDS vs IPS 차이점

|구분|IDS|IPS|
|---|---|---|
|성격|•Positive & Reactive 탐지와 패턴 등록 후 반응|•Active & Pro-Active<br>•공격전 사전 차단|
|목적|•침입 여부 탐지|•침입 방지 목적, 침입 탐지 후 적극적 대응|
|분석 방법|•시그니처 DB기반 패턴 매칭 방법<br>•알려진 공격 패턴 탐지|•시스템 Call에 대해 정책, Rule DB기반 비정상 행위 방지|
|대응 방법|•관리자에게 경고<br>•ESM 통해 Firewall Rule Set 변경|•알려지지 않은 공격탐지<br>•자원 접근 차단|

### 📌 IPS 유형

|유형|특징|
|---|---|
|호스트 기반 IPS|•커널 이벤트를 가로채 처리(Trusted OS)하는 방식<br>•커널과 독립적(Signature + 행동기반분석 알고리즘)<br>•특정규칙에 위배되는 Event Filtering<br>•SW제품, 취약한 Application Program 보호|
|네트워크 기반 IPS|•실시간 패킷 처리<br>•오탐지, 오용공격, 변형공격 탐지<br>•세션기반 탐지 지원 시스템<br>•Network Line상에 위치|

## 📘 13. ESM (Enterprise Security Management)

### 📌 개념 및 배경

|구분|내용|
|---|---|
|개념|•방화벽·IDS·VPN 등 각종 보안 제품을 통합 관리하고 개별 침입에 종합 대응하기 위해 요소제품 간 인터페이스·교환 메시지 포맷을 표준화한 지능형 통합 보안 관제 시스템|
|목적|•기업 보안 정책 반영하여 다종·다수 보안 시스템을 통합관제/운영/관리 → 기업보안 목적 효율적 실현|
|진화|•ESM → SIEM(데이터 분석·대응 기능 강화, 맞춤형 대시보드·Alert 관리, 예: ArcSight, Splunk)|

### 📌 배경 및 필요성

|구분|설명|
|---|---|
|보안 제품의 전문화|•Network 보안: 방화벽, VPN, IDS, Scanner 등<br>•System 보안: 바이러스 백신, HIDS, 이메일 보안 솔루션 등|
|관리의 어려움|•다양한 보안 제품·다양한 회사<br>•보안 정책 없거나 일관된 보안 정책 적용 어려움<br>•보안 제품 전문 인력 부족|
|기존 보안 제품의 한계성|•방화벽 한계: Open Port 대응 어려움, 내부자 해킹, 능동적 감시 어려움<br>•IDS 한계: 실제적 차단 어려움<br>•각각의 시스템보다 여러 시스템이 서로 보완하며 동작 필요|
|운영의 중요성|•보안은 도입보다 관리와 효과적인 운영이 더 중요|

### 📌 주요 기능

|측면|세부 기능|
|---|---|
|정보 수집|•통합 로그 보기, 개별 시스템 로그 보기, 실시간 모니터링, 통합 및 개별 모니터링<br>•이벤트 필터링, 시스템 모니터링|
|정보 가공|•보안 시스템 위험도 판별, 보안 현황 검색 및 출력, 리포트 및 통계 기능|
|자동 대응|•방화벽·IDS 연동을 통한 차단, 경보 및 추적 기능, 자동 대응 기능|
|정책 관리|•단일 뷰를 통한 통합 정책 관리<br>•사용자 정의에 따른 오브젝트별 정책 설정|
|운영 관리|•사용자 등급별 관리, 네트워크 장애 처리(NMS 기능), 에이전트 등록 및 관리<br>•구성관리, 성능관리, 장애관리, 이력관리, 자산관리 등|

## 📘 17. Block Chain

| 허위 정보 전파 공격          | •공격자가 다수의 노드를 장악하여 악의적인 노드를 추가하여 특정 노드를 블록체인 네트워크로부터 고립시키는 공격                                                                                      |
| 재진입 공격<br>(스마트컨트렉트)  | •스마트 계약을 구성하는 코드의 특정 거래가 합의 및 검증되는 시간 간격을 이용한 공격<br>•특정 거래의 잔고 차감이 수행되기 전에 새로운 요청에 대한 잔고 확인이 실행되고, 이전 거래에 의한 잔고 차감 전에 현재의 잔고를 기준으로 다시 거래가 수행되도록 유도 |
| 시빌 공격 (Sybil attack) | •공격자가 다수의 가짜 신원이나 가짜 노드를 만들어 네트워크에서 마치 여러 독립 참여자인 것처럼 행동하는 공격                                                                                      |
**혼동 포인트 정리:**

|구분|시빌 공격|이클립스 공격|
|---|---|---|
|수단|가짜 **신원/노드** 대량 생성|실제 노드로 피해 노드 **포위**|
|목적|네트워크 **전체** 신뢰 붕괴|**특정 노드 1개** 고립|
|피해 범위|네트워크 전체|타겟 노드만|
|키워드|가짜, 다수, 압도|고립, 피어 연결 편향, 포위|
