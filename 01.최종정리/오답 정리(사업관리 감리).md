정답 **② 결함 식별 (Defect Detection)**

PMBOK에서 변경 요청(Change Request)의 유형은:

|선지|변경 요청 유형 해당 여부|
|---|---|
|① 시정 조치 (Corrective Action)|✅ 해당|
|② **결함 식별 (Defect Detection)**|❌ 해당 없음 → **정답**|
|③ 예방 조치 (Preventive Action)|✅ 해당|
|④ 결함 수정 (Defect Repair)|✅ 해당|

**변경 요청 (Change Request) 유형 4가지**

| 유형                        | 내용                    |
| ------------------------- | --------------------- |
| **시정 조치 (Corrective Action)** | 현재 편차를 바로잡기 위한 조치     |
| **예방 조치 (Preventive Action)** | 미래 문제 발생을 막기 위한 선제 조치 |
| **결함 수정 (Defect Repair)**     | 발견된 결함을 고치는 조치        |
| **업데이트 (Update)**             | 공식 문서/계획 갱신           |

**결함 관련 개념 구분**

|개념|속하는 곳|
|---|---|
|결함 식별 (Defect Detection)|품질 통제 (Control Quality) 활동|
|결함 수정 (Defect Repair)|변경 요청 유형|

**변경 요청이 발생하는 프로세스**

**통합 변경 통제 수행 흐름**

|단계|내용|
|---|---|

**SV = EV - PV**

**50/50 법칙:** 시작하면 50%, 완료하면 나머지 50% 인정

**PV 계산 (4개월 말 기준 계획상 상태)**

|활동|계획 상태|PV|
|---|---|---|
|A|완료 예정 → 100%|200만원|
|B|시작됐지만 미완료 → 50%|100만원|
|C|시작됐지만 미완료 → 50%|100만원|
|D|미시작|0|
|E|미시작|0|
|**합계**||**400만원**|

**EV 계산 (실제 진척 기준)**

| 활동     | 실제 상태             | EV        |
| ------ | ----------------- | --------- |
| A      | 완료 → 100%         | 200만원     |
| B      | 3/4 진척, 미완료 → 50% | 100만원     |
| C      | 2/4 진척, 미완료 → 50% | 100만원     |
| D      | 미시작               | 0         |
| E      | 미시작               | 0         |
| **합계** |                   | **400만원** |

**SV = 400 - 400 = 0원 → ①**

**TCPI = (BAC - EV) / (BAC - AC)**

= (5 - 2) / (5 - 3) = 3/2 = **1.5 → ③**

**동기부여 이론 전체 정리**

**내용이론 (What - 무엇이 동기를 주는가)**

| 이론      | 학자                   | 핵심 내용                                                                       |
| ------- | -------------------- | --------------------------------------------------------------------------- |
| 욕구단계설   | 매슬로우 (Maslow)        | 생리→안전→사회→존경→자아실현 5단계. 하위 욕구 충족 후 상위로 이동. 단계 건너뜀 불가                          |
| ERG이론   | 앨더퍼 (Alderfer)       | 존재(Existence)·관계(Relatedness)·성장(Growth) 3단계. 매슬로우와 달리 **좌절-퇴행 가능**         |
| 동기-위생이론 | 허즈버그 (Herzberg)      | 동기요인(성취·인정·성장)은 만족 유발. 위생요인(급여·환경·정책)은 불만족 예방만. **만족↔불만족이 별개의 축**           |
| X·Y이론   | 맥그리거 (McGregor)      | X이론: 인간은 게으르고 강제해야 함. Y이론: 인간은 자율적이고 책임감 있음. **맥그리거는 Y이론 지지**               |
| 성취동기이론  | 맥클리랜드 (McClelland)   | 성취(nAch)·권력(nPow)·친교(nAff) 3욕구. 욕구는 **학습·경험으로 형성**. 탁월한 관리자 = nPow↑ nAff↓   |
| 자기결정이론  | 다니엘 핑크 (Daniel Pink) | 자율성(Autonomy)·숙련(Mastery)·목적(Purpose) 3요소. 외재적 보상보다 **내재적 동기**가 창의적 업무에 효과적 |

**허즈버그 핵심 포인트**

| 요인                | 충족 시                | 미충족 시                 |
| ----------------- | ------------------- | --------------------- |
| 동기요인 (성취, 인정, 성장) | 만족                  | **불만족 아님 → 그냥 만족 없음** |
| 위생요인 (급여, 환경, 정책) | **만족 아님 → 불만족만 없앰** | 불만족                   |

**과정이론 (How - 어떻게 동기가 생기는가)**

|이론|학자|핵심 내용|
|---|---|---|
|기대이론|브룸 (Vroom)|동기 = 기대 × 수단성 × 유인가. **셋 중 하나라도 0이면 동기 0**|
|공정성이론|아담스 (Adams)|자신의 투입/산출 비율을 타인과 비교. 불공정 느끼면 **행동 변화**|
|강화이론|스키너 (Skinner)|긍정적 결과 → 행동 반복. 부정적 결과 → 행동 감소. **과거 결과가 미래 행동 결정**|

**구분 핵심 요약**

| 구분       | 핵심 질문            | 해당 이론                                |
| -------- | ---------------- | ------------------------------------ |
| **내용이론** | **"뭐가" 동기를 주는가** | **매슬로우, 앨더퍼, 허즈버그, 맥그리거, 맥클리랜드, 핑크** |
| **과정이론**     | **"어떻게" 동기가 만들어지는가** | **브룸, 아담스, 스키너**                         |

**구분 핵심 요약**

내용이론 = **"뭐가"** 동기를 주는지 (요인 자체)

과정이론 = **"어떻게"** 동기가 만들어지는지 (메커니즘)

PMBOK에서 리스크 유형은:

| 유형          | 내용                               |
| ----------- | -------------------------------- |
| 개별 리스크      | 특정 활동에 영향을 주는 리스크                |
| 전체 프로젝트 리스크 | 프로젝트 전반에 영향                      |
| **가변성 리스크** | 불확실한 변동 가능성. 계획 대비 실제 값 변동 불확실   |
| **모호성 리스크** | 미래에 대한 불확실한 지식. 지식 부족, 뭔지 모르는 상태 |
| **출현 리스크**  | 현재 식별 불가, 새롭게 나타난 리스크            |
| **사건 리스크**  | 발생/미발생이 불확실한 특정 사건               |
PMI 리스크 관리 표준서 기준으로 리스크는 이렇게 분류됨:

|구분|유형|설명|
|---|---|---|
|**사건 리스크** (Event Risk)|-|발생 여부가 불확실한 특정 사건 (ex. 핵심인력 이탈)|
|**비사건 리스크** (Non-event Risk)|가변성(변동성) 리스크|알고 있지만 정도가 불확실|
||모호성(Ambiguity) 리스크|무엇이 불확실한지조차 모름|

RAG 개념으로 풀면:

RAG = 검색 증강 생성. LLM이 **외부 문서를 검색**해서 답변 생성. 추가학습(파인튜닝) 안 함.

|선지|RAG 체크리스트 해당?|
|---|---|
|① 벡터DB 구성 위한 데이터 확보|✅ RAG 핵심|
|**② 추가학습을 위한 컴퓨팅 자원 확보**|❌ 파인튜닝 모델 체크리스트|
|③ 검색결과 신뢰성 및 성능 모니터링|✅ RAG 핵심|
|④ 데이터 유출 방지 및 보안 대책|✅ RAG 해당|

정답 **②** 추가학습은 RAG가 아니라 파인튜닝 모델에서 필요한 거야.

**정답: ④ 인지적 평가 이론(Cognitive Evaluation Theory)**

|선택지|판정|근거|
|---|---|---|
|① 공정성 이론(Equity Theory)|❌|자신의 투입/산출 비율을 타인과 비교해 불공정하다고 느낄 때 동기 변화 → 외적 보상과 내적 동기 상충 개념 없음|

|③ 목표 설정 이론(Goal-Setting Theory)|❌|구체적이고 도전적인 목표 설정이 동기를 높인다는 이론(Locke) → 보상 도입 효과와 무관|
|④ 인지적 평가 이론(Cognitive Evaluation Theory)|✅|**내적 동기로 움직이던 사람에게 외적 보상을 도입하면 내적 동기가 감소(구축 효과)**하는 현상을 설명|

**핵심 개념 — 인지적 평가 이론**

Deci & Ryan(1971)이 제시한 이론으로, 핵심 키워드는 **"구축 효과(Crowding-out Effect)"** 또는 **"과정당화 효과(Overjustification Effect)"**

```
내적 동기(성취감·책임감·흥미)로 일하던 사람
          ↓ 외적 보상(돈·칭찬·인센티브) 도입
"이제 보상 때문에 하는 일이구나" 인식 전환
          ↓
내적 동기 감소 → 보상이 없으면 안 하게 됨
```

**동기 이론 비교 요약**

|이론|제안자|핵심 키워드|
|---|---|---|
|인지적 평가 이론|Deci & Ryan|외적 보상 → 내적 동기 감소|
|공정성 이론|Adams|투입/산출 비율 비교, 불공정 해소|
|목표 설정 이론|Locke|구체적·도전적 목표 → 동기↑|
|욕구 이론|Maslow / McClelland|욕구 단계·성취·권력·친화|
|기대 이론|Vroom|기대×수단×유의성|

**15. 다음에서 설명하는 리더십이론으로 가장 적절한 것은 무엇인가?**

> 이 이론은 역량(Competence)과 헌신(Commitment)을 두 가지 주요 변수로 사용하여 프로젝트 팀원의 성장을 측정한다. 역량은 능력, 지식 및 기술을 조합한 것이다. 그리고, 헌신은 개인이 가진 자신감과 동기에 관한 것이다. 개인의 역량과 헌신이 높아지면 리더십 스타일은 개인의 요구사항을 충족하기 위해 지시(Directing)에서 코칭(Coaching)으로 지원(Supporting)에서 위임(Delegating)으로 발전한다.

① Alistair Cockburn의 Transactional leadership 모델 
② Browaeys와 Price의 Servant Leadership 모델 
③ Karen Whittleworth와 Andrew Gilbert의 OSCAR 모델 
**==④ Ken Blanchard의 Situational Leadership 모델==**

**정답: ④**

| 선택지                                      | 판정  | 이유                                                                    |
| ---------------------------------------- | --- | --------------------------------------------------------------------- |
| ① Transactional leadership               | ❌   | 거래적 리더십 = 보상·처벌로 동기부여. 역량/헌신 변수 없음                                    |
| ② Servant Leadership                     | ❌   | 섬기는 리더십 = 팀원 우선 지원. 4단계 스타일 변화 없음                                     |
| ③ OSCAR 모델                               | ❌   | 코칭 대화 프레임워크 (Outcome·Situation·Choices·Actions·Review). 리더십 스타일 분류 아님 |
| ④ Situational Leadership (시츄에이션 상황적 리더쉽) | ✅   | 역량+헌신으로 팀원 발달단계 측정 → 지시→코칭→지원→위임 4단계 스타일이 핵심                          |

**켄 블랜차드(Ken Blanchard) 상황적 리더십 핵심 정리**

|발달단계|역량|헌신|리더십 스타일|
|---|---|---|---|
|D1|낮음|높음|지시 (Directing)|
|D2|낮음~중간|낮음|코칭 (Coaching)|
|D3|중간~높음|가변적|지원 (Supporting)|
|D4|높음|높음|위임 (Delegating)|

3번 찍었는데 OSCAR는 코칭 대화 모델이라 리더십 스타일 분류랑은 다른 개념이야. 다음 ㄱㄱ?

**22. 프로젝트의 일정관리 기법에 관한 설명으로 가장 부적합한 것은 무엇인가?**

① 주공정 경로(critical path)에 포함된 활동에 제한된 자원(resource)을 추가적으로 우선 배정한다. 
② 자원 제약을 특별히 고려하여 일정계획을 수립하는 다른 방법으로 주공정연쇄법(critical chain method)가 있다. 
③ 활동의 연관관계를 조정하여 둘 이상의 활동을 병행함으로써 일정을 단축하는 방법을 공정중첩 단축법(fast tracking)이라 한다. 
④ 프로젝트 예산에 여유가 있는 경우 공정압축법(crashing)보다는 공정중첩 단축법을 먼저 시도하는 것이 바람직하다.

**정답: ④**

|선택지|내용|판정|
|---|---|---|
|①|critical path 활동에 자원 우선 배정|✅ 옳음|
|②|자원 제약 고려 → critical chain method|✅ 옳음|
|③|활동 병행으로 일정 단축 → fast tracking|✅ 옳음|
|④|예산 여유 시 crashing보다 fast tracking 먼저|❌ **반대** → 정답|

**해설**

|기법|방식|특징|
|---|---|---|
|Fast Tracking (공정중첩)|순차 활동을 병행|**비용 추가 없음**, 재작업 위험 있음|
|Crashing (공정압축)|자원 추가 투입|**비용 증가**, 일정 단축 확실|

**23. 다음의 설명으로 가장 적절한 것은 무엇인가?**

> 이 측정은 주어진 시간에 작업 중인 작업 항목 수를 나타낸다. 프로젝트팀이 진행 중인 항목 수를 관리 가능한 크기로 제한하는 데 사용된다.

① 작업 진척 (work in process) 
② 리드 타임 (lead time) 
③ 주기 시간 (cycle time) 
④ 대기열 크기 (queue size)

**정답: ①**

| 선택지           | 판정  | 이유                            |
| ------------- | --- | ----------------------------- |
| ① 작업 진척 (WIP) | ✅   | "작업 중인 항목 수 제한" = WIP 제한 그 자체 |
| ② 리드 타임       | ❌   | 요청~완료까지 전체 소요 시간              |
| ③ 주기 시간       | ❌   | 작업 시작~완료까지 걸린 시간              |
| ④ 대기열 크기      | ❌   | 대기 중인 항목 수, 작업 중이 아님          |

**WIP 제한**은 칸반(Kanban)의 핵심 개념. 동시에 너무 많은 작업 하면 집중도 떨어지고 완료 속도 느려지니까 진행 중 항목 수를 제한함.

| --------------------- | --- | -------------------------------------- |

```
SOW(범위기술서) → WBS 작성 → PERT/CPM(일정 네트워크) 도출
```

②는 순서가 완전히 거꾸로 된 함정. PERT/CPM은 WBS 작성 **후에** 만드는 거야.

**11. AI기업에서 운영중인 클라우드 시스템의 장애 위험 발생 확률은 25%, 장애 발생 시 예상되는 손실금액은 4,000만원이다. 시스템 이중화 구성 도입에 200만원이 들며, 도입 후 위험 발생확률은 10%로 감소하고 손실금액은 동일할 경우 이 대응의 위험감소 레버리지(RRL, Risk Reduction Leverage)는 얼마인가?**

① 1  
② 2.5  
③ 3  
④ 3.5

정답: ③

선택지 판정표

|선택지|판정|이유|
|---|---|---|
|① 1|❌|위험노출 감소액과 대응비용이 같을 때 값인데 여기선 아님|
|② 2.5|❌|계산식 적용 오류|
|③ 3|✅ 정답|위험감소 레버리지 = 위험노출 감소액 ÷ 대응비용 = 600 ÷ 200 = 3|

|항목|내용|

|위험노출 감소액|1,000만원 - 400만원 = 600만원|
|대응비용|200만원|

|구분|계산|결과|
|---|---|--:|
|대응 전 위험노출액|0.25 × 4,000|1,000만원|
