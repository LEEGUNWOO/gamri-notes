### **파생 속성 (Derived Attribute)**

다른 속성들로부터 **계산되어 나오는 값**. 직접 저장하지 않아도 구할 수 있는 속성. UML에서 속성명 앞에 `/` 붙여서 표시.

![[Pasted image 20260408193100.png]]

예시:

- width=5, height=3이면 area는 굳이 저장 안 해도 5×3=15로 계산 가능
- 생년월일이 있으면 나이는 파생 속성
- 단가×수량 있으면 합계금액은 파생 속성**파생 속성 핵심 정리:**

| 구분        | 표기                  | 예시         |
| --------- | ------------------- | ---------- |
| 일반 속성     | `- width: int`      | 직접 저장      |
| 파생 속성     | `- /area: int`      | 다른 속성으로 계산 |
| static 속성 | `- count: int` (밑줄) | 클래스 공유     |

![[Pasted image 20260406121613.png]]

① 클래스 C3에 op1() 연산이 반드시 정의되어야 한다. 
② 클래스 C2에서 클래스 C3로의 관계는 의존관계가 아니라 연관관계가 되어야 한다. 
③ 클래스 C1에서 클래스 C2로의 연관관계는 C2 클래스 쪽에 '*' 등의 다중성이 명시되어야 한다. ④ 클래스 C1의 op12() 연산은 '-'가 아니라 '+'로 표시되어야 한다.

**정답: ③**

**풀이**

시퀀스 다이어그램 분석:

|시퀀스 내용|의미|
|---|---|
|loop [for each c2 in c2s]|C1이 **c2s라는 컬렉션**을 순회함|
|C1 → C2: op21()|C1이 C2 여러 개에 메시지 전송|
|C2 → C3: op1()|C2가 C3에 메시지 전송|
|C1 ← C2: op12()|C2가 C1에 반환|

**각 선택지 판정**

| 번호  | 판정    | 이유                                                                                                        |
| --- | ----- | --------------------------------------------------------------------------------------------------------- |
| ①   | ✗     | 시퀀스에서 C3가 받는 건 op1()인데, 클래스다이어그램에서 C3는 op20() 보유. 근데 op1()은 C4에 있고 C3→C4 상속관계. C3가 op1() 상속받으므로 별도 정의 불필요 |
| ②   | ✗     | C2→C3 점선 의존관계는 시퀀스에서 C2가 C3에 메시지 보내는 것과 일치. 의존관계 맞음                                                       |
| ③   | ==✓== | loop [for each c2 in **c2s**] → C1이 C2 컬렉션을 가짐. 클래스 다이어그램 C1→C2 연관관계에 C2 쪽 다중성(*****) 표시가 없음              |
| ④   | ✗     | op12()는 시퀀스에서 C2→C1 반환 메시지. C1 내부 연산이므로 가시성 '-'(private)도 가능                                              |

실무 코딩에서는 `List<C2>`, `C2[]` 이런 거 쓸 때만 다중성 생각하잖아.

근데 UML 클래스 다이어그램에서는 **"한 객체가 다른 객체를 몇 개 가질 수 있냐"** 를 표현하는 게 다중성이라서:

|상황|다중성|
|---|---|
|C2 하나만 가짐|`1`|
|C2 여러 개 가짐 (List, 배열 등)|`*`|
|C2 없을 수도 있고 하나일 수도|`0..1`|

결국 내부 구현이 List든 배열이든 상관없이 **"여러 개 참조 가능하면 *"** 이게 UML 규칙이야.

이 문제에서 `c2s`라는 이름 자체가 복수형(s 붙음)이고 for each 돌리니까 → 여러 개 → `*` 이렇게 읽는 거임.

**핵심 포인트**: `for each c2 in c2s`에서 **c2s = C2의 컬렉션**이라는 걸 읽어내는 게 관건. 컬렉션 순회 = 다중성 * 표시 필요.

# SOLID

좋아. 이미지 내용 기준으로 **SOLID 원칙 표**로 정리해줄게.

|원칙|영문|핵심 의미|핵심 포인트|이미지 예시 요지|
|---|---|---|---|---|
|단일 책임 원칙|SRP (Single Responsibility Principle)|객체는 **하나의 책임만** 가져야 함|**변경 이유가 오직 하나**여야 함|학생 클래스가 수강과목 추가/조회, DB 저장, 성적표 출력까지 다 맡으면 안 됨. 수강과목 추가/조회만 맡고 나머지는 분리|
|개방 폐쇄 원칙|OCP (Open-Closed Principle)|**확장에는 열려 있고, 변경에는 닫혀 있어야 함**|새 기능 추가 시 **기존 코드 수정 최소화**|클라이언트는 인터페이스 A에 의존하고 A1, A2, A3 같은 구현체를 추가해도 클라이언트 코드는 안 바뀜|
|리스코프 치환 원칙|LSP (Liskov Substitution Principle)|부모 타입 자리에 자식 타입을 넣어도 **의미가 유지**되어야 함|자식은 부모의 **행위 계약을 깨면 안 됨**|부모 인스턴스 대신 자식 인스턴스로 대체해도 프로그램 의미가 동일해야 함|
|의존 역전 원칙|DIP (Dependency Inversion Principle)|구체가 아니라 **추상에 의존**해야 함|자주 바뀌는 구현보다 **변화 적은 인터페이스/추상클래스**에 의존|클라이언트가 파일/네트워크/파이프 데이터 읽기 구현체가 아니라 추상 ‘데이터 읽기’에 의존|
|인터페이스 분리 원칙|ISP (Interface Segregation Principle)|인터페이스는 **클라이언트별로 작게 분리**해야 함|사용하지 않는 메서드에 **의존하지 않게** 해야 함|범용 인터페이스 하나를 여러 클래스가 억지로 공유하지 말고 필요한 기능만 분리|

|원칙|자주 나오는 함정|
|---|---|
|SRP|“기능이 여러 개면 무조건 SRP 위반”으로 단순화하면 위험. 핵심은 **변경 이유가 하나냐**다|
|OCP|“코드 수정이 아예 0이어야 한다”가 아님. 핵심은 **새 요구사항에 기존 코드 변경을 최소화**|
|LSP|단순 상속 여부 문제가 아니라 **행위 일관성** 문제|
|DIP|IoC, 훅(Hook), 프레임워크, 인터페이스 의존과 자주 연결됨|
|ISP|사용하지 않는 메서드까지 구현 강요하면 ISP 위반|

# Abstract Factory

![[Pasted image 20260409125637.png]]



- Abstract Factory 는 Factory Method 패턴과 유사하나 차이점은 객체 군(그룹)을 생성한다는 것임
- 문제를 보면, 객체의 생성은 하위 클래스인 ChefA 와 ChefB 가 담당하고 있음
- 그러나 Factory Method 패턴과의 차이점은 서브 팩토리 클래스는 하나의 객체를 생성하는 것이 아니라 여러 객체를 생성한다는 것
## ■ 핵심 차이 (시험 포인트)

|패턴|생성 대상|
|---|---|
|Factory Method|객체 1개|
|Abstract Factory|객체 “묶음(세트)”|

# 빌더 패턴 (Builder)

**조립하는 방식 복합객체 생성과정과 표현방법 분리**

```java
public class User {
    private String name;
    private int age;

    public static class MakeIt { <<---- 빌더 클래스>>
        private String name;
        private int age;

        public MakeIt withName(String name) {

            this.age = age;
            return this;

            user.name = this.name;
            user.age = this.age;
            return user;
        }
    }

    public static void main(String[] args) {
        User user = new User.MakeIt()
                .withName("Henry")
                .withAge(30)
                .make();   <<--- 각 항목을 넣고 조립함.>>
    }
}
```

# 싱글톤 패턴 (Singleton)

```java
public class Singleton {
    // 싱글톤 객체를 static 변수로 선언
    private static Singleton instance;
    private int msg;

    // 외부에서 생성자 호출 막기 
    private Singleton(int msg) { <---프라이빗 선언 호출막음>>
        this.msg = msg;
    }

    // 인스턴스를 전달
    public static Singleton getInstance(int msg) { <<--Static 선언 요기로 객체화함>>
        if (instance == null) {
            instance = new Singleton(msg);
        }
        return instance;  <<---- 한번 획득하면 다시 획득 안됨.
    }

    public void printMsg() {
        System.out.println(msg);
    }
}

class Main {
    public static void main(String[] args) {
        Singleton instance = Singleton.getInstance(1);
        Singleton instance2 = Singleton.getInstance(2);
        Singleton instance3 = Singleton.getInstance(3);

        instance.printMsg();
        instance2.printMsg();
        instance3.printMsg();  << 다 1씩 출력됨 >>
    }
}
```

**해당 객체가 전체에서 1개만 생성됨을 보장함**

# 프로토타입 패턴 (Prototype Pattern)
![[Pasted image 20260409131300.png]]- Prototype Pattern : **미리 만들어진 객체를 복사(clone)하여** 객체를 생성하는 패턴. 다수 객체 생성 시 객체 생성비용을 줄일 수 있음

```java
class Resume implements Cloneable {
    private String name;
    private String skill;

    public Resume(String name, String skill) {
        this.name = name;
        this.skill = skill;
    }

    public void setName(String name) {
        this.name = name;
    }

    public void setSkill(String skill) {
        this.skill = skill;
    }
```

# Command pattern과 Adapter pattern

다음 클래스 다이어그램은 Button이 눌렸을 때 기존에 구현된 LEDLight 클래스를 이용하여 ILamp 인터페이스를 구현하는 동작에 대한 설계를 보여 준다. 이 설계에서 적용되고 있는 2개의 디자인 패턴으로서 가장 적절한 것은?
![[Pasted image 20260406120230.png]]
① Strategy pattern과 Proxy pattern 
② State pattern과 Adapter pattern 
③ Command pattern과 Proxy pattern **(인터페이스가 바뀜 이건 아답타임)**
④ Command pattern과 Adapter pattern

**정답: ④ Command pattern과 Adapter pattern**

**풀이**

다이어그램을 두 부분으로 쪼개서 보면:

**① Command Pattern 부분**

|역할|클래스|
|---|---|
|Invoker|Button|
|Command 인터페이스|Command (execute())|
|ConcreteCommand|LightOnCommand|
|Receiver|Lamp|

Button이 Command 인터페이스를 통해 LightOnCommand를 실행 → 전형적인 Command 패턴

**② Adapter Pattern 부분**

|역할|클래스|
|---|---|
|Target 인터페이스|ILamp (turnOn, turnOff)|
|Adapter|Lamp|
|Adaptee|LEDLight (switchOn, switchOff)|

LEDLight는 switchOn/switchOff라는 다른 인터페이스를 가지고 있는데, Lamp가 이를 감싸서 ILamp 인터페이스(turnOn/turnOff)로 변환해줌 → 전형적인 Adapter 패턴

**③이 오답인 이유**

Proxy는 동일한 인터페이스를 유지하면서 접근제어/캐싱 등을 추가하는 패턴인데, 여기서 Lamp는 인터페이스를 **변환**하고 있으므로 Proxy가 아니라 Adapter임.

**선택지 판정**

| 번호  | 판정  | 이유                       |
| --- | --- | ------------------------ |
| ①   | ✗   | Strategy/Proxy 둘 다 아님    |
| ②   | ✗   | State 패턴 근거 없음           |
| ③   | ✗   | Command는 맞지만 Proxy가 틀림   |
| ④   | ✓   | Command + Adapter 정확히 맞음 |



# **Template Method, Strategy** 

![[Pasted image 20260406215846.png]]

핵심 차이 하나만 잡아.

**Template Method** → 뼈대는 부모, **살은 자식**이 채움. 상속으로 해결.
**Strategy** → 알고리즘을 **통째로 갈아끼움**. 인터페이스+구현체로 해결.
**State** → 객체의 **상태가 바뀌면 행동도 자동으로 바뀜**. 상태 객체를 교체하는 방식.


| 구분       | Strategy           | State               |
| -------- | ------------------ | ------------------- |
| 목적       | 알고리즘 교체            | 상태에 따른 행동 변화        |
| 교체 주체    | **클라이언트**가 선택해서 교체 | **객체 스스로** 상태 전이    |
| 구현체들의 관계 | 서로 독립              | 서로 전이 관계 있음         |
| 예시       | 정렬 방식 선택           | 신호등(빨강→노랑→초록 자동 전이) |

이 문제에서 State가 아닌 이유 → SlowScheduler/FastScheduler는 **클라이언트가 선택**하는 거지, 엘리베이터가 스스로 상태 바꾸는 게 아니야.

# Adapter 

![[Pasted image 20260406221940.png]]


**정답: ③**

| 역할      | 설명                                    | 해당      |
| ------- | ------------------------------------- | ------- |
| Target  | Client가 사용하는 인터페이스, request() 정의      | **(가)** |
| Adapter | Target 상속 + Adaptee 위임으로 request() 구현 | **(나)** |
| Adaptee | 실제 기능 보유, specificRequest() 제공        | **(다)** |

|선택지|판정|이유|
|---|---|---|
|①|❌|Client가 쓰는 게 adaptee가 될 수 없음|
|②|❌|adapter가 target을 상속해야 하는데 구조 불일치|
|③|⭕|구조 완벽 일치|
|④|❌|adaptee가 target 상속하는 건 말이 안 됨|

핵심 암기: **Client → Target ← Adapter → Adaptee** 이 흐름이야. Adapter가 가운데서 둘을 연결하는 번역기 역할.

|항목|판정|이유|
|---|---|---|
|(가) class → interface 실체화|⭕|class가 interface 구현하는 건 UML 기본|
