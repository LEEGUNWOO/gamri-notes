**정답: ③** 

|선지|판정|이유|
|---|---|---|
|① Son은 FatherA, FatherB 기능 모두 사용 가능|✅ 맞음|다중 상속의 기본 특성 — 부모 클래스 멤버 전부 상속|
|② 이름 충돌(Name Collision) 가능성 있음|✅ 맞음|FatherA, FatherB에 동일 메서드명 존재 시 충돌 발생|
|③ Diamond Problem 발생, **Java는 virtual 키워드로 제어**|❌ **틀림**|`virtual`은 C++ 키워드 — Java에는 존재하지 않음|
|④ 구조 복잡성으로 유지보수 어려움|✅ 맞음|다중 상속의 일반적 단점|

**③이 왜 틀렸나 — 언어별 Diamond Problem 해결 방식 비교**

|언어|다중 클래스 상속|Diamond Problem 해결 방법|
|---|---|---|
|**C++**|허용|`virtual` 키워드로 가상 상속(Virtual Inheritance) 사용|
|**Java**|**불허**|클래스 다중 상속 자체를 막음 → 인터페이스(Interface)로만 다중 구현 가능|
|**Python**|허용|MRO(Method Resolution Order, C3 선형화 알고리즘)로 해결|

Java는 애초에 **클래스 다중 상속을 언어 스펙에서 금지**해서 Diamond Problem이 발생할 수 없음. `virtual` 키워드 자체가 Java에 존재하지 않음.

**다이아몬드 구조 한눈에**

```
GrandFather
  /      \
FatherA  FatherB
  \      /
    Son          ← 이 구조가 다이아몬드(◇)
```

Son이 GrandFather의 멤버를 **FatherA 경로**로도, **FatherB 경로**로도 받게 되어 "어느 쪽 것을 써야 하나?" 충돌 → 이게 Diamond Problem.

C++은 `virtual inheritance`로 GrandFather 멤버를 한 번만 포함하도록 제어하고, Java는 그냥 이 구조 자체를 못 만들게 막은 것.

**정답: ②**

| 선지                            | 판정       | 이유                                 |
| ----------------------------- | -------- | ---------------------------------- |
| ① 직교성 (Orthogonality)         | ❌ 틀림     | 컴포넌트 간 독립성 원칙 — 이 코드와 무관           |
| ② DRY (Don't Repeat Yourself) | ✅ **정답** | `length`가 중복 데이터                   |
| ③ 가역성 (Reversibility)         | ❌ 틀림     | 기술/설계 결정을 쉽게 되돌릴 수 있어야 한다는 원칙 — 무관 |
| ④ 예광탄 (Tracer Bullet)         | ❌ 틀림     | 전체 시스템을 관통하는 최소 구현 먼저 만들자는 원칙 — 무관 |

**왜 DRY 위반인가**

문제 조건: **"길이란 끝에서 시작을 뺀 값"**

```java
class Line {
    Point start;
    Point end;
    double length;  // ← 이게 문제
}
```

`length`는 **이미 `start`와 `end`로부터 계산 가능한 값**임

```java
// 언제든 이렇게 구할 수 있음
double length = end - start;
```

즉 `length`를 필드로 따로 저장하면:

|문제점|설명|
|---|---|
|**지식 중복**|start/end에 이미 담긴 정보를 length에 또 저장|
|**동기화 문제**|start나 end가 바뀌면 length도 수동으로 갱신해야 함|
|**버그 위험**|두 값이 불일치하는 순간 발생|

DRY 원칙대로라면 `length`는 필드가 아니라 **메서드**로 구현해야 함:

```java
double getLength() {
    return end - start;  // 계산해서 반환
}
```

좋아. 이 문제 기준으로 선지들을 헷갈리지 않게 딱 구분해서 정리해줄게.

선지 내용 정리

| 개념                          | 한글 뜻          | 핵심 의미                                             | 시험 함정                                      |
| --------------------------- | ------------- | ------------------------------------------------- | ------------------------------------------ |
| Orthogonality               | 직교성           | 한 요소를 바꿔도 다른 요소에 영향이 최소가 되도록 서로 독립적으로 설계하는 것      | “클래스 필드가 여러 개 있으니 결합 문제인가?” 하고 직교성으로 오답 유도 |
| DRY (Don’t Repeat Yourself) | 반복 금지 / 중복 제거 | 동일한 지식, 로직, 데이터는 한 곳에만 존재해야 한다는 원칙                | “변수 하나 더 둔 것뿐”이라고 가볍게 보면 놓침                |
| Reversibility               | 가역성           | 지금의 설계/구현 선택을 나중에 다른 방식으로 쉽게 바꿀 수 있게 하는 성질        | ‘유연한 설계’라는 말에 끌려 찍기 쉬움                     |
| Tracer Bullet               | 예광탄           | 전체 시스템을 끝까지 관통하는 얇은 기능을 먼저 만들어 방향과 연결을 검증하는 개발 방식 | “실행 가능한 간단 버전” 같은 느낌으로 헷갈리기 쉬움             |

## 정답: ② Command 패턴

|선택지|판정|이유|
|---|---|---|
|① Adapter 패턴|❌|호환되지 않는 인터페이스를 변환하는 패턴. 여기선 인터페이스 변환 목적이 아님|
|**② Command 패턴**|**✅**|**요청을 객체로 캡슐화하여 호출자(Button)와 수신자(Lamp/Heater)를 분리**|
|③ Bridge 패턴|❌|추상화와 구현을 분리하는 패턴. 계층 분리 목적이 다름|
|④ Strategy 패턴|❌|알고리즘 교체가 목적. 구조가 유사해 헷갈리지만 의도가 다름|

## Command 패턴 핵심 구조 (이 문제 기준)

```
[Invoker]     [Command Interface]    [Receiver]
 Button   →→→   Request.execute()  →→→  Lamp.turnOn()
                                    →→→  Heater.start()
```

|Command 패턴 역할|이 문제의 클래스|
|---|---|
|Invoker (호출자)|Button|
|Command Interface|Request (interface)|
|ConcreteCommand|LampOnRequest, HeaterStartRequest|
|Receiver (실행자)|Lamp, Heater|

## Strategy와 헷갈리는 이유 & 구분법

|구분|Command|Strategy|
|---|---|---|
|**목적**|요청을 객체화, 실행 취소/큐잉 가능|알고리즘 교체|
|**핵심**|**Receiver가 별도 존재** (Lamp, Heater)|전략 객체 자체가 로직 보유|
|**키워드**|execute, undo, queue, log|알고리즘 변경, 정책 교체|

이 문제에서 `LampOnRequest`가 직접 로직을 갖지 않고 **Lamp에게 위임**하는 게 보이지? 그게 Command의 핵심 — Receiver 분리야.

왜 Command냐 한 줄로  
Button이 직접 Lamp.turnOn(), Heater.start()를 호출하지 않고  
Request.execute()를 통해 명령 객체에 위임하기 때문.

**각 패키지 Ca / Ce 계산**

|패키지|Ca (팬인)|Ce (팬아웃)|I = Ce/(Ca+Ce)|
|---|---|---|---|
|Pa|0|2 (q→, r→)|2/(0+2) = **1.0**|
|Pb|0|1 (s→)|1/(0+1) = **1.0**|
|Pc|3 (Pa, Pb, Pd)|1 (→Pd)|1/(3+1) = **0.25**|
|Pd|1 (Pc→)|0|0/(1+0) = **0.0**|

**선택지 판정**

| 번호  | 내용                        | 판정       | 이유                                             |
| --- | ------------------------- | -------- | ---------------------------------------------- |
| ①   | Pa, Pb의 I = 1             | ✅        | Ca=0이므로 I=1                                    |
| ②   | Pc의 Ca=3, Ce=1            | ✅        | Pa, Pb, Pd에서 들어오고, Pd로 하나 나감                   |
| ③   | **I가 가장 낮은 건 Pd, 값=0.25** | ❌ **정답** | Pd의 I는 0.25가 아니라 **0.0**                       |
| ④   | SDP 만족                    | ✅        | Pc(0.25)→Pd(0.0)은 안정→더 안정 의존이라 SDP 만족, 이건 맞는 말 |

### 정답: ③

③은 **두 가지가 동시에 틀림**:


  **I** (불안전성) 공식 기억법: **"Ce 분의 전체"** → I = Ce / (Ca + Ce)

- I=0 : 완전 안정 (아무도 안 의존, 나는 의존받기만 함)
- I=1 : 완전 불안정 (아무도 안 받아줌, 나만 의존함)

|번호|Code Smell|Extract Method로 해결?|이유|

## 문제 86 - C++ 상속 접근제한자

### 코드 분석

`Class A : B` → 상속 접근제한자 **생략** = C++에서 기본값 **private 상속**

### private 상속 규칙

|B의 원래 접근자|private 상속 후 A에서|A가 접근 가능?|
|---|---|---|
|public|**private으로 변경**|내부적으로는 가능|
|protected|**private으로 변경**|내부적으로는 가능|
|private|접근 불가 (상속과 무관)|❌|

### 선택지 판정

| 번호  | 내용                                | 판정       | 이유                                             |
| --- | --------------------------------- | -------- | ---------------------------------------------- |
| ①   | 상속 접근제한자가 생략됨                     | ✅        | **==`class`는 생략 시 private 상속이 기본값==**          |
| ②   | **B의 public 메서드를 A가 public으로 상속** | ❌ **정답** | private 상속이므로 B의 public은 A에서 **private**으로 변경됨 |
| ③   | B의 private은 A가 접근 불가              | ✅        | 상속 방식과 무관하게 private은 자식도 접근 불가                 |
| ④   | B의 protected는 A가 접근 가능            | ✅        | private 상속 시 A 내부에서는 접근 가능 (단 외부에선 불가)         |

### 핵심 포인트

C++ `class` 키워드 → 상속 접근제한자 생략 시 **private 상속** Java는 상속 접근제한자 개념 자체가 없으니 C++ 특수 개념으로 기억할 것

## 문제 54 - MC/DC

**==테스트 갯수는 : 조건문 + 1 = 조건문 3개 + 1 = 4==**

조건: `(x>0 && y>0) || z>10` = **(A && B) || C**

### MC/DC 요건: 각 조건이 **독립적으로** 결과를 바꾸는 쌍 필요

**A(x>0) 독립 쌍 찾기** → **==B,C 고정, A만 변경, 결과 변경==**

|-|A|B|C|결과|
|---|---|---|---|---|
|나|T|T|F|T|
|바|F|T|F|F|

→ A만 T→F, 결과 T→F ✅

**B(y>0) 독립 쌍 찾기** → **==A,C 고정, B만 변경, 결과 변경==**

|-|A|B|C|결과|
|---|---|---|---|---|
|나|T|T|F|T|
|라|T|F|F|F|

→ B만 T→F, 결과 T→F ✅

**C(z>10) 독립 쌍 찾기** → ==**A,B 고정, C만 변경, 결과 변경**==

|-|A|B|C|결과|
|---|---|---|---|---|
|마|F|T|T|T|
|바|F|T|F|F|

→ C만 T→F, 결과 T→F ✅

| 연산  | 독립 쌍 만들려면 나머지 조건을                   |
| --- | ----------------------------------- |
| AND | **T로 고정** (나머지가 F면 어차피 전체 F라 영향 없음) |
| OR  | **F로 고정** (나머지가 T면 어차피 전체 T라 영향 없음) |

이 문제 **(A && B) || C** 기준으로:

- A 독립 쌍 → AND 안에 있으니 B=**T**, OR이니 C=**F**
- B 독립 쌍 → AND 안에 있으니 A=**T**, OR이니 C=**F**
- ==C 독립 쌍 → OR 안에 있으니 (A&&B)=**F** → A나 B 중 하나를 F로==

## ISO/IEC 9126 vs 25010 품질 특성 정리

### ISO/IEC 9126 (구버전) - 6개 특성

|품질 특성|핵심 의미|
|---|---|
|기능성 (Functionality)|요구 기능 제공, Inter-operability는 여기 하위|
|신뢰성 (Reliability)|고장 없이 안정적 작동|
|사용성 (Usability)|쉽게 이해하고 사용|
|효율성 (Efficiency)|적은 자원으로 큰 성과|
|유지관리성 (Maintainability)|쉽게 변경/수정|
|이식성 (Portability)|다른 환경에서도 동작|

### ISO/IEC 25010 (신버전) - 8개 특성 + 부특성

| 품질 특성                           | 부특성                                                                                                                                                                    |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 기능 적합성 (Functional suitability) | 완전성(completeness), 정확성(correctness), 적절성(appropriateness)                                                                                                              |

## 복합시스템(System of Systems, SoS) 특성 정리

### 정의

두 개 이상의 요소가 독립적으로 관리되는 시스템. 독립적 목적을 가진 여러 하위 시스템들이 상호작용하여 개별 합보다 더 큰 기능을 발휘하는 대규모 복잡 시스템.

### 5대 특성

| 특성                                 | 핵심 내용                                            |
| ---------------------------------- | ------------------------------------------------ |
| 운영적 독립성 (Operational independence) | 구성 요소가 단순 컴포넌트가 아닌 독립적으로 운영 가능한 시스템, 각자 독립적으로 진화 |
| 관리적 독립성 (Managerial independence)  | 각 부분이 서로 다른 조직/부서에 의해 소유·관리, 규칙과 정책이 서로 다름       |
| 진화적 개발 (Evolutionary development)  | 단일 프로젝트로 개발되지 않고 시간이 흐르면서 점진적으로 진화               |
| **창발성 (Emergence)**                | **개별 시스템 수준에서 드러나지 않던 특성이 전체 구성 이후에야 출현**        |
| 지리적 분산 (Geographical distribution) | 구성 요소가 서로 다른 조직에 속해 지리적으로 분산, 외부 네트워크가 핵심 구성 요소  |

## 2025년 29번 — UML Sequence Diagram 패턴

**정답: ② Master-Slave**

**다이어그램 흐름 분석**

```
UI → Main: service(LargeData[])   // 큰 데이터 요청
Main → Main: split(data)           // data1, data2로 분할
  par (병렬)
    Main → Controller1: do(data1)  // 동시 처리
    Main → Controller2: do(data2)  // 동시 처리
Main → Main: merge(subResult1, subResult2)  // 결과 합산
Main → UI: result                  // 최종 반환
```

| 선택지                | 판정  | 근거                                                              |
| ------------------ | --- | --------------------------------------------------------------- |
| ① Pipe and Filter  | ❌   | 데이터가 파이프라인처럼 순차적으로 흘러가는 구조. 병렬 분배 없음                            |
| ② Master-Slave     | ✅   | Master(Main)가 작업을 분할→Slave(Controller×2)에 병렬 위임→결과 수집·병합        |
| ③ Front Controller | ❌   | 모든 요청을 단일 진입점이 받아 분배하는 구조. 웹 MVC의 DispatcherServlet 같은 것(병렬 없음) |
| ④ MVC              | ❌   | UI/비즈니스/데이터 역할 분리 구조. 병렬 분할·병합 개념 없음                            |

**Master-Slave 패턴 핵심**

|구성요소|이 문제에서|역할|
|---|---|---|
|Master|:Main|작업 분할, Slave 조율, 결과 병합|
|Slave|:Controller (×2)|동일한 작업을 병렬로 처리|
|키워드|**split → par → merge**|분할-병렬처리-병합 3단계|

**`par`** 키워드가 보이면 병렬 실행 = Master-Slave 의심하면 돼 👍

**정답: ③ 파이프와 필터 뷰 (Pipe and Filter View)**

|선택지|판정|근거|
|---|---|---|
|① Peer to Peer View|❌|동등한 컴포넌트들이 서로 직접 통신하는 구조|

```
cat file.txt | grep "error" | sort | uniq
```

**키워드 매핑**

```
"일련의 흐름"        → Pipe
"중간에 처리"        → Filter  
"변환이 이루어짐"    → Filter
"최종 목적지"        → Pipe 끝단
```

## 2025년 44번 — UML 클래스 다이어그램

**정답: ④번**

**코드 분석**

```java
Staff extends Person      → 상속 (Generalization)
Staff implements Service  → 실체화 (Realization)
Task t = new Task()       → 의존 (Dependency)
```

**UML 화살표 규칙**

|관계|선 종류|화살표|방향|
|---|---|---|---|
|상속 (extends)|실선|빈 삼각형 ▷|Staff → Person|
|실체화 (implements)|**점선**|빈 삼각형 ▷|Staff → Service|
|의존 (new 생성)|**점선**|열린 화살표 →|Staff → Task|

**선택지 비교**

|-|Staff→Service|Staff→Person|Staff→Task|
|---|---|---|---|
|①|실선+화살표 ❌|점선+삼각형 ❌|실선 ❌|
|②|실선+화살표 ❌|점선+삼각형 ❌|점선 ✅|
|③|점선+화살표 ❌|실선+삼각형 ✅|실선 ❌|
|④|**점선+화살표** ✅|**실선+삼각형** ✅|**점선** ✅|

**핵심 암기**

```
실체화(implements) = 점선 + 빈삼각형
상속(extends)      = 실선 + 빈삼각형
의존(new)          = 점선 + 열린화살표
```



**의존 vs 연관 차이**

```java
// 연관 (Association) - 필드로 가짐
class Staff {
    Task task;  // 멤버변수로 보유
}

// 의존 (Dependency) - 메서드 안에서만 생성
class Staff {
    public void doWork() {
        Task t = new Task();  // 메서드 안에서만 존재
    }
}
```

이 문제는 **메서드 안에서** `new Task()` 한 거야.

```
메서드 끝나면 t 사라짐 → 일시적 사용 → 의존(점선)
필드로 가지고 있으면  → 지속적 보유 → 연관(실선)
```

**한 줄 정리**

```
Task가 Staff 안에 항상 살아있냐? → 연관 (실선)
Task가 메서드 실행중에만 존재?  → 의존 (점선)
```

이 문제는 doWork() 안에서만 쓰니까 점선 맞아 👍

#### 2025년 49번 — 리팩토링 기법

**정답: ④ Replace Type Code with Subclasses**

|선택지|판정|근거|
|---|---|---|
|① Pull Up Method|❌|자식 클래스의 공통 메서드를 부모로 올리는 것|
|② Replace Type Code with Class|❌|타입코드를 클래스로만 대체 — 서브클래스 분리 없음|
|③ Replace Parameter with Method|❌|파라미터 대신 메서드 호출로 대체하는 것|
|④ Replace Type Code with Subclasses|✅|switch(typecode) → 각 타입을 **서브클래스로 분리** + @Override|

**핵심 변화 포인트**

```
Before: switch(_typecode) { case LINE / RECT / OVAL }
After:  Shape(abstract) → ShapeLine / ShapeRectangle / ShapeOval
                           각자 draw() 오버라이드
```

**②vs④ 구분법**

```
Type Code with Class      → 타입코드만 클래스화, 서브클래스 X
Type Code with Subclasses → 타입코드별로 서브클래스 생성 ← 이 문제
```

**②vs④ 딱 한 줄 차이**

```
② Replace Type Code with Class
   타입코드 자체를 클래스로 포장
   → Shape 하나에 TypeCode 클래스만 따로 뺌
   → switch 여전히 존재 가능

④ Replace Type Code with Subclasses  
   타입코드별로 클래스를 아예 쪼갬
   → ShapeLine / ShapeRectangle / ShapeOval 분리
   → switch 완전히 제거
```

**비유**

```
② 직급코드를 Enum으로 만든 것
④ 직급코드 없애고 사원/대리/과장 클래스로 분리한 것
```

코드에서 **서브클래스(@Override)가 생겼으면 무조건 ④** 👍

**정답: ③ Scenario testing**

| 선택지                         | 판정  | 근거                                       |
| --------------------------- | --- | ---------------------------------------- |
| ① State transition testing  | ❌   | 상태 전이 다이어그램 기반 — 시스템 상태 변화에 집중           |

| ③ Scenario testing          | ✅   | **사용자-시스템 간 입출력 시퀀스** 모델 기반, E2E 테스트에 활용 |
| ④ Requirement-based testing | ❌   | 요구사항 명세 기반 — 요구사항 충족 여부 검증               |

**키워드 매핑**

```
"사용자 사이에 발생하는 입/출력 시퀀스" → 사용자 행동 흐름 = Scenario
"End-to-End 기능 테스트"               → Scenario testing 대표 활용처
```

**테스트 기법 한 줄 정리**

|기법|키워드|
|---|---|
|State transition|상태 전이, 상태머신|
|Decision table|조건 조합, 참/거짓|
|**Scenario**|**사용자 흐름, E2E**|
|Requirement-based|요구사항 추적|

**차이는 목적이야**

```
Decision Table  → 조건 조합별 결과가 맞냐?  (블랙박스)
Coverage        → 코드/경로를 얼마나 실행했냐? (화이트박스)
```

**비유**

```
Decision Table → 메뉴판 경우의 수 다 주문해봄
Coverage       → 주방 안 레시피 코드를 다 실행해봄
```

겉으로 보이는 표 형태는 비슷한데 안을 보냐 밖을 보냐 차이야 👍

**Decision Table Testing**

```
조건들의 조합마다 결과가 뭔지 표로 정리해서 테스트
```

**예시 — 할인 정책**

|-|케이스1|케이스2|케이스3|케이스4|
|---|---|---|---|---|
|회원여부|Y|Y|N|N|
|10만원 이상|Y|N|Y|N|
|**할인적용**|**O**|**O**|**O**|**X**|

**이런 거 테스트할 때 씀**

```
조건 A (T/F)
조건 B (T/F)  → 조합 다 만들어서 결과 검증
조건 C (T/F)
```

**키워드**

```
조건 조합 + 참/거짓 + 표 = Decision table
```

## 2025년 41번 — ISO 22989 AI 신뢰성

**정답: ② AI Resilience**

| 선택지                  | 판정  | 근거                                              |
| -------------------- | --- | ----------------------------------------------- |
| ① AI Robustness      | ❌   | **(견고)**노이즈·변형 입력에도 성능 유지 — **버텨내는** 능력         |
| ② AI Resilience      | ✅   | **(복원)**사고 후 **빠르게 복구** + 장애 중에도 저하된 수준으로 운영 가능 |
| ③ AI Controllability | ❌   | **(제어)**인간이 AI를 **제어·개입**할 수 있는 능력              |
| ④ AI Explainability  | ❌   | **(설명)**AI 결정을 **설명**할 수 있는 능력                  |

**Robustness vs Resilience 구분**

```
Robustness  = 맞아도 안 쓰러짐 (방어)
Resilience  = 쓰러져도 다시 일어남 (복구)
```

**비유**

```
Robustness = 맷집 좋은 복서 (맞아도 버팀)
Resilience = 쓰러져도 일어나는 복서 (복구력)
```

**키워드**

```
복구 + 장애 중 저하 운영 = Resilience
```

**26. MDD(Model Driven Development)의 개념과 구성 요소 및 관련 표준에 대한 설명으로 가장 적절하지 않은 것은?**

① MDD는 특정 플랫폼에 종속되지 않는 모델(PIM)을 생성하고, 이를 특정 플랫폼에 종속적인 모델(PSM)로 변환하여 소스 코드를 자동으로 생성하는 방식이다. 
② PIM(Platform Independent Model)은 비즈니스 로직과 시스템의 기능을 기술하며, 기술 사양이나 플랫폼 변화에 영향을 받지 않는 추상화된 모델이다. 
③ PSM(Platform Specific Model)은 특정 미들웨어나 운영체제(Java, .NET 등)의 상세 사양을 포함하며, PIM으로부터 자동 또는 반자동으로 변환될 수 있다. 
④ MDD의 핵심 표준인 MOF(Meta-Object Facility)는 모델 간의 변환 규칙을 정의하는 언어이며, 가독성을 높이기 위해 텍스트 기반의 요구사항 명세서 작성을 필수 원칙으로 한다.

**정답: ④**

**MDD 전체 개념 정리**

MDD = 코드를 직접 짜는 대신 **모델을 먼저 만들고 코드를 자동 생성**하는 개발 방식.

**핵심 흐름:**

```
Transformation (QVM) : CIM → PIM → PSM → 코드
```

| 모델  | 영문                            | 설명                         |
| --- | ----------------------------- | -------------------------- |
| PIM | Platform Independent Model    | 플랫폼 독립적 모델 설계. 비즈니스 로직만    |
| PSM | Platform Specific Model       | 특정 플랫폼(Java, .NET 등) 종속 모델 |

**관련 표준:**

|표준|역할|
|---|---|
|MOF (Meta-Object Facility)|**메타모델 정의 표준**. 모델을 정의하는 모델|
|QVT (Query/View/Transformation)|**모델 간 변환 규칙** 정의 언어|
|UML|PIM 표현에 주로 사용|
|XMI|모델을 XML로 교환하는 표준|

**④가 틀린 이유:**

MOF는 **메타모델 정의** 표준이지 모델 간 변환 규칙 언어가 아님. 모델 간 변환은 **QVT**가 담당. 또한 텍스트 기반 요구사항 명세서 작성은 MDD 원칙과 무관함.

**28. 헥사고날 아키텍처(Hexagonal Architecture)의 구성 요소와 설계 원칙에 대한 설명으로 가장 적절하지 않은 것은?**

① 애플리케이션의 핵심 비즈니스 로직을 중심(내부 영역)에 두고, 데이터베이스나 UI와 같은 기술적 세부 사항을 외부 어댑터로 분리하여 비즈니스 로직의 순수성을 보존한다. 
② Inbound Adapter는 외부의 요청을 받아 내부 서비스가 구현한 Inbound Port를 호출하며, 대표적인 예로 웹 컨트롤러나 CLI 도구가 있다. 
③ Domain Entity는 비즈니스 핵심 규칙을 담고 있으며, 데이터의 영속성 처리를 위해 인프라 영역에 정의된 데이터베이스 어댑터의 구체적인 구현체에 직접 의존한다. 
④ Outbound Port는 내부 영역에서 외부 시스템과 상호작용하기 위해 정의한 인터페이스이며, 의존성 역전 원칙(DIP)을 통해 내부 영역이 외부 기술 환경으로부터 독립되도록 돕는다.

**정답: ③**

|선택지|판정|이유|
|---|---|---|
|① 비즈니스 로직 중심, 외부 기술 분리|✅|헥사고날 핵심 원칙 맞음|
|② Inbound Adapter → Inbound Port 호출|✅|맞음|
|③ Domain Entity가 DB 어댑터 구현체에 직접 의존|❌|**의존성 역전 위반**|
|④ Outbound Port = DIP로 내부-외부 분리|✅|맞음|

**헥사고날 구조 핵심:**

```
외부(UI, DB, API)
    ↓ Adapter
    ↓ Port (인터페이스)
[Domain Entity / Business Logic]  ← 여기가 중심
    ↓ Port (인터페이스)
    ↓ Adapter
외부(DB, 외부 API)
```

Domain Entity는 **인터페이스(Port)에만 의존**해야 함. 구체적인 DB 어댑터 구현체에 직접 의존하면 헥사고날의 핵심인 **의존성 역전 원칙(DIP) 위반**이야.


**헥사고날 아키텍처 = 포트와 어댑터 패턴**

핵심 아이디어:

> 비즈니스 로직을 가운데 두고, 외부 세계(DB, UI, API)와는 **포트(인터페이스)를 통해서만** 연결

**구조 그림:**

```
      [REST API]  [CLI]  [테스트]
           ↓        ↓       ↓
      Inbound Adapter (외부→내부)
           ↓
      Inbound Port (인터페이스)
           ↓
    ┌──────────────────┐
    │  Domain Entity   │  ← 비즈니스 핵심
    │  Use Case        │  ← 여기가 육각형 중심
    └──────────────────┘
           ↓
      Outbound Port (인터페이스)
           ↓
      Outbound Adapter (내부→외부)
           ↓
      [DB]  [외부API]  [메일]
```

**왜 헥사고날(육각형)이냐?**

포트가 여러 방향으로 뚫려있는 모양이 육각형처럼 생겨서 ㅋㅋ

**클린 아키텍처랑 뭐가 달라?**

사실 거의 같은 개념. 클린 아키텍처는 레이어를 동심원으로, 헥사고날은 육각형으로 표현한 차이. 핵심은 **의존성이 항상 안쪽(도메인)을 향해야 한다**는 거야.

**시험 포인트:**

- Domain Entity → 외부 구현체에 **직접 의존 금지**
- 반드시 **Port(인터페이스)** 를 통해서만 접근
- Inbound = 외부→내부, Outbound = 내부→외부

**29. 다음은 ISO/IEC 25010:2011에서 제시하는 이식성(Portability) 하위 속성에 대한 설명이다. (가), (나)에 해당하는 속성의 연결로 가장 적절한 것은?**

> (가) 시스템이 서로 다른 하드웨어, 운영체제, 미들웨어 등 다른 실행/운영 환경으로 이전될 수 있는 정도 
> (나) 특정 환경에서 제품을 설치하거나 제거하는 데 필요한 노력의 정도

① (가) Adaptability, (나) Installability 
② (가) Installability, (나) Adaptability 
③ (가) Replaceability, (나) Installability 
④ (가) Adaptability, (나) Replaceability

**정답: ①**

**ISO/IEC 25010 이식성(Portability) 하위 속성 정리:**

|속성|한글|핵심 설명|
|---|---|---|
|Adaptability|적응성|다른 **환경**(HW, OS, 미들웨어)으로 이전 가능한 정도 → **(가)**|
|Installability|설치성|**설치·제거**에 필요한 노력 → **(나)**|
|Replaceability|대체성|동일 환경에서 **다른 SW로 교체** 가능한 정도|

**함정 포인트:**

- Adaptability = 환경 **이전** (다른 플랫폼으로 옮기기)
- Replaceability = SW **교체** (다른 소프트웨어로 바꾸기)
- 이 둘 헷갈리면 틀림

네가 3번 썼는데, Replaceability는 교체 가능성이지 환경 이전이 아니야.

**31. Agile 방법론에서 작업의 흐름(Flow)을 최적화하기 위해 Kanban/Flow 기반에서 사용하는 '흐름 지표(Flow Metrics)' 그룹에 속하지 않는 것은?**

① Lead Time: 고객의 요청이 백로그에 등록된 시점부터 최종적으로 가치가 인도될 때까지의 전체 소요 시간을 측정한다. 
② Cycle Time: 작업 항목이 실제 개발 단계(In Progress)에 착수된 시점부터 완료될 때까지의 순수 작업 시간을 측정한다. 
③ Throughput: 특정 단위 시간(주, 월 등) 동안 시스템을 통해 최종 완료된 작업 항목의 개수를 측정하여 흐름의 속도를 파악한다. 
④ Velocity: 하나의 스프린트 동안 팀이 완료한 작업량(주로 스토리 포인트의 총합)을 의미하며, 향후 스프린트의 계획 수립과 팀의 생산 능력을 예측하는 데 사용한다.

**정답: ④**

|선택지|판정|이유|
|---|---|---|
|① Lead Time|✅ Flow Metrics|Kanban 핵심 지표|
|② Cycle Time|✅ Flow Metrics|Kanban 핵심 지표|
|③ Throughput|✅ Flow Metrics|Kanban 핵심 지표|
|④ Velocity|❌|**Scrum 지표**. Kanban에는 스프린트 자체가 없음|

Kanban = 연속 흐름 → Lead Time, Cycle Time, Throughput Scrum = 스프린트 단위 → **Velocity**

- 과도한 추상화 = 리팩토링 아니고 오버엔지니어링

**33. AI 프로젝트에서 사용자의 잠재적 요구사항 수집과 혁신적인 문제 해결을 위해 최근 많이 활용되고 있는 "Design Thinking" 기법의 일반적인 5단계 수행 절차를 올바르게 나열한 것은?**

> 가) 공감 단계에서 발견한 사용자의 페인 포인트(Pain Point)와 니즈를 분석하여, 해결해야 할 핵심 과제를 'POV(Point of View)' 문장으로 정립한다. 
> 나) 선별된 아이디어를 바탕으로 저비용의 물리적/디지털 모델을 신속하게 제작하여 가설을 시각화하고 구체화한다. 
> 다) 인터뷰, 관찰, 사용자 여정 지도(Customer Journey Map) 등을 활용하여 사용자의 입장에서 경험을 이해하고 잠재된 요구사항을 포착한다. 
> 라) 제작된 결과물을 실제 사용자 환경에서 검증하고 피드백을 수렴하여 모델을 정교화하거나 이전 단계로 회귀(Iteration)한다. 
> 마) '어떻게 하면 우리가 ~할 수 있을까?(HMW)'라는 질문을 던지며 브레인스토밍, 마인드맵 등을 통해 가급적 많은 해결책을 발산한다.

① 가→다→마→나→라 
② 다→마→가→라→나 
③ 다→가→마→나→라 
④ 마→다→가→나→라

**정답: ③**

**Design Thinking 5단계:**

| 단계       | 영문        | 설명                           | 해당    |
| -------- | --------- | ---------------------------- | ----- |
| 1. 공감    | Empathize | (발산)사용자 관찰·인터뷰로 경험 이해        | **다** |
| 2. 정의    | Define    | (수렴)Pain Point 분석, POV 문장 정립 | **가** |
| 3. 아이디어  | Ideate    | (발산)HMW 질문, 브레인스토밍           | **마** |
| 4. 프로토타입 | Prototype | (수렴)저비용 모델 제작                | **나** |
| 5. 테스트   | Test      | (검증)실제 환경 검증, 피드백, 반복        | **라** |

```
다(공감) → 가(정의) → 마(아이디어) → 나(프로토타입) → 라(테스트)
```

**34. SSPL(Software and System Product Line) 방법론의 공학적 프로세스와 주요 개념에 대한 설명으로 가장 적절하지 않은 것은?**

① Domain Engineering은 제품 라인 전체에서 공통적으로 사용될 Core Assets을 개발하는 과정으로, 요구사항, 아키텍처, 컴포넌트, 테스트 계획 등을 포함한다. 
② Application Engineering은 도메인 공학에서 구축된 핵심 자산을 재사용하고, 특정 고객이 요구한 개별 제품의 특화된 기능을 추가하여 최종 제품을 생산하는 과정이다. 
③ Variability 관리는 제품 라인의 핵심으로, 공통점과 차이점을 식별하여 설계에 반영하며, Binding Time은 반드시 컴파일 시점으로 고정되어야 한다. 
④ 제품 라인 도입의 주요 목적은 시장 출시 시간 단축, 개발 비용 절감, 제품 간의 품질 일관성 유지이며, 초기 핵심 자산을 구축하기 위한 초기 투자 비용이 높다는 단점이 있다.

**정답: ③**

|선택지|판정|이유|
|---|---|---|
|① Domain Engineering|✅|Core Assets 개발 맞음|
|② Application Engineering|✅|재사용 + 특화 기능 추가 맞음|
|③ Binding Time = 컴파일 시점 고정|❌|Binding Time은 컴파일만이 아님|
|④ 초기 투자 비용 높음|✅|맞음|

**③이 틀린 이유:**

Binding Time(바인딩 시점)은 가변성이 확정되는 시점으로 여러 종류가 있음:

|Binding Time|예시|
|---|---|
|컴파일 시점|#ifdef 조건부 컴파일|
|링크 시점|라이브러리 선택|
|실행 시점|설정 파일, 런타임 파라미터|
|설치 시점|인스톨러 옵션|

**반드시 컴파일 시점으로 고정**이라고 한 게 틀린 것. 다양한 시점에 바인딩 가능함.

**SSPL (소프트웨어 및 시스템 제품 계열) 정리**

**핵심 2개 프로세스:**

|프로세스|영문|역할|
|---|---|---|
|도메인 공학|Domain Engineering|공통 자산 개발. 전체 제품군에 재사용될 것들 미리 만들기|
|응용 공학|Application Engineering|공통 자산 재사용 + 개별 제품 특화 기능 추가|

**핵심 개념:**

|개념|영문|설명|

|가변성|Variability|제품들 간의 차이점. 이걸 잘 관리하는 게 핵심|
|바인딩 시점|Binding Time|가변성이 확정되는 시점 (컴파일/링크/설치/실행 시점 모두 가능)|

① Student PC와 Smartphone은 스테레오타입 «device»로 선언된 노드로, 시스템 내부의 소프트웨어가 실행될 수 있는 물리적인 하드웨어 자원을 의미한다. 
② Web Server와 App Server 사이의 통신 경로(Communication Path)에 명시된 AJP는 두 노드 간의 물리적 연결 규격이나 논리적인 통신 프로토콜을 나타낸다.

**정답: ③**

|선택지|판정|이유|
|---|---|---|
|① «device» = 물리적 하드웨어|✅|맞음|

|③ «node» = 소프트웨어 아티팩트|❌|**틀림**|
|④ JDBC = 통신 프로토콜|✅|맞음|

**③이 틀린 이유:**

UML에서 «node»는 **물리적 또는 논리적 실행 환경**을 의미함. «device»의 상위 개념이라 볼 수 있어.

- «device» → 물리적 하드웨어 노드
- «execution environment» → 소프트웨어 실행 환경 (Tomcat, Nginx 등)
- «node» → 둘 다 포함하는 상위 개념

Redis Cache가 «node»로 표현됐다고 해서 **소프트웨어 아티팩트라는 게 아님**. 물리적 서버일 수도 있고 논리적 실행 환경일 수도 있는 것. "반드시 아티팩트"라고 단정한 게 틀린 거야.

UML에서 **아티팩트(Artifact)** = 소프트웨어 개발 과정에서 생성되는 **물리적 파일**

쉽게 말하면:

```
코드 짜면 나오는 결과물들
.jar, .war, .exe, .dll
설정 파일 (.xml, .properties)
스크립트 (.sh, .sql)
```

**노드 vs 아티팩트 차이:**

|구분|의미|예시|
|---|---|---|
|노드 (Node)|실행 **환경** (하드웨어/소프트웨어)|서버, Tomcat, Docker 컨테이너|
|아티팩트 (Artifact)|실행 **파일**|app.war, redis.conf, schema.sql|

**비유:**

노드 = 집 (실행 환경) 아티팩트 = 집 안에 사는 사람/가구 (실제 파일)

Redis Cache는 **실행 환경(노드)** 이지 파일이 아님. 그래서 ③이 틀린 거야.

**37. DevOps 환경에서 팀의 성과와 인도 역량을 측정하기 위한 DORA(DevOps Research and Assessment) 4대 핵심 지표에 대한 설명으로 가장 적절하지 않은 것은?**

① Deployment Frequency: 조직이 상용(Production) 환경에 코드를 얼마나 자주 배포하는지를 측정하며, 소프트웨어 인도 속도와 민첩성을 나타내는 주요 척도이다. 
② Lead Time for Changes: 신규 요구사항이 분석 및 설계 단계에 진입한 시점부터 상용 환경에 배포되어 가치가 전달되기까지의 전체 소요 시간을 의미한다. 
③ Change Failure Rate: 전체 배포 횟수 중 서비스 중단이나 장애를 유발하여 롤백이나 핫픽스 등 후속 조치가 필요했던 배포의 비율을 측정한다. 
④ Time to Restore Service: 상용 환경에서 장애나 서비스 중단이 발생했을 때, 이를 인지하고 다시 정상 서비스 상태로 복구하기까지 소요된 시간을 측정한다.

**정답: ②**

|선택지|판정|이유|
|---|---|---|
|① 배포 빈도|✅|맞음|
|② Lead Time = 분석·설계 진입 시점부터|❌|시작점이 틀림|
|③ 변경 실패율|✅|맞음|
|④ 서비스 복구 시간|✅|맞음|

**②가 틀린 이유:**

Lead Time for Changes는 **코드 커밋(commit) 시점**부터 상용 배포까지의 시간임. 분석·설계 단계 진입 시점이 아님.

**DORA 4대 지표 정리:**

| 지표                                  | 측정 내용             |
| ----------------------------------- | ----------------- |
| Deployment Frequency (배포빈도)         | 배포 **빈도**         |
| Lead Time for Changes (변경 리드타임)     | **커밋→배포** 소요 시간   |
| Change Failure Rate (변경실패율)         | 배포 후 **장애 발생 비율** |
| Time to Restore Service (서비스 복구 시간) | 장애 발생 후 **복구 시간** |

| 선택지                         | 판정  | 이유               |
| --------------------------- | --- | ---------------- |
| ① 단순화된 인터페이스로 편의성 도모        | ✅   | 퍼사드 핵심 목적 맞음     |
| ② 결합도 낮춰 변경 영향 최소화          | ✅   | 퍼사드 효과 맞음        |
| ③ 서브시스템 기능 캡슐화 보호           | ✅   | 퍼사드 특징 맞음        |
| ④ 계층 간 유일한 진입점 정의 물리적 접근 차단 | ❌   | 이건 **프록시** 패턴 설명 |

**프록시 vs 퍼사드:**

**3가지 패턴 비교:**

| 패턴               | 목적                          | 핵심            |
| ---------------- | --------------------------- | ------------- |
| 퍼사드 (Facade)     | 복잡한 서브시스템을 **단순하게** 묶어주는 창구 | 여러 객체 → 하나로   |
| 프록시 (Proxy)      | 객체에 대한 **접근 제어**            | 하나 → 하나 (대리인) |
| 메디에이터 (Mediator) | 객체들 간의 **복잡한 상호작용 중재**      | 여러 객체 ↔ 여러 객체 |

**직관적 비유:**

- 퍼사드 = **안내데스크**. 복잡한 내부 부서 몰라도 됨
- 프록시 = **대리인**. 원본 객체 앞에 서서 접근 통제
- 메디에이터 = **교통정리 경찰**. 여러 차(객체)가 서로 직접 통신 안 하고 경찰 통해서 소통

**이 문제는 왜 퍼사드냐:**

Client → Agent Manager → (LLM, VectorDB, API)

Client가 복잡한 서브시스템들을 Agent Manager 하나로 접근. 전형적인 퍼사드.

메디에이터였으면 서브시스템들이 서로 Agent Manager를 통해 **상호 통신**해야 함.

> 다. 보고서 생성 기능은 데이터 입력 후 가끔씩 비정상적으로 작동할 수 있다. 
> 
> 라. 검색 결과는 사용자가 인식하기에 적절한 수준의 가독성을 유지하며 출력되어야 한다.

① 가, 나 ② 가, 라 ③ 나, 다 ④ 다, 라

**정답: ②**

**SRS 특성별 분류:**

| 사례  | 결여된 특성                | 이유                         |
| --- | --------------------- | -------------------------- |

| 추가  | 추적 가능성 (Traceability) | 요구사항이 어떻게 구현되는지 추적 가능 해야함. |

**46. 마이크로서비스 아키텍처(MSA)에서 로컬 트랜잭션의 순차적 실행을 통해 최종 일관성을 달성하는 ==Saga== 패턴의 두 가지 구현 방식에 대한 설명이다. (가)와 (나)에 해당하는 방식을 올바르게 연결한 것은?**

> (가) 각 참여 서비스가 자신의 로컬 트랜잭션을 완료한 후 특정 상태 변경을 알리는 신호를 방출하면, 이를 주시하고 있던 다음 단계의 서비스들이 연쇄적으로 반응하여 작업을 이어가는 방식이다. 비즈니스 흐름이 여러 서비스에 분산되어 존재하므로 개별 컴포넌트 간의 독립성은 높으나, 전체 공정의 복잡도가 높아질수록 현재 진행 상태를 추적하거나 디버깅하는 난이도가 급격히 상승한다. 
> 
> (나) 별도의 전담 객체가 비즈니스 워크플로우의 실행 순서를 정의하고, 각 참여 서비스가 수행해야 할 구체적인 작업 내용을 전달하며 응답을 수집하는 방식이다. 모든 복잡한 트랜잭션의 상태와 이력이 단일한 지점에 집약되어 있어 비즈니스 가시성이 높으며, 오류 발생 시 이전 단계들을 되돌리기 위한 보상 작업의 호출 시점도 이 지점에서 총괄적으로 판단한다.

① (가) Orchestration, (나) Choreography 
② (가) Choreography, (나) Orchestration 
③ (가) Choreography, (나) Two-Phase Commit 
④ (가) Try-Confirm-Cancel, (나) Orchestration

**정답: ②**

|선택지|판정|이유|
|---|---|---|
|① 반대로 연결|❌||
|② Choreography / Orchestration|✅||
|③ Two-Phase Commit|❌|Saga 패턴 구현 방식 아님|
|④ Try-Confirm-Cancel|❌|Saga 패턴 구현 방식 아님|

**Saga 패턴이 뭐냐:**

MSA에서 여러 서비스에 걸친 트랜잭션을 처리하는 방법. DB가 서비스마다 다르니까 일반 트랜잭션 못 씀.

**두 가지 구현 방식:**

**Choreography (안무/코레오그래피)**

```
서비스A 완료 → 이벤트 발행
              → 서비스B가 듣고 반응
                → 이벤트 발행
                  → 서비스C가 듣고 반응
```

- 서로 이벤트로만 통신
- 중앙 지휘자 없음
- 독립성 높음
- 복잡해지면 흐름 추적 어려움

**Orchestration (오케스트레이션)**

```
Orchestrator(지휘자)
  → 서비스A에게 "이거 해"
  → 서비스B에게 "이거 해"
  → 서비스C에게 "이거 해"
```

- 중앙 지휘자가 순서 관리
- 비즈니스 흐름 한눈에 파악
- 단일 장애점 존재

**문제 지문 매핑:**

(가) = 각 서비스가 이벤트 발행 → 다음 서비스가 반응 = **Choreography**

(나) = 별도 전담 객체가 순서 정의, 명령 전달 = **Orchestration**

**정답: ② (가) Choreography, (나) Orchestration**


**한 줄 구분:**

- Choreography = 이벤트로 서로 반응 (지휘자 없음)
- Orchestration = 중앙 지휘자가 명령 (지휘자 있음)

**48. 다음 중 SysML의 블록 정의 다이어그램(Block Definition Diagram)이 수행하는 역할과 표현 범위에 대한 설명으로 가장 적절한 것은?**

> 
> 라. 특정 컨텍스트 내에서 블록의 인스턴스들이 커넥터를 통해 상호작용하는 협력 구조를 표현한다. 
> 
> 마. 제약 블록을 활용하여 시스템의 물리적 성질 간에 성립해야 하는 수학적 관계를 정의한다.

① 나, 다, 마 
② 가, 다, 라 
③ 가, 나, 다 
④ 다, 라, 마

**정답: ③**

**SysML 다이어그램 종류 구분:**

|항목|해당 다이어그램|이유|
|---|---|---|
|가. 실체 유형 정의, 분류체계|**BDD** ✅|블록 타입과 계층 정의|
|나. 구조적 연관, 다중도, 특수화|**BDD** ✅|블록 간 관계 및 상속 표현|
|다. 내부 속성, 작업, 수치값|**BDD** ✅|블록 내부 구획 상세 기술|
|라. 인스턴스 간 협력 구조|**IBD** ❌|Internal Block Diagram|
|마. 수학적 제약 관계|**Parametric Diagram** ❌|파라메트릭 다이어그램|

**핵심 구분:**

- BDD = 블록 **타입** 정의 **(클래스 다이어그램 같은 것)**
- IBD = 블록 **인스턴스** 간 연결 구조
- Parametric = 수식/수학적 제약

**[서비스 운영 조건]**

- 측정 기간: 1개월 (30일, 24시간 운영 기준)
- 서비스 수준 목표(SLO): 월간 가용성 99.9% 달성

- 나. 장애 A: DB 연결 오류로 인한 전면 중단 15분 발생
- 다. 장애 B: API 응답 지연으로 인한 부분 장애(가중치 50% 적용) 10분 발생

**풀이:**

```
30일 × 24시간 × 60분 = 43,200분
43,200 × (1 - 0.999) = 43,200 × 0.001 = 43.2분
```

```
가. 0분
나. 15분 (전면 중단 × 1.0)
다. 10분 × 0.5 = 5분 (부분 장애 × 가중치 50%)

사용한 시간 = 0 + 15 + 5 = 20분
```

**3단계: 남은 Error Budget**

```
43.2분 - 20분 = 23.2분
```

**정답: ② 23.2분**

|방법론|핵심 목적|
|---|---|
|ATAM|품질 속성 간 **트레이드오프** 분석|
|CBAM|**비용·편익** 기반 아키텍처 평가|
|SAAM|시나리오 기반 **변경가능성** 평가|
|ARID|특정 설계의 **적합성** 검토|
**ATAM(Architecture Tradeoff Analysis Method)에서 트레이드오프**

트레이드오프는 **하나의 아키텍처 결정이 두 개 이상의 품질속성에 동시에 영향을 미치는데, 한쪽을 올리면 다른 쪽이 내려가는 관계**를 말함.

**예시**

|아키텍처 결정|좋아지는 품질속성|나빠지는 품질속성|
|---|---|---|
|암호화 추가|보안성 ↑|성능 ↓|
|캐싱 적용|성능 ↑|일관성(정합성) ↓|
|컴포넌트 분리|변경용이성 ↑|성능 ↓ (네트워크 오버헤드)|
|이중화(Redundancy)|가용성 ↑|비용 ↓, 복잡성 ↑|

**ATAM에서 관련 개념 4가지**

|개념|설명|
|---|---|
|감도점 (Sensitivity Point)|하나의 품질속성에만 영향을 미치는 결정|
|트레이드오프점 (Tradeoff Point)|**두 개 이상** 품질속성에 상충되는 영향을 미치는 결정|
|위험 (Risk)|나쁜 결과를 초래할 수 있는 아키텍처 결정|
|비위험 (Non-risk)|좋은 결과가 예상되는 아키텍처 결정|

**핵심 구분**: 감도점은 "한 놈한테만 영향", 트레이드오프는 "여러 놈한테 상충되게 영향" 이게 시험 포인트임.

## 35번 풀이

**정답: ④ 시나리오 기법** ✅ (표시 확인)


### 핵심 개념 먼저

**BPMN**은 업무 프로세스의 **흐름(Flow)**을 시각적으로 모델링하는 표기법임. 이벤트 → 태스크 → 게이트웨이 → 이벤트로 이어지는 **업무 흐름도**.

### 선택지 판정 테이블

|번호|기법|판정|이유|
|---|---|---|---|
|①|설문 조사 기법|❌|광범위한 의견 수집용. 프로세스 흐름과 무관|
|②|사용자 관찰 기법|❌|현장 관찰로 암묵지 도출. BPMN 작성보다는 관찰 후 인터뷰 등으로 연결|
|③|프로토타이핑 기법|❌|시제품으로 피드백 수집. UI 위주, 프로세스 흐름 모델링과 직접 연관 없음|
|④|**시나리오 기법**|✅|**업무 시나리오(As-Is/To-Be 프로세스)를 BPMN으로 표현하며 요구사항 도출. 가장 자연스러운 조합**|


### 왜 ④인가

시나리오 기법 = **"누가, 언제, 무엇을, 어떤 순서로"** 하는지 기술하는 방법

BPMN = 바로 이 시나리오를 **풀(Pool)/레인(Lane)/태스크/게이트웨이**로 시각화하는 표기법

→ 시나리오를 작성하면서 동시에 BPMN 다이어그램으로 그리는 게 실무에서도 표준적인 업무 분석 방법임. 둘이 세트.

### 헷갈릴 수 있는 포인트

② **사용자 관찰**이 BPMN이랑 연결될 것 같지만, 관찰은 "데이터 수집 단계"이고 BPMN은 "모델링/표현 단계"라 직접적인 짝이 아님. 시나리오 기법은 처음부터 끝까지 BPMN과 함께 작성함.

![[Pasted image 20260404104402.png]]


**○ 이벤트** — 얇은 원=시작, 굵은 원=종료. 중간 이벤트(더블 링)도 있음.

**□ 태스크(Task)** — 실제 업무 활동. 사람이 하면 User Task, 시스템이 하면 Service Task.

**◇ 게이트웨이** — 흐름 분기/합류. X 안에 넣으면 배타적(XOR, 하나만 선택), + 이면 병렬(AND, 모두 실행). 

각 요소 눌러보면 설명 더 볼 수 있음. 시험에서 BPMN이 "시나리오 기법"과 짝꿍인 이유도 딱 저 다이어그램 보면 이해되지 — 업무 시나리오를 그대로 흐름으로 그린 거니까.

나선형 모델은 **대규모·고위험·장기 프로젝트**에 적합하고, ②처럼 "저비용·단기" 프로젝트에는 오히려 폭포수나 애자일이 더 맞음. 시험에서 나선형의 단점 포인트는 항상 **"관리 복잡, 비용 과다, 위험분석 전문가 필요"** 이 세 가지임.
