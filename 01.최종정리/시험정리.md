# **직무설계 4가지**

|구분|방향|핵심|예시|
|---|---|---|---|
|직무단순화 (Job Simplification)|축소|직무를 단순·반복 작업으로 분해, 전문화|컨베이어벨트 조립공|
|직무확대 (Job Enlargement)|수평적 확대|같은 수준의 다른 작업 추가, 단조로움 감소|조립만 하다가 검사도 추가|
|직무충실화 (Job Enrichment)|수직적 확대|책임·권한·자율성 부여, 내재적 동기부여|실행뿐 아니라 계획·평가권 부여|
|직무순환 (Job Rotation)|이동|여러 직무를 주기적으로 교체, 다기능화|영업→마케팅→기획 순환 근무|

**헷갈리는 구분 포인트**

|비교|직무확대|직무충실화|
|---|---|---|
|방향|수평 (옆으로)|수직 (위로)|
|추가되는 것|같은 수준의 일|책임·권한|
|동기부여 이론|해당 없음|**Herzberg 2요인**|
|효과|지루함 감소|성취감·보람|

**직무단순화 vs 직무확대**

직무단순화는 줄이는 것, 직무확대는 늘리는 것. 반대 방향이야.

# 📘 조직 구조 비교

| 구분          | 기능 조직 (Functional Organization) | 프로젝트 조직 (Projectized Organization) | 매트릭스 조직 (Matrix Organization) |
| ----------- | ------------------------------- | ---------------------------------- | ----------------------------- |
| 조직 구조       | 기존 부서 중심 (개발팀, 운영팀 등)           | 프로젝트 중심으로 독립 조직                    | 기능 + 프로젝트 혼합                  |
| 권한 중심       | 기능 관리자                          | 프로젝트 관리자                           | 기능 관리자 + 프로젝트 관리자             |

| 의사결정        | 기능 관리자                          | 프로젝트 관리자                           | 공동 결정                         |

| 커뮤니케이션      | 수직 구조                           | 프로젝트 중심                            | 복합 구조                         |
| 장점          | 전문성 유지                          | 의사결정 빠름                            | 자원 활용 효율                      |
| 단점          | 프로젝트 우선순위 낮음                    | 자원 중복 가능                           | 권한 충돌                         |

# 📘자원 조정 기법

| 기법                       | 특징                          |
| ------------------------ | --------------------------- |
| Resource Leveling 자원평준화  | 자원 초과 해결, 일정 늘어날 수 있음(자원고정) |
| Resource Smoothing 자원평활화 | 일정 유지, 자원만 조정 (일정고정)        |
### 1️) Resource Leveling (자원 평준화)

| 구분    | 내용                                              |
| ----- | ----------------------------------------------- |
| 정의    | **자원(Resource) 제약을 기준으로 일정(Schedule)을 조정하는 기법** |
| 목적    | 특정 시점에 **자원이 부족하거나 과다 배치된 문제 해결**               |
| 일정 변경 | **가능 (프로젝트 종료일이 늘어날 수 있음)**                     |
| 적용 대상 | 자원이 절대적으로 부족할 때                                 |
| 특징    | Critical Path(주경로)도 변경될 수 있음                    |
### 2) Resource Smoothing (자원 평활화)

| 구분    | 내용                                       |
| ----- | ---------------------------------------- |
| 정의    | **일정 여유(Float) 범위 내에서 자원을 균등하게 배치하는 기법** |
| 목적    | 자원 사용량의 **변동을 완화**                       |
| 일정 변경 | **불가 (프로젝트 종료일 유지)**                     |
| 적용 대상 | 일정 여유가 존재할 때                             |
| 특징    | Critical Path는 **절대 변경되지 않음**            |
## 📘 ==일정== 단축 기법 (무조건 일정 단축해야함)

| 기법            | 핵심 개념       |
| ------------- | ----------- |
| Crashing      | 비용 ↑ → 기간 ↓ |
| Fast Tracking | 순차 작업 병렬 수행 |

| 구분    | Crashing                  | Fast Tracking             |
| ----- | ------------------------- | ------------------------- |
| 개념    | **추가 자원을 투입하여 일정 ==압축==** | **순차 작업을 병렬로 수행하여 일정 단축** |
| 방법    | 인력·장비·비용 증가               | 작업을 겹쳐서 수행                |
| 비용    | **증가**                    | 증가하지 않을 수도 있음             |
| 위험    | 비교적 낮음                    | **재작업 위험 증가**             |
| 적용 대상 | Critical Path 작업          | 순차 의존성이 있는 작업             |
| 핵심 특징 | **==돈으로 시간 압축==**         | ==**동시 수행으로 시간 단축**==     |

# 📘  버지니아 변화 모델 (Virginia Satir Change Model)

| 단계         | 영문                             | 핵심 특징                                                 | 성과 수준 |
| ---------- | ------------------------------ | ----------------------------------------------------- | ----- |
| 1. 현재 상태   | Late Status Quo(변화전 상태)        | 익숙하고 예측 가능, 안정적. 규칙과 관행이 지배                           | 안정적   |
| 2. 외부 요소   | Foreign Element<br>(신규시스템도입시점) | 변화 촉발 사건 발생. **저항**·부정·회피 시작. 성과 하락 시작.Resistance(저항) | 저하 시작 |
| 3. 혼돈      | Chaos                          | 기존 방식 붕괴, 방향 상실. 불안·혼란 극대화. 역설적으로 창의적 아이디어 폭발         | 최저    |
| 4. 변환 아이디어 | Transforming Idea              | 외부 요소를 이해하는 돌파구 발견. 흥분과 희망 생김                         | 반등 시작 |
| 5. 실무 및 통합 | Practice & Integration         | 새 방식 적용·연습. 성과 빠르게 회복                                 | 회복·상승 |
| 6. 새로운 상태  | New Status Quo                 | 새 방식이 일상화. 성과 기존 이상                                   | 기존 이상 |

# 📘 시각 데이터

| 구분                                  | 설명                                                                                                                          |
| ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| 사이클 타임 차트                           | 이 도표는 시간 경과에 따른 작업 항목의 평균 사이클 타임을 보여준다. 사이클 타임 차트는 산점도 또는 막대차트로 표시할 수 있다.                                                   |

#  📘 컴퓨터 시스템 보안 설계 원칙 (표 정리)

| 구분                                    | 설명                                                                         |
| ------------------------------------- | -------------------------------------------------------------------------- |
| 최소 권한 (Least Privilege)               | 사용자는 수행해야 할 작업에 필요한 최소한의 권한만 가져야 하며, 더 이상 필요하지 않은 경우 해당 권한은 회수되어야 한다.      |
| 고장 안전 디폴트 (Fail-Safe Defaults)        | 명시적으로 허가되지 않은 모든 자원에 대한 접근은 기본적으로 거부되어야 한다.                                |
| 매커니즘의 효율성 (Economy of Mechanism)      | 시스템 설계는 가능한 한 **단순하게** 구성되어야 하며, **복잡성은 오류 및 취약점 발생 가능성을 증가**시킨다.          |
| 완벽한 조율 (Complete Mediation)           | 모든 접근 시도는 반드시 검사되어야 하며, 접근 시마다 권한 검증이 이루어져야 한다.                            |
| 열린 설계 (Open Design)                   | 보호 메커니즘은 공격자의 무지에 의존하지 않아야 하며, **공개되고 검증된 방식으로 설계**되어야 한다.                 |
| 권한 분리 (Separation of Privilege)       | 시스템은 **단일 조건에 의해 권한을 부여해서는 안 되며, 다중 조건을 만족해야 권한이 부여**되어야 한다.               |
| 최소 공통 메커니즘 (Least Common Mechanism)   | **공유 메커니즘의 수와 사용을 최소화**해야 하며, 공유 객체는 정보 흐름 및 의도되지 않은 상호작용에 대한 잠재적 위험을 가진다. |
| 사용의 편리성 (Psychological Acceptability) | 사용자가 일반적으로 시스템을 사용하는 과정에**서 자연스럽게 보안 메커니즘**이 적용되도록 설계되어야 한다.               |

# 📘 정성적 위험분석 vs 정량적 위험분석

## 1️⃣ 구분

| 구분     | 정성적 위험분석                        | 정량적 위험분석                                           |
| ------ | ------------------------------- | -------------------------------------------------- |
| 개념     | 전문가 판단·경험 기반으로 위험 수준을 상대적으로 평가  | 수치·금액 기반으로 위험을 계량화하여 평가                            |
| 표현     | 높음/중간/낮음 등 등급                   | 금액, 확률, 기대손실 등 수치                                  |
| 목적     | 위험 우선순위 결정                      | 비용-효과 분석, 투자 의사결정                                  |
| 정확도    | 상대적으로 낮음                        | 상대적으로 높음                                           |
| 속도     | 빠름                              | 느림 (데이터 필요)                                        |
| 데이터 요구 | 적음                              | 많음                                                 |
|        | 델파이법, 시나리오법, 순위결정법, 퍼지행렬법, 질문서법 | 수학공식 접근법, 확률 분포 추정법, 확 률지배, 몬테카를로 시뮬레이션, 과거자 료 분석법 |

# 📘 생체인증의 조건

|특성|설명|
|---|---|
|보편성|모든 사람이 가지고 있는 보편적 특징을 대상으로 해야 한다.|
|영구성|특징은 변하지 않아야 하며, 시간이 지나도 변형되지 않아야 한다.|
|독특성|동일한 특징을 가진 다른 사람이 존재하지 않아야 한다.|
|획득성|센서를 통해 쉽게 획득할 수 있고, 정량화가 가능해야 한다.|
|시스템 성능|처리 속도가 빠르고 처리량이 커야 하며, 알고리즘 및 시스템 구성의 효율성이 확보되어야 한다.|
|저항성|위조 및 변조가 불가능하거나 매우 어려워야 한다.|
|수용성|사용자가 인증 시스템 사용에 거부감을 느끼지 않아야 한다.|

| 메시지 압축 (Message Digest) | 임의의 길이의 평문을 고정된 길이의 출력값으로 변환해야 한다.                                       |
| 계산의 용이성                 | 입력값 x가 주어지면 H(x)는 계산적으로 쉽게 구할 수 있어야 한다.                                  |
| 일방향성 (One-Way)          | 메시지(평문)에서 해시값을 구하는 것은 쉽지만, 해시값에서 원래 메시지를 구하는 것은 매우 어려워야 한다. (역상 저항성)     |
| 약한 충돌 회피성               | 입력값 x가 주어졌을 때 H(x') = H(x)인 x' (x ≠ x)를 찾는 것은 계산적으로 불가능해야 한다. (제2역상 저항성) |
| 강한 충돌 회피성               | H(x) = H(x')인 서로 다른 두 입력값 x, x'를 찾는 것은 계산적으로 불가능해야 한다. (충돌 저항성)          |

# 📘 디피-헬만 (Diffie-Hellman) 키 교환

## 1️⃣ 개념

|구분|내용|
|---|---|
|목적|공개된 채널에서 안전하게 **공유 비밀키 생성**|
|방식|이산로그 문제 기반|
|특징|중간에서 값이 보여도 **비밀키는 계산 불가**|

## 2️⃣ 계산 절차

|단계|설명|
|---|---|
|공통값 설정|큰 소수 p, 생성자 g 공개|
|A 선택|사용자 A가 비밀값 a 선택|
|B 선택|사용자 B가 비밀값 b 선택|
|A 공개값|A = g^a mod p|
|B 공개값|B = g^b mod p|
|A 계산|K = B^a mod p|
|B 계산|K = A^b mod p|

## 1️⃣ Feistel 구조 vs SPN 구조

|구분|Feistel 구조|SPN 구조|
|---|---|---|
|구조 방식|평문을 좌(L0), 우(R0)로 나누어 처리 후 교차|Substitution Layer + Permutation Layer 반복|
|연산 방식|함수 F 적용 후 XOR 연산으로 좌우 교환|S-box 치환 + P-box 순열|
|Confusion / Diffusion|동일 수준 확보 위해 SPN보다 더 많은 반복 필요|적은 반복으로 Confusion / Diffusion 달성|
|병렬 처리|어려움|가능|
|복호화|암호화 구조와 동일 (복호기 별도 불필요)|복호화 시 별도 역연산 필요|
|특징|구조 단순, 구현 용이|구조 복잡, 성능 우수|
## 2️⃣ 알고리즘 예시

| 구분      | 알고리즘                                         |
| ------- | -------------------------------------------- |
| Feistel | DES, SEED, CAST, Blowfish, Twofish, RC5, RC6 |
| SPN     | IDEA, Rijndael(AES), ARIA                    |

## 3️⃣ 핵심 차이 요약

|구분|내용|
|---|---|
|Feistel|XOR + 좌우 교환 구조, 복호화 동일|
|SPN|치환 + 순열 구조, 복호화 별도 필요|

| ---- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| SSID | Service Set Identifier  <br>무선랜에서 네트워크 이름을 의미하며 기본적인 접속 제어 기능 제공  <br>AP가 비콘 신호에 SSID를 포함하여 브로드캐스트  <br>SSID만으로는 보안 기능이 약하며 접속 허용/차단 기준으로는 취약 |

# 📘 SSL 프로토콜 구성

| 구성요소                      | 기능                                                                                                             |
| ------------------------- | -------------------------------------------------------------------------------------------------------------- |
| Record Layer Protocol     | 분배된 대칭키를 이용하여 데이터 암호화 수행  <br>TCP 위에서 동작하며 Handshake에서 협의된 알고리즘 사용  <br>데이터를 적절한 크기로 분할 → 압축 → 암호화 → 무결성 검증 수행 |

| Alert Protocol            | 오류 및 경고 상황을 상대방에게 알림  <br>SSL 통신 중 문제가 발생했을 때 해당 내용을 전달                                                        |

👉 **Handshake > ChangeCipherSpec** > Record

# 📘 침입탐지시스템 (IDS) – 네트워크 기반 vs 호스트 기반

| 구분    | 네트워크 기반 IDS (NIDS)                  | 호스트 기반 IDS (HIDS)                 |
| ----- | ----------------------------------- | --------------------------------- |
| 설치 위치 | 네트워크 구간 (스위치, 라우터 등)                | 개별 서버/PC 내부                       |

| 성능 영향 | 네트워크 장비 성능 영향                       | 호스트 자원 사용                         |

# 📘 OWASP LLM Top 10 (2025) — 영문 / 한글 / 개념

## 1️⃣ LLM01: Prompt Injection (프롬프트 인젝션)

- **정의:** 사용자 입력으로 시스템 프롬프트/정책을 우회하거나 모델 동작을 변경하는 공격
    
- **핵심:** 지시 탈취 / 정책 무력화
    

## 2️⃣ LLM02: Sensitive Information Disclosure (민감 정보 노출)

- **정의:** 모델이 학습 데이터, 프롬프트, API 키 등 민감 정보를 출력하는 문제
    
- **핵심:** 데이터 유출
    

## 3️⃣ LLM03: Supply Chain (공급망 취약점)

- **정의:** 외부 모델, 데이터셋, 라이브러리 등 의존 요소를 통한 공격
    
- **핵심:** 외부 의존 위험
    

## 4️⃣ LLM04: Data and Model Poisoning (데이터 및 모델 오염)

- **정의:** 학습 데이터, 파인튜닝, RAG 등을 통해 모델을 악의적으로 오염
    
- **핵심:** 응답 왜곡 / 백도어
    

## 5️⃣ LLM05: Improper Output Handling (부적절한 출력 처리)

- **정의:** LLM 출력값을 검증 없이 실행/처리하는 문제
    
- **핵심:** 출력 → 코드/명령 실행 위험
    

## 6️⃣ LLM06: Excessive Agency (과도한 권한 부여)

- **정의:** LLM에 시스템 제어, 결제, 실행 권한 등을 과도하게 부여
    
- **핵심:** 자동 실행 위험
    

## 7️⃣ LLM07: System Prompt Leakage (시스템 프롬프트 유출)

- **정의:** 내부 시스템 프롬프트 또는 정책이 외부로 노출되는 문제
    
- **핵심:** 내부 정책 노출
    

## 8️⃣ LLM08: Vector and Embedding Weaknesses (벡터/임베딩 취약점)

- **정의:** 벡터DB, 임베딩, RAG 과정에서 발생하는 보안 취약점
    
- **핵심:** 검색 기반 데이터 오염/유출
    

## 9️⃣ LLM09: Misinformation (허위 정보)

- **정의:** 모델이 부정확하거나 잘못된 정보를 생성하고 이를 신뢰하는 문제
    
- **핵심:** 환각(Hallucination) + 잘못된 판단 유도
    

## 🔟 LLM10: Unbounded Consumption (무제한 자원 소비)

- **정의:** 과도한 요청/토큰 사용으로 비용 및 자원을 무한 소비하게 하는 공격
    
- **핵심:** 비용 폭증 / 자원 고갈

## 1️⃣ A01: Broken Access Control (접근 통제 실패)

- **정의:** 사용자가 권한을 넘어 데이터나 기능에 접근하는 문제
    
- **핵심:** 권한 검증 실패
    

## 2️⃣ A02: Security Misconfiguration (보안 설정 오류)

- **정의:** 잘못된 설정, 기본값 사용, 불필요한 서비스 노출
    
- **핵심:** 설정 실수
    

## 3️⃣ A03: Software Supply Chain Failures (소프트웨어 공급망 실패)

- **정의:** 외부 라이브러리, 의존성, 빌드/배포 과정에서 발생하는 취약점
    
- **핵심:** 공급망 공격
    

## 4️⃣ A04: Cryptographic Failures (암호화 실패)

- **정의:** 민감 데이터 보호를 위한 암호화가 적절히 적용되지 않은 경우
    
- **핵심:** 데이터 보호 실패
    

## 5️⃣ A05: Injection (인젝션)

- **정의:** 사용자 입력이 코드나 쿼리로 실행되는 취약점
    
- **핵심:** SQL, OS, LDAP Injection
    

## 6️⃣ A06: Insecure Design (취약한 설계)

- **정의:** 설계 단계에서 보안이 고려되지 않아 발생하는 구조적 문제
    
- **핵심:** 설계 결함
    

## 7️⃣ A07: Authentication Failures (인증 실패)

- **정의:** 인증 및 세션 관리의 취약점으로 계정 탈취 가능
    
- **핵심:** 인증 취약
    

## 8️⃣ A08: Software or Data Integrity Failures (무결성 실패)

- **정의:** 코드 또는 데이터의 변조를 방지하지 못하는 문제
    
- **핵심:** 무결성 훼손
    

## 9️⃣ A09: Security Logging & Alerting Failures (로깅 및 알림 실패)

- **정의:** 공격 탐지 및 대응을 위한 로그와 모니터링 부족
    
- **핵심:** 탐지 실패
    

## 🔟 A10: Mishandling of Exceptional Conditions (예외 처리 실패)

- **정의:** 오류나 예외 상황을 적절히 처리하지 못해 발생하는 취약점
    
- **핵심:** 에러 처리 취약

# 📘 네트워크 필수 명령어 세트

## 1️⃣ ping

- 목적: **연결 여부 확인**
    
- 방식: ICMP Echo Request/Reply
    
- 핵심: “통신 되냐 안되냐”
    

## 2️⃣ tracert

- 목적: **경로 확인**
    
- 방식: TTL + ICMP
    
- 핵심: “어디서 끊기냐”
    

## 3️⃣ ipconfig

- 목적: **내 PC 네트워크 정보 확인**
    
- 확인 가능:
    
    - IP 주소
        
    - 서브넷 마스크
        
    - 게이트웨이
        
    - DNS
        

👉 `/all` 옵션 자주 나옴

## 4️⃣ nslookup

- 목적: **도메인 → IP 변환 확인**
    
- 핵심: DNS 조회
    

👉 “DNS 문제냐?” 판단할 때 사용

## 5️⃣ netstat

- 목적: **현재 연결 상태 확인**
    
- 확인:
    
    - 열린 포트
        
    - 연결 상태 (ESTABLISHED 등)
        

👉 `/an` 옵션 자주 나옴

# 📘 인공지능 ISO/IEC 42001 vs ISO/IEC 23894

| 구분     | ISO/IEC 42001 (AIMS)                                   | ISO/IEC 23894 (Risk Management)                        |
| ------ | ------------------------------------------------------ | ------------------------------------------------------ |
| 정의     | 인공지능 경영시스템 (Artificial Intelligence Management System) | 인공지능 위험 관리 (Artificial Intelligence - Risk Management) |
| 성격     | 요구사항(Requirements) 기반 표준                               | 지침(Guidelines) 기반 표준                                   |
| 인증 가능성 | 인증 가능 (Certifiable)                                    | 인증 불가 (가이드로 활용)                                        |
| 주요 목적  | 조직의 AI 거버넌스 체계 구축 및 대외 신뢰성 증명                          | AI 수명 주기 전반의 위험 관리 프로세스 구체화                            |
| 기반 표준  | ISO/IEC 27001 (정보보호 경영시스템)                             | ISO 31000 (위험 관리 가이드라인)                                |
| 핵심 범위  | 정책, 리더십, 자원 관리, 성과 평가 등 조직 전체                          | 위험 식별, 분석, 평가, 처리 등 리스크 관리 실무                          |
| 적용 시점  | 조직의 AI 도입 및 운영 전략 수립 시                                 | AI 개발, 배포, 운영 등 실무 프로세스 단계                             |
| 비유     | 건축물의 전체 설계도 및 관리 규정                                    | 안전한 건축을 위한 상세 안전 시공 매뉴얼                                |

# **3단계 스키마 아키텍처**

| 구분     | 내용                                        |
| ------ | ----------------------------------------- |
| 외부 스키마 | 사용자/응용 프로그램이 보는 뷰. 여러 개 존재 가능             |
| 개념 스키마 | DB 전체 논리 구조. 1개만 존재                       |
| 내부 스키마 | 물리적 저장 구조 기술 (레코드 형식, 인덱스, 접근 경로). 1개만 존재 |


**매핑:**

외부 ↔ 개념 = **논리적 데이터 독립성** 개념 ↔ 내부 = **물리적 데이터 독립성**

**핵심 함정:** 내부 스키마 ≠ 물리 장치. 시험에서 이걸 혼동하게 만드는 선택지 자주 나옴.

# 📘 라우팅 프로토콜 종류

라우팅 프로토콜은 크게 두 가지 유형으로 나뉜다:

1. **내부 라우팅 프로토콜 (Interior Gateway Protocol, IGP)**

2. **외부 라우팅 프로토콜 (Exterior Gateway Protocol, EGP)**

### 1️⃣ **내부 라우팅 프로토콜 (IGP)**

- 같은 **자율 시스템(AS)** 내에서 라우팅을 처리하는 프로토콜
- **라우팅 정보**를 **자기 네트워크 안**에서만 교환
###### AS (Autonomous System) = 자율 시스템(**"하나의 관리 주체가 운영하는 라우터 집합")**

| 프로토콜                                                   | 설명             | 특징                           | 예시                   |
| ------------------------------------------------------ | -------------- | ---------------------------- | -------------------- |
| **RIP (Routing Information Protocol)**                 | 거리 벡터 라우팅 프로토콜 | 홉 수(hop count) 기반, 최대 홉 수 15 | 작은 네트워크에서 사용         |
| **OSPF (Open Shortest Path First)**                    | 링크 상태 라우팅 프로토콜 | Dijkstra 알고리즘 사용, 최단 경로 계산   | 대규모 네트워크에서 사용        |
| **EIGRP (Enhanced Interior Gateway Routing Protocol)** | 하이브리드 라우팅 프로토콜 | 거리 벡터 + 링크 상태, 빠른 수렴         | Cisco 네트워크 장비에서 사용   |
| **IS-IS (Intermediate System to Intermediate System)** | 링크 상태 라우팅 프로토콜 | OSPF와 유사, 대규모 네트워크에서 사용      | 대형 ISP와 기업 네트워크에서 사용 |

### 2️⃣ **외부 라우팅 프로토콜 (EGP)**

- **다른 자율 시스템(AS)** 간의 라우팅을 처리하는 프로토콜

|프로토콜|설명|특징|예시|
|---|---|---|---|
|**BGP (Border Gateway Protocol)**|경로 벡터 라우팅 프로토콜|AS 간 라우팅, 경로 선택 기준으로 AS 경로, 정책 기반|인터넷 상의 라우팅에 사용|

## 🔥 각 라우팅 프로토콜의 특징

### **RIP (Routing Information Protocol)**

- **거리 벡터** 프로토콜로, **최대 15 홉까지** 라우팅 가능.

- 가장 오래된 라우팅 프로토콜이지만, 작은 네트워크에서 유용하다.

- **단점**: 느리고, 비효율적인 경로 계산.

### **OSPF (Open Shortest Path First)**

- **링크 상태** 프로토콜로, **최단 경로 우선(Dijkstra)** 알고리즘을 사용.

- 대규모 네트워크에서 효율적이며, **빠르고 정확한 수렴**이 가능.

- **단점**: 설정과 관리가 복잡할 수 있음.

### **EIGRP (Enhanced Interior Gateway Routing Protocol)**

- **하이브리드** 프로토콜로, **거리 벡터 + 링크 상태**를 결합한 방식.

- **빠른 수렴**과 **효율적인 경로 계산**이 특징.

- **Cisco 장비에서만** 사용 가능.

### **BGP (Border Gateway Protocol)**

- **경로 벡터** 프로토콜로, **자율 시스템(AS)** 간의 경로 정보를 교환.

- **인터넷**에서 주로 사용되며, **정책 기반 라우팅**을 지원.

- **단점**: 설정이 복잡하고, **수렴 속도**가 느림.

## 🔥 라우팅 프로토콜 비교

|특성|RIP|OSPF|EIGRP|BGP|
|---|---|---|---|---|
|유형|거리 벡터|링크 상태|하이브리드|경로 벡터|
|수렴 속도|느림|빠름|빠름|느림|
|확장성|낮음|높음|높음|매우 높음|
|복잡성|낮음|중간|중간|높음|
|네트워크 크기|소규모|대규모|중대규모|대규모|

# 📘 HBM vs HBF (핵심 비교)


■ HBM (High Bandwidth Memory)

- 정의: 3D 적층 구조 기반 고대역폭 DRAM
- 특징: TSV 적층 / 초고속 / 저전력
- 용도: GPU, AI 연산, HPC
- 핵심: 속도 중심 메모리

■ HBF (High Bandwidth Flash)

- 정의: 고대역폭 NAND Flash (HBM 구조 응용)
- 특징: 대용량 / HBM 대비 저속 / 적층 구조
- 용도: AI 저장/추론 보조
- 상태: 초기/개발 단계

| 항목     | HBM       | HBF        |
| ------ | --------- | ---------- |
| 메모리 종류 | DRAM      | NAND Flash |
| 속도     | 매우 빠름     | 상대적으로 느림   |
| 용량     | 작음        | 매우 큼       |
| 용도     | 연산용 (GPU) | 저장/추론 보조   |
| 상태     | 상용화       | 개발/초기 단계   |

# 📘 IOT 통신 프로토콜 CoAP / MQTT / XMPP

■ CoAP (Constrained Application Protocol)

- 구조: REST 기반 요청/응답
    
- 전송: UDP
    
- 특징: HTTP 경량화, 저전력, 저사양 디바이스
    
- 메시지: Confirmable / Non-confirmable
    
- 신뢰성: 낮음 (재전송으로 보완)
    
- 용도: 센서, IoT 디바이스
    

■ MQTT (Message Queuing Telemetry Transport)

- 구조: Pub/Sub (브로커 중심)
    
- 전송: TCP
    
- 특징: 초경량, 저전력, 연결 유지
    
- 구성: Publisher / Broker / Subscriber
    
- QoS: 0 (최대 1회), 1 (최소 1회), 2 (정확히 1회)
    
- 용도: IoT, 모바일, 실시간 데이터
    

■ XMPP (Extensible Messaging and Presence Protocol)

- 구조: 메시지 스트림 (클라이언트-서버)
    
- 전송: TCP
    
- 특징: XML 기반, 확장성 높음
    
- 통신: Message / Presence / IQ
    
- 확장: Pub/Sub 가능 (XEP-0060)
    
- 용도: 채팅, IoT, 메시징 시스템
    

■ 핵심 비교

| 항목  | CoAP         | MQTT                | XMPP   |
| --- | ------------ | ------------------- | ------ |
| 구조  | REST         | Pub/Sub             | 메시지    |
| 전송  | UDP          | TCP                 | TCP    |
| 특징  | HTTP 유사      | 브로커 기반              | XML 확장 |
| 신뢰성 | 낮음           | QoS 지원              | 높음     |
| 적합  | 초저전력         | IoT 실시간             | 메시징/확장 |
| 방식  | 요청/응답 (REST) | **발행/구독** (Pub/Sub) | 메시지 교환 |

**한 줄 암기**

- **CoAP** = IoT용 경량 HTTP (UDP)
- **MQTT** = IoT 메시지 발행/구독 (브로커 필요)
- **XMPP** = XML 기반 메시지 (채팅 앱 기원)

시험에서 **MQTT 발행/구독 구조** 가 제일 자주 나와.

■ 시험 포인트

- CoAP → UDP + REST
    
- MQTT → 브로커 + QoS
    
- XMPP → XML + 확장성 + IQ
    

■ 한줄  
CoAP=UDP REST / MQTT=브로커 PubSub / XMPP=XML 메시징 + 확장

# 📘 TCP 연결/종료 절차

■ 3-Way Handshake (연결 설정)

| 단계  | 송신자 → 수신자  | 내용                |
| --- | ---------- | ----------------- |
| 1   | 클라이언트 → 서버 | SYN (연결 요청)       |
| 2   | 클라이언트 ← 서버 | SYN + ACK (요청 수락) |
| 3   | 클라이언트 → 서버 | ACK (확인)          |

■ 핵심

- 연결을 “확립”하는 과정
    
- 양쪽 모두 송수신 준비 상태 확인
    

■ 한줄  
SYN → SYN/ACK → ACK

■ 4-Way Handshake (연결 종료)

| 단계  | 송신자 → 수신자  | 내용          |
| --- | ---------- | ----------- |
| 1   | 클라이언트 → 서버 | FIN (종료 요청) |
| 2   | 클라이언트 ← 서버 | ACK (확인)    |
| 3   | 클라이언트 ← 서버 | FIN (종료 요청) |
| 4   | 클라이언트 → 서버 | ACK (확인)    |

■ 핵심

- 연결을 “종료”하는 과정
    
- 송신/수신이 각각 독립적으로 종료됨 (Full Duplex)
    

■ 한줄  
FIN → ACK → FIN → ACK

📌 시험 포인트

|구분|3-way|4-way|
|---|---|---|
|목적|연결 설정|연결 종료|
|특징|동시 확인|순차 종료|
|이유|상태 동기화|양방향 독립 종료|

📌 핵심 한줄  
연결은 3번 / 종료는 4번

### CPU 스케줄링

**스케줄링 목표**

|목표|설명|
|---|---|
|공평성 vs 우선순위|가장 중요한 상반된 두 목표|
|CPU 유휴시간 최소화|처리량 극대화|
|응답시간 보장|신속한 response time|
|반환시간 단축|제출~종료 경과시간(turn around time)|
|기아 방지|에이징(aging) 기법 사용|

**비선점 vs 선점**

|구분|비선점|선점|
|---|---|---|
|CPU 강제회수|불가|가능|
|전환 조건|프로세스 종료 또는 자발적 반납|시간할당 초과, 높은 우선순위 프로세스 등장|
|오버헤드|적음|많음|
|적합|일괄처리 시스템|시분할/대화식 시스템|

| ------------- | -------------------- | --------------- |
| FCFS          | FIFO                 | 가장 단순           |

| HRN           | 응답률 가장 높은 것 먼저       | SJF 기아현상 보완     |

# 📘  TCP 혼잡제어 알고리즘 정리



### 1. AIMD (Additive Increase / Multiplicative Decrease)

- 혼잡 윈도우(cwnd) 크기를 늘릴 때는 **1씩 선형 증가** (1 → 2 → 3 → 4 → ...)
- 혼잡이 감지되면 윈도우 크기를 **절반으로 감소**
- 늘어날 때는 **+1**, 줄어들 때는 **×1/2**
- **공평한 방식** — 네트워크에 늦게 참여한 노드도 시간이 지나면 네트워크 평균에 수렴
- **단점:** 윈도우 크기를 올리는 속도가 너무 느림

### 2. 느린 출발 (Slow Start)

- cwnd 값이 **1에서 시작**하여 **2, 4, 8…** 순으로 **지수적으로 증가**
- 한계치(ssthresh)에 도달할 때까지 지수 증가, 혼잡 감지 시 윈도우 크기를 **1로 감소**
- AIMD의 단점(선형 증가로 인한 속도 지연)을 개선

### 3. TCP 혼잡제어 정책

|정책|설명|
|---|---|
|**Tahoe**|가장 기본. Slow Start → 혼잡 감지 시 AIMD(혼잡 회피) 방식으로 전환|
|Reno / New Reno / Cubic|Tahoe를 발전시킨 정책들|

- 공통 전제: **혼잡 발생 시 윈도우 크기를 줄여 혼잡을 회피**

### 4. Tahoe & Reno정책 동작 흐름

```
cwnd = 1 (시작)
  ↓
[Slow Start 구간] 지수 증가
  ↓
ssthresh 도달
  ↓
[Congestion Avoidance 구간] 선형 증가 (AIMD)
  ↓
혼잡 감지 → cwnd = 1, ssthresh = W(loss) / 2 로 재설정
  ↓
다시 Slow Start부터 반복
```


### 5. 혼잡 감지 조건

- ==**타임아웃** 발생  -> Tahoe,Reno : cwnd 1로 감소== 
- ==**3번의 중복 ACK** 수신  -> Tahoe : cwnd 1로 감소, **Reno : cwnd 1/2로 감소 (임계값이랑 동일해지기때문에 SlowStart 뛰어넘음 왜냐 ssthresh = cwnd/2로 동일함)==**


### 6. Slow Start 임계점 (ssthresh)

- **의미:** 해당 지점까지만 Slow Start를 사용하겠다는 기준값
- **설정 방법:** 송신측이 ssthresh 값을 **혼잡 윈도우 크기의 절반**으로 정함)

## 📘  블루투스 / BLE / 지그비 / 비콘


### 블루투스 (Bluetooth Classic)

- 표준: IEEE 802.15.1
- 주파수: 2.4GHz ISM 대역
- 거리: 일반적으로 **수 미터 ~ 수십 미터** (출력에 따라 다름)
- 속도: 최대 3Mbps (EDR 기준)
- 토폴로지: 피코넷(Piconet) — 마스터 1 + 슬레이브 최대 7 
- **스캐터넷 (Scatternet)**블루투스서 피코넷 여러 개 연결한 거.
- 용도: 헤드셋, 키보드, 마우스, 스피커

### BLE (Bluetooth Low Energy)

- 블루투스 4.0부터 도입
- 속도: 규격상 1Mbps, 실효 처리량은 그보다 낮음
- 거리: 일반적으로 **~10m**, 블루투스 5 기준 저전력 모드에서 ~40m
- Classic 대비 **전력 소모 대폭 감소**, 간헐적 소량 데이터 전송에 최적
- 용도: 스마트워치, 헬스케어 센서, 비콘

### 지그비 (ZigBee)

- 표준: IEEE 802.15.4 기반
- 주파수: 2.4GHz
- 거리: **최대 100m** (직접 시퀀스 확산 스펙트럼 방식)
- 속도: **최대 250kbps**
- 특징: **초저전력, 메시(Mesh)** / 스타 / 트리 토폴로지 지원
- 용도: 스마트홈 센서, 공장 자동화, 스마트 조명

### 비콘 (Beacon)

- BLE 기반 **일방향 신호 브로드캐스트**
- 대표 규격: iBeacon(Apple), Eddystone(Google)
- 거리: **~5m~50m (안정적 거리 20~30m, 최대 70m~100m)**
- 용도: 실내 측위, 근거리 마케팅
- 특징: 송신만 하고 응답 안 받음 → 전력 소모 극히 낮음

|---|---|---|---|---|

## SDN (Software Defined Networking)

### 한 줄 정의

**네트워크 제어(Control)와 데이터 전달(Data)을 분리**해서 소프트웨어로 네트워크를 중앙 집중 관리하는 기술


### 기존 네트워크 vs SDN

| 구분      | 기존 네트워크   | SDN       |
| ------- | --------- | --------- |
| **제어 위치**   | **각 장비에 분산**  | **중앙 컨트롤러**   |
| 설정 방식   | 장비별 개별 설정 | 소프트웨어로 일괄 |
| 유연성     | 낮음        | 높음        |
| 대표 프로토콜 | OSPF, BGP | OpenFlow  |

==**컨트롤 플랜(제어기)과 데이터 플랜(하드웨어 스위치)을 나누자 컨트롤플랜을 장비마다가 아닌 1개만 중앙에서 관리하여 모든 데이터플랜을 제어하자!**==

SDN에서 데이터 플레인 장비를 **스위치**라고 부르는 이유는

기존 라우터는 **스스로 경로 판단** (3계층 라우팅)을 했는데, SDN에서는 그 판단 기능을 컨트롤러가 다 가져가버리니까 남은 장비는 그냥 **패킷 전달만 하는 스위치 역할**만 하게 되는 거야.

즉

- **기존 라우터** = 경로판단 + 패킷전달 (둘 다)
- **SDN 데이터플레인 장비** = 패킷전달만 (스위치처럼)

### 3계층 구조

```
┌─────────────────────────┐
│     Application Layer    │  ← 네트워크 앱 (방화벽, 로드밸런서 등)
├─────────────────────────┤
│      Control Layer       │  ← SDN 컨트롤러 (뇌)
│   (SDN Controller)       │    경로 계산, 정책 결정
├─────────────────────────┤
│  Infrastructure Layer    │  ← 스위치·라우터 (근육)
│  (Data Plane)            │    단순 패킷 전달만 수행
└─────────────────────────┘
```

**계층 간 통신** 거야.

```
[Application Layer]  ← 앱들 (방화벽, 로드밸런서 등)
        ↕  Northbound API  (컨트롤러 ↔ 앱)
[Control Layer (= Network Control Layer)]      ← SDN 컨트롤러
        ↕  Southbound API  (컨트롤러 ↔ 스위치)
[Data Layer (=Infrastructure Layer)]         ← 스위치들
```

- 컨트롤러 ↔ 앱 : **Northbound API**
- 컨트롤러 ↔ 장비 : **Southbound API (OpenFlow)**


### 핵심 개념

**Control Plane (제어 평면)**

- 경로 결정, 라우팅 테이블 계산
- SDN에서는 컨트롤러가 담당

**Data Plane (데이터 평면)**

- 실제 패킷 전달
- SDN에서는 스위치가 담당
- 컨트롤러 지시에 따라 **Flow Table** 기반으로 동작

**OpenFlow**

- 컨트롤러 ↔ 스위치 간 표준 프로토콜
- 컨트롤러가 스위치의 Flow Table을 직접 제어


### 시험 포인트

- **핵심 = Control/Data Plane 분리**
- **컨트롤러 = 중앙집중식 뇌** (OSPF처럼 분산 아님)
- **OpenFlow = Southbound API 대표**
- **장점** : 유연성↑, 중앙관리, 프로그래밍 가능
- **단점** : 컨트롤러 단일장애점(SPOF), 컨트롤러 과부하 위험

| 구분      | 기술                | 핵심                    |
| ------- | ----------------- | --------------------- |
| 가상화     | NFV, 하이퍼바이저, 컨테이너 | 물리 자원을 논리적으로 추상화      |
| 제어방식 변경 | **SDN**           | Control/Data Plane 분리 |


근데 시험에서 SDN이 가상화 챕터에 같이 묶여서 나오는 이유는 **"네트워크 자원을 소프트웨어로 유연하게 다룬다"** 는 맥락에서 NFV랑 함께 묶이기 때문이야.

**==NFV(네트워크 기능 가상화)랑 SDN은 자주 같이 나오는데**==

- ==**SDN = 네트워크 제어를 소프트웨어로**==
- ==**NFV = 네트워크 장비를 소프트웨어==**

## OpenFlow

**정의:** 제어 평면과 데이터 평면을 분리하고 이들을 연결하는 표준 인터페이스

**동작 방식:** 컨트롤러가 스위치에 명령 → 스위치는 명령에 따라 패킷을 전송·수정·폐기. 스위치에 등록된 플로우 엔트리 없는 패킷은 컨트롤러에 문의 후 처리

### Flow 테이블 구조

|구성|설명|
|---|---|
|**조건 (Match fields)**|처리할 조건. 물리적 포트번호 ~ L4 TCP/UDP 포트까지. 이 정보를 **튜플(tuple)** 이라고 부름|
|**처리 (Actions)**|전송·폐기·지정 필드값 수정 등 처리 방법|
|**통계 (Counters)**|플로우 엔트리별 송수신 트래픽 양·패킷 수·바이트 수. 경로 우회 등에 활용|

### Action 종류

1. **Forward packet to port(**s)
2. **Encapsulate** and forward to controller
3. **Drop packet**
4. Send to normal processing **pipeline**
5. **Modify Fields**

### OpenFlow 테이블 처리

- 스위치 내 플로우 테이블에 번호 부여, **0번부터 시작**
- 모든 입력 패킷은 **0번 플로우 테이블부터 비교**
- Table 0 → Table 1 → ... → Table n → Packet Out

## 하이퍼바이저 (Hypervisor)

**정의:** 호스트 컴퓨터에서 다수의 OS를 동시 실행시키는 가상 플랫폼 = VMM(Virtual Machine Monitor)

### Type 1 vs Type 2

|구분|Type 1 (베어메탈)|Type 2 (호스티드)|
|---|---|---|
|구조|하드웨어 → 하이퍼바이저 → VM|하드웨어 → OS → 하이퍼바이저 → VM|
|동작|하드웨어 위에서 직접 구동|일반 프로그램처럼 OS 위에서 실행|
|성능|높음|낮음|
|설치|어려움 (드라이버 직접 설정)|쉬움|
|제품|Xen, ESX 서버|VMware, 버추얼PC|

### 전가상화 vs 반가상화

| 구분      | 전가상화                          | 반가상화                      |
| ------- | ----------------------------- | ------------------------- |
| 가상화 범위  | 하드웨어 **전체**                   | 하드웨어 **일부**               |
| OS 수정   | 불필요                           | 커널 **일부 수정** 필요           |
| 하드웨어 제어 | 게스트OS가 직접 통제하는 것처럼 동작         | 일부는 하이퍼바이저가 통제            |
| 성능      | CPU-VT 기술 필요, I/O 복잡 → **낮음** | I/O 단순, RT OS 지원 → **높음** |
| 주요 제품   | VMware ESX 서버                 | XENExpress                |
| 특이사항    | RT OS 지원 **불가** **(특권모드 필수)** | **HyperCall로** 기능 활용      |


| -    | Ring 0 | 비고                                |
| ---- | ------ | --------------------------------- |
| 전가상화 | 하이퍼바이저 | 게스트 OS 몰래 대신 처리(**특권 내가 하드웨어인척**) |
| 반가상화 | 하이퍼바이저 | 게스트 OS가 알고 **하이퍼콜**               |


### 도커

**컨테이너 실행/관리**

- 이미지로 컨테이너 띄우고, 시작/중지/삭제 관리
- 각 컨테이너는 독립된 프로세스로 격리 실행

**네트워킹**

- 컨테이너 간 통신을 위한 가상 네트워크 구성
- 포트 매핑으로 외부 접근 제어

**볼륨(스토리지) 관리**

- 컨테이너는 꺼지면 데이터 사라짐
- 볼륨으로 호스트와 데이터 공유·영속화

근데 시험 관점에서 도커의 핵심 포인트는 **VM vs 컨테이너 차이**야.

|구분|VM|컨테이너(Docker)|
|---|---|---|
|가상화 대상|하드웨어|OS 프로세스|
|OS|게스트OS 별도 필요|호스트OS 공유|
|크기|무거움 (GB)|가벼움 (MB)|
|시작속도|느림|빠름|
|격리수준|높음|낮음|

## 컨테이너 오케스트레이션 (Container Orchestration)

**정의:** 다수의 컨테이너를 자동으로 배포·확장·관리·복구하는 기술

### 주요 기능

|기능|설명|
|---|---|
|자동 배포|컨테이너를 적절한 서버에 자동 배치|
|오토스케일링|부하에 따라 컨테이너 수 자동 증감|
|로드밸런싱|트래픽을 컨테이너들에 분산|
|자가복구|죽은 컨테이너 자동 재시작|
|서비스 디스커버리|컨테이너 간 자동으로 서로 찾아 통신|


프로비저닝 및 배포 / 구성 및 일정 조정 / 리소스 할당 / 컨테이너 가용성 / 워크로드 밸런싱 기반 컨테이너 스케일링·제거 / 로드 밸런싱 및 트래픽 라우팅 / 컨테이너 상태 모니터링 / 애플리케이션 설정 / 컨테이너 간 보안 유지

### 주요 기능

| 기능               | 설명                                                            |
| ---------------- | ------------------------------------------------------------- |
| **서비스 디스커버리**    | 컨테이너 생성·배치·이동 시 IP·Port 정보 자동 업데이트로 서비스 탐색 지원 **수많은 컨테이너 검색** |
| **스케일링 - 로드밸런싱** | 컨테이너의 컴퓨팅 자원 사용량 설정 및 자동 배분                                   |
| **스케일링 - 스케줄링**  | 늘어난 컨테이너를 적합한 서버에 배포, 서버 다운 시 다른 서버에서 자동 구동                   |
| **클러스터링**        | 여러 서버를 묶어 하나의 서버처럼 사용, 가상네트워크로 산재된 서버 연결                      |
| **로깅/모니터링**      | 여러 서버를 한 곳에서 상태 모니터링 및 로그 관리                                  |
### 대표 도구

|도구|특징|
|---|---|
|**Kubernetes (k8s)**|업계 표준, 구글 개발, 가장 많이 사용|
|Docker Swarm|도커 내장, 설정 단순, 기능 제한적|
|Apache Mesos|대규모 클러스터, 복잡|

### Kubernetes 핵심 구조

```
Master Node (제어)
├── API Server   ← 모든 요청 처리
├── Scheduler    ← 컨테이너 배치 결정
└── etcd         ← 클러스터 상태 저장

Worker Node (실행)
└── Pod          ← 컨테이너 실행 최소 단위
```


### 구성 요소

| 구성              | 설명                                                                     |
| --------------- | ---------------------------------------------------------------------- |
| **Master**      | 설정 환경 저장, 클러스터 전체 관리. API서버·스케줄러·컨트롤러매니저·etcd로 구성. kubectl로 세팅         |
| **Worker Node** | 마스터 명령을 받아 실제 작업 수행. Pod들이 배치되는 서비스 컴포넌트                               |
| **Pod**         | 컨테이너 **배포·관리 단위. 하나 이상의 컨테이너 포함**. 같은 Pod 내 **컨테이너는 로컬통신 가능 + 디스크 공유** |
| **ReplicaSet**  | 여러 Pod를 필요한 수만큼 미리 실행시켜 두는 장치(복제해서 대기시킴)                               |
| **kubelet**     | 각 노드에서 Pod **상태 관리 에이전트**                                              |
| **kube-proxy**  | 각 노드의 네트워크 규칙 관리                                                       |


### Master 내부 구성

|컴포넌트|역할|
|---|---|
|API Server|모든 요청의 진입점|
|Scheduler|컨테이너 배치 결정|
|Controller-Manager|클러스터 상태 유지|
|etcd|클러스터 설정·상태 저장 (분산 키-값 저장소)|

### 핵심 특징

**Fault-tolerance:** 기존 서비스 중단 없이 점진적 수시 업데이트 가능. 특정 컨테이너 장애 발생 시 즉시 복제 컨테이너 생성해 서비스 유지

### 시험 포인트

- **오케스트레이션 = 컨테이너 자동 관리**
- **Kubernetes = 사실상 표준**
- **Pod = k8s에서 컨테이너 묶음 단위**
- **도커 단독 vs 오케스트레이션** 차이 → 도커는 단일 호스트, 오케스트레이션은 **멀티 호스트** 관리

## NFV (Network Functions Virtualization)

**정의:** 물리 네트워크 장비(방화벽, 로드밸런서, IDS 등)의 기능을 **소프트웨어로 가상화**하여 범용 서버에서 실행하는 기술

### 기존 방식 vs NFV

|구분|기존|NFV|
|---|---|---|
|장비|전용 하드웨어|범용 서버 (x86)|
|기능|장비에 종속|소프트웨어로 구현|
|확장|장비 추가 구매|소프트웨어 배포|
|비용|고가|저렴|
|유연성|낮음|높음|

### IaaS / PaaS / SaaS 기본 구조

| **IaaS** | 인프라(서버,스토리지,네트워크) | AWS EC2, Azure VM           |
| **PaaS** | 실행환경+인프라          | Google App Engine           |
| **SaaS** | 애플리케이션 전체         | Gmail, Office365            |
| **FaaS** | 함수 실행환경 제공        | AWS Lambda, Azure Functions |
| **BaaS** | 백엔드 기능을 API로 제공   | Firebase, AWS Amplify       |


### 서버리스 = BaaS + FaaS

**BaaS (Backend as a Service)**

- 로그인·DB·파일저장·푸시알림 등 대중적 백엔드 기능을 API로 제공
- 내 코드 없이 갖다 쓰기만 함

**FaaS (Function as a Service)**

- 내가 짠 코드를 함수 단위로 올리면 호출될 때만 실행
- 실행 시간·횟수만큼만 과금
- 제약 : 실행시간 제한, 메모리 제한, 무상태

**실제 사용 패턴**

```
대중적 기능 (로그인, DB, 알림) → BaaS 갖다 쓰기
기업 고유 비즈니스 로직       → FaaS로 함수 직접 작성
```

### PaaS vs 서버리스 차이

|구분|PaaS|서버리스|
|---|---|---|
|배포 단위|애플리케이션 전체|함수 단위|
|과금|서버 켜있는 시간|실행될 때만|
|스케일링|개발자 설정|자동|
|서버 관리|일부 개발자|완전 사업자|

### 시험 포인트

- **서버리스 = BaaS + FaaS**
- **FaaS = 함수 단위 실행, 호출시만 과금**
- **BaaS = 흔한 백엔드 기능 API 제공**
- 서버리스는 PaaS보다 한 단계 더 추상화
- 복잡한 비즈니스 로직은 FaaS로 직접 구현

## API 게이트웨이 (API Gateway)

**정의:** 클라이언트와 백엔드 서비스 사이에서 **모든 API 요청의 단일 진입점** 역할을 하는 서버

### 구조

```
클라이언트
    ↓
API Gateway  ← 단일 진입점
    ↓ ↓ ↓
서비스A  서비스B  서비스C  (마이크로서비스들)
```

### 주요 기능

|기능|설명|
|---|---|
|**인증/인가**|토큰 검증, 접근 권한 확인|
|**라우팅**|요청을 적절한 서비스로 전달|
|**로드밸런싱**|트래픽 분산|
|**요청/응답 변환**|프로토콜 변환, 데이터 포맷 변환|
|**속도 제한 (Rate Limiting)**|과도한 요청 차단|
|**로깅/모니터링**|요청 기록 및 추적|
|**캐싱**|응답 캐싱으로 성능 향상|

### MSA에서 API Gateway가 필요한 이유

MSA는 서비스가 수십~수백 개로 분산되어 있어서 클라이언트가 각 서비스를 직접 호출하면

- 서비스 주소를 다 알아야 함
- 인증을 서비스마다 각각 처리해야 함
- 관리가 복잡해짐

→ API Gateway 하나로 **통합 관리**

### 시험 포인트

- **단일 진입점 (Single Entry Point)** 이 핵심
- **MSA와 항상 묶어서** 나옴
- 인증·라우팅·로드밸런싱·모니터링 기능 담당
- SDN의 컨트롤러랑 역할이 비슷한 느낌 (중앙에서 제어)

## 프록시 (Proxy)

**정의:** 클라이언트와 서버 사이에서 **대신 통신해주는 중계 서버**

### 구조

```
클라이언트 → 프록시 서버 → 인터넷 → 목적지 서버
```

### 종류

|구분|위치|역할|
|---|---|---|
|**Forward Proxy**|클라이언트 앞|클라이언트 대신 요청. IP 숨김, 접근제어|
|**Reverse Proxy**|서버 앞|서버 대신 응답. 로드밸런싱, 서버 IP 숨김|

### Forward vs Reverse 한 줄 차이

- **Forward** = 클라이언트 신원 숨김 (내가 누군지 서버가 모름)
- **Reverse** = 서버 신원 숨김 (어느 서버인지 클라이언트가 모름)

### 주요 기능

- 캐싱 (같은 요청 반복 시 저장된 응답 반환)
- 접근 제어 (특정 사이트 차단)
- 로드밸런싱 (Reverse Proxy)
- 보안 (실제 서버 IP 노출 방지)

### API Gateway vs Reverse Proxy

|구분|Reverse Proxy|API Gateway|
|---|---|---|
|목적|트래픽 중계·분산|API 통합 관리|
|기능|단순|인증·변환·모니터링 등 복잡|
|대상|일반 서버|주로 MSA|


```
Reverse Proxy
    + 인증/인가
    + 라우팅
    + Rate Limiting
    + 로깅/모니터링
    + 요청변환
= API Gateway
```

그래서 API Gateway를 **"스마트 Reverse Proxy"** 라고 보면 딱 맞아.

Reverse Proxy는 그냥 대신 받아서 넘겨주는 거고, API Gateway는 넘겨주면서 **이것저것 다 처리**하는 거야.


### 시험 포인트

- **Reverse Proxy = 로드밸런서 역할도 함**
- **API Gateway는 Reverse Proxy의 확장판** 느낌
- 캐싱으로 서버 부하 감소가 자주 출제 포인트

## 하둡 (Hadoop)

**정의:** 아파치 재단의 빅데이터 저장·분석·활용 기술. 대용량 데이터 처리를 위한 대규모 분산 컴퓨팅 지원 프레임워크

### 빅데이터 처리 순서

수집(Collecting) → 저장(Store) → 처리·분석(Analysis) → 시각화(Visualizing)

### 핵심 2가지 (시험 必)

**HDFS (Hadoop Distributed File System)**

- 파일을 **블록(64M) 단위**로 나눠 다중 노드에 분산 저장
- **리플리케이션** : 블록을 여러 노드에 복제 (기본 3개) → 장애 대응
- **Namenode** : 메타데이터 관리, 클라이언트 접근 처리 (단일 마스터)
- **Datanode** : 실제 파일 블록 저장
- 순차처리 특성 (스트리밍 접근)

**MapReduce**

- 구글이 제안한 분산 처리 프레임워크
- **Map** : 흩어진 데이터를 (key, Value) 형태로 분류·묶기
- **Shuffle** : 정렬, 파티셔닝
- **Reduce** : 중복 제거하고 원하는 데이터 추출
- 순서 : Map → Shuffle → Reduce
          |                                |
### 하둡 버전 비교

| 구분   | Hadoop 1.0    | Hadoop 2.0                                          |
| ---- | ------------- | --------------------------------------------------- |
| 처리방식 | 배치만 가능        | 배치·실시간·스트리밍 모두                                      |
| 자원관리 | JobTracker 단독 | **YARN** (Resource Manager + Application Master 분리) |
| 확장성  | 4천대 이상 불가     | 최대 10000개 노드                                        |
| 처리엔진 | MapReduce만    | Spark, Storm 등 다양한 엔진 지원                            |

### Storm (실시간 처리)

**정의:** 실시간 데이터 처리를 위한 분산 환경 기반 스트리밍 시스템

|구성|설명|
|---|---|
|**Nimbus**|마스터 노드, 중앙 컨트롤러|
|**Supervisor**|워커 노드, 실제 작업 실행|
|**Zookeeper**|노드 상태 모니터링·공유|
|**Spout**|데이터 소스 (Kafka 등에서 데이터 수신)|
|**Bolt**|실제 데이터 처리 수행|

### 하둡 3.0 핵심

**Erasure Coding (EC)**

- 기존 : 1G 데이터 → 3개 복제 → 3G 사용
- EC : 1G 데이터 → n등분 → 패리티 셀 생성 → **1.5G만 사용**
- 저장 효율 대폭 향상, Fault Tolerance 유지

### 시험 포인트

- **HDFS + MapReduce = 하둡의 핵심 2가지**
- **Map → Shuffle → Reduce 순서**
- **Hive = SQL 유사 (HiveQL)**, MapReduce 대체
- **YARN = Hadoop 2.0 자원관리 플랫폼**
- **Storm = 실시간**, MapReduce = 배치
- **Namenode = 단일 마스터 → SPOF 주의**

## NoSQL

**정의:** Not Only SQL. SQL만 고집하지 말고 서비스에 특화된 DB를 선택하자는 아이디어. RDBMS 대체가 아닌 **RDBMS가 적합하지 않은 곳에서의 대안**

### RDBMS 한계 (NoSQL 등장 이유)

- 급격한 데이터 증가를 스케일업으로 감당 어려움
- 고비용
- 모든 데이터가 트랜잭션·강한 일관성이 필요한 건 아님
- 비정형 데이터 처리 요구 증가

### NoSQL 특징

- 고정된 스키마 없음 → 확장성 뛰어남
- 원자적 트랜잭션 지원 어려움
- 조인 지원 안 함 → 앱에서 직접 구성
- 무결성보다 **확장성** 우선

### NoSQL 4가지 유형

| 유형          | 구조                      | 대표 DB            |
| ----------- | ----------------------- | ---------------- |
| **키-값 스토어** | Key : Value             | Redis            |
| **컬럼형**     | Key : 여러 Column-Value   | HBase, Cassandra |
| **문서형**     | Key : JSON/XML Document | MongoDB, CouchDB |
| **그래프형**    | 노드와 노드 간 관계             | Neo4J            |

### CAP 이론 (시험 必)

**분산 컴퓨팅 환경은 3가지 특성 중 2가지만 만족 가능**

|특성|설명|
|---|---|
|**C (Consistency)**|어느 노드로 접근해도 같은 데이터|
|**A (Availability)**|노드 장애 시에도 정상 요청 처리 가능|
|**P (Partition Tolerance)**|네트워크 장애 시에도 서비스 수행 가능|

**CAP 조합별 대표 DB**

|조합|대표|
|---|---|
|CA|RDBMS (MySQL, PostgreSQL)|
|CP|MongoDB, HBase, BigTable|
|AP|Cassandra, CouchDB, Riak|
|CP (키-값)|Redis, MemcacheDB|

### 시험 포인트

- **NoSQL = RDBMS 대체 아님**, 보완적 관계
- **CAP 이론 = 3가지 중 2가지만 가능**
- **키-값 → Redis / 컬럼 → HBase, Cassandra / 문서 → MongoDB / 그래프 → Neo4J**
- **컬럼형 = 열 중심**, 관계형 = 행 중심

## 가용성 계산

### **==가용성== 공식**

```
가용성 = MTTF / (MTTR + MTTF) × 100
```

### 용어 정리

| 용어       | 풀네임                       | 의미                                         |
| -------- | ------------------------- | ------------------------------------------ |
| **MTTF** | Mean Time to Failure      | 정상 가동 시작 후 다음 고장까지 **평균 사용 시간**            |
| **MTTR** | Mean Time to Repair       | 고장을 **수리하는 평균 시간**                         |
| **MTBF** | Mean Time Between Failure | 고장 시작 후 다음 고장까지 **평균 고장 간격** = MTTF + MTTR |

### 예제 풀이

**문제:** 40시간 운영 → 10시간 수리 → 20시간 운영 → 10시간 수리

```
MTTF = (40 + 20) / 2 = 30시간  (정상 운영 평균)
MTTR = (10 + 10) / 2 = 10시간  (수리 시간 평균)

가용성 = 30 / (10 + 30) = 30 / 40 = 0.75 = 75%
```

### 시험 포인트

- **공식 = MTTF / (MTTR + MTTF)**
- **MTTF ↑, MTTR ↓ → 가용성 ↑**
- **MTBF = MTTF + MTTR** (고장 간격)
- 여러 번의 데이터 나오면 **평균값** 먼저 구하고 공식 적용

## 고가용성 HA (High Availability)

**정의:** 서버·네트워크·프로그램 등 정보시스템이 오랜 기간 지속적으로 정상 운영 가능한 성질

**5 nines (99.999%)** = 1년에 5분 15초 이하 장애시간 허용 → 고품질 데이터센터 목표

### HA 구성방식 3가지

| 구분  | Hot Standby                                 | Mutual Takeover                                   | Concurrent Access                              |
| --- | ------------------------------------------- | ------------------------------------------------- | ---------------------------------------------- |
| 방식  | Active + Standby 구성. 장애 시 Standby로 Failover | 2개 시스템이 각각 고유 기능 수행하다 장애 시 상대 자원 인수하여 2개 업무 동시 수행 | 다수 시스템이 동일 기능 수행 (로드밸런싱 유사). 장애 시 Failover 불필요 |
| 장점  | Failover 후에도 동일 성능 보장                       | 자원 효율적 이용                                         | Failover 불필요                                   |
| 단점  | 자원 낭비 (하나는 항상 대기)                           | 장애 후 한 서버에 업무 몰릴 수 있음                             | 로드밸런싱 위한 추가 장비 필요                              |

### HA 다운타임 (암기)

| 가용성         | Nines           | 1년 중 다운타임  |
| ----------- | --------------- | ---------- |
| 90%         | one nine        | 36.53일     |
| 99%         | two nines       | 3.65일      |
| **99.9%**   | **three nines** | **8.77시간** |
| 99.99%      | four nines      | 52.60분     |
| **99.999%** | **five nines**  | **5.26분**  |
| 99.9999%    | six nines       | 31.56초     |

### 시험 포인트

- **5 nines = 99.999% = 1년에 5분 15초** 이하
- **Hot Standby = Active/Standby** 구조
- **Mutual Takeover = 장애 시 상대 자원 인수**
- **Concurrent Access = 로드밸런싱 유사, Failover 불필요**
- 용량산정 3가지 방법 이름과 특징

## 성능 계산

### 용어 정리

| 용어                          | 설명                             |
| --------------------------- | ------------------------------ |
| **응답시간 (Response Time)**    | 사용자가 요청 보내고 응답 받을 때까지 시간       |
| **Think Time**              | 응답 받은 후 다음 요청 전까지 대기 시간        |
| **요청간격 (Request Interval)** | 응답시간 + Think Time              |
| **활성사용자 (Active User)**     | 지금 응답시간에 속해 있는 사용자             |
| **동시사용자 (Concurrent User)** | PC 앞에 앉아 있는 사용자 (요청간격에 속한 사용자) |
| **TPS**                     | 초당 처리 건수                       |

### 핵심 공식 (Little's Law)

```
TPS = 활성사용자 / 평균응답시간(초)
TPS = 동시사용자 / 서비스요청간격(초)
TPS = 총 방문자 / 총 시간(초)

활성사용자 = TPS × 응답시간
동시사용자 = TPS × 요청간격
```

**관계식**

- Concurrent Users(동시사용자) = Active Users(활성사용자 ) + Inactive Users(대기 사용자)
- TPS = Concurrent Users(동시사용자) ÷ Request Interval(요청간격)

### 예제

**햄버거 가게 : 1시간에 360명 방문, 평균 식사시간 10분**

```
TPS = 360명 ÷ 3600초 = 0.1 TPS
평균응답시간 = 600초 (10분)
활성사용자 = 0.1 × 600 = 60명
```

### 시험 포인트

- **활성사용자 = TPS × 응답시간**
- **동시사용자 = TPS × 요청간격**
- **요청간격 = 응답시간 + Think Time**
- 동시사용자 > 활성사용자 (Think Time만큼 더 넓음)

## 스토리지 DAS / NAS / SAN

| 구분   | DAS                     | NAS                      | SAN                     |
| ---- | ----------------------- | ------------------------ | ----------------------- |
| 풀네임  | Direct Attached Storage | Network Attached Storage | Storage Area Network    |
| 연결   | 서버에 **직접** (SCSI, FC)   | **LAN** (TCP/IP)         | **전용망** (Fiber Channel) |
| 접근단위 | 블록                      | **파일**                   | 블록                      |
| 공유   | 불가 (서버 각각 관리)           | 가능 (파일서버 통해)             | 가능 (**FC Switch** 통해)   |
| 속도   | 빠름                      | 보통 (접속 증가시 저하)           | **빠름** (고속전송)           |
| 확장성  | 낮음 (연결 수 한계)            | 높음                       | 높음                      |
| 비용   | 저렴                      | 중간                       | 고가                      |
| 특징   | 서버가 직접 파일시스템 관리         | 파일서버가 중간 매체 역할           | 데이터 전용망, 장거리 연결 가능      |
**SCSI (Small Computer System Interface)** 서버와 스토리지 장치 연결하는 인터페이스 프로토콜. 옛날부터 쓰던 표준.

**FC (Fibre Channel)** SCSI 프로토콜을 광케이블로 전송하는 고속 스토리지 네트워크. **SAN(Storage Area Network)** 구성할 때 쓰는 핵심 기술.

한줄 요약:

- SCSI → 스토리지 연결 프로토콜
- FC → 그걸 고속 광케이블로 구현한 것
- 
### NAS 통신 경로

```
서버 → TCP/IP(LAN) → 파일서버 → SCSI/FC → Storage
```

### 시험 포인트

- **NAS = 파일 단위, SAN = 블록 단위**
- **NAS = LAN 공유, SAN = 전용망**
- **NAS 단점 = 접속 증가시 성능 저하**
- **IP SAN** = SAN을 FC 대신 IP로 구현 (저렴)
- DAS → NAS → SAN 순으로 공유·확장성 높아짐

##  오픈소스 라이센스

개념: 소스코드가 공개된 SW / 라이센스 종류에 따라 일부 권한 제한 가능

### 라이센스별 특징

| 라이센스       | Copyleft | 핵심 특징                                                |
| ---------- | -------- | ---------------------------------------------------- |
| MIT        | 없음       | BSD 기반 / 소스 공개 의무 없음 / 가장 자유로움                       |
| BSD        | 없음       | 수정 시 소스 공개 불필요 / 배포 시 저작권 표시                         |
| Apache 2.0 | 없음       | 소스 공개 불필요 / 배포 시 Apache 저작권 표시 / 특허권 명시              |
| LGPL 2.1   | 약함       | 라이브러리 링크 시 소스 공개 불필요 / GPL보다 완화                      |
| MPL 2.0    | 약함       | 모질라 공중 라이센스 / 수정 파일만 공개 / 실행파일은 독점 배포 가능             |
| CDDL       | 약함       | MPL 유사 / 썬 마이크로시스템즈가 MPL 개조                          |
| EPL / CPL  | 약함       | IBM이 이클립스 등에서 만든 라이센스 / GPL 유사하나 완화                  |
| GPL 2.0    | 강함       | 수정·링크 시 소스 공개 필수 / 2차 저작물도 GPL 적용                    |
| GPL 3.0    | 강함       | GPL 2.0 + 설치정보 제공 의무 + 특허 개선자 면책 조항                  |
| AGPL 3.0   | 가장 강함    | **네트워크 서비스 제공 시에**도 소스 공개 의무<br>(**네트워크 블루홀** 문제 해결) |

**네트워크 블루홀(Network Loophole)** 이었네요!

### 배경

```
GPL 2.0/3.0 시절:
소프트웨어를 배포할 때만 소스 공개 의무

→ 근데 웹서비스는?
→ 서버에서 실행만 하고 배포 안 함
→ GPL 적용 안 됨
→ 소스 공개 안 해도 됨
```

**이게 네트워크 루프홀 (블루홀)**

### AGPL 3.0이 해결한 것

```
GPL:     배포 시 소스 공개
AGPL:    배포 + 네트워크 서비스 제공 시도 소스 공개
```

MongoDB (구버전) = AGPL
→ 내가 MongoDB로 SaaS 만들어서 서비스하면
→ 내 서비스 소스코드도 공개해야함
→ AWS같은 클라우드 기업들이 싫어한 이유

### TTA 표준 분류 기준

|분류|종류|해당 라이센스|
|---|---|---|
|허용적 (Permissive)|라이센스 변경 허용|Apache 2.0, BSD-2, BSD-3, MIT|
|약한 Copyleft (Weak)|라이센스 변경 허용|EPL 1.0, EUPL 1.0, LGPL 2.1/3.0, MPL 2.0, MS-PL|
|강한 Copyleft (Strong)|동일 라이센스 적용 필수|GPL 2.0, GPL 3.0, AGPL 3.0|

### Copyleft 강도 순서

없음(MIT = BSD = Apache) < 약함(LGPL = MPL) < 강함(GPL) < 가장 강함(AGPL)

## 분산처리 시스템

개념: 중앙집중식과 대비 / 분산된 시스템이 작업 처리 후 결과 상호 교환 / 사용자는 전체를 하나의 컴퓨터로 인식

### 분산처리 시스템의 투명성 (Transparency)

목적: 다수의 컴퓨터를 마치 한 대의 컴퓨터처럼 보이게 하는 것

| 투명성          | 영문            | 핵심 내용                       |
| ------------ | ------------- | --------------------------- |
| 접근 투명성       | Access        | 원격/내부 관계없이 동일한 접근 방식으로 접속   |
| 위치 투명성       | Location      | 자원이 어느 컴퓨터에 있는지 알 필요 없이 이용  |
| 병행(동시수행) 투명성 | Concurrency   | 다수 트랜잭션 동시 수행해도 결과 일관성 유지   |
| 중점(복사) 투명성   | Replication   | 동일 자원이 여러 곳에 존재해도 하나로 보임    |
| 고장(장애) 투명성   | Failure       | 구성 요소 장애 발생해도 서비스 정상 제공     |
| 이주(이동) 투명성   | Migration     | 자원이 다른 컴퓨터로 이동해도 사용자는 의식 못함 |
| 성능 투명성       | Performance   | 시스템 재구성해도 사용자는 의식 못함        |
| 규모 투명성       | Scaling       | 구성 요소 추가/제거 등 규모 변화를 의식 못함  |
| 이기종 투명성      | Heterogeneity | 다른 H/W·S/W 구성이어도 사용자는 의식 못함 |

|구분|내용|

### 기존 컴퓨터 vs 양자컴퓨터

### 학습 모드 & 알고리즘 매핑

|학습 방식|딥러닝 알고리즘|사용 분야|

|비지도학습|오토인코더|-|

|생성모델|GAN|이미지 생성|

### 학습 단위

| 용어         | 설명                                                  |
| ---------- | --------------------------------------------------- |
| epoch      | 전체 데이터셋을 신경망이 한 번 통과한 횟수 / 작으면 과소적합, 크면 과적합         |
| batch size | 전체 데이터를 나눈 소그룹의 데이터 수 / 크면 학습 느림·메모리 부족, 작으면 훈련 불안정 |
| iteration  | 1 epoch을 마치는데 필요한 **미니배치** 수 (= 스텝)                 |

### CNN 핵심

| 개념          | 설명                                                    |
| ----------- | ----------------------------------------------------- |
| Convolution | Input × Filter → Feature Map 생성                       |
| Stride      | 필터 순회 간격                                              |
| Padding     | Feature Map 크기 축소 방지 / 외곽을 0으로 채움                     |
| Pooling     | Feature Map 크기 축소 / Max·Average·Min Pooling           |
| 출력크기 공식     | **O = (I - F + 2P) / S + 1**  (전체-필터+(2*패딩)/스트라이더) +1 |

### GAN 문제점

|문제|설명|
|---|---|
|Vanishing Gradients|Discriminator가 너무 잘 학습 → Generator gradient 소실|
|Mode Collapse|Generator가 다양한 데이터 못 만들고 비슷한 것만 생성|

DCGAN: GAN의 fully-connected를 convolution으로 대체 / 배치 정규화로 학습 안정화

### RNN 핵심

|구분|내용|
|---|---|
|특징|순환구조 / 과거 정보를 현재 학습에 반영 / 시계열·순서 데이터에 특화|
|활성화 함수|tanh (하이퍼볼릭 탄젠트) / ReLU 쓰면 오버플로우 발생|
|한계|기울기 소실 / 과적합 / 병렬처리 어려움 / 장기 의존성 학습 어려움|

### LSTM (RNN 한계 보완)

|게이트|역할|
|---|---|
|망각 게이트 (Forget)|셀 정보 지울지 말지 결정 / 0=지움, 1=기억|
|입력 게이트 (Input)|새로운 정보 반영 방법 결정|
|출력 게이트 (Output)|cell state에서 output으로 보낼 부분 결정|

GRU = LSTM 간소화 버전

### LSTM 게이트 순서

```
1. 망각 게이트 (Forget Gate) 
   → 이전 정보 얼마나 버릴지

2. 입력 게이트 (Input Gate)
   → 새 정보 얼마나 받을지

3. 셀 상태 업데이트 (Cell State)
   → 망각 + 입력 반영해서 업데이트

4. 출력 게이트 (Output Gate)
   → 다음으로 얼마나 내보낼지
```

### GPU (Graphics Processing Unit)

**개요**

- 원래 그래픽 처리용으로 개발 → 계산 능력 강화되어 병렬 데이터 처리 보조프로세서로 사용

**NVIDIA 주요 기술**

|기술|설명|
|---|---|
|통합된 그래픽 및 계산구조|그래픽/일반 계산 모두 가능한 통합 하드웨어 구조|
|CUDA|일반 프로그래밍 언어로 GPU 프로그램 작성 가능한 병렬컴퓨팅 플랫폼|

**GPU 내부 구성요소**

| 구성요소                          | 설명                                                |
| ----------------------------- | ------------------------------------------------- |
| 호스트(host)                     | 일반적인 CPU 혹은 컴퓨터                                   |
| 디바이스(device)                  | GPU와 같은 병렬 프로세서                                   |
| SM (Streaming Multiprocessor) | 실제 코드 수행, 다수의 SP(스트리밍 프로세서)로 구성, 데이터 캐시+공유기억장치 존재 |
| 스케줄러/디스패치 유니트                 | 호스트로부터 들어오는 병렬 프로그램 코드를 SM에 할당                    |

**단일 SM 구성요소**

| 구성요소              | 설명                                                                      |
| ----------------- | ----------------------------------------------------------------------- |
| 코어 (CUDA 코어)      | 오퍼랜드 레지스터 + FPU(부동소수점) + ALU(정수 산술논리) + 결과 큐                            |
| 왑 (Warp)          | SM이 한번에 수행할 수 있는 스레드 묶음, 각 스레드에 순차적 ID 부여, 왑스케줄러가 스레드에 코어 할당 (32개 SIMD) |
| 레지스터 세트           | 32K개 × 32비트                                                             |
| 공유 기억장치/L1 데이터 캐시 | SM 내 스레드 공유                                                             |

**CUDA 프로그램 구성요소**

| 구성요소      | 설명                                |
| --------- | --------------------------------- |
| 호스트 코드    | ANSI C 기반 순자적 프로그램                |
| 디바이스 코드   | ANSI C + 커널 함수로 구성                |
| 커널 함수     | 디바이스가 실행하는 병렬 처리 부분               |
| 스레드       | 커널 함수의 인스턴스, GPU의 SP(CUDA 코어)가 실행 |
| 블록(block) | 스레드들의 묶음 → 하나의 SM에 할당             |
| 그리드(grid) | 병렬 커널 코드에 존재하는 스레드 전체             |

**쓰레드(최소작업단위) -> 블록(쓰레드묶음) -> 그리드(블록묶음)**


**CUDA 실행 과정**

1. 호스트가 순차적 코드 실행
2. 커널 함수 호출 → 병렬 커널 코드가 디바이스로 전송
3. 디바이스에서 커널이 그리드(블록+스레드) 생성 → SM에 할당 → SP가 스레드 실행
4. 모든 스레드 실행 완료 → 그리드 종료 → 결과값 호스트로 전송
5. 호스트가 다음 순차 코드 실행, 커널 만나면 2~4 반복

**GPU 기억장치 계층**

| 구분                                             | 범위          | 설명                                      |
| ---------------------------------------------- | ----------- | --------------------------------------- |
| 지역 기억장치 (local memory) **쓰레드 레지스터**            | 스레드 개별      | 각 스레드가 사용, 주로 레지스터 세트로 구현               |
| 공유 기억장치 (shared memory) **SM내 공유 블록단위로 분할 할당** | 블록 내 스레드 공유 | 같은 블록에 포함된 스레드들이 공동으로 사용                |
| 전역 기억장치 (global memory)                        | 전체 그리드 공유   | 모든 응용프로그램의 그리드가 공유, 호스트↔GPU 데이터 교환에도 사용 |

### 가상 기억장치 (메모리 - 디스크)

**개요**

|개념|설명|
|---|---|
|가상주소|프로세스가 참조하는 프로그램 상의 주소|
|사상(Mapping)|가상주소 → 물리주소 변환 과정|
|DAT|논리적 가상주소를 물리적 주소로 바꾸는 과정, 하드웨어로 구현|

**개발 이유:** 저렴한 주기억장치 확장, 산재 배당, 오버레이 자동 해결, 단편화 제거

**가상기억장치 관리 정책**

|정책|설명|
|---|---|
|반입(fetch)|언제 디스크→주기억장치로 가져올지. 요구반입(demand fetch) vs 예상반입|
|배치(placement)|페이지를 주기억장치 어느 곳에 적재할지|
|교체(replacement)|페이지 부재 시 어떤 페이지프레임을 디스크로 내보낼지|
|배당(allocation)|각 프로세스에 얼마만큼의 주기억장치를 할당할지|


### 페이징 기법

**기본 개념**

- 디스크/주기억장치를 고정 크기로 분할
- 디스크 단위 = 페이지, 메모리 단위 = 페이지프레임(블록)
- 가상주소 V = (p, d) → p: 페이지번호, d: 변위

**주소 사상 방법**

|방법|특징|
|---|---|
|직접 사상(Direct Mapping)|PMT에서 페이지번호 p 찾아 실제주소 r = p' + d 계산|
|연관 사상(Associative Mapping)|캐시/연관기억장치로 PMT 구성, 병렬 검색으로 빠름, 비용/하드웨어 복잡|
|직접/연관 혼합|자주 참조되는 페이지는 연관사상표에서 먼저 검색, 없으면 PMT 검색|


### 세그멘테이션 기법

- 고정 크기 아닌 **다양한 크기의 세그먼트**로 분할
- 가상주소 V = (s, d) → s: 세그먼트번호, d: 변위
- SMT(Segment Mapping Table): PMT와 유사, 크기필드 추가
- 변위 d가 크기 초과 시 → 세그먼트 오버플로우 에러


### 페이지 교체 알고리즘

| 알고리즘                | 핵심                                     | 특징                                   |
| ------------------- | -------------------------------------- | ------------------------------------ |
| FIFO                | 가장 오래된 페이지 교체                          | 프레임 늘려도 페이지부재 증가하는 이상(anomaly) 현상 발생 |
| LRU                 | 가장 오랫동안 참조 안된 페이지 교체                   | 스택 활용, 참조 국부성 활용                     |
| LFU                 | 참조횟수 가장 적은 페이지 교체                      | 카운터 오버헤드                             |
| 이차기회(Second Chance) | FIFO + 참조비트 확인, 1이면 한번 기회 줌(비트 0으로 바꿈) | FIFO 개선                              |



### 스래싱 (Thrashing)

| 항목  | 내용                                                                      |
| --- | ----------------------------------------------------------------------- |
| 정의  | 너무 자주 페이지 교체 발생, 실제 처리시간보다 페이지 교체시간이 더 많아지는 현상                          |
| 원인  | 다중 프로그래밍 정도 심화 → 프로세스당 페이지프레임 부족 → CPU 이용률 저하 → OS가 다중 프로그래밍 더 증가 → 악순환 |
| 방지책 | Page Fault Frequency + Working Set 사용                                   |

**Page Fault Frequency:** 부재율 상한 초과 시 프레임 추가 배당, 하한 미만 시 프레임 회수

**Working Set:** 최근 델타t 동안 참조된 페이지들의 집합, 자주 참조되는 페이지 주기억장치에 유지

### 교착상태 (Deadlock)

**발생 4가지 조건 (모두 성립해야 발생)**

| 조건       | 설명                              |
| -------- | ------------------------------- |
| 상호배제     | 한 프로세스가 사용 중인 자원은 다른 프로세스 사용 불가 |
| 점유 및 대기  | 자원 점유한 채로 다른 자원 대기              |
| 비선점      | 자원을 강제로 빼앗을 수 없음                |
| 환형대기(순환) | A→B 자원 요구, B→A 자원 요구 (순환)       |

**해결 방법**

|방법|설명|
|---|---|
|예방(Prevention)|4가지 조건 중 하나라도 발생 못하게 미리 부정|
|회피(Avoidance)|발생 가능성 인정, 안전상태일 때만 자원 배당. **은행원 알고리즘** 대표적|
|탐지 및 회복(Detection & Recovery)|발생 허용, 탐지되면 프로세스 제거 또는 재시작|

**교착상태 예방 방법**

|조건|예방법|
|---|---|
|상호배제|읽기 전용 공유 자원 사용|
|점유 및 대기|실행 전 필요 자원 모두 할당 (자원 낭비), 또는 자원 없을 때만 요청 가능 (기아 가능)|
|비선점|모든 자원 선점 허용, 못 받으면 기존 자원 반납 (자원 낭비)|
|환형대기|자원에 고유번호 할당, 번호 순서대로만 요청 (자원 낭비)|
