| 품질 특성                           | 부특성                                      | 의미                                                       |
| ------------------------------- | ---------------------------------------- | -------------------------------------------------------- |
| 기능 적합성 (Functional suitability) | 완전성 (completeness)                       | 요구된 기능이 **빠짐없이** 제공되는 정도                                 |
|                                 | 정확성 (correctness)                        | 기능이 **올바른 결과**를 내는 정도                                    |
|                                 | 적절성 (appropriateness)                    | 기능이 사용자의 업무 목적 달성에 **도움이 되도록 적합하게** 제공되는 정도              |
| 신뢰성 (Reliability)               | 성숙성 (Maturity)                           | 장애나 실패가 적고 **안정적으로 동작**하는 정도                             |
|                                 | 가용성 (Availability)                       | 필요할 때 시스템을 **사용 가능**한 정도                                 |
|                                 | 결함허용성 (Fault tolerance)                  | 일부 오류나 장애가 발생해도 **서비스를 계속 유지**하는 정도                      |
|                                 | 복구성 (Recoverability)                     | 장애 후 데이터·상태·서비스를 **복원**할 수 있는 정도                         |
| 사용성 (Usability) 유용성             | 적절 인식성 (Appropriateness recognizability) | 사용자가 이 시스템이 **내 목적에 맞는지 바로 파악**할 수 있는 정도                 |
|                                 | 학습성 (Learnability)                       | 배우기 쉬운 정도                                                |
|                                 | 운영성/운용성 (Operability)                    | 사용자가 기능을 **쉽게 조작·통제**할 수 있는 정도                           |
|                                 | 사용자 오류 방지성 (User error protection)       | 사용자의 실수 발생을 **줄이거나 예방**해 주는 정도                           |
|                                 | UI 심미성 (User interface aesthetics)       | 화면과 인터페이스가 **보기 좋고 일관적**인 정도                             |
|                                 | 접근성 (Accessibility)                      | 다양한 사용자, 특히 제약이 있는 사용자도 **이용 가능**한 정도                    |
| 성능 효율성 (Performance efficiency) | 시간 반응성 (Time behaviour)                  | 응답시간, 처리시간 등 **속도 측면 성능**                                |
|                                 | 자원 효율성 (Resource utilisation) 요소활용       | CPU, 메모리, 네트워크 등 자원을 **효율적으로 사용하는 정도**                   |
|                                 | 수용성 용량 (Capacity)                        | 동시 사용자 수, 데이터량, 처리량 등 **감당 가능한 한계 규모**                   |
| 유지관리성 (Maintainability)         | 모듈성 (Modularity)                         | 구성요소가 잘 분리돼 한 부분 변경이 다른 부분에 **영향을 덜 주는 정도**              |
|                                 | 재사용성 (Reusability)                       | 자산이나 구성요소를 **다른 시스템/기능에 다시 활용**할 수 있는 정도                 |
|                                 | 분석성 (Analysability)                      | 결함 원인, 변경 영향 등을 **분석하기 쉬운 정도**                           |
|                                 | 변경용이성 (Modifiability)                    | 수정·개선·확장이 쉬운 정도                                          |
|                                 | 시험용이성 (Testability)                      | 테스트 케이스 설계, 실행, 결과 확인이 **쉬운 정도**                         |
| 이식성 (Portability)               | 적응성 (Adaptability) 적용성                   | **다른 환경**에서도 **쉽게 맞춰 적용**할 수 있는 정도                       |
|                                 | 설치성 (Installability)                     | 설치·배포·제거가 쉬운 정도                                          |
|                                 | 대체성 (Replaceability)                     | 다른 제품을 대체하거나 다른 제품으로 **대체되기 쉬운 정도**                      |
| 보안성 (Security)                  | 기밀성 (Confidentiality)                    | 인가되지 않은 대상에게 정보가 **노출되지 않는 정도**                          |
|                                 | 무결성 (Integrity)                          | 정보나 기능이 **허가 없이 변경·훼손되지 않는 정도**                          |
|                                 | 부인방지 (Non-repudiation)                   | 행위 수행 사실을 나중에 **부인하지 못하게** 하는 정도                         |
|                                 | 접근추적성/책임추적성 (Accountability)             | 누가 무엇을 했는지 **추적·책임 귀속**이 가능한 정도<br>**내계좌로 누가 접근? 으로 외워** |
|                                 | 인증성 (Authenticity)                       | 사용자·시스템·데이터 출처가 **진짜임을 확인**할 수 있는 정도                     |
| 호환성 (Compatibility)             | 공존성 (Co-existence)                       | 동일 환경에서 다른 시스템과 **충돌 없이 함께 운영**되는 정도                     |
|                                 | 상호운용성 (Interoperability)                 | 다른 시스템과 **정보를 주고받고 활용**할 수 있는 정도                         |

| 성능 효율성 | **빠르고, 덜 먹고, 많이 처리하냐**                           |

| 보안성    | **새지 않고, 안 바뀌고, 부인 못 하고, 추적되고, 진짜 맞냐**           |
| 호환성    | **같이 살고, 서로 통하냐**                                |

## 📘 감리 제안서 기술평가 기준 (별표2)

| 평가항목           | 세부평가요소                                                                                                                                   |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **점검내용**       | • 감리대상사업 특성을 고려하여 주요 위험요소·예상문제점 도출 및 감리 수행 주안점·전략 제시 여부<br>• 위험요소 및 예상문제점에 대해 단계별 점검방안을 구체적·적정하게 제시하였는지 여부                               |
| **점검방법**       | • 과업이행여부, 기술적용계획표 등 필수점검사항에 대한 점검방법을 구체적으로 제시하였는지 여부<br>• 점검결과의 객관성·타당성 확보를 위한 **점검기법·방법**을 구체적으로 제시하였는지 여부                              |
| **감리일정 및 절차**  | • 사업 단계별 감리일정 및 세부 감리절차를 적정하게 제시하였는지 여부<br>• 각 단계별 시정조치확인에 투입되는 감리인력·기간·수행방법 등을 구체적·적정하게 제시하였는지 여부                                       |
| **감리인력 구성**    | • 사업 위험도 및 특성을 고려한 분야별 감리원·특정분야 인력 배치 수준의 적정성 여부<br>※ 요구하지 않은 상주감리원·특정분야 인력 제안, 요구 투입공수 초과 제안은 평가 제외<br>• 상근감리원의 비율                      |
| **총괄감리원**      | • 총괄감리원이 업무 수행에 필요한 기술자격·유사업무·감리수행경력 등 전문성을 갖추었는지 여부                                                                                     |
| **각 분야별 감리인력** | • 투입 감리인력(요구하지 않은 상주감리원·특정분야 투입인력 제외)의 담당분야 관련 업무 또는 감리수행경력 전문성 여부<br>• 투입 감리원(총괄감리원 포함)의 계속교육 이수실적(교육 종류 및 시간)                          |
| **기타 지원사항 등**  | • 감리수행절차 또는 감리보고서 품질향상을 위한 감리법인 차원의 체계적 교육훈련·품질관리 수행 여부<br>• **시험·진단·점검 자동화 도구 및 기법 제안의 적정성 여부**<br>• 발주자가 요청한 기타사항(감리범위 등)에 대한 적정 제안 여부 |

# 감리원 자가점검
| 점검항목     | 설명                                                                   |
| -------- | -------------------------------------------------------------------- |
| 윤리적 갈등상황 | 양심에 따라 공정하고 **합법적이며 사회 통념에 반하지 않는 지**를 스스로 점검                        |
| 의사결정시    | **결정권과 권한**을 적절하게 사용하는 지를 점검                                         |
| 금품 및 향응  | **부득이한 식사나 편의, 부당한 지시, 알선 청탁** 등 공정한 직무수행 저해요소를 점검                   |
| 비밀 유지    | 감리시 수령한 어떠한 문서와 **정보도 유출하지 않으며 보관하지 않는지** 점검                         |
| 이해상충 금지  | 감리 수행중 다른 개**인적인 업무 수행이나 부당한 이익, 금품 수수** 등을 점검                       |
| 직무윤리     | **잘못된 일처리에 대한 시정, 사행성 행위** 등의 직무윤리 준수 점검                             |
| 조직 문화    | 감리원 상호간 **무시나 폭언**을 하는 행위나 성별/종교/의모/학벌/장애 등의 **차별 등** 감리팀의 조직 문화를 점검 |
| 성희롱      | 감리시 **성적 언동과 신체적 접촉을** 금지하고 성희롱 행동여부에 대해 점검                          |
| 자산 보호    | 감리시 **수령받은 물품을 보전후 반납**하며 고객 용품을 함부로 사용하지 않는 등 고객 자산에 대한 사용 적정여부를 점검 |

# **생체인식 오류율 4가지 정리**

| 용어                               | 의미                                     | 방향      |

| **FAR** (False Acceptance Rate)  | 비정상 사용자를 허용 (타인인데 들어감)                 | 보안↓ 편의↑ |

| **CER** (Crossover Error Rate)   | **FRR=FAR이 되는 교차점**, 생체인식 시스템 성능 비교 기준 |         |

# 개인정보 비식별화 기법 

|기법|핵심 한 줄|보완 관계|
|---|---|---|
|k-익명성|나와 똑같은 레코드가 k-1개 이상 존재|기본|
|ℓ-다양성|동질 집합 내 민감 정보가 ℓ개 이상 달라야 함|k-익명성 보완|
|t-근접성|동질 집합의 분포가 전체 분포와 t 이하 차이|ℓ-다양성 보완|

**진화 순서**: k-익명성 → ℓ-다양성 → t-근접성

**왜 발전했냐**

k-익명성은 같은 레코드 k개 보장하지만 민감 정보가 다 같으면 뚫림 → ℓ-다양성으로 보완
ℓ-다양성은 다양하지만 분포 편중되면 뚫림 → t-근접성으로 보완


**전제: 전체 데이터 질병 분포 - 암10% 당뇨60% 고혈압30%**

**k-익명성 (k=3)**

준식별자(나이+성별)가 같은 레코드 3개 이상 → 개인 특정 불가

| 나이  | 성별  | 질병  |
| --- | --- | --- |
| 30  | 남   | 암   |
| 30  | 남   | 암   |
| 30  | 남   | 암   |

✅ k-익명성 만족 (3개) / ❌ 질병이 다 암 → 추론 가능

**ℓ-다양성 (ℓ=3)**

동질집합 내 민감정보(질병) 종류가 3개 이상

| 나이  | 성별  | 질병  |
| --- | --- | --- |
| 30  | 남   | 암   |
| 30  | 남   | 당뇨  |
| 30  | 남   | 고혈압 |

✅ 질병 3종류 / ❌ 분포가 전체랑 달라 → 통계적 추론 가능

**t-근접성 (t=0.2)**

동질집합 분포와 전체 분포 차이가 t 이하

|나이|성별|질병|
|---|---|---|
|30|남|당뇨|
|30|남|당뇨|
|30|남|고혈압|

이 집합: 당뇨67% 고혈압33% ← 전체 분포(당뇨60% 고혈압30%)랑 유사 ✅

| 기법    | 뭘 보호하냐           |
| ----- | ---------------- |
| k-익명성 | 개인 특정 방지 (사람 수)  |
| ℓ-다양성 | 민감정보 획일화 방지 (종류) |
| t-근접성 | 통계적 추론 방지 (분포)   |

# PERT 핵심 계산식

| 공식 종류           | 계산식              | 특징                                                     |
| --------------- | ---------------- | ------------------------------------------------------ |
| **삼각분포**        | ((O + M + P)/3)  | 단순평균, 모든 값에 동일 가중치, 빠르고 간단함 >> μ                       |
| **PERT (베타분포)** | ((O + 4M + P)/6) | M에 높은 가중치, 현실적 예측에 유리  >> μ                            |
| **표준편차** (PERT) | ((P - O)/6)      | 예측의 불확실성 정도 측정 >> σ                                    |
| 분산 (Variance)   | ((P - O)/6)²     | 분산은 최소값(O)과 최대값(P)을 사용하여 작업 시간의 **변동성**을 측정하는 데 사용됩니다. |

#### 🔥 **정규분포에서 표준편차에 따른 확률 값**

| **시그마 (σ)**   | **범위** | **확률 값 (%)**    |
| ------------- | ------ | --------------- |
| **1시그마 (1σ)** | μ ± 1σ | **68.23%**      |
| **2시그마 (2σ)** | μ ± 2σ | **95.5%**       |
| **3시그마 (3σ)** | μ ± 3σ | **99.7%**       |
| **4시그마 (4σ)** | μ ± 4σ | **99.9937%**    |
| **5시그마 (5σ)** | μ ± 5σ | **99.99994%**   |
| **6시그마 (6σ)** | μ ± 6σ | **99.9999998%** |
# Z-Score

PERT(Program Evaluation and Review Technique) 분석에서 주공정 경로(critical path)의 분산이 4.0이고 기대 소요시간이 20일이다. 프로젝트가 계획된 기간 내에 완료될 확률이 95%가 되려면 프로 젝트 기간을 얼마로 계획해야 하는가? (단, 표준 정규분포에서 누적확률이 95%가 되는 z 값은 1.65로 가정한다.)

μ = PERT 값

**공식 : z = (T - μ) / σ )   ==Z스코어  =  프로젝트 기간 - 기대 소요시간(PERT) / 표준편차**==

문제에서 Z스코어를 1.65로 줌 분산은 표준편차의 제곱이므로 분산 4는 표준편차 2임(2의2승은 4)
(1) 1.65 = X-20 / 2
(2) 1.65 * 2  =  X-20 (표준편차 이항)
(3) 3.3 + 20 = X (기대소요시간 이항)
(4) 23.3 = X

# 트랜스포머

| 종류                             | 설명                                                      |
| ------------------------------ | ------------------------------------------------------- |
| **Self-Attention**             | 같은 시퀀스 내에서 단어끼리 관계 파악. "나는 밥을 먹었다"에서 각 단어가 서로를 얼마나 참조하나 |
| **Multi-Head Self-Attention**  | Self-Attention을 여러 개 병렬로 돌림. 다양한 관점에서 동시에 관계 파악         |
| **Cross-Attention**            | **서로 다른 두 시퀀스 간 어텐션. 인코더(입력문장) → 디코더(출력문장) 연결할 때 사용**   |
| **Multi-Head Cross-Attention** | Cross-Attention을 여러 개 병렬로 돌림                            |

트랜스포머 구조:

- **인코더** → Multi-Head Self-Attention
- **디코더** → Multi-Head Self-Attention + Multi-Head Cross-Attention


## Self-Attention 3요소

| 벡터        | 역할                  | 비유       |
| --------- | ------------------- | -------- |
| Q (Query) | 현재 단어가 다른 단어에게 묻는 것 | 검색어      |
| K (Key)   | 각 단어가 가진 정보의 레이블    | 검색 인덱스   |
| V (Value) | 실제 보유한 의미 정보        | 검색 결과 내용 |

## 동작 순서

```
1. Q × K → 단어 간 유사도 점수 계산
2. Softmax → 점수를 확률로 변환 (합=1)
3. × V → 유사도 가중치로 Value 합산
4. 출력 = 문맥을 반영한 단어 벡터
```

## 예시

"나는 사과를 먹었다" 에서 **"먹었다"** 기준:

```
Q (먹었다가 던지는 질문)
= "나랑 관련있는 단어 누구야?"

K (각 단어의 명찰)
나는   → "주어입니다"
사과를 → "목적어입니다"
먹었다 → "동사입니다"

Q × K (유사도 계산)
먹었다 × 나는   = 0.1 (별로 관련없음)
먹었다 × 사과를 = 0.8 (많이 관련있음)
먹었다 × 먹었다 = 0.5 (자기 자신)

V (실제 내용)
나는   → [나, 주체, 1인칭...]
사과를 → [과일, 목적어, 빨간색...]
먹었다 → [동작, 과거, 섭취...]

최종 출력
= 0.1 × V(나는) + 0.8 × V(사과를) + 0.5 × V(먹었다)
= 사과를 위주로 문맥 반영된 "먹었다" 벡터
```

**"먹었다"가 사과를 먹은 거라는 문맥을 이해한 거임** ㅋ

## Temperature (온도)

**개념:** Softmax 출력 확률 분포를 얼마나 평탄하게/뾰족하게 만들지 조절

```
다음 단어 후보:
"사과" → 60%
"바나나" → 30%
"자동차" → 10%
```

**Temperature 낮음 (0에 가까울수록):**

```
"사과" → 95%
"바나나" → 4%
"자동차" → 1%
→ 항상 사과만 고름. 예측 가능하고 딱딱함
```

**Temperature 높음 (1 이상):**

```
"사과" → 40%
"바나나" → 35%
"자동차" → 25%
→ 뭐가 나올지 모름. 창의적이지만 엉뚱할 수 있음
```

**실용 기준:**

|값|특성|용도|
|---|---|---|
|0.0~0.3|결정적, 보수적|코드 생성, 사실 답변|
|0.7~1.0|자연스러움|일반 대화|
|1.0 이상|창의적, 엉뚱함|창작, 브레인스토밍|

# 📘 EAC 공식 3가지 (시험 핵심)

| 상황                               | 공식                                                         | 의미                    |
| -------------------------------- | ---------------------------------------------------------- | --------------------- |
| **현재 비용 성과(CPI)가 계속 유지된다고 가정**⭐⭐ | **EAC = BAC / CPI**                                        | 가장 자주 출제되는 공식         |
| **앞으로는 계획대로 진행된다고 가정**⭐⭐         | **EAC = AC + (BAC − EV)**                                  | 과거 문제 무시, 남은 일은 정상 진행 |
| **비용과 일정 영향 모두 반영**⭐             | **EAC = AC + (BAC − EV) / (CPI × SPI)**                    | 일정 영향까지 고려            |
| **완료시차이**                        | **VAC = BAC - EAC**                                        |                       |
| **남은 필요 비용**                     | **ETC** =**EAC - AC**                                      | 남은 필요 비용              |
| **완료성과지수**                       | **TCPI = (BAC-EV) / (BAC-AC) **<br><br>(BAC-EV) / (EAC-AC) | 남은 작업 / 남은 예산         |


# 시간으로 구하기

| 항목                | 값                        |
| ----------------- | ------------------------ |
| BAC               | 100만원                    |
| 사업 기간             | 40주                      |
| ==**AT (실제경과)**== | ==**20주**==              |
| 완료율               | 40%                      |
| AC                | 60만원                     |
| EV                | 100 × 40% = **40만원**     |
| PV                | 100 × (20/40) = **50만원** |

**풀이**

PV rate = 100만원 ÷ 40주 = **2.5만원/주**

ES = EV ÷ PV rate = 40 ÷ 2.5 = **16주** → "40만원어치 일은 원래 16주면 끝났어야 했다"

SPI(ES) = ES ÷ AT = 16 ÷ 20 = **0.8** → 20주 걸려서 할 일을 16주치밖에 못했으니 일정 80%밖에 못따라감

 ES=16주, SPI=0.8

**핵심 공식**

- ES = EV / PV rate
- SPI(ES) = ES / AT (1보다 작으면 일정 지연)

#   NoSQL 샤딩(Sharding) — 초압축 정리

### 🔥 1️⃣ 샤딩(Sharding)이란?

> 대용량 데이터를 **여러 서버에 ==수평 분할==(Horizontal Partitioning)** 하여 저장하는 기법

- 하나의 거대한 테이블을
- 여러 개의 물리적 노드에 나눠 저장
  
📌 목적
- 확장성(Scalability)
- 성능 향상
- 트래픽 분산

## 📌수평 분할 vs 수직 분할

| 구분    | 수평 분할 (Horizontal Partitioning) | 수직 분할 (Vertical Partitioning) |
| ----- | ------------------------------- | ----------------------------- |
| 분할 기준 | **행(row)** 기준                   | **열(column)** 기준              |
| 샤딩 여부 | ✅ 샤딩                            | ❌ 샤딩 아님                       |
| 예시    | 고객 1~1000 / 1001~2000           | 이름/주소 테이블 분리                  |
### 🔹 2️⃣ 샤딩 방식 3가지

| 방식        | 원리            | 장단점                      |
| --------- | ------------- | ------------------------ |
| Range     | 값 범위별 분산      | 단순 / **Hotspot(몰림)** 위험  |
| Hash      | hash(key) % N | 균등 분산 / 범위조회 불리          |
| Directory | 중앙 메타서버 관리    | 별도 매핑 테이블로 관리 → 유연하지만 복잡 |

### 🔹 3️⃣ 샤딩 키 (Shard Key)

좋은 조건 (어떤걸 기준으로 쪼갤 것인지?)

- 균등 분포
- 자주 조회 조건
- 변경 거의 없음
  **샤딩 성패 = 샤딩 키 선택**

### 🔹 4️⃣ 샤딩 vs 레플리케이션

|구분|샤딩|레플리케이션|
|---|---|---|
|데이터|서로 다름|동일 복제|
|목적|쓰기 확장|읽기 확장|
|효과|용량 증가|가용성 증가|

# 주소지정방식 정리표

| 영문명                          | 한글명             | 설명                                          |
| ---------------------------- | --------------- | ------------------------------------------- |
| Implicit Addressing          | 묵시적 주소지정 방식     | 명령어 자체에 피연산자가 포함되거나 AC, Stack Top 등으로 자동 결정 |

# 📘 **IPV4 vs IPV6 헤더 구성 비교**

#### 1️⃣ **IPV4 헤더 구성 (기본 20바이트)**

| 필드                                     | 크기    | 설명                             |
| -------------------------------------- | ----- | ------------------------------ |
| **버전 (Version)**                       | 4비트   | **IPv4**일 경우 4                 |
| **헤더 길이 (Header Length)**              | 4비트   | 헤더의 길이를 **32비트 워드** 단위로 표시     |
| **서비스 타입 (Type of Service, TOS)**      | 8비트   | 패킷의 **우선순위** 및 **서비스 요구사항**    |
| **전체 길이 (Total Length)**               | 16비트  | **패킷의 전체 길이** (헤더 + 데이터)       |
| ==**식별자 (Identification)**==           | 16비트  | 패킷을 **식별**하는 번호                |
| ==**플래그 (Flags)**==                    | 3비트   | 패킷의 **분할 여부**를 나타내는 플래그        |
| ==**조각 오프셋 (Fragment Offset)**==       | 13비트  | 패킷의 **조각화**를 위한 오프셋 값          |
| **TTL (Time to Live)**                 | 8비트   | 패킷이 **네트워크에서 살아 있는 시간** (초 단위) |
| **프로토콜 (Protocol)**                    | 8비트   | **상위 계층 프로토콜** (TCP, UDP 등)    |
| **헤더 체크섬 (Header Checksum)**           | 16비트  | 헤더의 **오류 검출**                  |
| **출발지 IP 주소 (Source IP Address)**      | 32비트  | 출발지의 **IPv4 주소**               |
| **목적지 IP 주소 (Destination IP Address)** | 32비트  | 목적지의 **IPv4 주소**               |
| **옵션 (Options)**                       | 가변 크기 | **옵션 필드** (옵션을 포함할 경우에만 존재)    |

- **기본 크기**: **20바이트**
- **옵션** 포함 시 **최대 60바이트**까지 확장

#### 2️⃣ **IPV6 헤더 구성 (고정 40바이트)**

| 필드                                  | 크기    | 설명                                |
| ----------------------------------- | ----- | --------------------------------- |
| **버전 (Version)**                    | 4비트   | **IPv6**일 경우 6                    |
| **트래픽 클래스 (Traffic Class)**         | 8비트   | **서비스 품질** 및 **우선순위** 정의          |
| **플로우 레이블 (Flow Label)**            | 20비트  | 특정 흐름에 대해 **처리 우선순위**를 지정         |
| **페이로드 길이 (Payload Length)**        | 16비트  | **데이터 부분**의 길이                    |
| **다음 헤더 (Next Header)**             | 8비트   | 상위 계층 프로토콜 (TCP, UDP 등)           |
| **홉 제한 (Hop Limit)**                | 8비트   | 패킷이 **네트워크에서 살아 있는 시간** (TTL과 동일) |
| **출발지 주소 (Source IP Address)**      | 128비트 | 출발지의 **IPv6 주소**                  |
| **목적지 주소 (Destination IP Address)** | 128비트 | 목적지의 **IPv6 주소**                  |

- **기본 크기**: **40바이트** 

- **옵션**은 **확장 헤더**로 분리하여 처리
## Pv6 확장 헤더 – Next Header 값 정리

| 값 (Decimal) | 헤더 이름                                | 설명                                    |
| ----------- | ------------------------------------ | ------------------------------------- |
| 0           | Hop-by-Hop Options Header            | 모든 라우터가 반드시 검사해야 하는 옵션 정보를 전달하는 확장 헤더 |
| 43          | Routing Header                       | 패킷이 특정 경로를 따라 전달되도록 경로 정보를 지정         |
| 44          | Fragment Header                      | IPv6 패킷을 분할(fragmentation)할 때 사용되는 헤더 |
| 50          | Encapsulating Security Payload (ESP) | IPsec에서 데이터 암호화 및 무결성 보호 제공           |
| 51          | Authentication Header (AH)           | IPsec에서 패킷 인증 및 무결성 검증 제공             |
| 60          | Destination Options Header           | 목적지 노드에서만 처리하는 옵션 정보 전달               |
| 59          | No Next Header                       | 다음 헤더가 없음을 의미                         |

# 4계층 SCTP QUIC 비교

| 구분    | SCTP                       | QUIC                        |
| ----- | -------------------------- | --------------------------- |
| 계층    | 전송계층 (L4)                  | UDP 기반 (L4+L7 사이)           |
| 기반    | 자체 프로토콜                    | UDP                         |
| 전송 방식 | 메시지 기반                     | 스트림(멀티플렉싱)                  |
| 신뢰성   | 있음 (재전송) SYN플루딩 방어         | 있음 (UDP 위 구현)               |
| 핵심 특징 | Multi-stream, Multi-homing | 0-RTT, Multiplexing, TLS 내장 |
| 목적    | TCP 보완 (통신 안정성)            | TCP 대체 (웹 성능 개선)            |
| 사용    | 통신/신호 시스템                  | HTTP/3                      |
간단하게 정리해줄게.

**SCTP** (Stream Control Transmission Protocol)

- TCP + UDP 장점 합친 전송계층 프로토콜
- **멀티스트리밍** (여러 스트림 동시 전송)
- **멀티호밍** (여러 IP 주소 동시 사용, 장애시 자동전환)
- 주로 통신망(SS7 over IP) 에서 씀

**QUIC** (Quick UDP Internet Connections)

- Google이 만든 프로토콜, HTTP/3 기반
- UDP 위에서 동작하는데 TCP처럼 신뢰성 보장
- **핸드셰이크 빠름** (0-RTT 또는 1-RTT)
- **HOL 블로킹 없음** (TCP의 단점 해결)
- 지금 크롬, 유튜브 다 씀

한 줄 요약:

- SCTP → **TCP 업그레이드판** (통신망용)
- QUIC → **UDP 기반 빠른 TCP** (웹용)

# ARQ

|구분|Stop-and-Wait|Go-Back-N|Selective Repeat|
|---|---|---|---|
|재전송|1개|오류 이후 전부|오류난 것만|
|송신 윈도우|1|2^m - 1|2^m / 2|
|수신 윈도우|1|**1**|2^m / 2|
|효율|최악|중간|최고|
|복잡도|단순|중간|복잡|
|시퀀스 비트|1비트|m비트|m비트|

m = 시퀀스 번호에 사용하는 비트 수

예를 들어 m=3이면:

- 시퀀스 번호 = 0~7 (2³=8개)
- Go-Back-N 송신 윈도우 = 2³-1 = 7
- Selective Repeat 송신 윈도우 = 2³/2 = 4

## 📘  데이터 압축 기법 정리

#### 무손실 압축 (Lossless)

| 기법                          | 한글명           | 설명                                                  |
| --------------------------- | ------------- | --------------------------------------------------- |
| **Run-length Coding**       | 실행(반복) 길이 부호화 | 연속 반복되는 정보를 횟수+값으로 표현 (TEAAAAAAAAAALOV → TEA@12LOV) |
| **Dictionary Coding (LZW)** | 사전 부호화        | 반복 어구를 사전에 등록 후 위치 정보로 치환                           |
| **Huffman Coding**          | 허프만 코딩        | 출현 빈도 기반으로 각 단위 정보에 비트 수를 효율적으로 할당                  |
| **Arithmetic Coding**       | 산술 부호화        | 메시지 전체를 구간으로 나누어 이진으로 인코딩                           |

#### 손실 압축 (Lossy)

| 기법                                  | 한글명             | 설명                                                   |
| ----------------------------------- | --------------- | ---------------------------------------------------- |
| **Delta Modulation**                | 델타 변조<br>(예측부호) | 양자화된 표본값을 그대로 저장하지 않고, 기준 표본값과 예측값의 **차이값**만 저장      |
| **DCT** (Discrete Cosine Transform) | 이산 코사인 변환(예측부호) | 수학적 변환(Transform Coding) 사용. JPEG, MPEG, H.261 등에 활용 |
| **Quantization**                    | 양자화             | 미리 정의된 quantum 정보를 이용하여 주어진 데이터를 코딩                  |

# Kubernetes 핵심 구조 (도커 오케스트레이션)



```
Master Node (제어)
├── API Server        ← 모든 요청 진입점
├── Scheduler         ← 컨테이너 배치 결정
├── Controller-Manager ← 클러스터 상태 유지
└── etcd              ← 클러스터 설정·상태 저장

Worker Node (실행)
├── kubelet           ← 마스터 명령 받아 Pod 실행
├── kube-proxy        ← 네트워크 규칙, 로드밸런싱
├── Container Runtime ← 실제 컨테이너 실행
└── Pod               ← 컨테이너 묶음, 최소 배포 단위
```

### 가용성 계산

### 고가용성 HA (High Availability)

#### HA 구성방식 3가지

| 구분  | Hot Standby                                 | Mutual Takeover                                   | Concurrent Access(미러사이트)                       |
| --- | ------------------------------------------- | ------------------------------------------------- | ---------------------------------------------- |
| 방식  | Active + Standby 구성. 장애 시 Standby로 Failover | 2개 시스템이 각각 고유 기능 수행하다 장애 시 상대 자원 인수하여 2개 업무 동시 수행 | 다수 시스템이 동일 기능 수행 (로드밸런싱 유사). 장애 시 Failover 불필요 |
| 장점  | Failover 후에도 동일 성능 보장                       | 자원 효율적 이용                                         | Failover 불필요                                   |
| 단점  | 자원 낭비 (하나는 항상 대기)                           | 장애 후 한 서버에 업무 몰릴 수 있음                             | 로드밸런싱 위한 추가 장비 필요                              |


**가용성 공식**

```
가용성 = MTTF / (MTTR + MTTF) × 100
```

#### 용어 정리

| 용어       | 풀네임                       | 의미                                         |
| -------- | ------------------------- | ------------------------------------------ |
| **MTTF** | Mean Time to Failure      | 정상 가동 시작 후 다음 고장까지 **평균 사용 시간**            |
| **MTTR** | Mean Time to Repair       | 고장을 **수리하는 평균 시간**                         |
| **MTBF** | Mean Time Between Failure | 고장 시작 후 다음 고장까지 **평균 고장 간격** = MTTF + MTTR |

#### 예제 풀이

**문제:** 40시간 운영 → 10시간 수리 → 20시간 운영 → 10시간 수리

```
MTTF = (40 + 20) / 2 = 30시간  (정상 운영 평균)
MTTR = (10 + 10) / 2 = 10시간  (수리 시간 평균)

가용성 = 30 / (10 + 30) = 30 / 40 = 0.75 = 75%
```

#### 시험 포인트

- **공식 = MTTF / (MTTR + MTTF)**
- **MTTF ↑, MTTR ↓ → 가용성 ↑**
- **MTBF = MTTF + MTTR** (고장 간격)
- 여러 번의 데이터 나오면 **평균값** 먼저 구하고 공식 적용

HA 다운타임 (암기)

| 가용성         | Nines           | 1년 중 다운타임  |
| ----------- | --------------- | ---------- |
| 90%         | one nine        | 36.53일     |
| 99%         | two nines       | 3.65일      |
| **99.9%**   | **three nines** | **8.77시간** |
| 99.99%      | four nines      | 52.60분     |
| **99.999%** | **five nines**  | **5.26분**  |
| 99.9999%    | six nines       | 31.56초     |

# 📘 컴퓨터 시스템 보안 설계 원칙 (표 정리)

| 구분                                    | 설명                                                                     |
| ------------------------------------- | ---------------------------------------------------------------------- |
| 최소 권한 (Least Privilege)               | 사용자는 수행해야 할 작업에 필요한 최소한의 권한만 가져야 하며, 더 이상 필요하지 않은 경우 해당 권한은 회수되어야 한다.  |
| 고장 안전 디폴트 (Fail-Safe Defaults)        | 명시적으로 허가되지 않은 모든 자원에 대한 접근은 기본적으로 거부되어야 한다.                            |
| 매커니즘의 효율성 (Economy of Mechanism)      | 시스템 설계는 가능한 한 단순하게 구성되어야 하며, 복잡성은 오류 및 취약점 발생 가능성을 증가시킨다.              |
| 완벽한 조율 (Complete Mediation)           | 모든 접근 시도는 반드시 검사되어야 하며, 접근 시마다 권한 검증이 이루어져야 한다.                        |
| 열린 설계 (Open Design)                   | 보호 메커니즘은 공격자의 무지에 의존하지 않아야 하며, 공개되고 검증된 방식으로 설계되어야 한다.                 |
| 권한 분리 (Separation of Privilege)       | 시스템은 단일 조건에 의해 권한을 부여해서는 안 되며, 다중 조건을 만족해야 권한이 부여되어야 한다.               |
| 최소 공통 메커니즘 (Least Common Mechanism)   | 공유 메커니즘의 수와 사용을 최소화해야 하며, 공유 객체는 정보 흐름 및 의도되지 않은 상호작용에 대한 잠재적 위험을 가진다. |
| 사용의 편리성 (Psychological Acceptability) | 사용자가 일반적으로 시스템을 사용하는 과정에서 자연스럽게 보안 메커니즘이 적용되도록 설계되어야 한다.               |

| 계산의 용이성                 | 입력값 x가 주어지면 H(x)는 계산적으로 쉽게 구할 수 있어야 한다.                                      |
| 일방향성 (One-Way)          | 메시지(평문)에서 해시값을 구하는 것은 쉽지만, 해시값에서 원래 메시지를 구하는 것은 매우 어려워야 한다. (**역상 저항성)**     |
| 약한 충돌 회피성               | 입력값 x가 주어졌을 때 H(x') = H(x)인 x' (x ≠ x)를 찾는 것은 계산적으로 불가능해야 한다. (**제2역상 저항성**) |
| 강한 충돌 회피성               | H(x) = H(x')인 서로 다른 두 입력값 x, x'를 찾는 것은 계산적으로 불가능해야 한다. **(충돌 저항성)**          |

## 📘 무선 보안 기술 정리 (SSID, WEP, WPA, WPA2, WPA3)

| 구분   | 내용                                                                                                                                            |
| ---- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| SSID | Service Set Identifier  <br>무선랜에서 네트워크 이름을 의미하며 기본적인 접속 제어 기능 제공  <br>AP가 비콘 신호에 SSID를 포함하여 브로드캐스트  <br>SSID만으로는 보안 기능이 약하며 접속 허용/차단 기준으로는 취약 |

| WPA3 | SAE 인증 방식 적용 (PSK 개선)  <br>GCMP-256(128) 암호화 사용  <br>PMF (Protected Management Frames) 적용                                                     |

# 📘 암호화 블록

| 모드          | 한글명               | 동작 방식              | 특징                | 시험 포인트           | 오류 전파          |
| ----------- | ----------------- | ------------------ | ----------------- | ---------------- | -------------- |
| **ECB**     | 전자 코드북            | 각 블록을 독립적으로 암호화    | 동일 평문 → 동일 암호문    | 패턴 노출 / 보안 취약    | 1블록만 영향        |
| **CBC**     | 암호 블록 연쇄          | 이전 암호문과 XOR 후 암호화  | 패턴 숨김             | 병렬 처리 불가         | 2블록 영향         |
| **CFB**     | 암호 피드백            | 이전 암호문을 입력으로 사용    | 블록 암호를 스트림처럼 사용   | 오류 전파 발생         | 일부(2개이상) 비트 영향 |
| **OFB**     | 출력 피드백            | 키 스트림 생성 후 XOR     | 스트림 암호처럼 동작       | 오류 전파 없음         | 해당 블록만 영향      |
| **CTR**     | 카운터 모드            | 카운터 값을 암호화 후 XOR   | 병렬 처리 가능          | 고속 암호화           | 해당 블록만 영향      |
| ==**GCM**== | ==갈루아 카운터 모드==    | ==CTR + GHASH 인증== | ==기밀성 + 인증(무결성)== | ==AEAD 모드==      | 블록 영향          |
| ==**CCM**== | ==카운터 + CBC-MAC== | ==CTR + CBC-MAC==  | ==기밀성 + 인증(무결성)== | ==무선 네트워크에서 사용== | 블록 영향          |


👉 암호화 알고리즘으로 복호화 가능 : CFB OFB CTR 
👉 암호화 복호화 병렬처리 : ECB CTR
👉 복호화만 병렬처리 가능 : CBC CFB

# 신뢰성 공학

| 구분      | POFOD(온디멘드 고장확률)                 | ROCOF(고장발생비율)                 |
| ------- | -------------------------------- | ----------------------------- |
| 풀네임     | Probability Of Failure On Demand | Rate Of Occurrence Of Failure |
| 의미      | 요청 시 실패할 확률                      | 단위 시간당 실패 발생 횟수               |
| 측정 단위   | 확률 (0~1)                         | 실패횟수/시간                       |
| 적합한 시스템 | 호출 빈도 낮은 시스템                     | 지속적으로 운영되는 시스템                |
| 예시      | 원전 제어 시스템 (요청할 때 잘 동작하냐)         | 통신 시스템 (1시간에 몇 번 죽냐)          |

**암기 포인트**

- POFOD = **요청할 때** 실패 확률 → 비행기 엔진, 원전처럼 평소엔 안 쓰다가 결정적 순간에 써야 하는 것
- ROCOF = **시간당** 실패 횟수 → 서버, 통신망처럼 항상 돌아가는 것

**한줄 구분**: 언제 쓰냐가 불규칙하면 POFOD, 항상 돌면 ROCOF

**SCTP (Stream Control Transmission Protocol)**

|구분|내용|
|---|---|
|정의|TCP + UDP 장점 결합한 전송계층 프로토콜|
|연결 방식|4-way 핸드셰이크 (TCP 3-way보다 보안 강화)|
|특징|멀티호밍, 멀티스트리밍 지원|

**4-way 핸드셰이크**

| 순서  | 방향    | 메시지         | 역할                 |
| --- | ----- | ----------- | ------------------ |
| 1   | 클라→서버 | INIT        | 연결 요청              |
| 2   | 클라←서버 | INIT ACK    | 응답 + **쿠키 전송**     |
| 3   | 클라→서버 | COOKIE ECHO | **쿠키 돌려줌** (신원 확인) |
| 4   | 클라←서버 | COOKIE ACK  | 연결 완료              |

**TCP vs SCTP 비교**

| 구분              | TCP   | SCTP                  |
| --------------- | ----- | --------------------- |
| 핸드셰이크           | 3-way | **4-way**             |
| 멀티호밍            | 미지원   | **지원** (여러 IP 동시 사용)  |
| 멀티스트리밍          | 미지원   | **지원** (여러 스트림 동시 전송) |
| HOL 블로킹         | 발생    | 스트림별 독립으로 감소          |
| SYN Flooding 방어 | 취약    | 쿠키 메커니즘으로 방어          |

**쿠키 메커니즘이 핵심**: TCP는 **SYN Flooding** 공격에 취약한데 SCTP는 COOKIE ECHO/ACK 단계로 방어함. **디도스도 방어함**

# Apriori

연관규칙 마이닝에서 6-항목집합 {Apple, Banana, Cherry, Grape, Melon, Tomato}가 빈발(frequent)로 판명되었을 때, 이 6-항목집합에 대해 고려할 수 있는 연관규칙은 모두 몇 개인가?

① 6개 ② 15개 ③ 32개 ④ 62개

**정답: ④ 62개**

**공식**

n개 항목집합의 연관규칙 수 = ==**2^n - 2**==

- 전체 부분집합 수 = 2^6 = 64
- 공집합(∅) 제외 = -1
- 전체집합 제외 = -1
- **64 - 2 = 62개**

**왜 2를 빼냐**

연관규칙은 `A → B` 형태인데 A와 B 둘 다 공집합이면 안 됨.
- A = ∅ 인 경우 (B = 전체집합) → 제외
- B = ∅ 인 경우 (A = 전체집합) → 제외
그래서 2^n - 2 = **62개**.

# [TF-IDF (Term Frequency - Inverse Document Frequency)]

■ 정의
= 단어 중요도 = 많이 나오고(문서 내) / 흔하지 않을수록(전체 문서)

■ 공식
= TF-IDF = TF × IDF
= TF = 특정 문서 내 단어 빈도
= IDF(역 문서 빈도 ) = log10(전체 문서 수 / 해당 단어 포함 문서 수)
많이 나올수록 **중요도 낮춤**

예시) 전체문서 수 10개 , '데이터'가 포함된 문서 수 5개
      문서 1에 나타난 전체단어수 100개
      문서 1에 나타난 '데이터' 출현 횟수 4회


TF(특정 문서 내 단어 빈도) =  4/100 = 0.4

IDF(log10(전체 문서 수 / 해당 단어 포함 문서 수)) =  LOG10(10/5) = 0.3
(계산 어려우니 하단 암기)

TF-IDF = 0.4 * 0.3 = 0.012

| 값      | 암기        |
| ------ | --------- |
| ==log₁₀2== | ==**0.3**==   |
| log₁₀3 | **0.477** |
| log₁₀4 | **0.6**   |
| log₁₀5 | **0.699** |
| log₁₀8 | **0.9**   |
==log₁₀2 = 0.3이니 이 기준으로 거듭제곱==

4 = 0.6 
8 = 0.9
16 = 1.2

## RAG 아키텍처 정리

LLM이 답변 생성 시 **외부 지식 베이스에서 관련 정보를 검색**해서 프롬프트에 추가함으로써, 모델 재학습 없이 더 정확한 답변을 생성하는 아키텍처.
### 3단계 흐름

```
[사용자 질문]
      ↓
1. Retrieval (검색)
   - 질문을 벡터로 변환
   - 벡터DB에서 유사 문서 검색
   - 주로 벡터DB 사용 (Pinecone, Chroma 등)
      ↓
2. Augmentation (증강)
   - 검색된 문서 + 원래 질문 결합
   - LLM에 넣을 프롬프트 생성
      ↓
3. Generation (생성)
   - LLM이 프롬프트 받아서 답변 생성
   - 재학습 없음
```

### 핵심 특징 정리

| 항목      | 내용                              |
| ------- | ------------------------------- |
| 목적      | LLM 재학습 없이 최신/외부 데이터 반영         |
| 검색 DB   | 벡터DB (RDB/SQL 아님)               |
| 학습 여부   | 없음 (Fine-tuning과 다름)            |
| 프롬프트 결합 | Augmentation 단계 (Generation 아님) |
| 장점      | 환각(Hallucination) 감소, 최신 정보 반영  |

### Fine-tuning vs RAG 비교

|구분|Fine-tuning|RAG|
|---|---|---|
|방식|모델 재학습|외부 검색 연동|

## 임베디드 소프트웨어 개념 정리

## N²SF (국가 망 보안체계) 정보등급 분류

(출처: 국정원 N²SF 가이드라인 1.0, 2025)

|등급|영문|대상 정보|보안 수준|
|---|---|---|---|
|C|Classified (기밀)|비밀, 안보/국방/외교/수사 등 기밀정보, 국민 생명/안전 직결 정보|엄격 통제|
|S|Sensitive (민감)|개인/국가 이익 침해 가능 비공개 정보, 로그/백업 등|제한 접근|
|O|Open (공개)|C, S 이외 모든 정보|활용 우선|

**적용 절차 5단계:**

```
준비 → C/S/O 등급분류 → 위협식별 → 보안대책 수립 → 적절성 평가/조정
```

**6개 보안통제 항목:** 권한 / 인증 / 분리 및 격리 / 통제 / 데이터 / 정보자산

## JWT (JSON Web Token)

**정의:** 사용자 인증 정보를 JSON 형태로 담아 서명한 토큰

**구조: Header.Payload.Signature (점으로 구분)**

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyIjoiaWsifQ.abc123xyz
      Header              Payload          Signature
```

## 구성요소

|구성|내용|예시|
|---|---|---|
|Header|알고리즘 타입|{"alg":"HS256","typ":"JWT"}|
|Payload|실제 데이터(클레임)|{"user":"이건우","exp":1234567}|
|Signature|Header+Payload를 비밀키로 서명|HMACSHA256(header.payload, secret)|

## 특징

```
장점:
- 서버가 세션 저장 안 해도 됨 (Stateless)
- 분산 환경에 유리

단점:
- 토큰 탈취되면 만료 전까지 막을 방법 없음
- Payload는 Base64 인코딩이라 누구나 볼 수 있음
  (암호화 아님 → 민감정보 넣으면 안 됨)
```

**시험 포인트:** Payload는 암호화가 아닌 **Base64 인코딩**이라 디코딩하면 내용 볼 수 있음

# 프로그램 포트폴리오

| 구분        | 정의                                             | 핵심          |
| --------- | ---------------------------------------------- | ----------- |
| **프로젝트**  | 독특한 산출물을 만들기 위한 **한시적** 노력                     | 기간 有, 단일 목표 |
| **프로그램**  | 개별 관리 시 얻을 수 없는 **시너지**를 위해 통합 관리하는 관련 프로젝트 묶음 | 상호의존성 관리    |
| **포트폴리오** | 전략적 목표 달성을 위해 묶은 프로젝트·프로그램·운영 업무의 집합           | 전략적 우선순위 관리 |

**계층 구조**

```
포트폴리오 (전략)
├── 프로그램 A (시너지)
│   ├── 프로젝트 1
│   └── 프로젝트 2
└── 프로젝트 3 (단독)
```

**핵심 차이**

|항목|프로그램|포트폴리오|
|---|---|---|
|관리 초점|**시너지·상호의존성**|**전략 정렬·투자 우선순위**|
|구성요소 간 관계|관련있음|관련 없어도 됨|
|성공 기준|편익 실현|전략 목표 달성|
|담당자|프로그램 관리자|포트폴리오 관리자|

**암기 포인트**

- 프로그램 = **시너지** 키워드
- 포트폴리오 = **전략** 키워드
- 포트폴리오는 서로 무관한 것들도 묶일 수 있음

이름 유래가 웃긴데:

> 19세기 골턴이 키 연구하다가 "키 큰 부모의 자녀는 평균으로 되돌아온다(회귀한다)"는 걸 발견해서 붙인 이름

근데 지금은 그 의미 아니고 그냥 **"예측 모델"** 의미로 쓰임.

**종류**

| 종류          | 내용                              |     |
| ----------- | ------------------------------- | --- |
| **선형 회귀**   | Y = aX + b (직선으로 예측)            |     |
| **다중 회귀**   | Y = a₁X₁ + a₂X₂ + ... (변수 여러 개) |     |
| **로지스틱 회귀** | 0 또는 1 예측 (분류용, 이름만 회귀임)        |     |

**표본추출(Sampling) 기법 정리**

|기법|핵심|키워드|
|---|---|---|
|**속성 표본추출** (Attribute)|결과의 **적합성 여부** 판단|Yes/No, 준거성 시험, 규정 위반 확인|
|**변수 표본추출** (Variable)|**단위당 평균·비율** 추정|연속적 등급, 적합성 달성 정도|
|**색출 표본추출** (Discovery)|위반율 **극단적으로 낮을 때**|하나라도 발견되면 통제 부적정 판단|
|**무작위 표본추출** (Random)|**난수** 활용 선별|컴퓨터 난수 생성|

**암기 포인트**

- 속성 = **있냐 없냐** (이분법)
- 변수 = **얼마나** (연속값)
- 색출 = **제로 기대** (하나라도 나오면 아웃)
- 무작위 = **난수**

감리 시험에서 색출 표본추출이 가장 자주 헷갈리는 포인트. "위반율 극단적으로 낮은 경우" + "하나라도 발견되면 통제 부적정" 이 두 개 세트로 외워.

# 📘 리스크 유형 (가변성 vs 모호성)

| 구분  | 가변성 리스크             | 모호성 리스크          |
| --- | ------------------- | ---------------- |
| 정의  | 결과는 알지만 값이 변동하는 리스크 | 결과 자체가 불확실한 리스크  |
| 특징  | 범위 내 변동 (확률 분포 존재)  | 정보 부족 / 불확실성 큼   |
| 원인  | 자연 변동성, 성능 편차       | 요구사항 불명확, 기술 불확실 |
| 대응  | 분석, 예측, 버퍼 설정       | 정보 수집, 명확화, 탐색   |

|구분|유형|설명|
|---|---|---|
|**사건 리스크** (Event Risk)|-|발생 여부가 불확실한 특정 사건 (ex. 핵심인력 이탈)|
|**비사건 리스크** (Non-event Risk)|가변성(변동성) 리스크|알고 있지만 정도가 불확실|
||모호성(Ambiguity) 리스크|무엇이 불확실한지조차 모름|

**PMBOK 7판 불확실성 성과영역 - 모호성 (Ambiguity)**

**불확실성 4가지 유형 (VUCA)**

| 유형                   | 내용                                      | 대응                |
| -------------------- | --------------------------------------- | ----------------- |
| **일반 불확실성**          | 모든 프로젝트에 존재, 결과 불명확<br>Unknown Unknowns | 리스크 관리<br>        |
| **모호성** (Ambiguity)  | 원인·의미 불명확                               | 점진적 구체화, 프로토타이핑   |
| **복잡성** (Complexity) | 인적·환경 요인으로 관리 어려움                       | 분리, 시뮬레이션, 다양한 관점 |
| **변동성** (Volatility) | 빠르고 예측 불가한 변화                           | 대안 분석, 예비비        |

**모호성 2가지 유형**

| 유형          | 내용                               | 대응              |
| ----------- | -------------------------------- | --------------- |
| **개념적 모호성** | 의미 자체가 불명확 → "이게 무슨 뜻?"          | 정의 수립, 가정 도전    |
| **상황적 모호성** | 결과가 불명확 → "어떻게 될지 모름"<br>여러가지 옵션 | 점진적 구체화, 프로토타이핑 |

# 공격기법

| 공격 기법             | 설명                                                                                |
| ----------------- | --------------------------------------------------------------------------------- |
| Drive by Download | 웹 브라우저 취약점을 이용해 웹 페이지를 변조, 보안 패치 안 된 사용자가 모르는 사이에 악성코드를 다운로드·설치하게 하는 공격(**제로데이**) |
| Ransomware        | 중요 파일을 암호화한 후 복호화를 조건으로 금품을 요구하는 악성프로그램 (ex. 크립토라커)                               |
| Watering hole     | 특정 대상이 자주 접속하는 사이트의 취약점을 파악해 악성코드를 심어두고 공격 대상이 접속하기를 기다리는 공격                      |
| APT               | 지능형 지속 공격. 장기간 은닉하며 정보를 수집·유출, 주로 산업 기반 시설을 목표로 하는 공격                             |
**제로데이 취약점 (Zero-day Vulnerability)**

**이름의 의미**

패치가 나오기까지 "0일" 남았다는 뜻. 즉 벤더도 모르는 취약점.

|유형|정의|특징|
|---|---|---|
|**Virus**|프로그램에 기생하여 복제|숙주 필요|
|**Worm**|네트워크 통해 자체 확산|숙주 불필요|
|**Trojan Horse**|정상 프로그램으로 위장|사용자 실행 유도|
|**Backdoor**|인증 절차 없이 접근 가능한 통로|원격 제어 가능|
|**Rootkit**|관리자 권한을 숨기고 유지|탐지 회피|
|**Ransomware**|파일 암호화 후 금전 요구|데이터 인질|
|**Logic Bomb**|특정 조건에서 실행되는 악성코드|시간·이벤트 트리거|

ARP 리다이렉트 과 **ARP 스푸핑이랑 차이**

| 구분  | ARP 스푸핑        | ARP 리다이렉트                       |
| --- | -------------- | ------------------------------- |
| 타겟  | 특정 호스트         | 게이트웨이                           |
| 방식  | 피해자 테이블 오염     | 피해자한테 "게이트웨이 MAC = 내 MAC" 하고 속임 |
| 목적  | 호스트 간 트래픽 가로채기 | 외부로 나가는 모든 트래픽 가로채기             |

**흐름**

```
피해자 → "인터넷 가야지, 게이트웨이로"
       → 근데 ARP 테이블에 게이트웨이 MAC이 공격자 MAC
       → 공격자한테 감
       → 공격자가 진짜 게이트웨이로 포워딩 (피해자는 모름)
```

사실 기법 자체는 ARP 스푸핑이랑 동일한데 **타겟을 게이트웨이로 잡느냐** 차이야.

# 동기부여 이론 전체 정리**

**내용이론 (What - 무엇이 동기를 주는가)**

|이론|학자|핵심 내용|
|---|---|---|
|욕구단계설|매슬로우 (Maslow)|생리→안전→사회→존경→자아실현 5단계. 하위 욕구 충족 후 상위로 이동. 단계 건너뜀 불가|
|ERG이론|앨더퍼 (Alderfer)|존재(Existence)·관계(Relatedness)·성장(Growth) 3단계. 매슬로우와 달리 **좌절-퇴행 가능**|
|동기-위생이론|허즈버그 (Herzberg)|동기요인(성취·인정·성장)은 만족 유발. 위생요인(급여·환경·정책)은 불만족 예방만. **만족↔불만족이 별개의 축**|
|X·Y이론|맥그리거 (McGregor)|X이론: 인간은 게으르고 강제해야 함. Y이론: 인간은 자율적이고 책임감 있음. **맥그리거는 Y이론 지지**|
|성취동기이론|맥클리랜드 (McClelland)|성취(nAch)·권력(nPow)·친교(nAff) 3욕구. 욕구는 **학습·경험으로 형성**. 탁월한 관리자 = nPow↑ nAff↓|
|자기결정이론|다니엘 핑크 (Daniel Pink)|자율성(Autonomy)·숙련(Mastery)·목적(Purpose) 3요소. 외재적 보상보다 **내재적 동기**가 창의적 업무에 효과적|

**허즈버그 핵심 포인트**

| 요인                | 충족 시                | 미충족 시                 |
| ----------------- | ------------------- | --------------------- |
| 동기요인 (성취, 인정, 성장) | 만족                  | **불만족 아님 → 그냥 만족 없음** |
| 위생요인 (급여, 환경, 정책) | **만족 아님 → 불만족만 없앰** | 불만족                   |

**과정이론 (How - 어떻게 동기가 생기는가)**

|이론|학자|핵심 내용|
|---|---|---|
|기대이론|브룸 (Vroom)|동기 = 기대 × 수단성 × 유인가. **셋 중 하나라도 0이면 동기 0**|
|공정성이론|아담스 (Adams)|자신의 투입/산출 비율을 타인과 비교. 불공정 느끼면 **행동 변화**|
|강화이론|스키너 (Skinner)|긍정적 결과 → 행동 반복. 부정적 결과 → 행동 감소. **과거 결과가 미래 행동 결정**|

# 실체화뷰
```sql
-- Oracle 기준
CREATE MATERIALIZED VIEW mv_sales
BUILD IMMEDIATE           -- 즉시 데이터 채움
REFRESH FAST              -- 변경분만 갱신
ON COMMIT                 -- 커밋 시 자동 갱신
AS
SELECT dept_id, SUM(amount) AS total
FROM sales
GROUP BY dept_id;
```

**주요 옵션:**

| 구분      | 옵션        | 내용                |
| ------- | --------- | ----------------- |
| BUILD   | IMMEDIATE | 생성 시 즉시 데이터 채움    |
| BUILD   | DEFERRED  | 나중에 수동으로 채움       |
| REFRESH | FAST      | 변경분만 갱신 (빠름)      |
| REFRESH | COMPLETE  | 전체 재실행 (느림)       |
| REFRESH | FORCE     | FAST 안되면 COMPLETE |
| 시점      | ON COMMIT | 커밋 시 자동           |
| 시점      | ON DEMAND | 수동 갱신             |

#### [요구정의단계 감리]

- **목적**: 계약문서 과업내용 → 요구사항정의서에 적정 반영 여부 점검
- **시기**: 분석단계 완료 시점 (일반적 개발방법론에서 별도 단계 아님)
- **사업자 역할**: 요구사항정의서 제출 + 세부 항목별 과업내용 반영 대비표 작성·발주자 확인 후 감리법인 제출 (감리 실시 전까지)
- **감리법인 역할**: 과업내용이 요구사항정의서에 적정 반영 여부 점검 → 결과를 감리수행결과보고서에 명시 → 발주자 제출

#### [설계단계 감리]

- **목적**: 설계 산출물의 과업내용 반영 여부 점검
- **시기**: 설계단계 완료 시점 (반복적 개발 시 중요도 높은 영역 포함, 반복 차수 정하여 수행)
- **사업자 역할**: 설계 산출물 제출 + 대비표 작성·제출
- **감리법인 역할**:
    1. 설계 산출물이 과업내용을 적정하게 반영하는지 여부
    2. 대비표가 세부 검사항목별 **적합·부적합 판정** 가능하도록 구체화되었는지 여부
- **보고**: 적합·부적합으로 명시하여 발주자 제출

> ⭐ **대비표**: 요구사항추적표 기반으로 요구사항ID 단위 → 설계내용·검사대상·검사기준 추적 문서  
> 기능요구사항 → 테스트 시나리오로 구체화 가능

#### [종료단계 감리]

- **목적**: 과업이행 여부 최종 점검
- **시기**: 사업자가 **통합시험 완료 후** + 시정조치 확인 기간이 검사 이전 종료 가능한 시점
- **감리법인 역할**: 검사 전까지 세부 검사항목별 과업내용 이행의 적합·부적합 점검 → 보고서 제출
- 과업이행 확인 = **통합시험·연계시험 관점**에서 수행

## 알파 vs 베타 테스트

|구분|알파 테스트|베타 테스트|
|---|---|---|
|수행자|**개발사 내부** 사용자|**외부** 실제 사용자|
|장소|개발사 **내부** 환경|사용자 **실제** 환경|
|시점|베타 **이전**|정식 출시 **이전**|
|목적|내부 품질 검증|실사용 환경 검증|
|통제|개발팀이 **통제**|통제 **어려움**|

**비유:**

> **알파** = 식당 오픈 전에 직원들끼리 음식 먹어보는 것 **베타** = 단골손님 몇 명 불러서 시식 행사 하는 것 **정식출시** = 일반 영업 시작

**시험 핵심:**

> 알파 = **내부**, 베타 = **외부** 둘 다 **인수테스트** 의 종류임

## One-Hot Encoding

**개념:**

> 범주형 데이터를 숫자로 변환하는 방법 해당 카테고리만 1, 나머지는 전부 0 (비트맵느낌)

**예시:**

색상 데이터가 있다면:

|원본|빨강|파랑|초록|
|---|---|---|---|
|빨강|1|0|0|
|파랑|0|1|0|
|초록|0|0|1|

**왜 쓰냐:**

> 그냥 숫자로 하면 (빨강=1, 파랑=2, 초록=3) 모델이 "초록 > 파랑 > 빨강" 이라는 **순서가 있다고 착각**함 One-Hot 쓰면 순서 없이 동등하게 처리

**단점:**

> 카테고리 많으면 컬럼 폭발 💥 예) 국가 200개면 컬럼 200개 생김

**시험 키워드:**

> 범주형 → 수치형 변환 희소행렬(Sparse Matrix) 발생 
> 차원의 저주(변수(컬럼) 수가 많아질수록 데이터가 희박해져서 분석이 오히려 망가지는 현상) 유발 가능

**Page Fault Frequency (PFF) = 페이지 폴트 발생 빈도 기반으로 프레임 수 조절하는 기법**

쓰레싱 해결책 중 하나야!

|상황|조치|
|---|---|
|PFF **높음** (폴트 많음)|프레임 **추가** 할당|
|PFF **낮음** (폴트 적음)|프레임 **회수**|

**워킹셋이랑 차이:**

|-|워킹셋|PFF|
|---|---|---|

|방식|필요한 페이지 예측|실제 폴트 발생 후 반응|
|성격|**예측적**|**반응적**|

**WAF (Web Application Firewall) = 웹 애플리케이션 방화벽**

**핵심:**

일반 방화벽이 IP/포트 기반이라면 WAF는 **HTTP/HTTPS 트래픽 내용을 분석**해서 웹 공격 차단

**막는 것들:**

|공격|설명|
|---|---|
|SQL Injection|DB 쿼리 조작|
|XSS|악성 스크립트 삽입|
|CSRF|위조 요청|
|파일 업로드 공격|악성 파일 업로드|

**일반 방화벽이랑 차이:**

|-|일반 방화벽|WAF|
|---|---|---|
|계층|3~4계층|**7계층(응용)**|
|기준|IP/포트|**HTTP 내용**|
|웹 공격 차단|❌|✅|

**배치 위치:**

```
인터넷 → WAF → 웹서버
```

웹서버 앞단에 두고 **모든 웹 트래픽을 검사**하는 거야 👍
