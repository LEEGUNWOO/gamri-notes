•컨설팅업무량 = 정보전략계획 수립 업무 가중치 × 정보전략계획 수립 난이도

### 📌 (2) 기능점수방식 소프트웨어 개발비 계산

#### 🔥 (2)-1/10 개요

기능점수(FP; Function Point) 방법은 **사용자 관점**("소프트웨어가 어떻게 구현되었는지"의 공급자 관점이 아니라 "사용자가 어떠한 기능을 요구했는지"의 수요자 관점에서 측정)에서의 사용자가 요구하고 사용자에게 인도되는 기능을 **정량적으로 산정**하는 소프트웨어 규모 측정방법

•ISO/IEC 14143(FSM; FunctionalSize Measurement)으로 SW Size에 대한 국제표준이며, 소프트웨어 개발, 유지관리 및 운영을 위한 비용과 자원 소요를 산정하는데 가장 중요한 요소<br>•개발은 물론 기획, 운영 등 전 수명주기에 걸쳐서 측정 가능

소프트웨어 기능은 사용자 관점에서 갖는 논리적 의미에 따라 크게 데이터 측면의 기능과 트랜잭션 측면의 기능으로 구분된다.

**기능점수 구조:** •데이터 기능 → 내부논리파일(ILF), 외부연계파일(EIF)<br>•트랜잭션 기능 → 외부입력(EI), 외부출력(EO), 외부조회(EQ)

**절차:** 1.사전준비 → 2.개발대상 SW 기능점수 산정 → 3.보정전 개발원가 산정 → 4.보정후 개발원가 산정 → 5.직접경비 및 이윤 산정 → 6.소프트웨어 개발비 산정

##### 기능점수 방식의 종류

##### 1) 기능점수 측정 범위 설정

##### 2) 애플리케이션 경계 설정

기능점수 측정 범위에 포함된 측정대상 애플리케이션과 다른 애플리케이션 또는 외부 사용자간의 경계를 설정 •**애플리케이션 경계** : 기능점수는 애플리케이션 외부의 사용자 관점에서 관찰되는 기능을 대상으로 측정한다. 따라서 특정 애플리케이션에 대한 기능점수를 측정하기 위해서는 해당 애플리케이션과 다른 애플리케이션의 경계 및 해당 애플리케이션과 외부 사용자 사이의 경계를 구분해야 하는데, 이를 애플리케이션 경계라고 한다. 경계를 정의할 때 반드시 사용자의 관점에서 해당 애플리케이션들이 명확히 구분되어야 한다. (예, MS사 제품의 경우, Word/Power-Point/Excel/Access 등 업무성격이 명확히 상이한 업무단위로 분리하는 것을 애플리케이션 경계라고 한다.)<br>•총 기능점수 = Σ(개별 애플리케이션별 기능점수)<br>•기능점수는 사용자 관점에서 측정되는 것으로 적용기술이나 방법과는 무관하게 측정되기 때문에 개발대상업무가 Client-Server 환경 또는 Web 환경 등에 따라 기능점수 측정결과에 차이가 발생해서는 안 된다.

##### 3) 데이터 기능점수 측정

사용자의 내부 및 외부 데이터 요구사항을 충족시키기 위한 기능량 •내부논리파일(ILF)과 외부연계파일(EIF)의 두 가지 유형이 있으며, 이들의 개수와 각각의 복잡도(가중치)에 의해 데이터 기능점수가 결정된다.

**데이터기능 식별기준** ▷측정 범위 내에서 논리적으로 관련되어 있고 사용자 식별 가능한 데이터나 제어정보를 식별한다.<br>▷애플리케이션에서 유지 또는 참조되지 않는 엔티티는 제외한다.<br>▷엔티티 종속성이 있는 엔티티들은 관련된 엔티티와 묶는다.(예: 게시판 테이블과 게시판 첨부 테이블이 존재할 때, 하나의 논리파일로 식별)<br>▷코드 데이터로 언급된 엔티티들은 제외한다.(예: 코드 데이터의 키, 값 속성은 사용자 관점에서는 불필요한 파일로 판단하여 제외)<br>▷사용자가 요구한 속성을 가지지 않고 기술적으로 도입된 엔티티들은 제외한다.<br>▷사용자가 요구하지 않은 추가적인 속성을 가진 관계 엔티티들과 단지 외래키만가지고 있는 관계 엔티티들은 제외한다.<br>▷측정 범위 내에서 존재하는 엔티티는 원칙적으로 **제3정규화 수준이 준수되어야 한다.**

**Step 1: 내부논리파일(ILF)과 외부연계파일(EIF)을 식별한다.** ▷**내부논리파일 (Internal Logical File: ILF)**: 사용자 관점에서 식별 가능한 논리적으로 연관된 하나의 데이터그룹 또는 제어정보로서 측정대상 애플리케이션 내부에서 유지되는 파일 (예: 애플리케이션 내부에서 유지되는 직원정보)<br>▷**외부연계파일 (External Interface File: EIF)**: 사용자 관점에서 식별가능한 논리적으로 연관된 하나의 데이터그룹 또는 제어정보로서 측정대상 애플리케이션 내부에서는 오직 참조만 되고 다른 애플리케이션에서 유지되는파일 (예: 측정대상 애플리케이션 내부에서 오직 참조용으로 사용하고 다른 애플리케이션 내부에서 유지되는 거래처주소정보)

*논리파일 식별대상에서 제외되어야할 데이터 그룹<br>-임시파일, - 식별된 논리파일의 다양한 물리적 복사파일, - 정렬 파일(Sort File), - 화면 또는 보고서 출력을 위한 추출파일<br>-기술적 이유로 도입된 코드파일, - 인덱스 파일, - 조인(join) 파일, - 키로만 구성된 관계파일(Relational File), - 일반적인 백업파일

**Step 2 : 내부논리파일(ILF)과 외부연계파일(EIF)별로 각각의 복잡도와 가중치를 산정한다.** (정통법과 간이법에 따라 가중치 산정방식이 다름)<br>▷**간이법** : 내부논리파일(ILF)은 평균복잡도 가중치 7.5, 외부연계파일(EIF)은 평균복잡도 가중치 5.4를 일률적으로 적용한다.<br>▷**정통법** : 내부논리파일(ILF)과 외부연계파일(EIF) 각각에 대해 데이터요소유형(DET)과 레코드요소유형(RET)을 식별하여 기능 복잡도에 따라 가중치를 결정한다.

•**레코드요소유형(RET)의 개수 판단 기준**<br>-내부논리파일이나 외부연계파일 내에 데이터 서브그룹이 없다면 내부논리파일이나 외부연계파일 자체를 하나의 레코드요소유형으로 식별<br>-서브그룹은 필수적 서브그룹(예: 정규직직원정보와 계약직직원정보)과 선택적 서브그룹(예: 게시판 첨부정보, 부양가족정보)으로 구분<br>-관계형 데이터베이스에서는 대부분 한 개의 ILF가 한개의 RET. (계층형 DB에서는 그룹별로 계산필요)

•**데이터요소유형(DET)의 개수 판단 기준**<br>-데이터요소유형(DET)은 사용자가 인식 가능하고 반복되지 않는 유일한 속성<br>-두 개의 서로 다른 애플리케이션이 동일한 내부논리파일/외부연계파일을 유지관리하거나 참조한다면, 각각의 애플리케이션이 사용하는 속성만 데이터요소유형으로 식별한다.<br>-서로 다른 내부논리파일/외부연계파일들끼리 관계를 맺기 위하여 사용된 속성은 각내부논리파일/외부연계파일에서 하나의 데이터요소유형으로 식별한다. (예: 외래 키)

•레코드요소유형(RET)과 데이터요소유형(DET)이 식별되면, 내부논리파일/외부연계파일의 복잡도와 가중치는 다음 표를 이용하여 산정 (표의 각 항목은 복잡도(가중치) - 정통법)

##### 4) 트랜잭션 기능점수 측정

트랜잭션 기능은 사용자에게 의미있는 데이터를 처리하는 기능을 제공하는 **단위프로세스의 집합** •외부입력(EI), 외부출력(EO), 외부조회(EQ) 기능으로 구분되어 기능점수로 산정

> **단위 프로세스** - 사용자의 기능적 사용자 요구사항을 바탕으로 사용자에게 의미를 주는 최소 단위활동 (예, 사원정보 등록, 사원정보 수정, 사원정보 삭제, 사원정보 상세조회 등)

**Step 1: 외부입력(EI), 외부출력(EO), 외부조회(EQ)를 식별한다.** ▷**외부입력(EI)** : 애플리케이션 경계 밖에서 들어오는 데이터나 제어정보를 처리하는 단위프로세스로서, 하나 이상의 논리파일을 변경하거나 시스템 동작을 변경함.<br>•하나 이상의 논리파일 변경 : △△등록, △△수정, △△삭제, △△설정, △△승인 등 단위프로세스 실행 시 특정 논리파일이 변경되는 활동<br>•시스템 동작 변경 : 특정 시점에 수행되던 단위프로세스가 실행 시점이 변경되는 활동(예:급여지급 주기가 격주에서 매월15일과 말일로 변경))<br>▷**외부출력(EO)** : 외부조회에 추가적인 처리로직을 포함하여 데이터나 제어정보를 애플리케이션 경계 밖으로 보여주는 단위프로세스로서, 처리로직은 하나 이상의 수학적 계산, 공식 또는 파생 데이터를 포함하거나, 하나 이상의 내부논리파일(ILF)을 유지(추가,수정,삭제)하거나, 시스템의 동작 변경을 수행해야 함(예 : 월간 판매실적 집계조회)<br>▷**외부조회(EQ)** : 데이터나 제어정보를 애플리케이션 경계 밖으로 보여주는 단위프로세스로서, 데이터나 제어정보를 단순히 검색하여 사용자에게 제공함. 외부조회는 수학적 계산이나 공식을 사용하지 않으며, 파생 데이터도 생성하지 않음. 처리 과정에서 내부논리파일(ILF)을 유지하지 않으며 시스템의 동작도 변경하지 않음. (예 : 게시판 조회, 게시판 목록조회)

**Step 2 : 외부입력(EI), 외부출력(EO), 외부조회(EQ) 각각의 복잡도와 가중치를 산정한다.** (정통법과 간이법에 따라 가중치 산정방식이 다름)<br>▷**간이법** : 평균복잡도 가중치(외부입력(EI) 4.0, 외부출력(EO) 5.2, 외부조회(EQ) 3.9)를 적용한다.<br>▷**정통법** : 외부입력(EI)/외부출력(EO)/외부조회(EQ) 각각에 대해 참조파일유형(FTR)과 데이터요소유형(DET)을 식별하여 복잡도(가중치)를 결정

•**참조파일유형(FTR)의 개수 판단 기준**<br>-참조파일유형(FTR)은 트랜잭션 기능에 의해 읽히거나 유지되는 내부논리파일 또는 트랜잭션 기능에 의해 읽히는 외부연계파일이다.<br>-외부입력/외부출력/외부조회 프로세스 처리 과정에서 참조되는 내부논리파일 및 외부연계파일 각각을 하나씩의 참조파일유형으로 식별한다. 수정과 조회가 동시에 적용되는 내부논리파일은 한번만 참조파일유형으로 식별한다.

•**데이터요소유형(DET)의 개수 판단 기준**<br>-경계를 넘나드는(진입 그리고/또는 진출) 모든 것을 검토한다.<br>-트랜잭션 기능이 수행되는 동안 경계로 넘나드는 사용자가 인식할 수 있는 유일하고 반복되지 않는 각 속성을 하나씩 데이터요소유형으로 식별.<br>-각 트랜잭션 기능에서 다수의 메시지가 존재하더라도 애플리케이션 응답 메시지를 보내는 기능에 대하여 트랜잭션 기능 당 오직 한번만 데이터요소유형으로 식별한다.<br>-동작을 시작하게 하는 여러 수단이 존재하더라도 트랜잭션 기능 당 오직 하나의 데이터요소유형으로 식별한다.<br>-다음은 데이터요소유형으로 식별하지 않는다.<br>　º보고서 제목, 화면이나 패널ID, 열 제목, 속성 제목과 같은 리터럴<br>　º날짜와 시간속성과 같은 애플리케이션이 생성하는 스탬프<br>　º페이지 설정 변수나 페이지 번호, 위치정보<br>　º"이전", "다음", "처음", "마지막" 그리고 이것을 그림을 이용하여 목록을 탐색하게 할 수 있는 기능과 같은 네비게이션 보조 수단<br>　º트랜잭션 기능에 의해 경계 내부에서 생성된 속성 그리고 경계를 넘지 않고 내부논리파일에 저장되는 속성<br>　º내부논리파일이나 외부연계파일로부터 검색되거나 참조되지만, 경계를 넘지않고 처리되는 속성

▷참조파일유형(FTR)과 데이터요소유형(DET)이 식별되면, 외부입력(EI)/외부출력(EO)/외부조회(EQ)의 각각에 대하여 다음 표를 이용하여 복잡도와 가중치를 산정한다

#### 🔥 인공지능(AI) 도입 방식 개요

• 인공지능(AI) 서비스 도입 사업비는 서비스 가격표 또는 견적서에 제시된 서비스 **총이용료**와 투입공수 방식의 커스터마이징 작업비용, 구축·개발비용에 따라 대가를 산정하는 방식이다.

• 서비스 총이용료는 서비스 가격표 또는 견적서에서 기업이 제시한 이용료를 기준으로 하며, 단위 이용료와 도입기간(월)에 따라 **연간 총이용료**를 산정한다.

• 커스터마이징 작업비용과 구축·개발비용은 작업 항목에 따른 서비스 기업의 견적가를 참고하여 투입공수방식으로 산정한다.

• ==AI 서비스 도입 사업비 = 서비스 총이용료 + 커스터마이징 작업비용 + 구축.개발비용==

### 📌 (3) 요율제 유지관리비 계산

**절차**: 1.사전 준비 → 2.유지관리 대상 SW개발비 재산정 → 3.유지관리 총점수(난이도 총점수)계산 → 4.유지관리 요율 계산 → 5.직접경비 계산 → 6.SW 유지관리비 산정

• 유지관리 대상 소프트웨어를 식별하고 해당 소프트웨어의 유지관리 범위와 요구사항을 정의

• 유지관리 대상 소프트웨어 개발비 = **현재 시점으로 재산정된 기능점수 × 현재 시점 FP단가 × 보정 계수** + 이윤

**(유지관리 계약 시점의 규모(기능점수)와 단가로 재산정)**

〈용역 소프트웨어 유지관리 난이도(TMP) 산정 평가표〉

|구분|단순|점수|보통|점수|복잡|점수|
|---|---|---|---|---|---|---|
|유지관리 횟수|연4회 이하|0|연 12회 이하|14|연12회 초과|27|
|시스템 사용자수|내부 25%이하 또는대국민 1만명 이하|0|내부 50%이하 또는대국민 10만명 이하|8|내부 50%초과 또는대국민 10만명 초과|18|
|시스템 중요도|업무 중요도(단순)|0|업무 중요도(보통)|17|업무 중요도(복잡)|31|
|타시스템 연계|연계 없음|0|1~2개|6|3개 이상|11|
|오류복구 신속성|12시간 초과|0|12시간 이내|6|6시간 이내|13|

> 유지관리 대상 소프트웨어는 유지관리 되어온 기간 동안 기능의 추가,변경,삭제 등으로, 해당 소프트웨어가 처음 개발되었을 때 규모와는 상이함. 따라서, 개발비는 유지관리 계약시점(현재)을 기준 기능점수로 재산정하며, 단가도 유지관리 계약시점(현재)의 단가를 적용한다.

> 시스템 중요도는 업무수행 영향도 및 국민생명, 안보/치안 위험수준, 국민생활에 대한 손실이나 장애 수준에 따라 판단하는 특성요소.(1, 2급은 복잡, 3급은 보통, 4, 5급은 단순)

• 대상 소프트웨어 시스템의 유지관리 요율 [%] = 10 + [5 × (난이도 총 점수 TMP ÷ 100)]

• 총점수(TMP)가 최저점인 0점일 경우 유지관리 요율은 10%가 되며, 총점수가 최고점 인 100점일 경우 유지관리 요율은 15%가 됨 (즉, 유지관리 요율은 10% ~ 15% 의 범위 임)

• 요율제 유지관리비 = 유지관리대상 소프트웨어의 **계약시점에 재산정된** 개발비 × 유지관리 요율(10~15%) + 직접경비

※ 유지관리 대상 소프트웨어의 규모를 **기능점수방식으로 산정하는것이 곤란하거나 불합리한 경우**(홈페이지 디자인과 같은 콘텐츠, 내부처리 복잡도가 현저히 높아 기능점수 적용이 불합리한 소프트웨어 등)에 대해서는 유지관리 활동에 **실제 투입되는 공수**를 별도로 산정하여 실비 계상할 수 있다.


### 📌 (4) 고정비/변동비방식 유지관리 및 운영비 계산

**절차**: 1.사전 준비 → 2.고정비 및 변동비 업무 구분 → 3.고정비 및 변동비 산정 → 4.직접경비 계산 → 5.SW 유지관리 및 운영비 산정

• 유지관리 및 운영 대상 소프트웨어의 업무에 대한 범위와 요구사항을 정의

• **고정비** : 기능 개선이 아닌 업무 -> **투입공수에 의한 방식을 사용하여 대가를 산정** : 적응유지관리, 수리유지관리(하자보수), 일상운영, 지원업무 / 무상하자보수는 제외

• **변동비** : 기능 개선 업무 (소프트웨어의 기능개선, 추가개발 등 기능 변경이 일어나는 업무) : 완전유지관리 -> 기본적으로 **기능점수를 이용하여 대가를 산정**

변동비는 해당 유지관리 및 운영사업을 통해 이루어지는 기능개선 추정량을 바탕으로 산정 **기능개선 추정량 = 계획 기능개선량 + 예측 기능개선량**

- 계획 기능개선량 : 해당기간 유지관리 및 운영사업에서 개선하기로 사전에 계획된 기능 개선량을 말함
- 예측 기능개선량 : 해당기간 유지관리 및 운영사업에서 요구될 것으로 예상되는 사전에 계획되지 않은 기능 개선량으로서 경험자료에 의해 예측 (예: CSR 자료)

- 변동비는 사업 이전에 기능개선 추정량을 대상으로 개발비 산정방식을 적용하여 산정하며, 이를 사전산정이라고 한다. 이후에 사업수행 과정에서 발생한 기능개선 실적량에 의해 변동비가 다시 산정될 수 있으며, 이를 사후산정이라고 한다. 사후산정량과 사전산정량에 차이가 발생할 경우 사후정산의 대상이 된다.

• 고정비 : 투입공수에 의한 산정 방식 • 변동비 : 기능점수에 의한 산정 방식 **(SW재개발비 산정방식을 적용)**

• SW 유지관리 및 운영비 = **변동비 산정(재개발대가) + 고정비 산정(투입공수방식 운영비) + 직접경비**


### 📌 (5) SLA기반 유지관리 및 운영비 정산법

**절차**: 1.서비스준비 → 2.서비스 정의 → 3.서비스 측정 및 기준 협약 → 4.서비스 평가 → 5.월별 대가 사후정산

• 이 방식은 사업이 수행 완료된 이후에 서비스수준 관리지표 및 목표수준에 따라 운영성과를 평가하고 이 결과에 의거하여 사업비를 정산하는 경우에 사용할 수 있는 방법

• 서비스수준관리(SLM: Service Level Management) 프로세스는 서비스 수준의 정의, 협상/계약,모니터링, 평가, 관리 그리고 서비스 수준을 개선하는 프로세스

• 서비스수준관리는 **서비스수준협약서(SLA:Service Level Agreement)**를 통하여 아웃소싱 사업에 적합한 서비스수준관리 지표 및 목표수준을 계약업체와 협의한다

• 서비스 발주기관과 서비스 사업자가 참여하는 담당 조직을 구성

• 서비스수준관리 활동과 관련된 목표 및 계획을 수립하여 필요한 교육 및 현황 조사를 수행

• SLA기반 대가산정의 기준이 되는 유지관리 및 운영비를 산정한다. 대가산정은 요율제,투입공수 방식, 또는 고정비/변동비 방식으로 업무 특성에 따라 산정하여 SLA기반 대가산정의 기준으로 삼는다.

• 서비스를 정의해 서비스 명세표 작성

• 서비스수준협약(SLA) 구조 결정 및 **서비스 측정항목 정의**

• 일반적으로 **서비스 측정항목은 1.적시성, 2.품질, 3.인력, 4.생산성, 5.만족도, 6.보안** 등에 대해 부문별로 발주기관의 업무 특성을 고려하여 선정한다

• 서비스 측정항목 정의서에는 측정항목에 대한 구체적인 산출 공식이 포함된 측정방법 과 측정도구 및 기법을 정의함

• 측정항목 초기값 조사 : 서비스 측정항목 초기 측정치 조사기간은 일반적으로 6개월이며, 초기 측정치 조사 기간을 확보하기 어려울 경우, 가능한 기간 동안 초기 측정치 조사를 실시하여 측정된 값을 기준으로 목표수준과 최소허용 수준을 상호 합의하에 임시로 설정 : 초기 측정치 조사 기간을 정할 때에는 특정시점의 서비스 사용이 집중되는지 등을 고려 : 초기 측정 조사 자료의 존재 여부에 따라 측정 여부를 결정하며, 기존에 측정 조사 자료가 있는 경우에는 생략 가능 : 측정항목 초기값 측정치 조사의 종료여부와 상관없이 서비스 평가방법 설정 활동 이후 의 단계를 계속 진행 : 초기값 측정 시범운영 기간은 6 ~ 9 개월이 적당하나 IT 아웃소싱 서비스 계약 이전과 관련하여 측정항목에 대한 측정 데이터 값이 확보되어 있을 경우, 기존 데이터 값을 검증만 하는 수준으로 측정기간을 협의 하에 단축

• 평가는 매월 실시하여 최소수준 이하의 서비스에 대해 일정 수준의 노력이 없으면 제재 (Penalty) 비용으로 부과함

• 서비스 성과가 높을 경우 기지불한 제재를 돌려받을 권리인 환급 제도를 적용

• 매월 제재 비용과 환급 금액을 상계 계산하여 연말에 최종 정산하여 상계 후 남은 제재 금액이 있을 시 징수

• SW사업 대가의 사후 정산이 가능한 경우 계약 종료시점 시 최종 보상금액을 사후정산 할 수 있음

[ 재개발 대상 소프트웨어의 기능 유형 ] *재개발=재사용+신규개발

• **재개발 소프트웨어 기능** = 신규개발 기능 +수정 없이 재사용 기능 + 수정 후 재사용 기능(기능변경)

- **신규개발 기능** : 신규로 추가되는 기능 (소프트웨어 개발비 산정과 동일하게 별도 산정)
    
- **수정 없이 재사용 기능** : **설계 및 코드의 변경이 일어나지는 않으나** 소프트웨어 **통합, 시험활동이** 반드시 요구되는 기능
    
- **수정 후 재사용 기능(기능변경)** : 기존 소프트웨어에 존재하는 기능 중 **분석, 설계, 구현, 시험 단계에서 변경이 일어나는 기능**이며, 기존 재사용 대상 기능으로부터 산정
    

> (재개발 전) 인사시스템 / 총무시스템(수정없이 재사용 대상) (재개발 후) 인사시스템 / [기능수정: 수정후 재사용 + 신규개발 기능] / 수정없이 재사용 기능 → 이 전체가 (재개발 대상 업무)

#### 🔥 신규개발/재사용 기능 규모의 산정

[신규개발 기능 규모의 산정] 신규개발 기능 규모 = 신규개발 데이터 규모 + 신규개발 트랜잭션 규모

[수정 없이 재사용 기능 규모의 산정] 수정 없이 재사용 기능 규모 = 수정 없이 재사용 대상 기능 규모 × 시험단계 비율 범위 내 적정 비율(0~25%)

[수정 후 재사용 기능 규모의 산정]

- 기능변경 규모를 산정하고 재사용 난이도를 적용하여 산정
- 기능변경의 식별 기준은 **데이터 기능 변경과 트랜잭션 기능 변경**으로 구분 　• 데이터기능 변경 규모 = Σ(재사용 데이터 기능(FP) × **기능변경 영향계수**) 　• 트랜잭션기능 변경 규모 = Σ(재사용 트랜잭션 기능(FP) × **기능변경 영향계수**)

> 〔참고〕시험단계 비율 범위 적용방법
> 
> - 0%는 재개발과 관련 통합이나 시험이 전혀 필요 없음
> - 25%는 재개발에 소요되는 통합 및 시험의 노력이 새로 개발하는 것과 동일한 정도로 소요됨을 의미

> 〈기능변경의 식별 기준〉 • 데이터기능 변경 : 기존 데이터 기능에 구조적인 변경 (예 : 필드의 추가 또는 삭제, 속성 변경 등) • 트랜잭션기능 변경 : 기존 트랜잭션 기능에서 사용자 화면의 필드가 추가 및 삭제로 변경되거나, 내부 처리로 직의 변경 또는 참조 되는 데이터의 변경이 있어야 함

[기능변경 영향계수는 기능 변경률에 기반하여 판단]

|기능 변경률|33% 이하|33% 초과 66% 이하|66% 초과|
|---|---|---|---|
|기능변경 영향계수|0.25|0.5|0.75|

• 데이터 변경률 = DET 변경률

• 트랜잭션 변경률 = (DET 변경률 + FTR 변경률) ÷ 2 (* 단, DET나 FTR 중 변경이 발생하지 않는 요소가 있을 경우에는 2로 나누지 않는다.)

• 데이터기능 변경 규모 = Σ(재사용 데이터 기능(FP) × 기능변경 영향계수) • 기능변경 영향계수 : 데이터요소유형(DET) 의 변경률을 확인하여 위의 표에서 찾아서 적용

- DET의 변경률 = (신규, 수정, 삭제 DET 수 / 기존 DET 수) × 100

• 트랜잭션기능 변경 규모 = Σ(재사용 트랜잭션 기능(FP) × 기능변경 영향계수) • 트랜잭션 기능변경 영향계수 : 데이터요소유형(DET)과 참조파일유형(FTR)의 변경률을 확인하여 위의 표에서 찾아서 적용

- DET 변경률 = (신규, 수정, 삭제 DET 수 / 기존 DET 수) × 100
- FTR 변경률 = (신규, 수정, 삭제 FTR 수 / 기존 FTR 수) × 100
- 트랜잭션 변경률 = (DET 변경률 + FTR 변경률) ÷ 2 (단, DET나 FTR 중 변경 발생하지 않는 요소 있을 경우 2로 나누지 않음)

#### 🔥 재개발 소프트웨어 규모 및 재개발원가 산정

• **재개발 소프트웨어 규모 계산**

- 재개발 소프트웨어 규모는 신규개발 기능 규모와 보정된 수정 후 재사용 기능 규모, 그리고 수정 없이 재사용 기능 규모를 합하여 다음과 같이 산정 재개발 소프트웨어 규모 = 신규개발 기능 규모 + 수정 후 재사용 기능 규모 + 수정없이 재사용 기능 규모

• **보정 전 재개발원가 산정**

- 산정된 재개발 소프트웨어 규모에 기능점수당 단가를 곱하여 보정전 재개발 원가를 구한다. (※ 기능점수당 단가는 대가 적용시점의 단가를 사용한다.)

• **보정 후 재개발원가 산정**

- 보정 전 재개발원가는 재개발하고자 하는 소프트웨어의 규모, 애플리케이션 복잡성 등과 같은 각 사업별로 발생할 수 있는 특성적 요인들을 고려하지 않은 재개발비용
- 사업의 수행 비용에 영향을 미칠 수 있는 특성적 요인들은 소프트웨어 개발비 대가산정 방식과 동일하게 보정계수를 적용하여 재개발원가를 보정
- 각 보정계수의 식별과 적용 기준에 대한 내용은 소프트웨어 개발비를 참고 보정후 재개발원가 = 보정전 재개발원가 × (규모 보정계수 × 연계복잡성수준 보정계수×성능요구수준 보정계수 × 운영환경 호환성 보정계수× 보안성수준 보정계수)

• 이윤은 보정후 재개발원가의 25% 범위 내

• 소프트웨어 재개발비 = 보정후 재개발원가 + **직접경비 + 이윤**

#### 🔥 예외사항

재개발 사업대가는 기능점수를 기반으로 작성되어 있다. 그러나 현장에서 다음과 같은 사례가 발생할 수 있다. 이 경우는 예외적으로 처리하는 것을 권고한다.

• **기능 변경 없이 사용자 인터페이스만 변경하는 경우**

기능의 변경 없이 사용자 인터페이스(User Interface, UI)만 변경하는 경우가 있다. 이런 경우에는 기능점수로 도출되어지지 않는다. 따라서 기능의 변경 없이 사용자 인터페이스만 변경하는 경우에는 별도의 **사용자 인터페이스에 소요되는 투입공수를 투입공수 기반으로 산정하여** 사업대가에 추가한다.

단, 기능변경과 사용자 인터페이스를 모두 변경하는 경우에는 재개발 사업대가를 적용하여 산정한다.

• **기능 변경 없이 내부 로직만 변경하는 경우**

데이터요소유형(DET)이나 참조파일유형(FTR)의 변경 없이 단순 로직만 변경하여 프로그램을 수정하는 경우가 있을 수 있다. 대부분의 경우에는 이러한 사항은 유지관리 사업에서 반영하여 처리되어지는 것이 일반적이다.

그러나 재개발 사업에서 기능 변경없이 단순 로직만 변경하는 경우에는 이 또한 기능점수에서 반영하기가 어렵다. 따라서 이런 경우에는 **내부 로직 변경에 소요되는 투입공수를 별도로 산정하여** 사업대가에 추가하여 재개발 사업대가를 산정한다.

• **기능 변경 없이 소프트웨어의 프레임워크를 변경하거나 프로그래밍 언어를 업그레이드하는 경우**

이 경우에도 기능 변경이 나타나지 않으므로 재개발 사업대가를 적용하기가 모호해 질 수 있다. 이 경우에는 소프트웨어 프레임워크 또는 프로그래밍 언어를 업그레이드 하고자하는 재개발 대상 소프트웨어의 규모에 수정영향계수를 곱하여 산정한다. 수정영향계수는 변경재사용 기능 규모를 산정할 때 적용한 0.25, 0.5, 0.75를 적용할 수 있다. 예를 들면, 동일한 프레임워크의 버전을 업그레이드하거나 유사한 프로그래밍 언어로 현재의 재사용 소프트웨어를 재개발 하는 경우에는 수정영향계수를 0.25로 적용하고, 상당히 다른 경우 프레임워크를 적용하거나 다른 종류의 프로그래밍 언어로 변경할 경우에는 수정영향계수를 0.75를 적용하여 재개발 사업대가를 적용한다. 프레임워크나 프로그래밍 언어의 변경 수준이 보통인 경우에는 0.5를 적용하여 재개발 기능 점수를 산정한다.

### 📌 (11) 데이터베이스 구축비

#### 🔥 개요

• 데이터베이스 구축 - 정보로서 가치가 있는 원시 자료를 이용자에게 유용한 형태로 가공·제작하는 일련의 작업과정

- 문헌 등을 문자 입력방식으로 디지털화 하는 경우, 페이지 단위의 스캐닝, 또는 촬영방식으로 디지털화하는 경우, 메타데이터 등의 형태로 구축하는 경우, 동영상, 음성 등의 자료를 디지털화 하는 경우 등

• 데이터베이스 구축비는 **투입공수에 의한 방식으로만 산정** - MD(Man-Days) 방식

• 투입인력의 직접인건비는 **학술연구용역 인건비 단가로 산정함을 원칙** 〈2019.2차, 자료입력원 단가 삭제됨〉

#### 🔥 절차 및 작업요소별 소요공수 산정

• 데이터베이스 구축 대상 자료를 확정하고, 목표한 데이터베이스 품질을 달성하기 위한 공정을 정의한다.

• 사업에 적용되는 작업요소를 식별한다.

• 작업요소별 대상 물량을 확인한다.

• 작업요소별 기초공수(MD: Man·Day)를 산정한다. **작업요소별 기초공수(MD) = 대상 물량 / 일일작업량**

• 작업요소별 소요공수(MD)를 산정한다. **작업요소별 소요공수(MD) = 작업요소별 기초공수 × 보정계수**
