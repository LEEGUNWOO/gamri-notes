<a id="acid-정리표"></a>
## 📘 ACID 정리표

| 요소  | 의미                | 설명             | 예시              |
| --- | ----------------- | -------------- | --------------- |
| A   | Atomicity (원자성)   | 전부 수행되거나 전부 취소 | 송금 중 오류 → 전체 롤백 |
| C   | Consistency (일관성) | 무결성 규칙 항상 만족   | 잔액 합계 유지        |
| I   | Isolation (격리성)   | 동시에 실행해도 독립적   | 동시 송금 충돌 방지     |
| D   | Durability (지속성)  | 커밋 후 영구 저장     | 시스템 장애 후에도 유지   |

---

<a id="한-줄-설명"></a>
## 📌 한 줄 설명

- **Atomicity** → 올 오어 낫싱

- **Consistency** → 규칙 위반 없음

- **Isolation** → 서로 간섭 없음

- **Durability** → 커밋은 영구

<a id="정규화-전체-요약표"></a>
## 📘 정규화 전체 요약표

### 1️⃣ 단계별 핵심 정리
| 단계       | 제거 대상  | 위반 구조                         | 판단 기준                                 | 핵심 키워드    |
| -------- | ------ | ----------------------------- | ------------------------------------- | --------- |
| **1NF**  | 반복/다중값 | 한 칸에 여러 값                     | 원자값인가?                                | 값은 하나     |
| **2NF**  | 부분 종속  | (복합키 일부 → 일반속성)               | 기본키 일부에 종속 존재?                        | 복합키 쪼개기   |
| **3NF**  | 이행 종속  | (키 → 비키 → 비키)                 | 비키가 비키를 결정하는가? (Prime 예외 존재)          | 중간단계 제거   |
| **BCNF** | 결정자 위반 | (비키 → 속성)                     | 모든 결정자가 후보키인가?                        | 결정자는 키    |
| **4NF**  | 다치 종속  | (X ↠ Y)<br>정확히는<br>X ↠ Y \| Z | **같은 A에서 B랑 C가 서로 상관없이 막 섞이면 = 다치종속** | 독립 다중값 분리 |
| **5NF**  | 조인 종속  | JD                            | 후보키 기반 분해인가?<br>쪼갯다가 합쳤더니 가짜생김        | 완전 분해     |

---

### 2️⃣ 종속 유형 정리

|종속 종류|기호|구조|제거 단계|
|---|---|---|---|
|함수 종속 (FD)|X → Y|하나 결정|2NF~BCNF|
|부분 종속|키 일부 → 속성|복합키 일부 종속|2NF|
|이행 종속|키 → 비키 → 비키|중간 단계 존재|3NF|
|비키 결정|비키 → 속성|결정자가 키 아님|BCNF|
|다치 종속 (MVD)|X ↠ Y|독립 다중값|4NF|

---

### 3️⃣ 3NF vs BCNF 비교

X - > Y (후보키면)

| 구분     | 3NF              | BCNF    |
| ------ | ---------------- | ------- |
| 결정자 조건 | 슈퍼키이거나 Y가 후보키 일부 | 무조건 후보키 |
| 엄격함    | 덜 엄격             | 더 엄격    |
| 시험 출제  | 매우 자주            | 고난도     |

---

### 4️⃣ 정규화 흐름 정리

|단계| 질문                    |
|---|---|
|1NF| 값이 원자값인가?             |
|2NF| 복합키 일부에만 종속되는 속성 있는가? |
|3NF| 비키가 또 다른 비키를 결정하는가?   |
|BCNF| 모든 결정자가 후보키인가?        |
|4NF| 독립적 다중값 존재하는가?        |

---

### 5️⃣ 이상 현상과 연결

|이상|원인|

<a id="armstrong-공리-armstrongs-axioms"></a>
## 📘 Armstrong 공리 (Armstrong’s Axioms)

### ■ 개요

함수 종속(Function Dependency)에서
**항상 성립하는 기본 추론 규칙 3가지**

---

### 🔵 1. 기본 공리 3개

| 구분  | 명칭 (한글 / 영어)              | 형식                   | 의미              | 시험 포인트        |
| --- | ------------------------- | -------------------- | --------------- | ------------- |
| ①   | 반사성(재귀) (Reflexivity)     | Y ⊆ X ⇒ X → Y        | 포함된 것은 항상 종속    | 자명 종속 근거      |
| ②   | 부가(첨가/확장성) (Augmentation) | X → Y ⇒ XZ → YZ      | 양쪽에 동일 속성 추가 가능 | 속성 추가해도 종속 유지 |
| ③   | 이행성 (Transitivity)        | X → Y, Y → Z ⇒ X → Z | 체인 종속 가능        | 이행 종속 근거      |

---

<a id="2-파생-규칙-derived-rules"></a>
## 🔵 2. 파생 규칙 (Derived Rules)

※ 전부 위 3개에서 유도됨

| 명칭                         | 형식                       | 유도 근거     | 시험 출제 빈도 |
| -------------------------- | ------------------------ | --------- | -------- |
| 분해 (Decomposition)         | X → YZ ⇒ X → Y AND X → Z | 반사성 + 이행성 | 매우 높음    |
| 결합(합집합) (Union)            | X → Y AND X → Z ⇒ X → YZ | 확장성 + 이행성 | 높음       |
| 의사이행 (Pseudo-Transitivity) | X → Y, YZ → W ⇒ XZ → W   | 확장 + 이행   | 중상       |

---

## 📘 폐포(Closure) 초간략 정리

### 1️⃣ 정의

```text
X⁺ = X로부터 연쇄적으로 얻을 수 있는 모든 속성
```

---

### 2️⃣ 계산 규칙 (이거 하나)

```text
LHS ⊆ 현재 폐포  →  RHS 추가
```

- 왼쪽 전부 있으면

- 오른쪽 가져온다

- 새로 추가되면 다시 처음부터 훑는다

- 더 이상 추가 없으면 끝

## 📘 최소커버 (Minimal Cover) 암기본

---

### 1️⃣ 정의

```text
의미는 동일하고
가장 단순한 함수종속 집합
```

---

### 2️⃣ 목적

- 정규화 분해 기준

- 불필요 종속 제거

- 중복 제거

---

### 3️⃣ 계산 순서 (무조건 이 순서)

|단계|작업|핵심|
|---|---|---|
|①|RHS 분해|X → YZ ⇒ X → Y, X → Z|
|②|LHS 축소|속성 제거 후 폐포 확인|
|③|FD 제거|FD 제거 후 폐포 확인|

---

### 4️⃣ LHS 축소 공식

```text
(X − a)⁺ 에 RHS 포함되면
a 제거 가능
```

---

### 5️⃣ FD 제거 공식

```text
FD 제거 후
LHS⁺에 RHS 포함되면 삭제 가능
```

---

### 6️⃣ 폐포 사용 위치

|위치|이유|
|---|---|
|LHS 축소|속성 제거 가능 여부 판단|
|FD 제거|유도 가능 여부 판단|

---

### 7️⃣ 핵심 암기 문장

```text
최소커버 = 쪼개고 → 줄이고 → 지우고
```

<a id="예제-최소커버--후보키-순차-정리"></a>
## 📘 [예제] 최소커버 + 후보키 순차 정리

### 🔷 문제

릴레이션 R(A, B, C, D, E, F, G)

함수종속 F:

```
1) AB → CE
2) A → B
3) C → D
4) D → E
5) BE → F
6) F → G
7) A → C
```

---

<a id="1-step-1--rhs-분해"></a>
## 1️⃣ STEP 1 — RHS 분해

### 🔎 왜?

최소커버는 **RHS가 단일 속성**이어야 판단이 정확하다.

### 적용

```
AB → CE
⇒ AB → C
⇒ AB → E
```

현재 FD:

```
AB → C
AB → E
A → B
C → D
D → E
BE → F
F → G
A → C
```

---

<a id="2-step-2--lhs-축소-줄이기"></a>
## 2️⃣ STEP 2 — LHS 축소 (줄이기)

#### 🔎 왜?

왼쪽 속성 중 불필요한 것을 제거하여 단순화.

---

### 🔹 (1) AB → C

이미 A → C 가 있다.

확장성(augmentation):
A→C 이면 AB→C 자동 성립.

👉 **AB→C 삭제 가능 (완전 중복)**

---

### 🔹 (2) AB → E

B 제거 가능 여부 확인

👉 A⁺ 계산

A → C
C → D
D → E

⇒ A⁺에 E 있음

즉 A만으로 E 도달 가능

👉 **AB→E ⇒ A→E 로 축소 가능**

---

현재 FD:

```
A → B
A → C
C → D
D → E
A → E
BE → F
F → G
```

---

<a id="3-step-3--fd-삭제-중복-제거"></a>
## 3️⃣ STEP 3 — FD 삭제 (중복 제거)

#### 🔎 왜?

이미 다른 경로로 유도 가능하면 불필요.

---

### 🔹 A → E 삭제 가능?

A → C
C → D
D → E

⇒ A → E 유도 가능 (이행)

👉 **A→E 삭제**

---

최종 최소커버:

```
A → B
A → C
C → D
D → E
BE → F
F → G
```

---

<a id="최소커버-결과"></a>
## ✅ 최소커버 결과

```text
{ A→B, A→C, C→D, D→E, BE→F, F→G }
```

---

<a id="4-step-4--후보키-구하기"></a>
## 4️⃣ STEP 4 — 후보키 구하기

---

### 🔷 1단계: RHS 모으기

RHS = {B, C, D, E, F, G}

전체 = {A,B,C,D,E,F,G}

```
전체 − RHS = {A}
```

👉 A는 반드시 후보키에 포함

---

### 🔷 2단계: A⁺ 계산

시작 {A}

A→B
A→C
C→D
D→E
이제 B,E 있음 → BE→F
F→G

⇒ A⁺ = 전체

---

<a id="후보키"></a>
## ✅ 후보키

```
{A}
```

---

<a id="핵심-흐름-요약"></a>
## 📌 핵심 흐름 요약

|단계|무엇을 했는가|왜 했는가|
|---|---|---|
|RHS 분해|AB→CE → AB→C, AB→E|표준화|
|LHS 축소|AB→E → A→E|불필요 속성 제거|
|삭제|A→E 제거|이행으로 유도 가능|
|후보키|전체−RHS → 폐포|최소 결정자 판별|

---

<a id="시험용-한-줄-정리"></a>
## 🔥 시험용 한 줄 정리

```
최소커버 = 분해 → 축소 → 삭제
삭제 판단은 항상 LHS 폐포
후보키는 (전체−RHS)부터 시작
```

<a id="view-뷰-정리"></a>
## 📘 View (뷰) 정리

---

### 1️⃣ 기본 개념

- **View = 가상 테이블 (Virtual Table)**

- 하나 이상의 기본 테이블(Base Table)로부터 정의

- 실제 데이터 저장하지 않음

- SELECT 문만 저장

---

### 2️⃣ View 목적

|목적|설명|
|---|---|
|보안|특정 컬럼/행만 노출|
|단순화|복잡한 쿼리 숨김|
|논리적 독립성|구조 변경 영향 최소화|

---

### 3️⃣ View 특징

- 실제 데이터 저장 ❌

- 기본 테이블 변경 시 자동 반영

- 인덱스 생성 불가 (일반 View 기준)

- DML 가능 여부는 조건에 따라 다름

---

<a id="4-갱신-가능한-view-조건-시험-핵심"></a>
## 4️⃣ 갱신 가능한 View 조건 (시험 핵심)

### ✅ 갱신 가능 조건

- 단일 기본 테이블 기반

- 기본 테이블의 PK 포함

- 집계 함수 없음

- GROUP BY 없음

- DISTINCT 없음

- UNION 없음

- 계산식 컬럼 없음

---

### ❌ 갱신 불가 View 유형

|유형|이유|
|---|---|
|조인 뷰|시험 기준 원칙적으로 불가|
|GROUP BY|집계 결과|
|DISTINCT|중복 제거|
|집계 함수|SUM, AVG 등|
|UNION|집합 결합|
|계산식 컬럼|원본 컬럼 아님|

---

### 🔥 시험 암기 공식

> GROUP BY / DISTINCT / 집계 / UNION
> → 무조건 갱신 불가

> 단일 테이블 → 갱신 가능
> 조인 → 시험 기준 불가

---

### 5️⃣ WITH CHECK OPTION

```sql
CREATE VIEW v_emp AS
SELECT * FROM emp
WHERE deptno = 10
WITH CHECK OPTION;
```

- 뷰 조건 위반하는 INSERT/UPDATE 금지

---

### 6️⃣ View vs Base Table

|구분|View|Base Table|
|---|---|---|
|데이터 저장|❌|⭕|
|물리적 공간|없음|있음|
|인덱스|불가|가능|
|DML|조건부 가능|가능|

---

### 7️⃣ Oracle 예외 (참고용)

- Key-Preserved Table 조건 시 조인 뷰 일부 UPDATE 가능

- 감리사 시험은 이론 기준 적용

<a id="관계대수-기초-정리표"></a>
## 📘 관계대수 기초 정리표
관계대수는 중복 제거가 기본임

|기호|한글|정의식(쓰는 법)|의미|예시|
|---|---|---|---|---|
|σ|선택|`σ_{조건}(R)`|조건 만족 행만 남김|`σ_{B=10}(R)`|
|Π|투영|`Π_{속성들}(R)`|지정 열만 남김 (중복 제거)|`Π_{A,B}(R)`|
|×|카티션곱|`R × S`|모든 행 조합|R 2개 × S 3개 = 6개|
|⋈|자연조인|`R ⋈ S`|공통 속성 자동 = 조인|`R(A,B) ⋈ S(B,C)`|
|⋈_{θ}|세타조인|`R ⋈_{R.B<S.B} S`|조건 만족 행 결합|`R ⋈_{R.B=S.B} S`|
|∪|합집합|`R ∪ S`|두 릴레이션 합침|스키마 동일 필요|
|−|차집합|`R − S`|R에만 있는 튜플|공통 스키마 필요|
|∩|교집합|`R ∩ S`|공통 튜플|`R − (R − S)` 가능|
|÷|나눗셈|`R(A,B) ÷ T(B)`|T의 모든 B 가진 A|“for all” 개념|
|⋉|세미조인|`R ⋉ S`|조인되지만 R만 반환|EXISTS 느낌|
|⟕|좌외부조인|`R ⟕ S`|R 전부 유지|NULL 발생 가능|
|⟖|우외부조인|`R ⟖ S`|S 전부 유지|NULL 발생 가능|
|⟗|완전외부조인|`R ⟗ S`|양쪽 전부 유지|FULL OUTER JOIN|

|표현|의미|
|---|---|
|R ⋈ S|자연조인|
|R ⋈_N S|자연조인|
|R ⋈_{조건} S|조건조인|
|R × S|카티션곱|

| --------- | ----------- | ------------------- | ------- | -------- | ------- | ------------- |
| 카티션 곱     | R × S       | ❌ 없음                | R+S     | ❌        | ❌       | 모든 조합         |

| 동등 조인     | R ⋈ S (조건=) | ⭕ = 조건              | R+S     | ❌        | ❌       | 세타조인의 특수형     |
| 자연 조인     | R ⋈ S       | 공통 속성 자동 =          | R+S-공통  | ⭕        | ❌       | 공통 속성 하나로 병합  |
| 왼쪽 외부 조인  | R ⟕ S       | ⭕                   | R+S     | ❌        | S쪽      | R 전부 유지       |
| 오른쪽 외부 조인 | R ⟖ S       | ⭕                   | R+S     | ❌        | R쪽      | S 전부 유지       |
| 완전 외부 조인  | R ⟗ S       | ⭕                   | R+S     | ❌        | 양쪽      | 모두 유지         |
| 세미 조인     | R ⋉ S       | ⭕EXISTS / IN        | R만      | 해당없음     | ❌       | S에 존재하는 R만 반환 |
| 역세미 조인    | R ⋊ S       | ⭕ NOT EXISTS/NOT IN | R만      | 해당없음     | ❌       | S에 없는 R 반환    |
| 자기 조인     | R ⋈ R       | ⭕                   | R+R     | ❌        | ❌       | 동일 릴레이션 조인    |
| 교집합 조인    | R ⋂ S       | 암묵적                 | 동일      | 해당없음     | ❌       | 동일 스키마 교집합    |
| 외부 세미 조인  | R ⟕⋉ S      | ⭕                   | R만      | 해당없음     | 가능      | 존재 여부 + 외부 유지 |

| 유형     | 한글명       | 영어                  | 무엇이 문제?   | 발생 원인           | 핵심 특징         |
| ------ | --------- | ------------------- | --------- | --------------- | ------------- |
| 오손 읽기  | 더티 리드     | Dirty Read          | 잘못된 값 읽음  | 미커밋 데이터 읽기      | 커밋 전 데이터      |
| 비반복 읽기 | 반복 불가능 읽기 | Non-Repeatable Read | 값 변경      | UPDATE 후 재조회    | 같은 행 값이 바뀜    |
| 유령 읽기  | 팬텀 리드     | Phantom Read        | 행 개수 변경   | INSERT / DELETE | 조건 결과 개수 변함   |
| 연쇄 읽기  | 캐스케이딩 리드  | Cascading Read      | 연쇄 롤백     | 미커밋 의존          | 롤백 전파         |
| 분실 갱신  | 로스트 업데이트  | Lost Update         | 업데이트 덮어쓰기 | 동시 UPDATE       | 한 트랜잭션 결과 사라짐 |

<a id="isolation-level-트랜잭션-격리-수준"></a>
## 📘 Isolation Level (트랜잭션 격리 수준)

### 🔥 격리 수준 표

|Isolation Level|Dirty Read|Non-Repeatable|Phantom|설명|
|---|---|---|---|---|
|READ UNCOMMITTED|⭕ 허용|⭕|⭕|가장 낮음|
|READ COMMITTED|❌|⭕|⭕|커밋된 것만 읽음|
|REPEATABLE READ|❌|❌|⭕|같은 행은 보호|
|SERIALIZABLE|❌|❌|❌|완전 직렬화|

---

### 🔎 단계별 이해

#### 1️⃣ READ UNCOMMITTED

- 미커밋 값도 읽음

- Dirty Read 발생

---

#### 2️⃣ READ COMMITTED

- 커밋된 데이터만 읽음

- Dirty Read 차단

- 하지만 다시 읽으면 값 바뀔 수 있음

---

#### 3️⃣ REPEATABLE READ

- 내가 읽은 행은 다른 트랜잭션이 못 바꿈 (UDATE DELETE 가능)

- 하지만 새로운 행 삽입은 막지 못함

- Phantom 발생 가능

---

#### 4️⃣ SERIALIZABLE

- 완전 고립

- 다른 트랜잭션이 조건 범위에 접근 못함 (조건범위에 3의 조건에 INSERT 도 못한다.)

- Phantom도 차단

---

<a id="1-로킹의-목적"></a>
## 1️⃣ 로킹의 목적

👉 **격리성(Isolation) 보장**
👉 동시 실행 시 데이터 일관성 유지

<a id="2-락-종류-lock-mode"></a>
## 2️⃣ 락 종류 (Lock Mode)

### 🔹 기본 락

| 락                 | 의미  | 특징      |
| ----------------- | --- | ------- |
| S (Shared) 공유락    | 읽기  | 여러 개 가능 |
| X (Exclusive) 베타락 | 쓰기  | 단독만 가능  |

#### 🔥 충돌표

| 요청 보유 | S   | X   |
| ----- | --- | --- |
| S     | 가능  | 대기  |
| X     | 대기  | 대기  |

👉 쓰기는 항상 혼자

---

<a id="3-의도-락-intention-lock"></a>
## 3️⃣ 의도 락 (Intention Lock)

계층 구조(DB→테이블→페이지→레코드) 때문에 필요

|락|의미|
|---|---|
|IS|하위에 S락 예정|
|IX|하위에 X락 예정|
|SIX|상위는 S, 하위 일부는 X|

---

### 🔥 의도락 충돌 핵심

| 요청 보유 | IS  | IX  | S   | X   |
| ----- | --- | --- | --- | --- |
| IS    | ✔   | ✔   | ✔   | ❌   |
| IX    | ✔   | ✔   | ❌   | ❌   |
| S     | ✔   | ❌   | ✔   | ❌   |
| X     | ❌   | ❌   | ❌   | ❌   |

✔ IS는 거의 안 막힘
✔ IX는 S랑 충돌
✔ X는 전부 충돌

---

<a id="4-로킹-단위-granularity"></a>
## 4️⃣ 로킹 단위 (Granularity)

|단위 작음|단위 큼|
|---|---|
|동시성 ↑|동시성 ↓|
|오버헤드 ↑|오버헤드 ↓|

👉 시험 단골 문장

---

<a id="5-2단계-로킹-2pl"></a>
## 5️⃣ 2단계 로킹 (2PL)

- 확장 단계(Growing): 락 획득만

- 축소 단계(Shrinking): 락 해제만

- 중간에 다시 획득 불가

### 종류

| 유형               | 특징                         | 직렬성 | Dirty Read 방지 | Cascading Rollback 방지 |
| ---------------- | -------------------------- | --- | ------------- | --------------------- |
| **Basic 2PL**    | Growing → Shrinking 단계만 지킴 | ⭕   | ❌             | ❌                     |
| **Strict 2PL**   | **X락을 COMMIT까지 유지**        | ⭕   | ⭕             | ⭕                     |
| **Rigorous 2PL** | **모든 락(S,X)을 COMMIT까지 유지** | ⭕   | ⭕             | ⭕                     |

<a id="2pl-유형-비교"></a>
## 2PL 유형 비교

| 구분             | Basic 2PL | Strict 2PL | Rigorous 2PL |
| -------------- | --------- | ---------- | ------------ |
| 2단계 구조         | ✔         | ✔          | ✔            |
| X락 커밋까지 유지     | ❌         | ✔          | ✔            |
| S락 커밋까지 유지     | ❌         | ❌          | ✔            |
| 연쇄 롤백 방지       | ❌         | ✔          | ✔            |
| 직렬 가능성 보장      | ✔         | ✔          | ✔            |
| **교착상태 발생 가능** | ✔         | ✔          | ✔            |

<a id="6-데드락"></a>
## 6️⃣ 데드락

서로 락을 기다리며 멈춤

### 🔥 데드락 4대 필요조건

|조건|의미|
|---|---|
|상호 배제 (Mutual Exclusion)|자원은 한 번에 하나만 사용|
|점유 대기 (Hold and Wait)|자원 잡고 다른 자원 기다림|
|비선점 (No Preemption)|강제로 빼앗지 못함|
|**순환 대기 (Circular Wait)**|서로가 서로를 기다림|

<a id="해결"></a>
## 🔥 해결

| 방법  | 시점                           | 핵심        |
| --- | ---------------------------- | --------- |
| 예방  | 사전 차단                        | 조건 제거     |
| 회피  | 발생 직전 (Wait-Die, Wound-Wait) | 안전 상태만 허용 |
| 탐지  | 발생 후 (Wait-for Graph)        | 그래프 분석    |
| 회복  | 탐지 후 (Rollback)              | 롤백        |
### 🔥 Wait-Die

- **늙은 트랜잭션이 기다림**

- 젊은 트랜잭션은 죽음(rollback)

### 🔥 Wound-Wait

- 늙은 트랜잭션이 젊은 것을 죽임 (은행원 알고리즘도 여기에 포함)

- 젊은 트랜잭션이 기다림

<a id="3-cap-이론"></a>
## 📘 3️⃣ CAP 이론

분산 시스템에서 동시에 만족할 수 없는 3가지 속성

| 요소                      | 한글    | 의미             |
| ----------------------- | ----- | -------------- |
| C (Consistency)         | 일관성   | 모든 노드가 동일한 데이터 |
| A (Availability)        | 가용성   | 항상 응답 가능       |
| P (Partition Tolerance) | 분할 허용 | 네트워크 분할에도 동작   |

---

<a id="핵심-원리"></a>
## 🔥 핵심 원리

> 분산 시스템에서는
> C, A, P를 동시에 완벽히 만족 불가

→ 반드시 **2개 선택**

---

<a id="4-cap-조합-유형"></a>
## 📘 4️⃣ CAP 조합 유형

| 유형  | 특징          | 예                          |
| --- | ----------- | -------------------------- |
| CP  | 일관성 + 분할 허용 | MongoDB (기본 설정)            |
| AP  | 가용성 + 분할 허용 | Cassandra                  |
| CA  | 일관성 + 가용성   | (이론적, 분산에선 어려움) 관계형 데이터베이스 |

※ 분산 시스템이면 P는 사실상 필수
→ 현실은 CP 또는 AP 선택

---

<a id="5-궁극적-일관성-eventual-consistency"></a>
## 📘 5️⃣ 궁극적 일관성 (Eventual Consistency)

> 시간이 지나면 결국 일관성 도달

✔ 즉시 일관성은 아님
✔ AP 계열에서 많이 사용

<a id="6-acid-vs-base"></a>
## 📘 6️⃣ ACID vs BASE

|ACID|BASE|
|---|---|
|즉시 일관성|궁극적 일관성|
|강한 트랜잭션|약한 일관성|
|RDB 중심|NoSQL 중심|

BASE =
Basically Available   시스템은 항상 응답을 준다(일부 노드 장애가 있어도)
Soft state  (데이터 상태가 시간이 지나면서 변할 수 있다. 즉시 일관성 유지 안 함)
**Eventually consistent (시간이 지나면 결국 일관된 상태가 된다.)**

<a id="nosql-샤딩sharding--초압축-정리"></a>
## 📘 NoSQL 샤딩(Sharding) — 초압축 정리

### 🔥 1️⃣ 샤딩(Sharding)이란?

> 대용량 데이터를 **여러 서버에 수평 분할(Horizontal Partitioning)** 하여 저장하는 기법

- 하나의 거대한 테이블을

- 여러 개의 물리적 노드에 나눠 저장

📌 목적

- 확장성(Scalability)

- 성능 향상

- 트래픽 분산

---

<a id="수평-분할-vs-수직-분할"></a>
## 📌수평 분할 vs 수직 분할

| 구분    | 수평 분할 (Horizontal Partitioning) | 수직 분할 (Vertical Partitioning) |
| ----- | ------------------------------- | ----------------------------- |
| 분할 기준 | **행(row)** 기준                   | **열(column)** 기준              |
| 샤딩 여부 | ✅ 샤딩                            | ❌ 샤딩 아님                       |
| 예시    | 고객 1~1000 / 1001~2000           | 이름/주소 테이블 분리                  |

---

### 🔹 2️⃣ 샤딩 방식 3가지

|방식|원리|장단점|
|---|---|---|
|Range|값 범위별 분산|단순 / Hotspot 위험|
|Hash|hash(key) % N|균등 분산 / 범위조회 불리|
|Directory|중앙 메타서버 관리|유연 / 메타 장애 위험|

---

### 🔹 3️⃣ 샤딩 키 (Shard Key)

좋은 조건

- 균등 분포

- 자주 조회 조건

- 변경 거의 없음

> 샤딩 성패 = 샤딩 키 선택

---

### 🔹 4️⃣ 샤딩 vs 레플리케이션

|구분|샤딩|레플리케이션|
|---|---|---|
|데이터|서로 다름|동일 복제|
|목적|쓰기 확장|읽기 확장|
|효과|용량 증가|가용성 증가|

<a id="조인-알고리즘-4대장"></a>
## 📘 조인 알고리즘 4대장
---

### 🔵 1️⃣ 중첩 루프 조인

**Nested Loop Join (NLJ, 중첩 루프 조인)**

#### 🔹 동작 원리

외부 테이블(R)의 각 튜플에 대해
내부 테이블(S)을 전체 스캔

```
for each r in R:
    for each s in S:
        if join condition:
            output
```

#### 🔹 비용 개념

- O(N × M)

- 작은 테이블 × 큰 테이블일 때 유리

#### 🔹 특징

- 정렬 필요 없음

- 인덱스 필요 없음

- 모든 조인 조건 가능 (=, <, > 등)

#### 🔹 튜닝 포인트

- 작은 테이블을 Outer로

- 내부 테이블에 인덱스가 있으면 Indexed NLJ로 전환

---

### 🟢 2️⃣ 인덱스 중첩 루프 조인

**Indexed Nested Loop Join (인덱스 중첩 루프 조인)**

#### 🔹 전제

- 내부 테이블 조인 키에 인덱스 존재

#### 🔹 동작

- 외부 테이블 1건

- 내부 테이블은 인덱스 탐색

#### 🔹 비용

- O(N × log M)

#### 🔹 특징

- 인덱스 없으면 불가능

- 선택도(selectivity) 높을수록 효과 큼

#### 🔹 시험 포인트

> “내부 테이블 조인 속성에 B-tree 인덱스 존재” → 이거

---

### 🟡 3️⃣ 정렬 합병 조인

**Sort-Merge Join (정렬 합병 조인)**

#### 🔹 전제

- 두 테이블이 조인 키 기준 정렬되어 있어야 함

#### 🔹 동작

- 두 정렬된 리스트를 병합하듯 진행

```
R pointer →
S pointer →
작은 쪽 이동
```

#### 🔹 비용

- 정렬 비용 O(N log N)

- 이미 정렬돼 있으면 O(N+M)

#### 🔹 특징

- 범위 조인(<, >) 가능

- 대용량에 안정적

- 정렬 상태면 매우 강력

#### 🔹 시험 키워드

- “오름차순 정렬”

- “정렬된 파일”

---

### 🔴 4️⃣ 해시 조인

**Hash Join (해시 조인)**

#### 🔹 전제

- 동등조인(=) 조건만 가능

#### 🔹 동작

1. 작은 테이블 해시 테이블 생성

2. 큰 테이블 순차 탐색하면서 매칭

#### 🔹 비용

- O(N + M)

#### 🔹 특징

- 대용량 동등조인 최강

- 정렬 필요 없음

- 메모리 사용

#### 🔹 시험 키워드

- “동등조인”

- “대용량”

- “해시 기반”

---

## 📊 종합 비교표 (감리사 암기판)

|구분|중첩 루프|인덱스 NLJ|정렬 합병|해시 조인|
|---|---|---|---|---|
|정렬 필요|❌|❌|✅|❌|
|인덱스 필요|❌|✅|❌|❌|

<a id="동시성-제어--낙관적-vs-비관적"></a>
## 📘 동시성 제어 – 낙관적 vs 비관적

### 🔥 먼저 직관부터

#### 🟢 낙관적(Optimistic)

> “설마 충돌하겠어?”
> 일단 그냥 진행하고, 마지막에 확인하자.

- 작업 중에는 락 거의 안 건다.

- 커밋 직전에 충돌 검사.

- 충돌 나면 → 롤백 후 재시도.

✔ 충돌이 거의 없으면 매우 빠름
❌ 충돌 많으면 계속 재시도 → 성능 폭락

---

#### 🔴 비관적(Pessimistic)

> “충돌 날 거야.”
> 시작할 때부터 락 걸고 안전하게 가자.

- 데이터를 읽거나 수정할 때 바로 락 획득.

- 다른 트랜잭션은 기다림.

✔ 충돌 많을 때 안정적
❌ 락 대기로 느려질 수 있음
❌ 교착상태 가능

---

<a id="부드러운-비교표"></a>
## 📊 부드러운 비교표

|구분|낙관적|비관적|
|---|---|---|
|기본 가정|충돌 적음|충돌 많음|
|락 시점|마지막 검증 단계|시작 시|
|작업 중 락|거의 없음|계속 유지|
|충돌 발생 시|재시도|대기|
|교착 상태|거의 없음|발생 가능|
|적합 환경|읽기 많고 수정 적음|경쟁 심한 환경|

---

                ↑
            충돌 검사
```

→ 충돌 나면 여기서 되돌아감.

```

```

→ 처음부터 잠가버림.

- 충돌 확률 낮음

### 🔴 비관적이 좋은 경우

<a id="이진분류-평가척도"></a>
## 📘 이진분류 평가척도

<a id="1-기본-구조--검색-결과-기준"></a>
## 📌 1️⃣ 기본 구조 — 검색 결과 기준

어떤 키워드를 검색했다고 하자.

|            | 실제 관련 문서 | 실제 비관련 문서 |
| ---------- | -------- | --------- |
| **검색됨**    | TP       | FP        |
| **검색 안 됨** | FN       | TN        |

- **TP** : 관련 문서를 제대로 검색

- **FP** : 관련 아닌데 검색됨 (노이즈)

- **FN** : 관련인데 못 찾음 (놓침)

- **TN** : 관련 아닌데 안 나옴

---

<a id="2-평가-지표를-검색-관점에서"></a>
## 📌 2️⃣ 평가 지표를 검색 관점에서

### 🔹 ① 정밀도 (Precision)

> 검색된 문서 중에 얼마나 정확했나

Precision = TP / (TP + FP)

- 검색 결과의 “순도”

- 광고, 노이즈 많으면 낮아짐

---

### 🔹 ② 재현율 (Recall)

> 전체 관련 문서 중 얼마나 찾아냈나

Recall = TP / (TP + FN)

- 중요한 문서를 놓치면 낮아짐

- “검색 커버리지”

---

### 🔹 ③ 특이도 (Specificity)

> 관련 아닌 문서를 잘 걸러냈나

Specificity = TN / (TN + FP)

- 불필요한 문서가 적을수록 높음

---

### 🔹 ④ 정확도 (Accuracy)

> 전체 문서 중 맞게 분류한 비율

Accuracy = (TP + TN) / 전체

⚠ 검색에서는 TN이 너무 많아 의미 약함

---

### 🔹 ⑤ 에러율

> 틀린 비율

Error Rate = (FP + FN) / 전체

---

### 🔹 ⑥ F1 Score

> 정밀도와 재현율의 균형 점수

- 둘 중 낮은 값에 끌림

- 둘 다 좋아야 높음 (조화평균)

---

<a id="3-한눈에-비교표"></a>
## 📌 3️⃣ 한눈에 비교표

|지표|공식|기준|의미|낮아지는 원인|
|---|---|---|---|---|
|**Precision (정밀도)**|TP / (TP + FP)|검색된 것 기준|검색 결과의 정확성|FP 증가|
|**Recall (재현율, 민감도)**|TP / (TP + FN)|전체 관련 기준|관련 문서를 얼마나 놓치지 않았는가|FN 증가|
|**Specificity (특이도)**|TN / (TN + FP)|전체 비관련 기준|비관련 문서를 잘 걸러냈는가|FP 증가|
|**Accuracy (정확도)**|(TP + TN) / (TP + TN + FP + FN)|전체 기준|전체 중 맞춘 비율|클래스 불균형|
|**Error Rate (에러율)**|(FP + FN) / (TP + TN + FP + FN)|전체 기준|전체 중 틀린 비율|FP 또는 FN 증가|
|**F1 Score**|2 × (Precision × Recall) / (Precision + Recall)|P + R 균형|정밀도·재현율 균형 점수|FP 또는 FN 증가|

---

<a id="4-검색-상황에서-감각"></a>
## 📌 4️⃣ 검색 상황에서 감각

- 광고/쓰레기 문서 많다 → Precision 낮음

- 중요한 문서 많이 빠졌다 → Recall 낮음

- 둘 중 하나만 높아도 F1 낮음

---

<a id="실전-예시"></a>
## 🎯 실전 예시

검색 결과 10개 중
5개가 진짜 관련 → Precision 50%

전체 관련 문서가 20개인데
5개만 찾음 → Recall 25%

→ F1은 25%에 가까운 값 (낮은 쪽에 끌림)

## ⭐ B* TREE (차수 3)

같은 데이터.

차이점은 overflow 처리.

예:

왼쪽 노드 overflow:

```

B-Tree → 바로 7 승격

B*Tree → 오른쪽 형제와 합쳐 재배치

예:

형제 노드:

```
[5 | 7] + [15]
```

재분배 후 균등

즉,

```
[5 | 7]   [15]
```

바로 승격 안 한다.

그래서 노드 점유율이 높다.

---

<a id="비교-다시-정리"></a>
## 🔵 비교 다시 정리

| 구분     | overflow 시 행동              |
| ------ | -------------------------- |
| B-Tree | 즉시 split                   |
| B+Tree | 즉시 split (리프 분할 + 키 복사)    |
| B*Tree | 형제에 여유 있으면 재분배 → 없으면 split |

<a id="확장성-해시-extendible-hash"></a>
## 📘 확장성 해시 (Extendible Hash)

---

<a id="1-기본-구조"></a>
## 🔵 1️⃣ 기본 구조

```text
확장성 해시 = Directory + Bucket
```

- **Directory(디렉토리)** : 해시 prefix → 버킷을 가리키는 포인터 테이블

- **Bucket(버킷)** : 실제 레코드를 저장하는 블록

---

<a id="2-핵심-용어-정리"></a>
## 🔵 2️⃣ 핵심 용어 정리

### ✅ GD (Global Depth, 전역 깊이)

```text
디렉토리가 사용하는 비트 수
```

- 디렉토리 크기 결정

- 공식:

```text
디렉토리 수 = 2^GD
```

예:

- GD=3 → 8개

- GD=4 → 16개

- GD=5 → 32개

---

### ✅ LD (Local Depth, 지역 깊이)

```text
특정 버킷이 구분에 사용하는 prefix 길이
```

- 그 버킷을 가리키는 디렉토리들의

- **공통 prefix 길이**

항상:

```text
LD ≤ GD
```

---

<a id="3-gd-vs-ld-차이-완전-고정"></a>
## 🔵 3️⃣ GD vs LD 차이 완전 고정

|구분|GD|LD|
|---|---|---|
|의미|전체 디렉토리 깊이|특정 버킷 깊이|
|범위|시스템 전체|개별 버킷|
|영향|디렉토리 크기 결정|공유 개수 결정|
|관계|항상 LD ≤ GD||

---

<a id="4-공유-개수-공식-시험-핵심"></a>
## 🔵 4️⃣ 공유 개수 공식 (시험 핵심)

```text
한 버킷을 가리키는 디렉토리 수 = 2^(GD - LD)
```

---

### 📌 패턴 (GD=4 기준)

|LD|공유 개수|
|---|---|
|4|1|
|3|2|
|2|4|
|1|8|
|0|16|

👉 LD가 1 줄어들 때마다 공유 수는 2배 증가
👉 LD = GD이면 1:1 대응

---

<a id="5-ld-구하는-방법"></a>
## 🔵 5️⃣ LD 구하는 방법

### ✔ 방법 1 : 공통 prefix 길이 찾기

예:

```
100
101
110
111
```

공통 prefix = `1`
→ LD = 1

---

예:

```
0100
0101
```

공통 prefix = `010`
→ LD = 3

---

### ✔ 방법 2 : 공식으로 역산

공유 개수 알 때:

```text
공유 개수 = 2^(GD - LD)
```

예:

- GD=5

- 공유=4

```
4 = 2^(5 - LD)
2 = 5 - LD
LD = 3
```

---

<a id="6-overflow-처리-규칙"></a>
## 🔵 6️⃣ overflow 처리 규칙

overflow 발생 시:

```
1️⃣ LD < GD
   → 버킷만 split
   → LD++

2️⃣ LD = GD
   → GD++
   → 디렉토리 2배
   → LD++
```

---

<a id="7-직관-고정"></a>
## 🔵 7️⃣ 직관 고정

```text
GD = 전체 배율
LD = 버킷 분화 정도
LD 작으면 많이 공유
LD 크면 적게 공유
LD=GD → 한 디렉토리만 가리킴
```

---

<a id="최종-암기-5줄"></a>
## 🔥 최종 암기 5줄

```text
1) 디렉토리 수 = 2^GD
2) LD ≤ GD
3) 공유 수 = 2^(GD - LD)
4) overflow & LD<GD → 버킷만 분할
5) overflow & LD=GD → GD 증가
```

---
좋다.
지금 헷갈리는 이유는 **이름은 많은데 계층 구조가 안 잡혀서**다.
딱 “위에서 아래로” 순서대로 정리해준다.
(시험용 핵심 키워드만)

## 🧱 0️⃣ 전체 큰 구조

<a id="1-hdd-하드디스크"></a>
## 1️⃣ HDD (하드디스크)

- 물리 저장 장치

- 그냥 컴퓨터 디스크

- 소프트웨어 아님

---

<a id="2-hadoop-하둡"></a>
## 2️⃣ Hadoop (하둡)

👉 **빅데이터 플랫폼 이름**

안에 여러 구성요소가 들어 있음.

HDFS + YARN + MapReduce

---

<a id="3-hdfs-hadoop-distributed-file-system"></a>
## 3️⃣ HDFS (Hadoop Distributed File System)

👉 **분산 저장 시스템**

- 여러 서버 디스크를 묶음

- 대용량 데이터 저장

- 블록 단위 저장

- 복제 기능 있음

📌 역할 = 저장

---

<a id="4-yarn-yet-another-resource-negotiator"></a>
## 4️⃣ YARN (Yet Another Resource Negotiator)

👉 **자원 관리 시스템**

- CPU, 메모리 배분

- 어떤 작업이 어디서 실행될지 관리

📌 역할 = 스케줄러 / 자원관리

---

<a id="5-mapreduce"></a>
## 5️⃣ MapReduce

👉 **하둡의 처리 방식**

- Map → Reduce

- 단계마다 디스크 기록

- 느림

📌 역할 = 구형 분산 처리

---

<a id="6-spark"></a>
## 6️⃣ Spark

👉 **고속 분산 처리 엔진 (하둡과 별개)**

- 메모리 기반 처리

- DAG 실행 구조

- RDD 사용

- 반복 연산 빠름

📌 역할 = 현대식 처리

---

<a id="7-rdd-resilient-distributed-dataset"></a>
## 7️⃣ RDD (Resilient Distributed Dataset)

👉 Spark의 데이터 추상화

- 분산된 데이터 집합

- 불변(immutable)

- 변환(Transformation) + 액션(Action)

📌 Spark에서 릴레이션 같은 개념

---

<a id="핵심-관계-한-줄씩"></a>
## 🔥 핵심 관계 한 줄씩

```text
HDFS = 저장
YARN = 자원관리
MapReduce = 구형 처리
Spark = 최신 처리
RDD = Spark의 데이터 구조
```

---

<a id="자주-나오는-시험-함정"></a>
## 🔵 자주 나오는 시험 함정

|문장|정답|
|---|---|
|HDFS는 처리 엔진이다|❌|
|Spark는 MapReduce 기반이다|❌|
|MapReduce는 Hadoop의 구성요소이다|⭕|
|Spark는 HDFS 없이 실행 가능하다|⭕|
|YARN은 자원관리 담당이다|⭕|

---

<a id="기억-방법"></a>
## 🧠 기억 방법

하둡은 세트다:

```text
HDFS + YARN + MapReduce
```

Spark는 별도 처리 엔진인데
HDFS를 저장소로 사용할 수 있다.

---

이제 구조가 이렇게 잡혀 있으면 된다:

```text
데이터 저장 → HDFS
자원 관리 → YARN
구형 처리 → MapReduce
고속 처리 → Spark
```
