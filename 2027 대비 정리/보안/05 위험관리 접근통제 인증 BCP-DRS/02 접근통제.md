## 📘 접근통제

### 📌 접근통제(Access Control)의 개념

자원(객체)에 대한 사용자(주체)의 접근을 통제하는 것이다. 자원에 대한 접근을 감시, 접근을 요구하는 이용자를 식별, 이용자의 요구가 정당한 것인지 확인, 보안정책에 근거하여 접근을 승인/거부, 접근 기록을 수행하는 HW, SW 및 행정적 관리를 총칭한다.

### 📌 접근통제 개념도

#### 🔥 구성요소

|구분|설명|
|---|---|
|**주체**|•자원에 접근하려는 사용자, 프로그램, 프로세스 (능동적 성격)|
|**객체**|•자원 자체, Data, 시스템, 프로그램 (수동적 성격)|
|**접근통제 매커니즘**|•접근통제 결정을 중재함, 모든 접근 요청이 통과되어야 하는 Single Point<br>•Reference Monitor라고도 불림, 방화벽이 하나의 예임<br>•특징: 우회 불가능해야 함, 부정 조작이 불가능해야 함|

흐름: 주체(인가자/비인가자) → 접근통제 매커니즘(데이터베이스 참조) → 객체(접근 허가/접근 거부)

### 📌 접근(Access)의 단계

|구분|설명|
|---|---|
|(단계1) Identification(식별)|•사용자가 시스템에게 본인이 누구라는 것을 밝히는 것, 신원을 밝히는 것<br>•예: ID|
|(단계2) Authentication(인증)|•시스템이 본인임을 주장하는 사용자가 그 본인이 맞다고 인정해 주는 것<br>•인증을 위해 사용자는 자신을 증명해야 함<br>•예: 패스워드, 토큰, 지문 등 생체인식, 서명|
|(단계3) Access Control mechanism|•접근 권한 유무를 판별|
|(단계4) Authorization(인가)|•접근을 승인하고, 사용자에게 해당 권한을 부여함<br>•ACL, Role|
|(단계5) Audit(감사)|•부여된 권한 내에서 작업이 진행되었는지를 감시|

### 📌 접근 통제의 구분 — 개념상의 분류

•접근통제 시스템: 시스템 자원에 대한 접근통제와 C.I.A 등의 보안서비스를 제공(방화벽, Secure OS 등) •접근통제 정책: 시스템 자원에 대한 사용자의 접근제한 권한/조건을 서술식으로 표현 •접근통제 매커니즘(기술): 정책을 시스템 구현상으로 표현(MAC, DAC, ACL 등) •접근통제 보안모델: 정확하고 간결하게 정책을 정규화(BLP, Biba, Clark & Wilson)

### 📌 접근 통제 정책

|구분|설명|
|---|---|
|**신분기반**(Identity-based) 접근통제 정책|•주체나 또는 그들이 속해있는 그룹들의 신분에 근거하여 객체에 대한 접근을 제한하는 방법<br>•**DAC**(Discretionary Access Control): 어떤 사용자가 어떤 행동을 할 수 있는지 각 타겟별로 목록 표현하는 IBP(Individual-Based Policy)와 다수의 사용자가 하나의 타겟에 대하여 동일한 허가를 부여받는 방식인 GBP(Group-Based Policy)로 분류|
|**규칙기반**(Rule-based) 접근통제 정책|•객체에 포함된 정보의 비밀성과 이러한 비밀성의 접근정보에 대하여 주체가 갖는 권한에 근거하여 객체에 대한 접근을 제한하는 방법<br>•**MAC**(Mandatory Access Control): MLP, CBP로 분류|
|**역할기반**(Role-based) 접근통제 정책|•중앙관리자가 주체와 객체의 상호관계를 통제하며 역할에 기초하여 자원에 대한 접근 허용 여부 결정<br>•**RBAC**(Role Based Access Control)의 적용원리: 권한관리, 계층적 역할분배, 최소권한 정책, 임무 분리, 객체 분류|

### 📌 접근 통제 유형 분석

|항목|DAC|MAC|RBAC|
|---|---|---|---|
|기반|ID기반 ACL(Access Control List)|보안 등급별 SL(Security Level)|Role 기반|
|중앙관리|구현 어려움|구현 용이|조직 사용자나 그룹별|
|개별보안|가능|적용 어려움|개별 룰 지정|
|장점|유연, 단순|정보 흐름 통제 가능|조직이나 경영 전략 유연|
|단점|효율성 저하, **트로이 목마**에 취약|개별적, 객체 단위 접근 불가|-|

### 📌 접근 통제 메커니즘

#### 🔥 ACL(Access Control List: 접근통제 목록)

주체가 디렉토리나 파일과 같은 특정 시스템 객체에 접근할 수 있는 허가받은 접근 종류들이 기록된 목록이다.

|사용자|권한(접근허가)|
|---|---|
|사용자A|읽기, 실행|
|사용자B|읽기, 쓰기, 실행|
|사용자C|접근 불가|

#### 🔥 CL(Capability List)

주체에게 허가된 자원 및 권한의 목록이다.

|객체|권한(접근허가)|
|---|---|
|X|읽기, 실행|
|Y|읽기, 쓰기, 실행|
|Z|접근 불가|

#### 🔥 [참고] 속성 기반 접근 제어(ABAC) — **2026년 기출**

•시간, 승인상태, 장애 여부 등의 속성에 따라 접근 권한을 통제 •주체, 객체, 환경 등의 다양한 '속성'과 '정책'을 기반으로 데이터나 리소스에 대한 접근을 동적으로 통제할 수 있기 때문에 유연성이 높아 클라우드 환경 등 IAM에서 널리 사용

### 📌 접근 통제 보안 모델
#### 🔥 Bell-Lapadula 모델(기밀성 모델)

군사용 보안구조의 요구사항을 충족 위하여 Bell과 Lapadula가 개발한 최초의 수학적 모델이다. 정보를 극비(Top Secret), 비밀(Secret), 미분류(Unclassified)로 분류함.

|구분|내용|
|---|---|
|상위레벨 읽기금지 정책(단순보안규칙, Simple Security Rule)|•No-read-up Policy(Simple Security Rule)<br>•인가받은 비밀등급이 낮은 주체는 비밀등급이 높은 객체를 읽지 못하는 보안정책|
|하위레벨 쓰기금지 정책(스타보안규칙, *-Security Rule)|•No-write-down Policy(Star Property Rule)<br>•인가받은 비밀등급 이하의 정보를 수정하지 못하게 하는 보안정책(읽기 가능, 쓰기 불가능)|
|==강한 스타 규칙(강한 스타보안규칙, Strong *-Security Rule)==|•==자신과 동일한 비밀등급의 객체에 대해서만 읽기/쓰기를 모두 허용하는 규칙==<br>•==Simple Security Rule과 Star Property Rule보다 더 엄격하게 통제, 등급이 다르면 읽기·쓰기 모두 불가==|

#### 🔥 Biba 모델(무결성 모델)

Biba Integrity 모델이라고도 하며, Bell-Lapadula 모델에서 불법 수정방지 내용을 추가로 정의한 무결성(Integrity) 모델이다. 낮은 비밀등급에서 높은 비밀등급으로 write를 하지 못하도록 함으로써 높은 무결성을 가진 데이터가 낮은 무결성을 가진 데이터와 합쳐져서 무결성이 오염되는 것을 방지함.

| 구분                                            | 내용                                                                                                    |
| --------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| 상위레벨 쓰기금지 정책(스타 무결성 규칙, Star Integrity Rule)  | •No-write-up Policy(* Star Integrity Property)<br>•주체는 자신의 무결성 접근등급이 객체의 무결성 접근등급보다 높을 경우에만 쓰기할 수 있음  |
| 하위레벨 읽기금지 정책(단순 무결성규칙, Simple Integrity Rule) | •No-read-down Policy(Simple Integrity Property)<br>•주체는 자신의 무결성 접근 등급보다 객체의 무결성 접근등급이 높은 경우에만 읽을 수 있음 |

#### 🔥 Clark and Wilson 모델(무결성 모델)

상업환경에 적합하게 개발된 불법 수정방지를 위한 보안 모델이다. 금융자산의 관리나 회계 등의 내용을 담고 있는 상업환경에서는 기밀성보다는 자료의 무결성이 더 중요하다. 주체는 객체에 대한 직접적인 접근을 가지지 않으며 객체는 오직 프로그램을 통하여 접근할 수 있음. 오직 프로그램을 통하여 접근될 수 있기 때문에 프로그램이 올바르게 디자인되면 무결성을 보장할 수 있음. 업무 분할로 중대한 기능을 둘로 분리하여 서로 다른 주체는 각 부분을 맡게 되고 이것은 객체의 무결성을 보호함.

비바 모델과 마찬가지로 무결성 보호 모델이지만 다른 관점에서 접근한다. 주체/프로그램/객체의 세 부분 관계를 사용한 무결성 보호 모델이다. 주체는 객체에 직접 접근하지 않고 객체가 프로그램을 통하여 접근하게 하는 것으로, 각 프로그램은 객체에 대하여 특정한 제한을 가지며 주체의 능력을 제한한다. 따라서 프로그램이 올바르게 작성됨으로써 객체의 무결성을 보호하는 수단을 제공한다. 또한 중대한 기능을 둘 이상의 부분으로 분리하여 서로 다른 주체가 각 부분을 완수하게 하는 것으로, 이것은 권한있는 주체가 객체에 대한 무허가 수정을 방지하여 객체의 무결성을 보호한다. 그리고 감사를 통해 객체에 대한 변경과 접근, 시스템 외부로부터의 입력을 추적한다.
