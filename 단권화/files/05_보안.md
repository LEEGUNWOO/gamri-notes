# # 📘 컴퓨터 시스템 보안 설계 원칙 (표 정리)

|구분|설명|
|---|---|
|최소 권한 (Least Privilege)|사용자는 수행해야 할 작업에 필요한 최소한의 권한만 가져야 하며, 더 이상 필요하지 않은 경우 해당 권한은 회수되어야 한다.|
|고장 안전 디폴트 (Fail-Safe Defaults)|명시적으로 허가되지 않은 모든 자원에 대한 접근은 기본적으로 거부되어야 한다.|
|매커니즘의 효율성 (Economy of Mechanism)|시스템 설계는 가능한 한 단순하게 구성되어야 하며, 복잡성은 오류 및 취약점 발생 가능성을 증가시킨다.|
|완벽한 조율 (Complete Mediation)|모든 접근 시도는 반드시 검사되어야 하며, 접근 시마다 권한 검증이 이루어져야 한다.|
|열린 설계 (Open Design)|보호 메커니즘은 공격자의 무지에 의존하지 않아야 하며, 공개되고 검증된 방식으로 설계되어야 한다.|
|권한 분리 (Separation of Privilege)|시스템은 단일 조건에 의해 권한을 부여해서는 안 되며, 다중 조건을 만족해야 권한이 부여되어야 한다.|
|최소 공통 메커니즘 (Least Common Mechanism)|공유 메커니즘의 수와 사용을 최소화해야 하며, 공유 객체는 정보 흐름 및 의도되지 않은 상호작용에 대한 잠재적 위험을 가진다.|
|사용의 편리성 (Psychological Acceptability)|사용자가 일반적으로 시스템을 사용하는 과정에서 자연스럽게 보안 메커니즘이 적용되도록 설계되어야 한다.|

# 📘 정성적 위험분석 vs 정량적 위험분석

## 1️⃣ 구분

|구분|정성적 위험분석|정량적 위험분석|
|---|---|---|
|개념|전문가 판단·경험 기반으로 위험 수준을 상대적으로 평가|수치·금액 기반으로 위험을 계량화하여 평가|
|표현|높음/중간/낮음 등 등급|금액, 확률, 기대손실 등 수치|
|목적|위험 우선순위 결정|비용-효과 분석, 투자 의사결정|
|정확도|상대적으로 낮음|상대적으로 높음|
|속도|빠름|느림 (데이터 필요)|
|데이터 요구|적음|많음|

---

## 2️⃣ 정성적 위험분석 기법

|기법|설명|
|---|---|
|델파이 기법 (Delphi)|전문가 의견을 반복적으로 수렴하여 합의 도출|
|브레인스토밍 (Brainstorming)|다양한 아이디어를 자유롭게 도출하여 위험 식별|
|체크리스트 (Checklist)|사전에 정의된 위험 목록을 활용하여 식별|
|시나리오 분석 (Scenario Analysis)|발생 가능한 상황을 가정하여 위험 분석|
|SWOT 분석|강점·약점·기회·위협을 기반으로 위험 분석|
|위험 매트릭스 (Risk Matrix)|발생 가능성과 영향도를 기준으로 위험 수준 분류|

## 3️⃣ 정량적 위험분석 기법

|기법|설명|
|---|---|
|기대손실 (Expected Loss, ALE)|자산가치 × 위협발생확률 × 손실비율|
|연간 예상 손실 (ALE)|SLE × ARO (Single Loss Expectancy × Annual Rate of Occurrence)|
|의사결정나무 (Decision Tree)|확률 기반으로 의사결정 결과 분석|
|몬테카를로 시뮬레이션|반복 난수 실험으로 위험 분포 분석|
|민감도 분석 (Sensitivity Analysis)|변수 변화가 결과에 미치는 영향 분석|
|손익분기 분석 (Break-even Analysis)|비용과 수익의 균형점 분석|

# 📘 생체인증의 조건

|특성|설명|
|---|---|
|보편성|모든 사람이 가지고 있는 보편적 특징을 대상으로 해야 한다.|
|영구성|특징은 변하지 않아야 하며, 시간이 지나도 변형되지 않아야 한다.|
|독특성|동일한 특징을 가진 다른 사람이 존재하지 않아야 한다.|
|획득성|센서를 통해 쉽게 획득할 수 있고, 정량화가 가능해야 한다.|
|시스템 성능|처리 속도가 빠르고 처리량이 커야 하며, 알고리즘 및 시스템 구성의 효율성이 확보되어야 한다.|
|저항성|위조 및 변조가 불가능하거나 매우 어려워야 한다.|
|수용성|사용자가 인증 시스템 사용에 거부감을 느끼지 않아야 한다.|

# 📘 SHA-3 해시 함수

## 1️⃣ 개요

| 항목  | 내용                                                                                 |
| --- | ---------------------------------------------------------------------------------- |
| 구성  | SHA3-224, SHA3-256, SHA3-384, SHA3-512 4개 해시 함수 + SHAKE128, SHAKE256 확장 출력 함수(XOF) |
| 구조  | 스펀지 구조 (Sponge Construction)                                                       |
| 특징  | 입력 메시지를 bitrate에 따라 흡수(absorbing) 후, 압착(squeezing) 과정을 통해 해시값 출력                   |
| 연산  | XOR 연산 기반 처리                                                                       |


## 2️⃣ SHA-3 종류별 출력 및 블록 크기

|종류|출력 크기 (bit)|블록 크기 (bit)|
|---|---|---|
|SHA3-224|224|1152|
|SHA3-256|256|1088|
|SHA3-384|384|832|
|SHA3-512|512|576|

# 📘 암호화 블록

| 모드          | 한글명               | 동작 방식              | 특징                | 시험 포인트           | 오류 전파     |
| ----------- | ----------------- | ------------------ | ----------------- | ---------------- | --------- |
| **ECB**     | 전자 코드북            | 각 블록을 독립적으로 암호화    | 동일 평문 → 동일 암호문    | 패턴 노출 / 보안 취약    | 1블록만 영향   |
| **CBC**     | 암호 블록 연쇄          | 이전 암호문과 XOR 후 암호화  | 패턴 숨김             | 병렬 처리 불가         | 2블록 영향    |
| **CFB**     | 암호 피드백            | 이전 암호문을 입력으로 사용    | 블록 암호를 스트림처럼 사용   | 오류 전파 발생         | 일부 비트 영향  |
| **OFB**     | 출력 피드백            | 키 스트림 생성 후 XOR     | 스트림 암호처럼 동작       | 오류 전파 없음         | 동일 비트 위치  |
| **CTR**     | 카운터 모드            | 카운터 값을 암호화 후 XOR   | 병렬 처리 가능          | 고속 암호화           | 해당 블록만 영향 |
| **GCM** | 갈루아 카운터 모드    | CTR + GHASH 인증 | 기밀성 + 인증(무결성) | AEAD 모드      | 블록 영향     |
| **CCM** | 카운터 + CBC-MAC | CTR + CBC-MAC  | 기밀성 + 인증(무결성) | 무선 네트워크에서 사용 | 블록 영향     |
👉 암호화 알고리즘으로 복호화 가능 : CFB OFB CTR 
👉 암호화 복호화 병렬처리 : ECB CTR
👉 복호화만 병렬처리 가능 : CBC OFB

# 📘 해시함수 요구사항

| 특징                      | 설명                                                                       |
| ----------------------- | ------------------------------------------------------------------------ |
| 메시지 압축 (Message Digest) | 임의의 길이의 평문을 고정된 길이의 출력값으로 변환해야 한다.                                       |
| 계산의 용이성                 | 입력값 x가 주어지면 H(x)는 계산적으로 쉽게 구할 수 있어야 한다.                                  |
| 일방향성 (One-Way)          | 메시지(평문)에서 해시값을 구하는 것은 쉽지만, 해시값에서 원래 메시지를 구하는 것은 매우 어려워야 한다. (역상 저항성)     |
| 약한 충돌 회피성               | 입력값 x가 주어졌을 때 H(x') = H(x)인 x' (x ≠ x)를 찾는 것은 계산적으로 불가능해야 한다. (제2역상 저항성) |
| 강한 충돌 회피성               | H(x) = H(x')인 서로 다른 두 입력값 x, x'를 찾는 것은 계산적으로 불가능해야 한다. (충돌 저항성)          |

# 📘 디피-헬만 (Diffie-Hellman) 키 교환

## 1️⃣ 개념

|구분|내용|
|---|---|
|목적|공개된 채널에서 안전하게 **공유 비밀키 생성**|
|방식|이산로그 문제 기반|
|특징|중간에서 값이 보여도 **비밀키는 계산 불가**|

## 2️⃣ 계산 절차

|단계|설명|
|---|---|
|공통값 설정|큰 소수 p, 생성자 g 공개|
|A 선택|사용자 A가 비밀값 a 선택|
|B 선택|사용자 B가 비밀값 b 선택|
|A 공개값|A = g^a mod p|
|B 공개값|B = g^b mod p|
|A 계산|K = B^a mod p|
|B 계산|K = A^b mod p|

## 1️⃣ Feistel 구조 vs SPN 구조

|구분|Feistel 구조|SPN 구조|
|---|---|---|
|구조 방식|평문을 좌(L0), 우(R0)로 나누어 처리 후 교차|Substitution Layer + Permutation Layer 반복|
|연산 방식|함수 F 적용 후 XOR 연산으로 좌우 교환|S-box 치환 + P-box 순열|
|Confusion / Diffusion|동일 수준 확보 위해 SPN보다 더 많은 반복 필요|적은 반복으로 Confusion / Diffusion 달성|
|병렬 처리|어려움|가능|
|복호화|암호화 구조와 동일 (복호기 별도 불필요)|복호화 시 별도 역연산 필요|
|특징|구조 단순, 구현 용이|구조 복잡, 성능 우수|
## 2️⃣ 알고리즘 예시

|구분|알고리즘|
|---|---|
|Feistel|DES, SEED, CAST, Blowfish, Twofish, RC5, RC6|
|SPN|IDEA, Rijndael(AES), ARIA|

## 3️⃣ 핵심 차이 요약

|구분|내용|
|---|---|
|Feistel|XOR + 좌우 교환 구조, 복호화 동일|
|SPN|치환 + 순열 구조, 복호화 별도 필요|

# 📘 무선 보안 기술 정리 (SSID, WEP, WPA, WPA2, WPA3)

|구분|내용|
|---|---|
|SSID|Service Set Identifier  <br>무선랜에서 네트워크 이름을 의미하며 기본적인 접속 제어 기능 제공  <br>AP가 비콘 신호에 SSID를 포함하여 브로드캐스트  <br>SSID만으로는 보안 기능이 약하며 접속 허용/차단 기준으로는 취약|
|WEP|Wired Equivalent Privacy  <br>무선 데이터 보안을 위한 초기 암호화 방식  <br>대칭키 구조 사용  <br>RC4 (40bit key) 암호 알고리즘 사용  <br>취약점 존재로 현재 사용 비권장|
|WPA|Wi-Fi Protected Access  <br>WEP 보완 기술  <br>RC4 (128bit key) 사용  <br>TKIP 적용하여 동적 키 사용|
|WPA2|AES-CCMP 암호화 사용  <br>인증 방식: IEEE 802.1X (기업) / PSK (개인)  <br>무결성: Michael 알고리즘|
|WPA3|SAE 인증 방식 적용 (PSK 개선)  <br>GCMP-256 암호화 사용  <br>PMF (Protected Management Frames) 적용|

# 📘 SSL 프로토콜 구성

| 구성요소                      | 기능                                                                                                             |
| ------------------------- | -------------------------------------------------------------------------------------------------------------- |
| Record Layer Protocol     | 분배된 대칭키를 이용하여 데이터 암호화 수행  <br>TCP 위에서 동작하며 Handshake에서 협의된 알고리즘 사용  <br>데이터를 적절한 크기로 분할 → 압축 → 암호화 → 무결성 검증 수행 |
| ChangeCipherSpec Protocol | ChangeCipherSpec 단계 시작을 알리는 신호  <br>이후부터 협의된 암호 알고리즘(비밀키, 메시지 다이제스트, 공개키 등)을 적용하여 통신 시작                        |
| Alert Protocol            | 오류 및 경고 상황을 상대방에게 알림  <br>SSL 통신 중 문제가 발생했을 때 해당 내용을 전달                                                        |
| Handshake Protocol        | 사용할 암호 알고리즘 결정 및 키 교환 수행  <br>Record Layer 상위에서 동작  <br>PKI 기반 사용자 인증 수행  <br>비밀키 생성 및 설정  <br>서버/클라이언트 인증 수행  |
👉 **Handshake > ChangeCipherSpec**

# 📘 포워드 프록시 vs 리버스 프록시

| 구분         | 포워드 프록시 (Forward Proxy) | 리버스 프록시 (Reverse Proxy) |
| ---------- | ----------------------- | ----------------------- |
| 위치         | 클라이언트 앞단                | 서버 앞단                   |
| 대상         | 내부 사용자 → 외부 서버          | 외부 사용자 → 내부 서버          |
| 목적         | 사용자 보호, 접근 제어           | 서버 보호, 부하 분산            |
| 숨기는 대상 | 클라이언트(IP)           | 서버(IP)              |
| 사용 예   | 기업 내부 인터넷 접속 통제     | 웹 서비스, CDN, 로드밸런서   |
| 대표 기능  | 캐싱, 필터링, 우회 접속      | 로드밸런싱, SSL 종료, 보안   |

# 👉 핵심 특징

|구분|내용|
|---|---|
|포워드 프록시|클라이언트가 프록시를 인지하고 사용|
|리버스 프록시|클라이언트는 프록시 존재를 모름 (서버처럼 보임)|


# 💣 시험 핵심 포인트

- 클라이언트 숨김 → 포워드 프록시
    
- 서버 숨김 → 리버스 프록시
    
- 로드밸런싱 → 리버스 프록시
    
- 접근 통제 → 포워드 프록시

# 📘 침입탐지시스템 (IDS) – 네트워크 기반 vs 호스트 기반

|구분|네트워크 기반 IDS (NIDS)|호스트 기반 IDS (HIDS)|
|---|---|---|
|설치 위치|네트워크 구간 (스위치, 라우터 등)|개별 서버/PC 내부|
|감시 대상|네트워크 트래픽 전체|시스템 내부 활동 (로그, 파일, 프로세스)|
|탐지 방식|패킷 분석|로그/파일 변경/시스템 호출 분석|
|장점|전체 네트워크 감시 가능  <br>실시간 트래픽 분석|내부 침입 탐지 가능  <br>암호화 트래픽 이후 분석 가능|
|단점|암호화 트래픽 분석 어려움  <br>내부 행위 상세 분석 어려움|각 호스트마다 설치 필요  <br>관리 부담 증가|
|탐지 범위|외부 공격, 네트워크 기반 공격|내부 침입, 권한 상승, 파일 변조|
|성능 영향|네트워크 장비 성능 영향|호스트 자원 사용|

# 📘 OWASP LLM Top 10 (2025) — 영문 / 한글 / 개념

## 1️⃣ LLM01: Prompt Injection (프롬프트 인젝션)

- **정의:** 사용자 입력으로 시스템 프롬프트/정책을 우회하거나 모델 동작을 변경하는 공격
    
- **핵심:** 지시 탈취 / 정책 무력화
    

## 2️⃣ LLM02: Sensitive Information Disclosure (민감 정보 노출)

- **정의:** 모델이 학습 데이터, 프롬프트, API 키 등 민감 정보를 출력하는 문제
    
- **핵심:** 데이터 유출
    

## 3️⃣ LLM03: Supply Chain (공급망 취약점)

- **정의:** 외부 모델, 데이터셋, 라이브러리 등 의존 요소를 통한 공격
    
- **핵심:** 외부 의존 위험
    

## 4️⃣ LLM04: Data and Model Poisoning (데이터 및 모델 오염)

- **정의:** 학습 데이터, 파인튜닝, RAG 등을 통해 모델을 악의적으로 오염
    
- **핵심:** 응답 왜곡 / 백도어
    

## 5️⃣ LLM05: Improper Output Handling (부적절한 출력 처리)

- **정의:** LLM 출력값을 검증 없이 실행/처리하는 문제
    
- **핵심:** 출력 → 코드/명령 실행 위험
    

## 6️⃣ LLM06: Excessive Agency (과도한 권한 부여)

- **정의:** LLM에 시스템 제어, 결제, 실행 권한 등을 과도하게 부여
    
- **핵심:** 자동 실행 위험
    

## 7️⃣ LLM07: System Prompt Leakage (시스템 프롬프트 유출)

- **정의:** 내부 시스템 프롬프트 또는 정책이 외부로 노출되는 문제
    
- **핵심:** 내부 정책 노출
    

## 8️⃣ LLM08: Vector and Embedding Weaknesses (벡터/임베딩 취약점)

- **정의:** 벡터DB, 임베딩, RAG 과정에서 발생하는 보안 취약점
    
- **핵심:** 검색 기반 데이터 오염/유출
    

## 9️⃣ LLM09: Misinformation (허위 정보)

- **정의:** 모델이 부정확하거나 잘못된 정보를 생성하고 이를 신뢰하는 문제
    
- **핵심:** 환각(Hallucination) + 잘못된 판단 유도
    

## 🔟 LLM10: Unbounded Consumption (무제한 자원 소비)

- **정의:** 과도한 요청/토큰 사용으로 비용 및 자원을 무한 소비하게 하는 공격
    
- **핵심:** 비용 폭증 / 자원 고갈

# 📘  OWASP Top 10 (2025) — 영문 / 한글 / 개념

## 1️⃣ A01: Broken Access Control (접근 통제 실패)

- **정의:** 사용자가 권한을 넘어 데이터나 기능에 접근하는 문제
    
- **핵심:** 권한 검증 실패
    

## 2️⃣ A02: Security Misconfiguration (보안 설정 오류)

- **정의:** 잘못된 설정, 기본값 사용, 불필요한 서비스 노출
    
- **핵심:** 설정 실수
    

## 3️⃣ A03: Software Supply Chain Failures (소프트웨어 공급망 실패)

- **정의:** 외부 라이브러리, 의존성, 빌드/배포 과정에서 발생하는 취약점
    
- **핵심:** 공급망 공격
    

## 4️⃣ A04: Cryptographic Failures (암호화 실패)

- **정의:** 민감 데이터 보호를 위한 암호화가 적절히 적용되지 않은 경우
    
- **핵심:** 데이터 보호 실패
    

## 5️⃣ A05: Injection (인젝션)

- **정의:** 사용자 입력이 코드나 쿼리로 실행되는 취약점
    
- **핵심:** SQL, OS, LDAP Injection
    

## 6️⃣ A06: Insecure Design (취약한 설계)

- **정의:** 설계 단계에서 보안이 고려되지 않아 발생하는 구조적 문제
    
- **핵심:** 설계 결함
    

## 7️⃣ A07: Authentication Failures (인증 실패)

- **정의:** 인증 및 세션 관리의 취약점으로 계정 탈취 가능
    
- **핵심:** 인증 취약
    

## 8️⃣ A08: Software or Data Integrity Failures (무결성 실패)

- **정의:** 코드 또는 데이터의 변조를 방지하지 못하는 문제
    
- **핵심:** 무결성 훼손
    

## 9️⃣ A09: Security Logging & Alerting Failures (로깅 및 알림 실패)

- **정의:** 공격 탐지 및 대응을 위한 로그와 모니터링 부족
    
- **핵심:** 탐지 실패
    

## 🔟 A10: Mishandling of Exceptional Conditions (예외 처리 실패)

- **정의:** 오류나 예외 상황을 적절히 처리하지 못해 발생하는 취약점
    
- **핵심:** 에러 처리 취약

# 📘 네트워크 필수 명령어 세트

## 1️⃣ ping

- 목적: **연결 여부 확인**
    
- 방식: ICMP Echo Request/Reply
    
- 핵심: “통신 되냐 안되냐”
    

## 2️⃣ tracert

- 목적: **경로 확인**
    
- 방식: TTL + ICMP
    
- 핵심: “어디서 끊기냐”
    

## 3️⃣ ipconfig

- 목적: **내 PC 네트워크 정보 확인**
    
- 확인 가능:
    
    - IP 주소
        
    - 서브넷 마스크
        
    - 게이트웨이
        
    - DNS
        

👉 `/all` 옵션 자주 나옴

## 4️⃣ nslookup

- 목적: **도메인 → IP 변환 확인**
    
- 핵심: DNS 조회
    

👉 “DNS 문제냐?” 판단할 때 사용

## 5️⃣ netstat

- 목적: **현재 연결 상태 확인**
    
- 확인:
    
    - 열린 포트
        
    - 연결 상태 (ESTABLISHED 등)
        

👉 `/an` 옵션 자주 나옴

# 📘 인공지능 ISO/IEC 42001 vs ISO/IEC 23894

| 구분     | ISO/IEC 42001 (AIMS)                                   | ISO/IEC 23894 (Risk Management)                        |
| ------ | ------------------------------------------------------ | ------------------------------------------------------ |
| 정의     | 인공지능 경영시스템 (Artificial Intelligence Management System) | 인공지능 위험 관리 (Artificial Intelligence - Risk Management) |
| 성격     | 요구사항(Requirements) 기반 표준                               | 지침(Guidelines) 기반 표준                                   |
| 인증 가능성 | 인증 가능 (Certifiable)                                    | 인증 불가 (가이드로 활용)                                        |
| 주요 목적  | 조직의 AI 거버넌스 체계 구축 및 대외 신뢰성 증명                          | AI 수명 주기 전반의 위험 관리 프로세스 구체화                            |
| 기반 표준  | ISO/IEC 27001 (정보보호 경영시스템)                             | ISO 31000 (위험 관리 가이드라인)                                |
| 핵심 범위  | 정책, 리더십, 자원 관리, 성과 평가 등 조직 전체                          | 위험 식별, 분석, 평가, 처리 등 리스크 관리 실무                          |
| 적용 시점  | 조직의 AI 도입 및 운영 전략 수립 시                                 | AI 개발, 배포, 운영 등 실무 프로세스 단계                             |
| 비유     | 건축물의 전체 설계도 및 관리 규정                                    | 안전한 건축을 위한 상세 안전 시공 매뉴얼                                |
