# 📘 **IPV4 vs IPV6 헤더 구성 비교**

#### 1️⃣ **IPV4 헤더 구성 (기본 20바이트)**

IPv4는 32비트, IPv6는 128비트라서 주소 공간이 훨씬 넓어진 거야.

| 필드                                     | 크기    | 설명                             |
| -------------------------------------- | ----- | ------------------------------ |
| **버전 (Version)**                       | 4비트   | **IPv4**일 경우 4                 |
| **헤더 길이 (Header Length)**              | 4비트   | 헤더의 길이를 **32비트 워드** 단위로 표시     |
| **서비스 타입 (Type of Service, TOS)**      | 8비트   | 패킷의 **우선순위** 및 **서비스 요구사항**    |
| **전체 길이 (Total Length)**               | 16비트  | **패킷의 전체 길이** (헤더 + 데이터)       |
| ==**식별자 (Identification)**==           | 16비트  | 패킷을 **식별**하는 번호                |
| ==**플래그 (Flags)**==                    | 3비트   | 패킷의 **분할 여부**를 나타내는 플래그        |
| ==**조각 오프셋 (Fragment Offset)**==       | 13비트  | 패킷의 **조각화**를 위한 오프셋 값          |
| **TTL (Time to Live)**                 | 8비트   | 패킷이 **네트워크에서 살아 있는 시간** (초 단위) |
| **프로토콜 (Protocol)**                    | 8비트   | **상위 계층 프로토콜** (TCP, UDP 등)    |
| **헤더 체크섬 (Header Checksum)**           | 16비트  | 헤더의 **오류 검출**                  |
| **출발지 IP 주소 (Source IP Address)**      | 32비트  | 출발지의 **IPv4 주소**               |
| **목적지 IP 주소 (Destination IP Address)** | 32비트  | 목적지의 **IPv4 주소**               |
| **옵션 (Options)**                       | 가변 크기 | **옵션 필드** (옵션을 포함할 경우에만 존재)    |

- **기본 크기**: **20바이트**
- **옵션** 포함 시 **최대 60바이트**까지 확장
- **플래그는 3비트:**

| 이름                      | 의미                         |
| ----------------------- | -------------------------- |
| 예약                      | 항상 0                       |
| DF (**Don't Fragment**) | 1이면 단편화 금지                 |
| MF (**More Fragments**) | 1이면 뒤에 조각 더 있음, 0이면 마지막 조각 |


#### 2️⃣ **IPV6 헤더 구성 (고정 40바이트)**

| 필드                                  | 크기    | 설명                                   |
| ----------------------------------- | ----- | ------------------------------------ |
| **버전 (Version)**                    | 4비트   | **IPv6**일 경우 6                       |
| **트래픽 클래스 (Traffic Class)**         | 8비트   | **서비스 품질** 및 **우선순위** 정의 TOS(이름만 바뀜) |
| **플로우 레이블 (Flow Label)**            | 20비트  | **특정 흐름(같은묶음)에** 대해 **처리 우선순위**를 지정  |
| **페이로드 길이 (Payload Length)**        | 16비트  | **데이터 부분**의 길이                       |
| **다음 헤더 (Next Header)**             | 8비트   | 상위 계층 프로토콜 (TCP, UDP 등)              |
| **홉 제한 (Hop Limit)**                | 8비트   | 패킷이 **네트워크에서 살아 있는 시간** (TTL과 동일)    |
| **출발지 주소 (Source IP Address)**      | 128비트 | 출발지의 **IPv6 주소**                     |
| **목적지 주소 (Destination IP Address)** | 128비트 | 목적지의 **IPv6 주소**                     |

- **기본 크기**: **40바이트** 

- **옵션**은 **확장 헤더**로 분리하여 처리
## Pv6 확장 헤더 – Next Header 값 정리

| 값 (Decimal) | 헤더 이름                                | 설명                                    |
| ----------- | ------------------------------------ | ------------------------------------- |
| 0           | Hop-by-Hop Options Header            | 모든 라우터가 반드시 검사해야 하는 옵션 정보를 전달하는 확장 헤더 |
| 43          | Routing Header                       | 패킷이 특정 경로를 따라 전달되도록 경로 정보를 지정         |
| 44          | Fragment Header                      | IPv6 패킷을 분할(fragmentation)할 때 사용되는 헤더 |
| 50          | Encapsulating Security Payload (ESP) | IPsec에서 데이터 암호화 및 무결성 보호 제공           |
| 51          | Authentication Header (AH)           | IPsec에서 패킷 인증 및 무결성 검증 제공             |
| 60          | Destination Options Header           | 목적지 노드에서만 처리하는 옵션 정보 전달               |
| 59          | No Next Header                       | 다음 헤더가 없음을 의미                         |

<a id="2-tcp-헤더-구성-transmission-control-protocol-header"></a>
## 📌 2️⃣ TCP 헤더 구성 (Transmission Control Protocol Header)

| 필드                    | 한글        | 설명                  |
| --------------------- | --------- | ------------------- |
| Source Port           | 출발지 포트    | 송신 프로세스 식별          |
| Destination Port      | 목적지 포트    | 수신 프로세스 식별          |
| Sequence Number       | 순서 번호     | 데이터 순서 보장           |
| Acknowledgment Number | 확인 번호     | 수신 확인               |
| **Data Offset(HLEN)** | 헤더 길이     | 헤더 크기 표시            |
| Flags (Control Bits)  | **제어 비트** | **SYN, ACK, FIN 등** |
| Window Size           | 윈도 크기     | 흐름 제어               |
| **Checksum**          | 체크섬       | 오류 검출               |
| Urgent Pointer        | 긴급 포인터    | 긴급 데이터 위치           |

✔ 연결지향(Connection-Oriented)
✔ 신뢰성 보장(Reliable)
✔ 순서 보장(Ordered)
✔ 20Byte
✔ 소스포트번호 16비트

<a id="3-udp-헤더-구성-user-datagram-protocol-header"></a>
## 📌 3️⃣ UDP 헤더 구성 (User Datagram Protocol Header)

|필드|한글|설명|
|---|---|---|
|Source Port|출발지 포트|송신 프로세스 식별|
|Destination Port|목적지 포트|수신 프로세스 식별|
|Length|길이|전체 데이터그램 길이|
|**Checksum**|체크섬|오류 검출|

✔ 비연결형(Connectionless)
✔ 신뢰성 없음(Unreliable)
✔ 순서 보장 없음
✔ 8Byte

|                                 | 정확성 (correctness)                        | 기능이 **올바른 결과**를 내는 정도                                    |
|                                 | 적절성 (appropriateness)                    | 기능이 사용자의 업무 목적 달성에 **도움이 되도록 적합하게** 제공되는 정도              |
| 신뢰성 (Reliability)               | 성숙성 (Maturity)                           | 장애나 실패가 적고 **안정적으로 동작**하는 정도                             |
|                                 | 가용성 (Availability)                       | 필요할 때 시스템을 **사용 가능**한 정도                                 |
|                                 | 결함허용성 (Fault tolerance)                  | 일부 오류나 장애가 발생해도 **서비스를 계속 유지**하는 정도                      |
|                                 | 복구성 (Recoverability)                     | 장애 후 데이터·상태·서비스를 **복원**할 수 있는 정도                         |
| 사용성 (Usability) 유용성             | 적절 인식성 (Appropriateness recognizability) | 사용자가 이 시스템이 **내 목적에 맞는지 바로 파악**할 수 있는 정도                 |
|                                 | 학습성 (Learnability)                       | 배우기 쉬운 정도                                                |
|                                 | 운영성/운용성 (Operability)                    | 사용자가 기능을 **쉽게 조작·통제**할 수 있는 정도                           |
|                                 | 사용자 오류 방지성 (User error protection)       | 사용자의 실수 발생을 **줄이거나 예방**해 주는 정도                           |
|                                 | UI 심미성 (User interface aesthetics)       | 화면과 인터페이스가 **보기 좋고 일관적**인 정도                             |
|                                 | 접근성 (Accessibility)                      | 다양한 사용자, 특히 제약이 있는 사용자도 **이용 가능**한 정도                    |
| 성능 효율성 (Performance efficiency) | 시간 반응성 (Time behaviour)                  | 응답시간, 처리시간 등 **속도 측면 성능**                                |
|                                 | 자원 효율성 (Resource utilisation) 요소활용       | CPU, 메모리, 네트워크 등 자원을 **효율적으로 사용하는 정도**                   |
|                                 | 수용성 용량 (Capacity)                        | 동시 사용자 수, 데이터량, 처리량 등 **감당 가능한 한계 규모**                   |
| 유지관리성 (Maintainability)         | 모듈성 (Modularity)                         | 구성요소가 잘 분리돼 한 부분 변경이 다른 부분에 **영향을 덜 주는 정도**              |
|                                 | 재사용성 (Reusability)                       | 자산이나 구성요소를 **다른 시스템/기능에 다시 활용**할 수 있는 정도                 |
|                                 | 분석성 (Analysability)                      | 결함 원인, 변경 영향 등을 **분석하기 쉬운 정도**                           |
|                                 | 변경용이성 (Modifiability)                    | 수정·개선·확장이 쉬운 정도                                          |
|                                 | 시험용이성 (Testability)                      | 테스트 케이스 설계, 실행, 결과 확인이 **쉬운 정도**                         |
| 이식성 (Portability)               | 적응성 (Adaptability) 적용성                   | 다른 환경에서도 **쉽게 맞춰 적용**할 수 있는 정도                           |
|                                 | 설치성 (Installability)                     | 설치·배포·제거가 쉬운 정도                                          |
|                                 | 대체성 (Replaceability)                     | 다른 제품을 대체하거나 다른 제품으로 **대체되기 쉬운 정도**                      |
| 보안성 (Security)                  | 기밀성 (Confidentiality)                    | 인가되지 않은 대상에게 정보가 **노출되지 않는 정도**                          |
|                                 | 무결성 (Integrity)                          | 정보나 기능이 **허가 없이 변경·훼손되지 않는 정도**                          |
|                                 | 부인방지 (Non-repudiation)                   | 행위 수행 사실을 나중에 **부인하지 못하게** 하는 정도                         |
|                                 | 접근추적성/책임추적성 (Accountability)             | 누가 무엇을 했는지 **추적·책임 귀속**이 가능한 정도<br>**내계좌로 누가 접근? 으로 외워** |
|                                 | 인증성 (Authenticity)                       | 사용자·시스템·데이터 출처가 **진짜임을 확인**할 수 있는 정도                     |
| 호환성 (Compatibility)             | 공존성 (Co-existence)                       | 동일 환경에서 다른 시스템과 **충돌 없이 함께 운영**되는 정도                     |
|                                 | 상호운용성 (Interoperability)                 | 다른 시스템과 **정보를 주고받고 활용**할 수 있는 정도                         |

적응성 (Adaptability) vs 공존성 (Co-existence)

|-|Adaptability|Co-existence|
|---|---|---|
|상위 특성|**이식성(Portability)**|**호환성(Compatibility)**|
|핵심|환경이 바뀌어도 잘 작동|다른 SW와 공유 환경에서 충돌 없이 공존|
|주어|**소프트웨어 자신**이 적응|**다른 소프트웨어들**이랑 같이 삼|
한 줄 요약:

- Adaptability → **"나 어디서든 잘 돼"**
- Co-existence → **"나 옆 프로그램이랑 안 싸워"**
- **이식성은 나 자신에 대한 이야기 호환성은 타인과의 이야기**

# **복원성 공학의 4가지 복원 활동**

| 복원 활동                     | 설명                                                                                                                                                                                                                                                    |
| ------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **인식**<br>(Recognition)   | • 시스템이나 시스템 운영자들은 시스템 장애를 일으킬 수 있는<br> 문제의 증상들을 인식할 필요가 있다<br>• 이러한 인식은 장애 발생 이전에 가능해야 한다                                                                                                                                                             |
| **저항**<br>(Resistance)    | • 문제의 증상 또는 사이버 공격의 징후들이 조기에 발견되는 경우,<br> 저항 전략은 시스템이 실패할 확률을 감소시킬 수 있다<br>• 저항 전략들은 **시스템의 중심 부분들을 분리**함에 초점을 두어<br> 다른 곳에서 발생된 문제에 영향을 받지 않게 된다<br>• 저항은 문제들을 가두기 위한 방어들이 시스템에 포함되어 있는<br> **적극적 저항**과 문제가 발견될 때 필요한 조치들이 수행되는<br> **반응성 저항**을 포함한다 |
| **복구**<br>(Recovery)      | • 장애가 발생하는 경우, 복구 활동의 목적은<br> **중심 시스템 서비스들이 빠르게 복구**되는 것을 보장하여<br> 시스템 사용자들이 장애에 의하여 심각한 영향을 받지 않게 된다                                                                                                                                                |
| **회복**<br>(Reinstatement) | • 이 마지막 활동에서, **모든 시스템 서비스들이 복구**되고<br> 정상적인 시스템 작업이 계속될 수 있다                                                                                                                                                                                         |

**암기 순서:** 인식 → 저항 → 복구 → 회복 (**인저복회**)

# COT = **추론 과정 명시화**, COVE = **자가 검증으로 환각 감소**

| 구분          | COT (Chain of Thought)          | COVE (Chain of Verification)            |
| ----------- | ------------------------------- | --------------------------------------- |
| **한글명**     | 사고 연쇄                           | 검증 연쇄                                   |
| **목적**      | 추론 정확도 향상                       | 환각(Hallucination) 감소                    |
| **핵심 아이디어** | 중간 추론 단계를 명시적으로 생성              | **생성한 답변을 스스로 검증 질문으로 재확인**             |
| **동작 방식**   | "단계적으로 생각해봐" → 중간 과정 포함해서 답변    | 초안 생성 → 검증 질문 생성 → 검증 질문에 답변 → 최종 답변 수정 |
| **프롬프트 예시** | "Let's think step by step"      | "위 답변이 맞는지 확인 질문을 만들고 각각 답해봐"           |
| **강점**      | 복잡한 수학/논리 문제에 효과적               | 사실 오류 자가 수정에 효과적                        |
| **약점**      | **추론 자체가 틀리면 자신 있게 오답 생성**      | 검증 질문 자체가 부실하면 효과 없음                    |
| **제안자**     | Google Brain (Wei et al., 2022) | Meta AI (Dhuliawala et al., 2023)       |

**COVE 동작 4단계**

```
① Draft    → "세종대왕은 1397년생이다"
② Plan     → "세종대왕 출생연도는?" "훈민정음 창제연도는?"
③ Execute  → 각 질문에 독립적으로 답변
④ Refine   → 검증 결과 반영해서 최종 답변 수정
```

| **다음 주자의 준비 부족** (Dropped Baton) | 일찍 끝내도 후속작업 자원이 준비 안 됨 → 시간 낭비                 |
| **자원 병목** (Resource Bottleneck)  | 희소 자원 가용성 문제로 일정 지연                            |
| **학생 증후군** (Student Syndrome)    | 목표일 임박해야 일 시작 → 문제 발생 시 즉시 지연으로 연결             |

# 의도형 로크

| 락   | 의미                       | 자손에 걸 수 있는 락     |
| --- | ------------------------ | ---------------- |
| IS  | 아래 어딘가(**일부**) S 걸 거야    | IS, S            |
| IX  | 아래 어딘가(**일부**) X 걸 거야    | IS, IX, S, X     |
| S   | 이 노드 전체 **읽기**           | 없음 (자손 이미 묵시적 S) |
| SIX | **전체** 읽으면서 **일부** 고칠 거야 | **X** , IX       |
| X   | 이 노드 전체 쓰기               | 없음 (자손 이미 묵시적 X) |

# 양립성 표 (같은 노드에 두 트랜잭션이 동시에 걸 수 있냐):

|-|IS|IX|S|SIX|X|
|---|---|---|---|---|---|
|IS|○|○|○|○|X|
|IX|○|○|X|X|X|
|S|○|X|○|X|X|
|SIX|○|X|X|X|X|
|X|X|X|X|X|X|

암기 팁:

- X 들어가면 다 충돌 (X는 독점)
- IS는 제일 약해서 X 빼고 다 허용
- SIX는 S+IX니까 둘 다 안 되는 거 합집합으로 다 안 됨

**혼돈(Confusion)과 확산(Diffusion)**

**혼돈 (Confusion)**

> 암호문에서 키를 알아내기 어렵게 하는 성질
   치환(Substitution)

비유: 원문 "A"를 "Z"로 바꿔치기 → 원래 뭐였는지 모르게

**확산 (Diffusion)**

> 암호문에서 원본 메시지를 알아내기 어렵게 하는 성질
> 전치(Permutation)

비유: 바꾼 결과를 전체에 흩뿌려서 패턴 자체를 없앰

| 구분  | 목적         | 방법               |
| --- | ---------- | ---------------- |
| 혼돈  | **키** 숨기기  | 치환(Substitution) |
| 확산  | **평문** 숨기기 | 전치(Permutation)  |

> AB → C, C → D, E → B
> 
① R1(A, C, E), R2(E, B), R3(C, D)

**E를 결정해주는 결정자가 없다면 E는 후보키의 일부임.**
**기준 메인테이블은 후보키를 살려놔야함 . E->B이므로 AE가 후보키가됨.** 

## 지문 복원

순서(순차) 고정 길이 레코드 저장 파일 구조에서 특정 레코드를 **이진 탐색**할 경우 걸리는 최대 시간으로 가장 가까운 것은?

> 전체 레코드: 200,000개 / 레코드 길이: 50byte / 블록 헤더: 24byte / 블록 크기: 1,024byte / 블록 읽기: 5ms

① 65ms ② 70ms ③ 10,000ms ④ 25,000ms

**정답: ②**

## 풀이

**Step 1. 블록당 레코드 수 (블로킹 인수)**

> 실제 데이터 공간 = 1,024 - 24 = 1,000 byte 블록당 레코드 수 = 1,000 ÷ 50 = **20개**

**Step 2. 전체 블록 수**

> 200,000 ÷ 20 = **10,000 블록**

**Step 3. 이진 탐색 최대 비교 횟수**

> log₂(10,000) = ? 2¹³ = 8,192 2¹⁴ = 16,384 → 올림해서 **14회**

**Step 4. 최대 탐색 시간**

> 14 × 5ms = **70ms**

## 선택지 판정

|선택지|판정|이유|
|---|---|---|
|① 65ms|❌|13회 × 5ms, 올림 안 한 경우|
|② 70ms|✅|14회 × 5ms|
|③ 10,000ms|❌|전체 블록 순차탐색 시간|
|④ 25,000ms|❌|전체 레코드 순차탐색 시간|

**핵심 공식:**

> 이진탐색 최대 횟수 = **⌈log₂(전체블록수)⌉ << 해밍코드 패리티 비트 구할때도 동일함**
