## 1. ISO/IEC 12207

**소프트웨어 생명주기 프로세스** 표준이다.  
소프트웨어의 **획득, 공급, 개발, 운영, 유지보수** 등 전 과정을 프로세스 관점에서 정리한 틀이다.  
시험에서는 “생명주기”라는 말이 나와도, **소프트웨어 중심인지 시스템 전체인지**를 구분해야 한다.  
**소프트웨어 중심이면 12207**이다. ISO는 12207을 “software life cycle processes”로 설명한다. ([ISO](https://www.iso.org/ics/35.080.html?utm_source=chatgpt.com "35.080 - Software"))

**핵심**

- 소프트웨어 전용 생명주기
    
- 개발뿐 아니라 운영·유지보수 포함
    
- **15288과 비교 출제** 자주 됨
    

**시험 포인트**

- “소프트웨어 생명주기 프로세스” → **12207**
    
- “시스템 전체”가 아니라 **SW 중심**
    

---

## 2. ISO/IEC 15288

**시스템 생명주기 프로세스** 표준이다.  
12207보다 범위가 넓고, 소프트웨어뿐 아니라 **하드웨어, 인력, 운영환경 등 시스템 전체**를 대상으로 한다.  
ISO는 15288을 “system life cycle processes”로 제시한다. ([ISO](https://www.iso.org/ics/35.080.html?utm_source=chatgpt.com "35.080 - Software"))

**핵심**

- 시스템 전체 관점
    
- 12207보다 상위/광범위
    
- **12207과 짝으로 외움**
    

**시험 포인트**

- “시스템 생명주기” → **15288**
    
- **12207 = SW / 15288 = System**
    

---

## 3. ISO/IEC 15504

**SPICE(Software Process Improvement and Capability dEtermination)** 계열의 **프로세스 평가 모델**이다.  
핵심은 **프로세스 능력수준(capability level)** 평가다.  
시험에서는 거의 항상 **수준 문제**로 연결된다.

**능력수준**

- Level 0: 불완전(Incomplete)
    
- Level 1: 수행(Performed)
    
- Level 2: 관리(Managed)
    
- Level 3: 확립(Established)
    
- Level 4: 예측(Predictable)
    
- Level 5: 최적화(Optimizing)
    

**핵심**

- 프로세스 평가
    
- 능력수준 0~5
    
- **구 SPICE 표준**
    

**시험 포인트**

- “SPICE” 나오면 → **15504**
    
- “능력수준/수행능력/capability” 나오면 바로 연결
    
- 후속 체계는 **330 계열**
    

---

## 4. ISO/IEC 33063

이 표준은 **테스트 프로세스 평가 모델**로 잡으면 된다.  
단순히 “330 계열”로만 외우면 애매하고, **29119의 테스트 프로세스를 평가하는 쪽**이라고 이해해야 안 헷갈린다.  
ISO 설명상 33063은 **테스트 프로세스 평가 모델(Test process assessment model)** 성격이다. ([ISO](https://www.iso.org/obp/ui/en/?utm_source=chatgpt.com "ISO/IEC 25010:2023(en), Systems and software engineering"))

**핵심**

- 테스트 프로세스를 평가
    
- 29119와 연결
    
- 15504의 후속 계열 맥락
    

**시험 포인트**

- **29119 = 테스트 표준**
    
- **33063 = 테스트 프로세스 평가**
    
- “테스트의 성숙도/수행능력 평가” 느낌이면 **33063**
    

---

## 5. ISO/IEC 29119

**소프트웨어 테스트 국제표준**이다.  
핵심은 **테스트 용어, 테스트 프로세스, 테스트 문서, 테스트 기법**이다.  
즉 테스트 자체를 체계화한 표준이다. ISO는 29119를 소프트웨어 테스트 표준 시리즈로 다룬다. ([ISO](https://www.iso.org/obp/ui/en/?utm_source=chatgpt.com "ISO/IEC 25010:2023(en), Systems and software engineering"))

**핵심**

- 테스트 개념/용어
    
- 테스트 프로세스
    
- 테스트 문서/기법
    

**시험 포인트**

- “테스트 표준” → **29119**
    
- “테스트 문서/프로세스” → **29119**
    
- “그 테스트 프로세스를 평가” → **33063**
    

---

## 6. ISO/IEC 9126

**구(舊) 소프트웨어 품질모델**이다.  
핵심은 이것 하나다.  
현재 실전 중심은 25010이고, 9126은 **이전 품질모델**로 비교용으로 본다.

**핵심**

- 구 품질모델
    
- 현재는 25010으로 이어짐
    

**시험 포인트**

- “예전 품질모델” → **9126**
    
- “신 품질모델”과 비교되면 → **25010**
    

---

## 7. ISO/IEC 25010

이건 **현행 품질모델의 핵심**이다.  
ISO는 SQuaRE 계열 안에서 **품질모델**을 제공한다고 설명한다. ([ISO](https://www.iso.org/obp/ui/en/?utm_source=chatgpt.com "ISO/IEC 25010:2023(en), Systems and software engineering"))  
시험에서는 사실상 **품질특성**이 전부라고 봐도 된다.

**핵심**

- 품질특성 정의
    
- 신 품질모델
    
- 품질 요구사항, 측정, 평가의 기준
    

**시험 포인트**

- “품질특성/품질모델” → **25010**
    
- 9126의 후속
    
- **25023(측정), 25040(평가)** 와 연결해서 이해
    

**연결**

- 25010 = 품질이 무엇인지 정의
    
- 25023 = 그 품질을 어떻게 측정할지
    
- 25040 = 그 품질을 어떻게 평가할지
    

---

## 8. ISO/IEC 14598

**소프트웨어 품질평가 구표준**이다.  
9126이 품질모델이라면, 14598은 **품질평가 절차** 쪽이다.  
현재는 **25040 계열로 연결**해서 이해하면 된다.

**핵심**

- 구 품질평가 표준
    
- 품질평가 절차/방법
    
- 25040의 이전 계열
    

**시험 포인트**

- “구 품질평가” → **14598**
    
- “현행 품질평가 프로세스” → **25040**
    

---

## 9. ISO/IEC 12119

**패키지 소프트웨어 품질 요구사항 및 시험** 표준이다.  
중요한 건 **패키지 소프트웨어 전용**이라는 점이다.  
ISO 설명에서도 delivered package software에 대한 요구사항/테스트 성격으로 이해하면 된다. ([ISO](https://www.iso.org/ics/35.080.html?utm_source=chatgpt.com "35.080 - Software"))

**핵심**

- 패키지 소프트웨어 대상
    
- 품질 요구사항 + 시험
    
- 범용 생명주기 표준 아님
    

**시험 포인트**

- “패키지 소프트웨어” → **12119**
    
- 개발 프로세스 표준과 혼동 금지
    

---

## 10. ISO/IEC 25000

**SQuaRE(Systems and software Quality Requirements and Evaluation)** 전체 틀이다.  
이 번호 하나로 세부 품질특성이나 측정지표를 직접 묻기보다는, **품질 요구사항·평가 표준군 전체 프레임워크**라는 점이 핵심이다. ISO는 25000을 “Guide to SQuaRE”로 제시한다. ([ISO](https://www.iso.org/ics/35.080.html?utm_source=chatgpt.com "35.080 - Software"))

**핵심**

- SQuaRE 전체 프레임워크
    
- 상위 안내 표준
    
- 25010, 25023, 25040을 묶는 틀
    

**시험 포인트**

- “SQuaRE” → **25000**
    
- 세부 내용은 하위 번호가 담당
    

---

## 11. ISO/IEC 25023

**품질 측정 지표** 표준이다.  
25010이 품질특성을 정의하면, 25023은 그 특성을 **정량적으로 측정하는 기준**이다.

**핵심**

- 품질 측정지표
    
- 25010의 품질특성과 직접 연결
    
- “수치화”가 핵심
    

**시험 포인트**

- “품질을 측정” → **25023**
    
- “품질특성 자체” → **25010**
    
- 모델과 측정 구분
    

---

## 12. ISO/IEC 25040

**품질 평가 프로세스** 표준이다.  
25010이 품질모델, 25023이 측정지표라면, 25040은 **평가 절차와 프로세스**다.  
ISO는 SQuaRE 안의 evaluation process 표준으로 다룬다. ([ISO](https://www.iso.org/obp/ui/en/?utm_source=chatgpt.com "ISO/IEC 25010:2023(en), Systems and software engineering"))

**핵심**

- 품질평가 절차/방법
    
- 14598의 후속 계열
    
- 25010, 25023과 연계
    

**시험 포인트**

- “품질평가 프로세스” → **25040**
    
- “구 품질평가” → **14598**
    

---

## 13. ISO/IEC 5055

**자동 정적분석 기반 소스코드 품질 측정** 표준이다.  
핵심은 **코드 수준**, **자동화**, **정적분석**이다.  
ISO는 아키텍처/코딩 관행 위반을 자동으로 분석해 품질을 측정하는 방향으로 설명한다. ([ISO](https://www.iso.org/obp/ui/en/?utm_source=chatgpt.com "ISO/IEC 25010:2023(en), Systems and software engineering"))

**핵심**

- 소스코드 품질
    
- 자동 정적분석
    
- 유지보수 위험/운영 위험과 연결
    

**시험 포인트**

- “정적분석” → **5055**
    
- “소스코드 품질 자동 측정” → **5055**
    
- 문서/프로세스 품질이 아니라 **코드 품질**

# 테스트

## 회귀 테스트 (Regression Test)

**정의**

> 소프트웨어 수정(버그 수정, 기능 추가, 변경 등) 이후에 **기존 기능이 정상적으로 동작하는지 다시 검증하는 테스트**


컴포넌트(단위) 테스트 , 통합테스트, 시스템테스트에서 수행된다
# 핵심 개념

| 항목    | 내용                         |
| ----- | -------------------------- |
| 목적    | 수정으로 인해 **기존 기능이 깨졌는지 확인** |
| 수행 시점 | 버그 수정, 기능 변경 후             |
| 대상    | **기존에 정상 동작하던 기능**         |
| 특징    | 이전 테스트 케이스를 다시 실행          |
## 블랙박스 테스트 vs 화이트박스 테스트

|구분|블랙박스 테스트 (Black-Box Test)|화이트박스 테스트 (White-Box Test)|
|---|---|---|
|관점|**외부 기능 중심**|**내부 구조 중심**|
|코드 이해 필요|필요 없음|필요함|
|테스트 기준|**입력 / 출력 결과**|**코드 흐름 / 로직**|
|수행자|주로 테스트 담당자|주로 개발자|
|테스트 대상|기능 요구사항|프로그램 내부 로직|
|테스트 설계|요구사항 기반|코드 구조 기반|

(출처: ISTQB Glossary of Testing Terms, IEEE Software Engineering Standards)


## 정적분석 5가지 결함부류

**정적 분석 도구(Static Analysis Tool)** 가 검출하는 대표적인 **5가지 결함 부류**는 다음과 같다.

(출처: **ISO/IEC 9126 / 25010 품질 모델 관련 정적 분석 항목, Pressman Software Engineering, SW Engineering Body of Knowledge(SWEBOK)**)

| 결함 부류                                   | 설명                            |
| --------------------------------------- | ----------------------------- |
| 데이터 결함 (Data Fault)                     | 변수 초기화 누락, 잘못된 데이터 참조, 미사용 변수 |
| 제어 결함 (Control Fault)                   | 잘못된 조건문, 무한 루프, 잘못된 분기        |
| 인터페이스 결함 (Interface Fault)              | 모듈 간 인터페이스 불일치, 매개변수 오류       |
| 입출력 결함 (Input/Output Fault)             | 입력 검증 부족, 출력 처리 오류.           |
| 저장/메모리 관리 결함 (Storage Management Fault) | 메모리 누수, 버퍼 오버플로우, 잘못된 메모리 사용  |

|상황|결함|
|---|---|
|입력 검증 없음|입출력 결함|
|출력 포맷 오류|입출력 결함|
|**값 갱신 없이 반복 출력**|**입출력 결함**|

---

### 핵심 정리

정적 분석 도구 결함 5가지  
데이터 / 제어 / 인터페이스 / 입출력 / 저장관리

정적 분석은 **코드를 실행하지 않고** 소스코드를 분석하여 이런 결함을 발견한다.  
(출처: IEEE Software Engineering Standard, SWEBOK)
